## 4.1 ChatService

**File:** `src/services/chatService.ts`

**Constructor:** `new ChatService(appRoot, options)` — sets `baseDir` to `<options.dataRoot || appRoot/data>/chat`, creates `workspaces/` and `artifacts/` dirs synchronously at startup, initializes in-memory `Map<convId, workspaceHash>` for fast lookup. `server.ts` passes `config.AGENT_COCKPIT_DATA_DIR`, so production installs can keep chat data outside replaceable app code while existing callers without `dataRoot` preserve the legacy `<appRoot>/data/chat` layout.

`ChatService` remains the public facade for conversation, workspace, memory, KB, Context Map, settings, queue, usage, artifact, and workspace-instruction operations. Pure attachment helpers stay in `src/services/chat/attachments.ts` and are re-exported where legacy imports still expect them from `chatService.ts`; generated artifact persistence lives in `src/services/chat/artifactStore.ts` behind `createConversationArtifact`; message-queue persistence and legacy queue normalization live in `src/services/chat/messageQueueStore.ts` behind the same public `getQueue`/`setQueue`/`clearQueue` methods; usage-ledger persistence/aggregation lives in `src/services/chat/usageLedgerStore.ts` behind `addUsage`, `getUsageStats`, and `clearUsageStats`; workspace KB and Context Map enablement/settings flags live in `src/services/chat/workspaceFeatureSettingsStore.ts`; workspace-instruction compatibility detection and pointer-file creation live in `src/services/chat/workspaceInstructionStore.ts` behind the existing instruction facade methods. This keeps helper/store behavior testable without changing the service API.

Chat API route registration is split by domain. `src/routes/chat.ts` owns dependency construction, stream lifecycle orchestration, and route composition. Focused routers own status/settings (`statusRoutes.ts`), CLI profile metadata/auth (`cliProfileRoutes.ts`), conversation/session/queue CRUD (`conversationRoutes.ts`), stream send/input/abort/status (`streamRoutes.ts`), backend goal mutations (`goalRoutes.ts`), conversation uploads and workspace file delivery (`uploadRoutes.ts`), local directory picker routes (`filesystemRoutes.ts`), workspace instruction routes (`workspaceInstructionRoutes.ts`), workspace file explorer routes (`explorerRoutes.ts`), read-only workspace Git status/diff routes (`gitRoutes.ts`), memory routes (`memoryRoutes.ts`), Context Map routes (`contextMapRoutes.ts`), and KB routes (`kbRoutes.ts`). Common Express helpers live in `src/routes/chat/routeUtils.ts`; goal lifecycle transcript formatting and runtime snapshots live in `src/services/chat/goalEventMessages.ts` so routes and stream processing persist the same `Message.goalEvent` shape. Runtime request validators and small shared response contracts live in focused files under `src/contracts/`; browser-safe wire types avoid importing server-only `src/types` where web/mobile clients use type-only references.

### Methods

| Method | Description |
|--------|-------------|
| `initialize()` | Runs migration if legacy `conversations/` dir exists, assigns deterministic server-configured `cliProfileId` values to vendor-only conversations, ensures matching settings profiles, then builds the convId→workspace lookup map. |
| `createConversation(title, workingDir, backend, model?, effort?, cliProfileId?)` | Creates entry in workspace index + empty session-1.json. Falls back to `_defaultWorkspace`. When `cliProfileId` is supplied without an explicit backend, resolves the profile from settings and stores the backend derived from `CliProfile.vendor + CliProfile.protocol` (`claude-code + interactive → claude-code-interactive`; `claude-code + standard → claude-code`; other vendors map directly). A conflicting explicit `backend` is rejected. When neither profile nor backend is supplied, resolves `settings.defaultCliProfileId` if present and derives the backend from that profile; otherwise it uses legacy `settings.defaultBackend` if present. If neither is configured, creation fails with a CLI-profile-required error instead of falling back to a provider. When only a backend is supplied, derives `cliProfileId` from the selected backend's physical server-configured profile and ensures settings contains it. `model` is the optional model alias to persist. `effort` is the optional adaptive reasoning level; when unsupported by the chosen model it falls back to `high` if available, then the first supported level, or is dropped if the model has no effort support. |
| `getConversation(id)` | Returns API-compatible object with messages, including `cliProfileId` when present, or `null`. |
| `listConversations(opts?)` | Scans all workspace indexes. Returns summaries sorted by `lastActivity` desc, each with `workspaceHash`, `workspaceKbEnabled` (defaults to `false` for legacy workspaces), and `cliProfileId` when present. Pass `{ archived: true }` to list only archived; default returns active only. |
| `resolveCliProfileRuntime(cliProfileId, fallbackBackend?)` | Loads settings and resolves runtime adapter identity. A present `cliProfileId` must exist and not be disabled; the returned `profile` is the CLI profile and `backendId` is derived from the profile's vendor/protocol unless the profile is legacy and lacks `protocol`, in which case a compatible fallback backend can preserve old conversation records. Without a profile id, returns `fallbackBackend`, then `settings.defaultCliProfileId` / `settings.defaultBackend` if configured, deriving the physical server-configured profile through the backend-to-vendor mapping when possible. Throws profile-specific errors for missing/disabled profiles and a CLI-profile-required error when no runtime is configured. |
| `renameConversation(id, newTitle)` | Updates title in workspace index and sets `titleManuallySet: true` on the conversation entry. The flag locks the title against all subsequent automatic mutations (`addMessage` first-message snapshot, `generateAndUpdateTitle`, `resetSession` "New Chat" stamp), so a manual rename survives session resets. Returns full conversation or `null`. |
| `archiveConversation(id)` | Sets `archived: true` on conversation entry in workspace index. Returns `true` or `false` if not found. Files remain on disk. |
| `restoreConversation(id)` | Removes `archived` flag from conversation entry. Returns `true` or `false` if not found. |
| `setConversationUnread(id, unread)` | Toggles the `unread` flag on the conversation's index entry. `unread:true` writes `unread: true`; `unread:false` deletes the field (so the on-disk file stays lean for the common read state). No-op (returns `true`) when the entry already matches the requested state. Returns `false` when the conversation isn't found. The `unread` field is also surfaced in `listConversations()` summaries so the v2 sidebar can render the unread dot on first paint. |
| `deleteConversation(id)` | Removes from index, deletes session folder + artifacts, removes from lookup map. Works on both active and archived conversations. |
| `updateConversationBackend(convId, backend)` | Updates backend field in workspace index and, during the CLI-profile transition, updates `cliProfileId` to the matching server-configured profile. When the backend actually changes, clears `conversation.usage.contextUsagePercentage` and the active session's `usage.contextUsagePercentage` before persisting so a prior backend's context-window snapshot (e.g. Kiro's `14.45%`) can't leak into the new backend's top-bar chip. |
| `updateConversationCliProfile(convId, cliProfileId, requestedBackend?)` | Resolves the profile, writes `convEntry.backend` to the profile's protocol-derived backend (or a compatible requested backend only for legacy profiles without an explicit protocol), writes `convEntry.cliProfileId`, and clears Kiro-style `contextUsagePercentage` when the effective backend changes. The chat route only calls this before the active session has messages; mid-session profile switching returns `409`. |
| `updateConversationModel(convId, model)` | Updates model field in workspace index. Pass `null` to clear. Reconciles the stored `effort` against the new model: preserve if supported, otherwise fall back to `high` if available, then the first supported level, or clear if the new model has no effort support at all. |
| `updateConversationEffort(convId, effort)` | Updates effort field in workspace index. Pass `null` to clear. Preserves supported values; unsupported requests fall back to `high` if available, then the first supported level. |
| `addMessage(convId, role, content, backend, thinking, toolActivity, turn, contentBlocks)` | Appends to active session + updates index metadata. | `createConversationArtifact(convId, input)` | Persists backend-generated artifacts into `data/chat/artifacts/{convId}/` and returns a `ConversationArtifact`. `input` accepts either `sourcePath` (copied from disk) or `dataBase64` (decoded; data URLs are accepted), plus optional `filename`, `mimeType`, `title`, and `sourceToolId`. Used by `processStream` for vendor-neutral `artifact` stream events. |

| `setMessagePinned(convId, messageId, pinned)` | Updates `Message.pinned` on the active-session message and returns `{ conversation, message }`, or `null` when the conversation/message is missing. `pinned:true` stores `true`; `pinned:false` deletes the field. The mutation writes only the session file and intentionally does not change workspace-index activity metadata. |
| `generateAndUpdateTitle(convId, userMessage)` | Resolves the conversation's `cliProfileId` to a vendor adapter, generates a new title via that adapter's `generateTitle()`, hard-cuts it to max 8 whitespace-separated words, and persists it. The clamp runs after the adapter (and after the fallback path) so every backend is bounded uniformly. Short-circuits before calling the adapter when `titleManuallySet` is true and returns the existing title — avoids both the wasted backend round-trip and clobbering a manual rename. | `resetSession(convId)` | Archives active session (endedAt plus immediate fallback summary), creates new session, resets title to "New Chat" **only when the title is auto-managed** — when `titleManuallySet` is true the existing title is preserved across the reset. `src/services/cliProfileAuthService.ts` owns remote account-auth jobs for CLI profiles. It supports Codex and Claude Code account profiles only; self-configured profiles are treated as already managed on the server, and Kiro remains unsupported until it has a safe dedicated profile/config directory. `profileWithAuthDefaults(settings, profileId)` looks up the profile and, when an explicit non-setup account profile has no `configDir`, assigns a deterministic directory under `data/cli-profiles/<slug>-<sha1>/` so future runtime spawns and the auth job use the same isolated home. For Welcome setup account profiles, the same helper removes stale `configDir`, `CODEX_HOME`, and `CLAUDE_CONFIG_DIR` values instead of assigning a private directory. `checkProfile(profile)` runs the vendor status command (`codex login status` or `claude auth status --json`) with the profile command/env/config directory and returns availability/authentication status plus redacted output; Codex is exit-code based, while Claude Code parses the JSON status body and requires `loggedIn: true` so a logged-out exit-0 response does not become a false verified state. On Windows those commands are built from the shared CLI resolver, so installer-managed Codex uses `node.exe <installDir>\cli-tools\node_modules\@openai\codex\bin\codex.js ...` and installer-managed Claude uses the package `claude.exe` before any npm `.cmd` shim fallback. The route-level `/cli-profiles/:id/test` wrapper also asks the registered vendor adapter for `getMetadata({ cliProfile })` so the profile-specific model catalog is warmed/read during the check. `startAuth(profile)` starts one in-memory job per profile using `codex login --device-auth` or `claude auth login --claudeai`, captures redacted stdout/stderr/info/exit events, and leaves recent job snapshots pollable by ID. A running auth job has a 15-minute timeout; timeout marks it `failed`, records an actionable message, and sends `SIGTERM` to the login process. When the login process exits successfully, the service polls the vendor status command before marking the job `succeeded`; Claude Code polling also requires `loggedIn: true`. Status timeout, nonzero status, or repeated logged-out Claude JSON leaves the job `failed` with the last redacted status output. `cancelJob(jobId)` marks a running job cancelled and sends `SIGTERM`; `shutdown()` cancels any running auth jobs during server shutdown.

`src/routes/chat/cliProfileRoutes.ts` also exposes Welcome/setup auth wrappers at `/cli-profiles/setup-auth/:vendor/test` and `/cli-profiles/setup-auth/:vendor/start` ([ADR-0060](adr/0060-use-cli-profile-auth-for-setup-login.md), [ADR-0064](adr/0064-use-system-cli-auth-for-welcome-setup.md), [ADR-0065](adr/0065-normalize-setup-profile-auth-homes-at-settings-boundary.md)). These routes intentionally do not call `profileWithAuthDefaults()`: for `codex` and `claude-code`, they reuse the first enabled system-home account profile for the vendor or create a first-run account profile (`setup-codex-account` / `setup-claude-code-account`) without `configDir`, then delegate to `checkProfile()` or `startAuth()`. Existing setup profiles from earlier releases that still have generated auth homes are migrated by settings normalization and by the direct profile-auth helper before any check/job so Welcome auth uses the user's normal CLI home and matches terminal `codex` / `claude` behavior no matter whether the UI calls the setup-auth route or the direct profile route; setup-auth route selection also ignores non-setup account profiles that still have `configDir`, `CLAUDE_CONFIG_DIR`, or `CODEX_HOME` so explicit isolated profiles remain isolated. If the current default profile is that vendor's server-configured profile, the created setup account profile becomes the default so the completed login is used by new conversations. On Windows, once system-home Claude Code auth verifies with `loggedIn: true`, `CliProfileAuthService` ensures the user's `%USERPROFILE%\.claude.json` contains `hasCompletedOnboarding: true`; this repairs the case where `claude auth status --json` succeeds but plain terminal `claude` still shows Claude Code's first-run login method selector. The onboarding write is skipped for isolated profiles with `CLAUDE_CONFIG_DIR`. `kiro` remains a `400` response because Kiro has no dedicated safe config-home override.

All methods are `async` except `getWorkspaceContext()`. `getWorkspaceMemoryPointer()` is `async` so it can `mkdir -p` the memory dir on first access.

### LocalAuthStore

**File:** `src/services/localAuthStore.ts`

`LocalAuthStore` owns the first-party single-owner auth state for the backend. It stores `owner.json` under `AUTH_DATA_DIR` (default `<AGENT_COCKPIT_DATA_DIR>/auth`) and writes it through `atomicWriteFile`, then chmods the file to `0600`. The directory is created with mode `0700`.

State shape:

```json
{
  "version": 1,
  "owner": {
    "id": "local-owner",
    "email": "owner@example.com",
    "displayName": "Owner User",
    "passwordHash": "scrypt$<salt>$<derived>",
    "createdAt": "2026-05-03T00:00:00.000Z",
    "updatedAt": "2026-05-03T00:00:00.000Z"
  },
  "policy": {
    "passkeyRequired": false
  },
  "passkeys": [],
  "recoveryCodes": [
    {
      "id": "uuid",
      "codeHash": "scrypt$<salt>$<derived>",
      "createdAt": "2026-05-03T00:00:00.000Z",
      "usedAt": "2026-05-03T00:00:00.000Z"
    }
  ]
}
```

Methods:

| Method | Description |
|--------|-------------|
| `hasOwner()` | Returns true when a valid owner state file exists. |
| `getOwner()` | Returns the owner record or null. |
| `getPolicy()` | Returns `{ passkeyRequired }`; throws `owner-missing` before setup. |
| `getRecoveryStatus()` | Returns `{ configured, total, remaining, createdAt }` for recovery-code health. |
| `listPasskeys()` / `getPasskeyByCredentialId(credentialId)` | Reads passkey metadata and stored credential public material. Lists are sorted by creation time descending. |
| `createOwner({ email, displayName, password })` | Creates the single owner. Fails when an owner already exists, when email/display name are blank or invalid, or when the password is shorter than 12 characters. Email is normalized to lowercase. |
| `verifyPassword(email, password)` | Verifies the submitted password against the stored scrypt hash with `crypto.timingSafeEqual`. Returns the owner on success, null on wrong credentials, and throws `owner-missing` when setup has not happened. |
| `regenerateRecoveryCodes(count?)` | Replaces all recovery codes with `count` new single-use codes (default 10, max 50), stores only scrypt hashes, and returns plaintext codes for one-time display. |
| `useRecoveryCode(code)` | Matches one unused recovery-code hash, marks it used, disables `passkeyRequired`, and returns the owner. Invalid or already-used codes return null. |
| `setPasskeyRequired(required)` | Updates policy. Enabling passkey-required throws `unsafe-policy` unless at least one passkey and one unused recovery code exist. |
| `createPasskey({ name?, credentialId, publicKey, counter, transports? })` | Stores verified WebAuthn credential public material with a display name and created timestamp. Duplicate credential ids and invalid inputs throw `invalid-input`. |
| `renamePasskey(id, name)` | Updates passkey display name; missing ids return null. |
| `updatePasskeyUsage(credentialId, counter)` | Updates authenticator counter and `lastUsedAt` after successful WebAuthn login; missing credentials return null. |
| `deletePasskey(id)` | Deletes a passkey; missing ids return null. Throws `unsafe-policy` instead of deleting the last passkey while passkey-required mode is enabled. |
| `resetOwnerAccess(opts)` | Local CLI reset helper. Can update owner identity/password, disable passkey-required mode, and regenerate recovery codes. |

The password and recovery-code hash format is `scrypt$<base64url salt>$<base64url derived key>`, using a 16-byte random salt and a 64-byte derived key. Plaintext passwords, setup tokens, and recovery codes after display are not stored in this file. Passkeys store only WebAuthn credential public material: credential id, public key, counter, transports, and timestamps.

### Durability primitives

Every JSON file ChatService writes (`workspaces/{hash}/index.json`, session files, the usage ledger, memory `snapshot.json`) goes through `atomicWriteFile` from `src/utils/atomicWrite.ts` — a sibling-tmp + `rename(2)` helper so readers never observe a torn file. Read-modify-write methods that mutate a workspace index are wrapped in `_indexLock.run(hash, ...)` (a `KeyedMutex` from `src/utils/keyedMutex.ts`) so concurrent mutators on the same workspace serialize FIFO and never lose updates; different workspaces run in parallel. `UsageLedgerStore` owns the shared ledger mutex and serializes ledger read-modify-write operations under the constant key `'__usage_ledger__'`. `generateAndUpdateTitle` and `generateAndStoreSessionSummary` keep slow backend adapter calls **outside** the lock (only the persist step acquires it) so a hung backend can't stall other writes on the workspace. `initialize()`'s `_buildLookupMap` catches `JSON.parse` failures per workspace, logs a warning, and keeps building — a single corrupt `index.json` degrades access to that workspace but cannot crash the server into a restart loop.

`src/services/streamJobRegistry.ts` applies the same atomic-write + keyed-mutex pattern to `data/chat/stream-jobs.json`, the chat router's operational registry for active CLI turns. The registry is not transcript history: it contains only accepted/preparing/running/abort-requested/finalizing jobs, and normal completion, explicit abort finalization, delete/archive cleanup, and startup reconciliation remove entries once a conversation message or terminal `streamError` is durable.

### Structured Logging

`src/utils/logger.ts` provides the server's structured logging foundation. It supports `error`, `warn`, `info`, and `debug` levels, reads the active threshold from `LOG_LEVEL` (default `info`), emits one line as `[level] message {json}`, and redacts metadata keys containing `token`, `secret`, `password`, `credential`, `authorization`, `cookie`, or `session` before writing. Child loggers attach stable bindings such as `{ subsystem: 'ws' }`.

The WebSocket server has started migrating diagnostic output to this logger. Connection lifecycle and operational events are emitted at `info`/`warn`/`error`; frame-level and stdin-length diagnostics are gated behind `debug`, and stdin content is not logged.

`src/services/streamJobSupervisor.ts` is the runtime owner layered on top of the registry. It owns the process-local `activeStreams` map, the process-local pending-send map, and the state-transition helpers used by the router. `POST /conversations/:id/message` calls `beginAcceptedTurn()` before profile/context setup, records the saved user message id after `ChatService.addMessage`, and calls `markRunning()` once a runtime stream attachment owns the backend iterator. `processStream` calls supervisor finalization/completion helpers before persisting terminal backend/server errors or emitting terminal `done`. The abort path records `abortRequested` before calling the backend abort handle. Delete/archive call `cleanupRuntimeConversation()` so runtime abort, map detachment, and durable active-job deletion use one path. Graceful shutdown calls `prepareForShutdown()` to mark active/pending jobs `finalizing` with `Interrupted by server shutdown` and then detaches runtime maps without deleting those durable jobs, leaving startup reconciliation to persist the terminal `streamError`.

On startup, `server.ts` calls `chatResult.reconcileInterruptedJobs()` immediately after `chatService.initialize()` and before binding the HTTP port. Reconciliation scans active jobs that are not attached to this process's runtime maps. If the referenced user message still exists, it appends a single assistant `streamError` using the recorded terminal reason (`abortRequested` or `terminalError`) or the default `Interrupted by server restart` with `source:'server'`. If the user message was never saved, the job is removed without adding transcript noise. The path is intentionally idempotent: an already-final terminal `streamError` with the same message/source prevents duplicate markers.

`src/services/sessionFinalizerQueue.ts` owns post-reset/archive cleanup jobs whose outputs are useful but not required before the reset/archive HTTP response returns. It persists per-workspace jobs in `workspaces/{hash}/session-finalizers.json` using the same atomic-write + keyed-mutex pattern. Job types are `session_summary`, `memory_extraction`, and `context_map_conversation_final_pass`; statuses are `pending`, `running`, `retrying`, `completed`, and `failed`. `enqueue()` writes the job, de-duplicates by `(type, payload.source, conversationId, sessionNumber)`, and schedules processing without awaiting execution. `start()` recovers any `running` jobs back to `pending` after a server restart, then drains in the background with bounded concurrency and retries. The chat router's handler updates archived session summaries, mirrors/extracts workspace memory, and runs conversation-only Context Map final passes. If a Context Map final pass collides with another workspace scan, it throws so the queue retries later instead of blocking reset/archive.

### Workspace Context Injection

When a new CLI session starts, the router prepends up to three bracketed pointer blocks to the outgoing user message (not stored in the conversation's message list). The first is always present; the second is added only when Memory is enabled for the workspace; the third is added only when Knowledge Base is enabled for the workspace.

```
[Workspace discussion history is available at {abs_workspace_path}/
Read index.json for all past and current conversations in this workspace with per-session summaries.
Each conversation subfolder contains session-N.json files with full message histories.
When the user references previous work, decisions, or discussions, consult the relevant session files for context.]

[Workspace memory is available at {abs_workspace_path}/memory/files/
Contains .md files with YAML frontmatter (type, name, description) followed by body text.
Read these when the user references preferences, feedback, decisions, project context, or prior work style.]

[Workspace knowledge base is available at {abs_workspace_path}/knowledge/
- state.json: pipeline state (raw uploads, digested entries, synthesis status).
- entries/*.md: digested knowledge entries with YAML frontmatter (title, tags, source).
- synthesis/*.md: cross-entry synthesis (created by the Dreaming stage).
Read these when the user references documents they've uploaded, domain knowledge, or asks questions the digested entries may cover.]
```

These prefixes are **only added on new sessions**. On resumed sessions the CLI already has them in its conversation history from the original first user message, so the pointers remain visible to the model across `--resume` without re-injection. Memory-enabled resumed messages do prepend one short write-rule reminder: if `memory_note` fails or is unavailable, the CLI must not create/edit local native memory files (`MEMORY.md`, `memory/*.md`, `.claude/**/memory/*.md`, or similar) as a fallback, and must report that the Memory note was not saved with the tool error instead. This reminder keeps older resumed CLI sessions aligned when Memory MCP behavior changes without re-sending the full workspace pointer. The memory pointer is produced by `chatService.getWorkspaceMemoryPointer(hash)`, which returns `null` when memory is disabled and otherwise `mkdir -p`s `memory/files/` before returning so the model never hits ENOENT on a brand-new workspace. The KB pointer is produced by `chatService.getWorkspaceKbPointer(hash)` with the same "return null when disabled / `mkdir -p` when enabled" contract, targeting `knowledge/entries/`.

### Workspace Memory

CLI backends may have their own memory systems (e.g. Claude Code stores memory under `~/.claude/projects/{sanitized}/memory/`). Agent Cockpit captures that memory at the **workspace** level so it survives CLI switches, session resets, and fresh conversations. Non-Claude CLIs that don't have a native memory system instead write memory via an MCP tool (`memory_note`) that the cockpit exposes to them — see **Memory MCP Server** below.

Memory is **opt-in per workspace** via a toggle stored on `WorkspaceIndex.memoryEnabled`. When the toggle is off, every code path that would read or write memory (capture, watcher, injection, MCP addendum) is skipped, and the memory store stays inert. Toggling is done via the workspace Settings modal (Instructions + Memory tabs).

**On-disk layout:** `memory/files/` is split into two subdirectories that correspond to the two write paths:
- `memory/files/claude/` — Claude Code native captures. Wiped and rewritten every time `saveWorkspaceMemory` is called, so the directory always mirrors the current state of the CLI's extraction output.
- `memory/files/notes/` — Entries written via the `memory_note` MCP tool and by post-session extraction. Preserved across captures. Each file has a timestamped, slug-based name (`note_<ts>_<slug>.md` or `session_<ts>_<slug>.md`).
The merged snapshot (`memory/snapshot.json`) combines both directories so callers of `getWorkspaceMemory(hash)` see a single file list with a `source` field (`cli-capture | memory-note | session-extraction`) on each entry.
Agent Cockpit-owned lifecycle metadata lives in `memory/state.json`, a versioned sidecar keyed by the same relative filenames. Each state record stores `entryId`, `filename`, `status` (`active | superseded | redacted | deleted`), `scope` (`workspace | user`), `source`, timestamps, optional source conversation, optional supersession links, optional confidence, and optional redaction metadata. Current write paths persist records only for files that exist; delete/clear prune records instead of retaining tombstones. Older workspaces without `state.json` still load because `getWorkspaceMemory` synthesizes active workspace metadata in returned `MemoryFile.metadata`, and the next memory write materializes the sidecar.

**Capture trigger (Claude Code):** On `POST /conversations/:id/reset`, after the session boundary is durable, the router enqueues a `memory_extraction` session-finalizer job when workspace Memory is enabled. The job resolves the ending conversation's CLI profile and calls `chatService.captureWorkspaceMemory(convId, endingBackend, endingProfile)` in the background. The ending backend's `extractMemory(workspacePath, { cliProfile })` is invoked and the resulting `MemorySnapshot` is persisted via `saveWorkspaceMemory` (which only wipes the `claude/` subtree, preserving notes). Capture is best-effort and retried by the finalizer queue; extraction or persistence errors are logged and never block the reset response. For Claude Code profiles with `configDir`, memory is read from `<configDir>/projects/...` instead of `~/.claude/projects/...`.

**Post-session extraction (all backends):** The same `memory_extraction` finalizer then calls `memoryMcp.extractMemoryFromSession({ workspaceHash, conversationId, messages })` with the archived transcript for every backend (including Claude Code). This spawns the globally-configured Memory CLI via `runOneShot` with a prompt that lists existing memory entries (for dedup context) and asks the CLI to extract any new durable memories. The CLI is expected to reply with either `NONE` or one or more frontmatter markdown entries separated by `===` delimiters. Before persistence, each extracted entry is passed through the same deterministic memory redaction used by governed `memory_note` writes; redacted extracted entries are saved with placeholders and receive `metadata.status:'redacted'` plus `metadata.redaction[]` reasons. Each new entry is then written to `notes/` via `addMemoryNoteEntry(hash, { source: 'session-extraction', ... })`. Claude Code still receives its native `captureWorkspaceMemory` first so explicit `#` saves are mirrored to `claude/`; extraction runs on top to catch incidental durable facts mentioned conversationally. When extraction saves entries, it emits a refresh-only `memory_update` (`sourceConversationId:null`, `displayInChat:false`) so Memory surfaces can refresh without adding a synthetic Memory bubble to any chat transcript. This path is best-effort and swallows all errors inside the extraction service; queue-level failures retry the whole memory finalizer.

**Real-time capture (MemoryWatcher):** Session-reset capture misses memories written during long sessions if the browser is closed before the user resets. To close that gap, `src/services/memoryWatcher.ts` wraps `fs.watch()` on the backend's memory directory and re-captures into workspace storage whenever a `.md` file changes. The router (`createChatRouter`) owns one `MemoryWatcher` instance. When a message is sent and a WebSocket is connected for the conversation, the router calls `adapter.getMemoryDir(workingDir, { cliProfile })`; if a directory is returned, `memoryWatcher.watch(convId, memDir, onChange)` is invoked alongside `processStream`. The `onChange` handler calls `chatService.captureWorkspaceMemory(convId, backendId, cliProfile)` — the same code path as session-reset capture — so there is a single write path into workspace memory storage. Change events are debounced (500ms default) to collapse bursts from Claude Code's extraction agent into a single re-snapshot. Non-`.md` files are ignored. The watcher is scoped to a processStream's lifecycle: it is unwatched in `processStream`'s `onDone` callback, in the `processStream().catch` branch, on `DELETE /conversations/:id`, on `PATCH /conversations/:id/archive`, and via `memoryWatcher.unwatchAll()` in `shutdown()`.

**`memory_update` WebSocket frame:** After each successful re-capture, the router emits a lightweight `memory_update` `StreamEvent` to every connected conversation in the same workspace: `{ type: 'memory_update', capturedAt, fileCount, changedFiles, sourceConversationId?, displayInChat?, writeOutcomes? }`. `changedFiles` is computed by maintaining a per-conversation `Map<convId, Map<filename, fingerprint>>` in the router (`memoryFingerprints`) for the conversation whose stream triggered the watcher; fingerprints are a cheap `length:hash` of the first 256 chars of file content (djb2). The diff returns any filename whose fingerprint changed or didn't exist in the previous frame. The map is cleared in every unwatch path (`onDone`, `.catch`, delete, archive, shutdown) so a re-watched conversation starts fresh and reports all files as changed on the first capture. The snapshot is still persisted even when no conversation in the workspace is connected, but no live frame fires in that case. Manual memory delete/clear/consolidation, `memory_note`, and post-session extraction use the same workspace-scoped live fan-out. Conversation-visible writes (`MemoryWatcher` and `memory_note`) set `sourceConversationId`; `broadcastMemoryUpdate()` sends `displayInChat: true` only to that source conversation and `false` to other same-workspace recipients, so idle chats refresh silently instead of accumulating Memory bubbles. Manual delete/clear/consolidation and post-session extraction send refresh-only frames with `displayInChat: false`. `memory_note` governed writes include `writeOutcomes` describing `saved`, `redacted_saved`, `superseded_saved`, `skipped_duplicate`, or `skipped_ephemeral` decisions; skip-only outcomes use `changedFiles: []` so the source chat can explain the decision without implying a file changed.

**Memory panel and review UI:** V2 handles `memory_update` in `web/AgentCockpitWeb/src/streamStore.js` by dispatching an `ac:memory-update` browser event for workspace surfaces and appending a synthetic in-memory message with `role: 'memory'` only when the frame has `displayInChat: true`. The synthetic message carries `{ capturedAt, fileCount, changedFiles, sourceConversationId, writeOutcomes }` metadata, is client-side only, is not persisted to backend session files, and does not survive a page reload or conversation switch. `web/AgentCockpitWeb/src/shell.jsx` renders it through `MemoryUpdateBubble`, an in-feed memory card with an expandable changed-file or governed-outcome preview and a **View update** action. Clicking the action opens `MemoryUpdateModal`, which fetches `GET /workspaces/:hash/memory`, filters the current snapshot by `changedFiles`, and shows changed memory entries plus any governed write outcomes. The modal's **View all memory items** action opens `WorkspaceSettingsPage` directly on the Memory tab.

The full Memory tab listens for `ac:memory-update` while open and silently refetches the Memory snapshot when the event hash matches; it groups files by type, supports per-entry expansion, exposes delete / clear-all actions, renders search + type/status filters, and exposes superseded-row **Restore**. It also owns the Memory Review schedule controls, the **Start new review** generation trigger, the in-progress review indicator, and the **Audit Current Review** navigation button for the newest persisted run. **Start new review** is disabled while the compact review status reports `latestRunStatus === 'running'` or while the manual start request is still in flight. The tab polls `getMemoryReviewSchedule(hash)` while open so a missed websocket event cannot leave the settings UI unaware of a running or completed generation. It does not render proposal/draft decisions inline; those live on the dedicated Memory Review page.

`web/AgentCockpitWeb/src/streamStore.js` handles `memory_review_update` by patching `conv.memoryReview` and dispatching `ac:memory-review-update`. `ComposerMemoryReviewIcon` appears only when the compact review status says pending items exist. Its popover lists draft/metadata/attention counts and has a single **Review** action. `web/AgentCockpitWeb/src/screens/memoryReview.jsx` loads pending runs or a specific run id, renders safe metadata items and draft before/after panes, and calls the review item endpoints to apply, dismiss, or regenerate. Draft operations also expose **Edit markdown** per operation; edited content updates the After diff locally and is sent on apply, while the server keeps the generated operation type/path/supersession metadata authoritative. Apply operations may create new `notes/` files, rewrite selected `notes/*` files, or patch lifecycle metadata, but scheduled and manual review generation never applies anything automatically and never rewrites `claude/*` captures. Dismissed items stay in the run as audit-only decisions and are shown as `Dismissed`; they are not applied and do not change future review eligibility. Regenerating a dismissed draft removes that dismissal by clearing `discardedAt`, refreshing the draft from the original advisory action, and returning the item to `pending`.

**Read access (pointer + search tool, not dump):** On `POST /conversations/:id/message` for a new session, the router calls `chatService.getWorkspaceMemoryPointer(hash)` and prepends the returned bracketed block to the outgoing user message alongside the existing workspace discussion-history pointer from `getWorkspaceContext(convId)`. The pointer is a 3-line text block that names the absolute path to `memory/files/` and tells the CLI to read its `.md` files when the user references preferences, feedback, decisions, project context, or prior work style. Gated on `getWorkspaceMemoryEnabled(hash)`: when the toggle is off, the pointer is skipped entirely. The memory **content** is never serialized into the system prompt — the CLI reads the files on demand via its own file tools, so (a) resumed sessions keep access because the pointer lives in the CLI's conversation history from the first user message, (b) mid-session additions (notes written from another tab via `memory_note`) are visible on the very next turn because each read fetches fresh content, and (c) the static token cost per spawn is bounded by the pointer length rather than the memory size. For every backend (Claude Code included), an additional **Memory MCP addendum** is appended to the system prompt that teaches the CLI to use `memory_search` for targeted recall and call `memory_note` when it encounters durable facts. That addendum also forbids falling back to native/local memory-file writes when `memory_note` fails; the chat route repeats this specific write-failure rule on resumed memory-enabled messages so stale CLI sessions do not save to native files after a Memory MCP processor failure. See the Memory MCP Server subsection below.

### Memory MCP Server

**File:** `src/services/memoryMcp/index.ts` (router + factory) and `src/services/memoryMcp/stub.cjs` (stdio shim).

Exposes `memory_search` and `memory_note` MCP tools to every CLI backend. `memory_search` gives CLIs targeted read access to relevant memory without dumping the whole store into the prompt. `memory_note` lets CLIs persist durable memory into workspace storage. Kiro has no native memory system; Claude Code does (via `#`) but gets both tools too, so incidental durable facts mentioned conversationally are still captured and future turns can retrieve them consistently.

**Architecture:**
1. **Stub (`stub.cjs`)** — A dependency-free CommonJS Node script that implements the minimal MCP protocol (`initialize`, `tools/list`, `tools/call`) over stdio. It exposes `memory_search({ query, limit?, type?, status?, include_content? })` and `memory_note({ content, type?, tags? })`, forwarding `tools/call` invocations to Agent Cockpit's HTTP endpoints over localhost via `X-Memory-Token`. `status` is scoped to `active` (default, active + redacted) or `all` (also include superseded/deleted) so CLIs can intentionally inspect historical memory without exposing the detailed lifecycle filter surface. The stub is spawned by the CLI's MCP host (Kiro's ACP `mcpServers` field, or Claude Code's `--mcp-config` flag) as `node <stub.cjs>` with `MEMORY_TOKEN`, `MEMORY_ENDPOINT`, and `MEMORY_SEARCH_ENDPOINT` in its env. For non-2xx note responses, the stub preserves the server-provided `message`/`error` text and returns an MCP tool result beginning with "Memory note was not saved" so hidden `memory_note` failures are readable in the chat/tool outcome.

2. **Session registry** — `createMemoryMcpServer` maintains an in-memory `Map<token, { conversationId, workspaceHash, createdAt, activeChatProfile? }>` and a reverse `Map<convId, token>`. `issueMemoryMcpSession(convId, hash, { activeChatRuntime? })` is **idempotent per conversation**: if a token is already cached for the same `{convId, workspaceHash}` pair, it is returned unchanged and the active chat profile metadata is refreshed; otherwise a fresh 24-byte hex token is minted and stored. Idempotency is load-bearing — the chat route calls this on every message, but the MCP stub is only spawned once per CLI session and captures its bearer token from its spawn-time env forever. Minting a fresh token on every message would revoke the live token the still-running stub is holding, causing subsequent `memory_search`/`memory_note` HTTP calls to fail with `401 Invalid or missing memory token`. Token rotation only happens at real lifetime boundaries: `revokeMemoryMcpSession(convId)` is called on session reset, conversation delete, and workspace-hash change. The optional active chat profile metadata is used only to explain Memory processor failures when the Memory CLI profile differs from the chat runtime profile.

3. **Search endpoint** — `POST /api/chat/mcp/memory/search` (mounted under the chat router at `/mcp/memory/search`). The handler validates `X-Memory-Token`, checks `getWorkspaceMemoryEnabled(hash)`, validates non-empty `query`, clamps `limit` to 1-20, optionally filters by type, maps `status:'active'` to active + redacted and `status:'all'` to active + redacted + superseded + deleted, and calls `chatService.searchWorkspaceMemory`. Results include filename, entry ID, type, source, lifecycle status, score, snippet, and content unless `include_content:false` is passed. Content returned through MCP is capped at 4000 characters with a `truncated` flag. Search is local BM25-style lexical matching and does not use embeddings/Ollama.

4. **Write endpoint** — `POST /api/chat/mcp/memory/notes` (mounted under the chat router at `/mcp/memory/notes`). The handler:
   - Validates `X-Memory-Token` against the session registry (`401` on mismatch)
   - Enforces `getWorkspaceMemoryEnabled(hash)` (`403` if disabled)
   - Deterministically redacts secret-looking raw note content before it is sent to the Memory CLI. The current detectors cover private-key blocks, Authorization bearer/basic values, common API token prefixes (OpenAI, GitHub, Slack, AWS access keys), `password`/`secret`/`token`/`api_key` assignments, and Luhn-valid payment-card numbers. Redacted writes are annotated in `memory/state.json` instead of preserving the original sensitive value.
   - Loads the current merged snapshot for dedup and supersession context
   - Resolves the **Memory CLI profile** from `Settings.memory.cliProfileId` (falling back to legacy `Settings.memory.cliBackend`, then `settings.defaultBackend`)
   - Calls the adapter's `runOneShot(prompt, { model, effort, timeoutMs: 90s, cliProfile })` with a prompt template that lists existing entries and asks the CLI to reply with exactly one JSON decision object: `saved`, `skipped_duplicate`, `skipped_ephemeral`, `redacted_saved`, or `superseded_saved`. Legacy `SKIP: <filename>` and frontmatter-only responses are still accepted for compatibility.
   - On skip decisions, returns `{ ok, skipped, outcome }` and emits a `memory_update` frame with empty `changedFiles` plus `writeOutcomes`. On saved decisions, writes the entry via `addMemoryNoteEntry(hash, { content, source: 'memory-note', filenameHint })`, patches `memory/state.json` with `sourceConversationId`, redaction metadata, and supersession links as needed, returns `{ ok, filename, outcome }`, and emits a workspace-scoped `memory_update` WS frame to connected conversations in the same workspace.
   - Keeps Memory MCP bearer/session auth failures, disabled-workspace failures, processor configuration failures, processor CLI auth failures, processor runtime failures, and processor bad-output failures distinct. Invalid/missing `X-Memory-Token` stays `401 { error: "Invalid or missing memory token" }`; disabled workspaces stay `403`; processor profile resolution/adapter availability returns `500` with `code:"memory_processor_unavailable"`; `runOneShot` authentication-looking failures return `502` with `code:"memory_processor_auth_failed"`; other `runOneShot` errors return `502` with `code:"memory_processor_runtime_failed"`; empty/invalid processor output returns `502` with `code:"memory_processor_bad_output"`. Processor failure responses include `message`, `statusLabel`, and a redacted `memoryProcessor` snapshot with Memory profile/backend, active chat profile/backend when known, mismatch flag, timestamp, and sanitized error text. Credential-looking paths and token values are redacted before returning or persisting error details.
   - Persists the last known processor status at `Settings.memory.lastProcessorStatus` ([ADR-0053](adr/0053-persist-memory-processor-status.md)). Successful write/skip decisions store `last_succeeded`; processor failures store the classified status plus redacted error details. Settings uses this status to show whether the selected Memory processor last succeeded, failed authentication, was unavailable, failed at runtime, or returned bad output.

5. **Consolidation and Memory Review endpoints** — workspace routes call `memoryMcp.proposeMemoryConsolidation(hash)`, `memoryMcp.applyMemoryConsolidation(hash, { summary, actions })`, `memoryMcp.draftMemoryConsolidation(hash, { action })`, and `memoryMcp.applyMemoryConsolidationDraft(hash, { summary, draft })` for low-level manual consolidation actions. Proposal uses the same configured Memory CLI profile as `memory_note`/post-session extraction, but the prompt is review-only: it lists active/redacted entries, with redacted bodies withheld, and asks for JSON actions. Proposal and draft-generation CLI calls use a 10-minute timeout because Memory Review can examine up to 50 memory entries and then draft exact file operations. The direct apply path is deliberately narrower than the proposal schema: only `mark_superseded` actions are automatically applied, and they update sidecar lifecycle metadata plus an audit file. Merge/split/normalize actions require a draft step. Draft generation rejects redacted/deleted/superseded sources, sends full non-redacted source content to the Memory CLI, and expects exact `create` or `replace` operations. Draft apply writes created notes through `addMemoryNoteEntry`, rewrites only selected `notes/*` files through `replaceMemoryNoteEntry`, patches supersession/redaction lifecycle metadata, writes an audit with draft operations, and emits a refresh-only `memory_update`.

   The durable Memory Review workflow wraps those primitives. `startMemoryReviewRun(hash, { source })` persists a `running` review file, starts proposal/draft generation in the background, and returns the running run immediately for HTTP callers that must avoid proxy timeouts ([ADR-0043](adr/0043-start-memory-reviews-asynchronously.md)). `createMemoryReviewRun(hash, { source })` uses the same start path but awaits the background completion for scheduler/worker callers that want the final run. Generation calls the proposal path, stores metadata-only actions as `safeActions`, generates drafts for merge/split/normalize actions, and persists the final run as `pending_review`, `completed`, or `failed`. Scheduled calls reuse an existing actionable run instead of generating another. Manual **Start new review** calls pass `replaceExisting:true`: before generating the new run, existing `running`, `pending_review`, and `failed` runs are marked `dismissed` and any unapplied items in them are marked `discarded`, so the manual action starts from the current memory snapshot rather than opening stale pending drafts. Review item apply/discard/regenerate helpers update the same run file, recompute status, and emit `memory_review_update`. Draft apply accepts an optional reviewed draft payload, but only the Markdown `content` values are taken from it; operation type, target filename, filename hint, and supersedes are copied from the persisted generated draft before validation/redaction. Source fingerprints prevent stale items from applying after the underlying memory changed. The frontend treats applied items as handled work: they remain visible only for a short confirmation window after apply and are hidden from the active review queue on later loads; discarded items remain visible because dismissed drafts can be regenerated.

6. **Wiring into backends** — When a message is sent for a memory-enabled workspace, the stream route delegates to the chat composition shell, which calls `memoryMcp.issueMemoryMcpSession` and passes the resulting array as `mcpServers` on `SendMessageOptions`. When KB is also enabled, `kbSearchMcp.issueKbSearchSession(convId, wsHash)` is called and the resulting `mcpServers` config is merged into the same array. When Context Map is enabled, `contextMapMcp.issueContextMapMcpSession(convId, wsHash)` is also merged in so the CLI receives Memory, KB, and Context Map MCP servers simultaneously when all three features are on. `KiroAdapter` forwards the merged array to ACP's `session/new` and `session/load` (env as an array of `{name, value}` pairs per the ACP spec). `ClaudeCodeAdapter` transforms the array into Claude Code's `--mcp-config` JSON shape via `mcpServersToClaudeConfigJson` (env as a plain `Record<string, string>`, keyed by server name) and passes it as a JSON string argument. These paths spawn the respective `stub.cjs` processes with the correct env vars. KB Search and Context Map tokens are revoked alongside Memory tokens on session reset and conversation delete.

7. **Memory CLI helper** — `BaseBackendAdapter.runOneShot(prompt, options?)` is the cross-backend primitive used by the MCP write handler, the post-session extraction path, KB digestion, and the OCR endpoint. The default base class implementation throws. `memory_search` does not use this helper because it is local lexical retrieval. `ClaudeCodeAdapter` overrides it with `execFile(profile.command || 'claude', ['--print', '-p', prompt, '--model', ...], { env })`, applies `cliProfile.configDir` as `CLAUDE_CONFIG_DIR`, and closes stdin immediately via `child.stdin?.end()` to prevent the Claude CLI's "no stdin data received" warning. `KiroAdapter` overrides it with an **ephemeral ACP session** (spawn `kiro-cli acp` → `initialize` → `session/new` → `[session/set_model]` → `session/prompt` → kill); the same `_acpOneShot` helper also backs `KiroAdapter.generateTitle` and `KiroAdapter.generateSummary` so every Kiro one-shot LLM call goes through ACP. Details in the KiroAdapter section below. ACP is required for Kiro because the alternative `kiro-cli chat --no-interactive` mode interleaves tool-call narration into the answer text whenever the agent decides to call a tool (which OCR forces, since it must `read` the image). Both backends respect a hard timeout (default 60s, adaptive per chunk for KB digestion) and throw a typed error on failure.

**Claude Code project path resolution:** `resolveClaudeProjectDir(workspacePath, configDir?)` first tries the exact sanitized match (`/workspace/path` → `-workspace-path`) and falls back to scanning `<config-root>/projects/` for directories whose name starts with the first 200 sanitized chars — this handles the hashed-suffix case Claude Code uses for long paths, where Bun's hash can't be reproduced in Node. Without a profile `configDir`, `<config-root>` is `~/.claude` (with the `HOME` env var preferred over `os.homedir()` so tests can sandbox the lookup). With a Claude Code profile `configDir`, `<config-root>` is that directory, matching `CLAUDE_CONFIG_DIR`. `resolveClaudeMemoryDir(workspacePath, configDir?)` appends `memory` to the resolved project directory for native memory capture. `ClaudeCodeAdapter.getMemoryDir(workspacePath, { cliProfile })` exposes this same path resolution (wrapped with `resolveCanonicalWorkspacePath` for worktrees) as a pure function the `MemoryWatcher` can call without reading the directory contents. `ClaudeCodeAdapter.getGoal()` uses the same project directory to find session JSONL transcripts for transcript-derived goal status.

**Worktree canonicalization:** Before the path lookup, `ClaudeCodeAdapter.extractMemory` and `getMemoryDir` both call `resolveCanonicalWorkspacePath(workspacePath)`, a pure-filesystem helper that detects git worktrees by looking for a `.git` **file** (not directory) containing a `gitdir:` pointer. When a worktree is detected, it reads the `commondir` file inside the worktree's metadata dir to locate the main repo's `.git` directory, and returns its parent as the canonical workspace path. This ensures all worktrees of one repo share a single memory store. Non-git workspaces (no `.git` entry) and main repos (`.git` is a directory) pass through unchanged, as do any worktrees whose metadata is malformed or whose resolved main repo no longer exists.

### Memory Review Scheduler

**File:** `src/services/memoryReview.ts`.

`MemoryReviewScheduler` is the background worker for scheduled Memory Review generation. `server.ts` starts it after `chatService.initialize()` alongside the KB Auto-Dream scheduler, and `createChatRouter.shutdown()` stops it during graceful shutdown.

The scheduler scans every 60 seconds. For each memory-enabled workspace it reads `WorkspaceIndex.memoryReviewSchedule`; `{ mode:'off' }` disables the workspace. Window schedules require valid `HH:mm` start/end times, support `daily`, `weekdays`, or `custom` day selection, and optionally evaluate against an IANA timezone. A run starts only when all gates pass:

- the current time is inside the configured window;
- no persisted scheduled run at or after `memoryReviewScheduleUpdatedAt` already belongs to the current active schedule window;
- `memoryMcp.isMemoryReviewRunning(hash)` is false;
- there is no pending/failed Memory Review (`hasPendingMemoryReview`);
- the current memory snapshot fingerprint differs from the newest non-running scheduled review at or after `memoryReviewScheduleUpdatedAt`, or no such scheduled review exists yet.

The scheduler calls `memoryMcp.createMemoryReviewRun(hash, { source:'scheduled' })` and does not await it on the scheduler loop. Failures are logged and persisted on the run when possible. Scheduled runs generate proposals/drafts only; applying/discarding remains a user action on the Memory Review page. Manual runs have `source:'manual'`; they are visible in the last-run UI but are deliberately excluded from scheduled-run guard and fingerprint history.

### Context Map Database

**File:** `src/services/contextMap/db.ts`.

`ContextMapDatabase` is the storage foundation for the Context Map feature (issue #281 / ADR-0044). It owns a per-workspace SQLite database expected at `workspaces/{hash}/context-map/state.db` once higher-level services start opening it. The wrapper turns on WAL mode and `foreign_keys = ON`, initializes `CONTEXT_MAP_DB_SCHEMA_VERSION = 2`, and seeds the default system entity type catalog (`person`, `organization`, `project`, `workflow`, `document`, `feature`, `concept`, `decision`, `tool`, `asset`).

The database is the canonical Context Map source of truth. It stores entity types, entities, aliases, facts, relationships, evidence refs/links, processor runs, source spans, conversation cursors, processor candidates, and audit events. Markdown-like content is stored only as field values such as `summary_markdown`, `notes_markdown`, and `statement_markdown`; there is no generated editable Markdown mirror.

`openContextMapDatabase(contextMapDir)` creates the directory and opens `<contextMapDir>/state.db`. The wrapper exposes CRUD for the storage foundation: entity type upsert/list/get, entity insert/list/get, entity sensitivity updates, alias insert/list, fact insert/list/get, relationship insert/list/get, evidence ref upsert/link/list, processor run insert/list/finish/get, source span insert/list/duplicate detection/get, conversation cursor upsert/list/get, candidate insert/list/status update/get, and audit event insert/list/get.

The route-facing Context Map settings slice lives on `ChatService`: `WorkspaceIndex.contextMapEnabled` is toggled independently of Memory and KB, and `WorkspaceIndex.contextMap` stores workspace-level processor mode/settings. The implemented REST routes expose enablement, settings, active-map browsing, entity detail/edit, Needs Attention reads, manual scan, manual scan stop, clear/reset, and candidate edit/apply/dismiss/restore governance.

**Files:** `src/services/contextMap/defaults.ts`, `src/services/contextMap/service.ts`, `src/services/contextMap/sourcePlanning.ts`, `src/services/contextMap/candidatePrimitives.ts`, `src/services/contextMap/autoApply.ts`, `src/services/contextMap/apply.ts`, `src/services/contextMap/mcp.ts`, `src/services/contextMap/stub.cjs`.

`ContextMapService` is the orchestration entrypoint. Source discovery/cursor planning is owned by `sourcePlanning.ts`; candidate type/predicate/fact/sensitivity primitives are owned by `candidatePrimitives.ts`; safe processor auto-apply policy is owned by `autoApply.ts`; active graph mutation remains in `apply.ts`; JSON repair and run metadata helpers remain in `jsonRepair.ts` and `pipelineMetadata.ts`. The split is internal only: scheduler behavior, persisted DB rows, run metadata, candidate ids, and route responses remain the same.

`DEFAULT_CONTEXT_MAP_SCAN_INTERVAL_MINUTES` is `5`, and `DEFAULT_CONTEXT_MAP_CLI_CONCURRENCY` is `1`. Context Map source selection is currently product-owned rather than user-configurable: conversation spans are always processed; `initial_scan` / `manual_rebuild` runs process every discovered workspace instruction, Markdown file, and bounded code-outline packet; and `scheduled` runs discover the same source set but only process changed, new, or previously missing source packets based on `source_cursors`. Root/spec instruction files are still loaded first so their candidates win de-duplication ties.

`ContextMapService` is the asynchronous Context Map processor. `processWorkspace(hash, opts?)` performs normal workspace scans. `processConversationSession(hash, conversationId, sessionNumber, { source })` is the narrower reset/archive finalizer path: it reads one archived session transcript, skips workspace source discovery, and writes a `session_reset` or `archive` run if there is processable transcript content. For an enabled workspace, the processor:

1. Opens the workspace Context Map DB through `ChatService.getContextMapDb(hash)`.
2. Lists non-archived conversations in that workspace for normal scans, or uses the single requested conversation/session for reset/archive finalizer scans.
3. Reads each active session through `getConversation(id)` for normal scans, or reads the archived session file through `getSessionMessages(id, sessionNumber)` for reset/archive finalizer scans.
4. Filters to processable user/assistant messages, excluding progress-turn assistant messages and durable stream-error markers.
5. Compares the active session against `conversation_cursors`.
6. Records a `context_runs` row with `source:'initial_scan'` when the workspace has no existing runs, otherwise `source:'scheduled'`, unless the caller supplied an explicit source such as `manual_rebuild`, `session_reset`, or `archive`.
7. Resolves the configured Context Map processor runtime through `ChatService.resolveCliProfileRuntime()`. Workspace settings with `processorMode:'override'` can override CLI profile/backend/model/effort; otherwise the global `Settings.contextMap` defaults are used, falling back to `Settings.defaultBackend`.
8. Sends each new conversation source span to the configured backend adapter via `runOneShot()` with `timeoutMs: 120000`, `allowTools:false`, the conversation working directory, the resolved CLI profile, and an `AbortSignal` owned by the active Context Map run. The prompt includes only the unprocessed conversation message range, asks for strict JSON `{ candidates: [...] }` output, and tells the model to use filenames/paths/local assets as evidence rather than entities.
9. Builds source packets from product-owned workspace sources for normal `initial_scan`, `manual_rebuild`, and `scheduled` workspace runs only; conversation-scoped reset/archive finalizer runs skip this step entirely. Workspace source packets include workspace instructions, known high-signal Markdown files under the workspace root, up to 120 scored `.md` files discovered recursively under the workspace root, and bounded code-outline packets for selected implementation/configuration files. Initial/manual runs extract every selected packet; scheduled runs compare each selected packet to `source_cursors` and extract only packets with no cursor, a changed stable source hash, or `status:'missing'`. Previously active cursors not found during discovery are marked `missing`, counted in `sourceCursorsMarkedMissing`, and sampled in `staleSources`; no existing graph/candidate/evidence rows are deleted. The recursive Markdown scan skips hard infrastructure/generated-state directories (`.git`, `node_modules`, and `data/chat`), ignores files over 1 MB, sorts files by deterministic path score/order, and truncates each source body to the Context Map source character limit before prompting. Selected Markdown files that become empty, unreadable, oversized, or shim-skipped are treated as unprocessable and can mark an existing cursor `missing`; lower-ranked recursive Markdown files outside the 120-file cap remain discovered/deferred so their cursors are not marked missing only because of the cap. Thin compatibility shims are skipped when the canonical source exists: short `CLAUDE.md` files that defer to `AGENTS.md` are not scanned separately, and root `SPEC.md` redirect/index files are not scanned separately when `docs/SPEC.md` exists. Code-outline source selection is likewise deterministic and bounded: it skips infrastructure/generated directories (`.git`, `node_modules`, `data`, `dist`, `build`, `coverage`, `.next`, `.turbo`, virtualenv/cache/vendor/target/tmp-style folders), ignores lock/minified/generated declaration/map files, ignores files over 300 KB, scores common manifests/configuration files, root/server/app/index entrypoints, route/API files, services/libs, frontend/mobile files, DB/store/repository files, schedulers/managers, and settings/workspace screens, then keeps the top 36 files in packets of six generated outlines. Code outlines include package manifest scripts/dependencies or static summaries of imports, top-level declarations, routes/endpoints, and config keys; they do not send full raw code dumps. Each packet uses its own strict JSON extraction prompt and a stable source hash. Source-packet prompts explicitly tell the model to first judge whether the source deserves any candidates, prefer empty output over weak extraction, treat the scanned file or code path as evidence rather than an entity, avoid ordinary filenames/local paths/source files/root folders/incidental assets, use the built-in entity type catalog unless separately proposing a `new_entity_type`, include concise durable facts on entities when the source supports them, use `feature` for user-facing capabilities/behavior areas/feature proposals, use `subjectName`/`predicate`/`objectName` for relationships, include `evidenceMarkdown` on every proposed relationship, avoid loose associations as relationships, include relationship candidates only for durable reporting, ownership, authorship, dependency, storage, workflow/tool usage, or implementation links, avoid duplicate project-prefixed/unprefixed names for the same concept, treat maintained specs/ADR collections/roadmaps/plans as `document` entities instead of workflows, put `sensitivity` directly on a new sensitive entity, and keep the output small enough that Context Map remains useful without normal manual review. The source prompt gives shape-specific guidance: blog posts/article drafts/essay-like sources should usually produce at most one durable document or concept; contact/profile files should usually produce the person plus only the most durable related items and may include up to two high-confidence durable relationships; workflow/runbook files should usually produce one workflow plus critical durable tools/rules and may include up to two high-confidence durable relationships to tools/documents/projects; code-outline packets should extract only stable implementation areas such as services, API surfaces, data stores, schedulers, backend adapters, frontend screens, mobile clients, MCP servers, build/runtime tooling, and durable test harnesses while avoiding entities for ordinary functions/classes/imports/route strings/files/directories/package names/dependencies. The deterministic source budget keeps at most four candidates from workspace instructions, three from `AGENTS.md`/`CLAUDE.md`, five from `README.md`, `SPEC.md`, and `docs/SPEC.md`, four from `workflows/*` and `context/contact-*`, three from `drafts/*`, two from blog/theme content under `repos/*`, five from other Markdown source packets, and eight from code-outline packets before synthesis. For source shapes that commonly contain durable edges (`context/contact-*`, `workflows/*`, `context/*`, README/SPEC sources), the cap reserves one slot for a strict evidence-backed relationship when the processor emitted one; code-outline packets reserve up to two relationship slots so entity candidates do not starve useful graph edges before synthesis.
10. Parses candidate types (`new_entity`, `entity_update`, `entity_merge`, `new_relationship`, `relationship_update`, `relationship_removal`, `new_entity_type`, `alias_addition`, `evidence_link`, `sensitivity_classification`, `conflict_flag`) into pending `context_candidates`. Before any candidates are persisted, deterministic normalization maps common noisy type slugs (`product`, `feature_proposal`, `capability`, `subsystem`, `backend`, `issue`, `github_issue`, `pull_request`, `architecture`, `security_policy`, `principle`, etc.) into the seeded built-in types, drops redundant `new_entity_type` candidates for built-in or aliased built-in types, preserves a custom type slug on entity candidates only when the same processor output includes a `new_entity_type` candidate for that slug, normalizes aliases plus canonical facts, merges alternate fact fields (`factsMarkdown`, `factMarkdown`, `keyFacts`, `durableFacts`, `factStatements`) into `payload.facts`, normalizes legacy relationship payload keys (`sourceName`/`fromName`, `targetName`/`toName`, `relationshipType`) into the applyable schema, normalizes common relationship predicates such as `supports_backend` -> `supports` and `is specified by` -> `specified_by`, and normalizes sensitivity labels such as `private_personal_data` and `secret_pointer_only` into supported sensitivity values. Source-path sensitivity correction can override obvious work-source mismatches to `work-sensitive` and obvious personal-source mismatches to `personal-sensitive`; `secret-pointer` is never downgraded. Relationship candidates must use a governed durable predicate from the product-owned predicate set (`supports`, `uses`, `depends_on`, `part_of`, `specified_by`, `documents`, `implements`, `stores`, `requires`, `references`, `relates_to`, and related stable lifecycle/ownership/configuration predicates); ambiguous structural labels such as `extends` and comparative or one-off predicates such as `preserves_data_unlike`, `compared_to`, `versus`, `not_*`, or `does_not_*` are dropped before application. Self-relationships are dropped both before and after endpoint resolution. The extraction prompt and scoring treat `implements` / `implemented_by` as implementation ownership/component relationships only; UI placement, navigation, and access details are facts unless the endpoint type is a concrete implementation component (`concept`), `tool`, `workflow`, or `project`. `part_of` edges into a project/root entity require at least `0.8` confidence so weak issue, bug, UI-state, or loose project associations become facts instead of broad graph edges. A deterministic guard then drops obvious file/path noise: `new_entity` candidates whose `document` or `asset` name looks like a local filename/path or matches the source file, root workspace-folder entity candidates, filename-like alias additions, relationships whose subject or object is a local filename/path, and `evidence_link` candidates without a concrete target kind/id. Sensitivity classifications are kept only when they target an already active entity; when they target a `new_entity` in the same processor output, the sensitivity is folded into that `new_entity` payload and the separate classification candidate is dropped; orphan classifications are dropped. The same cleanup pass canonicalizes duplicate entity variants across the current run, adds losing names as aliases on the winning `new_entity`, converts `new_entity` drafts into `entity_update` drafts when they match an already active entity by exact/canonical name, drops no-op update drafts whose source rescan repeats data already present on the active entity, drops model-emitted `entity_update` drafts that do not resolve to active entities, and resolves `new_relationship` endpoints against active entities, existing pending `new_entity` candidates, and kept same-run `new_entity` candidates. Resolved relationships are scored by predicate, subject/object entity-type compatibility, confidence, and relationship evidence; low-scoring or generic relationships are dropped, and when a rejected relationship has useful evidence and its subject is a same-run entity candidate, that evidence is folded into the subject candidate's `facts` instead of creating a weak edge.
11. When deterministic cleanup leaves at least eight candidate drafts, or at least three candidate drafts for a `scheduled` run, `ContextMapService` runs synthesis/ranking through the same configured processor with `timeoutMs: 180000`, `allowTools:false`, the workspace path, and the active run's abort signal. For small sets up to 36 candidates, the service runs one `single` synthesis pass. Larger sets are split into natural buckets (`conversation:*`, `context`, `context-contacts`, `workflows`, `repo-content`, `drafts`, or source-folder fallback) and chunks of at most 36 candidates. Each chunk pass receives compact candidate payloads, stable `sourceRef` ids, active entity names/aliases for context, and targets at most 10 full candidate outputs. Chunk passes are independent and may run concurrently, but their outputs and stage metadata are restored to original chunk order before the reduced set is cleaned again. When at least eight reduced candidates remain, or at least three reduced candidates for a `scheduled` run, the service ranks the reduced set, sends at most 50 compact summaries to a `final` arbiter pass, and asks for decisions rather than rewritten candidate payloads. The final arbiter returns `keepRefs`, `dropRefs`, `mergeGroups`, `typeCorrections`, and `relationshipToFactRefs`; the backend applies those decisions to the already-normalized candidates. The final prompt has a normal target of 34 or fewer candidates and explicitly tells the model not to fill the budget; the hard safety cap remains 45 candidates.
12. Synthesis output is parsed through the same candidate normalizers and deterministic cleanup runs again afterward. Candidate facts are normalized to string arrays before persistence; object-shaped fact values such as `{ markdown }`, `{ text }`, `{ value }`, `{ content }`, `{ statementMarkdown }`, or `{ summaryMarkdown }` and alternate fact fields such as `factsMarkdown` are converted to canonical readable fact strings so the UI/apply path never sees `[object Object]` fact values. Chunk synthesis attaches the primary cited `sourceSpan` back to every synthesized candidate and records extra cited spans in `payload.relatedSourceSpans` when multiple source units support a merged candidate. Final arbiter merge groups preserve the canonical candidate's source span, carry loser source spans into `relatedSourceSpans`, preserve losing names as aliases, can override the canonical name/type/summary, and can add durable facts. `relationshipToFactRefs` folds weak relationship evidence into facts on kept subject entities instead of persisting noisy graph edges. Same-name `new_entity` candidates with conflicting type slugs are resolved before persistence by choosing the highest-scoring type/name candidate, preserving weaker names as aliases, preserving weaker summaries as facts when useful, and carrying related source spans forward. After synthesis, low-value source-local entities such as assets, weak concepts/features/tools, and non-maintained documents can be folded into facts on a stronger same-source parent entity. The service also recovers up to 12 strict relationship candidates from the original extraction when synthesis kept both endpoints, the predicate is governed/non-generic, confidence is at least `0.84`, and evidence is present. Relationship candidates are then pruned to a maximum of 12 highest-value relationships so weak edge proposals do not dominate Needs Attention. Malformed synthesis JSON gets one repair pass (`timeoutMs: 90000`, `allowTools:false`) before fallback; if the repaired JSON parses, the stage records `repairAttempted:true` and `repairSucceeded:true` and continues without fallback. Failed repair, synthesis timeout, or invalid refs do not fail the scan; each failed chunk falls back to a ranked top subset of that chunk, and a failed final arbiter falls back to the ranked reduced set capped at 40 candidates before deterministic cleanup/folding. Empty synthesized output is allowed when the processor intentionally returns no candidates. Run metadata includes `candidateSynthesis` with `attempted`, `inputCandidates`, `outputCandidates`, `inputCandidateTypes`, `outputCandidateTypes`, `droppedCandidates`, `targetCandidates`, `hardMaxCandidates`, `openQuestions`, optional `recoveredRelationshipCandidates`, per-stage `stages[]` entries (`stage`, `chunkId`, input/output/drop counts, input/output candidate type counts, target/hard-cap counts where applicable, open questions, fallback, sampled error, and optional repair metadata), plus `fallback`, `errorMessage`, and `fallbackBound` when applicable. The report script warns when relationship candidates entered synthesis but none survived persistence.
13. In one transaction, inserts deterministic `source_spans`, updates `conversation_cursors`, updates successful workspace `source_cursors`, marks missing source cursors, and inserts pending candidates while skipping exact duplicate candidate ids. Source-packet candidate ids omit the volatile `sourceSpan.runId` from the identity payload, so a repeated manual scan over unchanged Markdown files does not recreate identical pending work. Before insertion, the extraction pass also de-duplicates semantically equivalent candidates within the same run, such as the same entity/type/relationship emitted from both README and SPEC source packets, and canonicalizes common project-prefixed/generic variants such as project specifications or ADR collections. Context Map JSON parsing first applies a deterministic local repair for the common model error where adjacent array values are emitted without a comma; this applies to extraction, chunk synthesis, and final arbiter output and avoids unnecessary repair/fallback work when the intended JSON is otherwise unambiguous. Malformed extraction JSON that still cannot be parsed gets one repair pass through the same Context Map JSON repair processor used by synthesis, with the extraction `{ candidates: [...] }` shape and no tools; successful repairs are parsed normally and record `extractionRepairs` metadata (`attempted`, `succeeded`, `failed`, sampled failures). Extraction failures are isolated per conversation span or source packet: unrepaired malformed JSON, invalid candidate shape, backend timeout, or backend error from one unit records `extractionUnitsFailed` / `extractionFailures` in run metadata and leaves that unit retryable, while successfully parsed or repaired units in the same run still commit their source spans/cursors/candidates. Failed source packets do not advance `source_cursors`, so the next scheduled or manual pass can retry them. When at least one extraction unit succeeds, the run finishes `completed` with a warning in `error_message`; when every attempted extraction unit fails, processor resolution fails, or the final transaction fails, the run is marked `failed` and no failed unit's cursor/span/candidates are committed. If the active run is stopped, the run is marked `stopped` instead, any in-flight one-shot processor call is aborted when the adapter supports it, and no cursors/source spans/candidates are committed for the interrupted range so the next interval or manual scan can retry it.
14. After insertion, `ContextMapService` attempts to auto-apply safe candidates. Auto-apply requires a still-pending candidate with `payload.sourceSpan` and a type-specific confidence threshold: `0.80` for safe additive `new_entity` candidates, `0.80` for durable `new_relationship` candidates whose endpoints already resolve without pending dependencies, `0.90` for conservative additive `entity_update` candidates, `0.94` for `alias_addition`, and `0.96` for `sensitivity_classification` / `evidence_link`. New entities must use a known built-in/existing entity type, include a summary/notes/facts body, and can auto-apply with `normal`, `work-sensitive`, or `personal-sensitive` sensitivity but never with `secret-pointer`; relationship candidates must include evidence, use a durable non-generic predicate, not connect an entity to itself, and have no pending endpoint-entity dependencies. Entity updates can auto-apply only when they are additive: facts, aliases, evidence, or filling an empty summary/notes field. They do not auto-apply when they rename an entity, change type/status/sensitivity, or overwrite an existing summary/notes field. Sensitivity classifications can auto-apply only when they move the resolved active entity to a more restrictive class (`normal` to work/personal/secret, or work/personal to secret); downgrades, no-ops, and work-vs-personal lateral changes stay pending. Each completed run re-evaluates both newly inserted candidates and existing pending candidates, so candidates left pending under an older policy can become active once they satisfy the current safe auto-apply rules. Auto-application uses `applyContextMapCandidate()` with `appliedBy:'processor'`, so it creates the same active graph/evidence rows as a manual apply and writes an audit event that identifies processor application. Candidates that do not meet the safe auto-apply rules, fail application, or require user judgment remain `pending` and are counted as Needs Attention in run metadata. Run metadata includes `candidatesInserted`, `candidatesAutoApplied`, `existingCandidatesAutoApplied`, `candidatesNeedingAttention`, and sampled `autoApplyFailures`.

Extraction and synthesis processor calls are bounded by process-wide queues in the server process. `Settings.contextMap.extractionConcurrency` (default 3, clamped 1..6) controls the global extraction cap; extraction spans/source packets run as ordered jobs across all active Context Map scans, and extraction JSON repair calls use the same extraction queue. `Settings.contextMap.synthesisConcurrency` (default 3, clamped 1..6) controls the global synthesis cap; synthesis chunk calls, final arbiter calls, and synthesis/arbiter JSON repair calls share that separate queue across all active scans. Queued work checks the active run's abort signal before starting, while active backend calls receive the same abort signal. Deterministic merge steps restore extraction job results, chunk outputs, and stage metadata to source/chunk order before persistence.

Run metadata also includes source discovery counters (`sourcePacketsDiscovered`, `sourcePacketsProcessed`, `sourcePacketsSkippedUnchanged`, `sourcePacketsSucceeded`, `sourceCursorsMarkedMissing`, and sampled `staleSources`) plus a `timings` diagnostic object. Timings record total, planning, source-discovery, extraction, synthesis, persistence, and auto-apply durations in milliseconds; extraction unit totals plus the 20 slowest conversation/source units; and synthesis stage durations for single, chunk, and final arbiter passes. The report script prints these fields plus source cursor status counts and missing-source samples in text and JSON output so scan performance and staleness work can identify slow phases or source drift without changing extraction behavior.

`ContextMapService.stopWorkspace(hash)` stops the active workspace run when one exists. It aborts the run's `AbortController`, marks the current `context_runs` row `stopped` with `Stopped by user`, emits a Context Map update, and returns `true`; otherwise it returns `false`. Stop is cooperative with the backend adapter but also checked between extraction units, so it is safe for scheduled, initial, manual, reset, and archive-triggered runs. Stop is a run-control action only: it does not disable Context Map, does not clear existing map/review data, and leaves the pending work retryable.

`ContextMapScheduler` is created by `createChatRouter()`, started from `server.ts`, and stopped during chat-router shutdown. It immediately checks enabled workspaces on start, then wakes every 60 seconds. Per check, it reads global `Settings.contextMap.scanIntervalMinutes` and `cliConcurrency`, then applies any per-workspace `WorkspaceIndex.contextMap.scanIntervalMinutes` override when deciding whether that workspace is due. Due workspace scans run with the configured global concurrency cap. Because `ContextMapService` persists source spans and cursors only after successful extraction, scheduler scans are idempotent and can safely run after restarts. The `PUT /context-map/enabled` route also starts `ContextMapService.processWorkspace(hash)` in the background when a workspace changes from disabled to enabled so initial setup does not depend on the manual scan button; errors are logged and the normal update fan-out reports started, stopped, failed, and successful runs.

`createChatRouter()` also enqueues `context_map_conversation_final_pass` session-finalizer jobs after resetting or archiving an enabled workspace conversation. Those jobs call `ContextMapService.processConversationSession(hash, convId, sessionNumber, { source:'session_reset' | 'archive' })` in the background, so reset/archive responses do not wait on Context Map processing. Finalizer passes reuse conversation cursor/source-span idempotency for the exact archived session range, skip workspace source discovery, and retry later when another Context Map run is already active for the workspace.

The route-facing active graph/needs-attention slice exposes compact active-map browsing, pending candidate governance, and active-graph apply/edit paths. `GET /workspaces/:hash/context-map/graph` returns entities plus active relationships for the Workspace Settings Active Map section, with local filtering by query, type, entity status, sensitivity, and limit, and without opening storage while the feature is disabled. The default entity status filter is `active`; `status=all` includes every lifecycle state. Query matching uses entity names and aliases for every entity, but only searches summary, notes, and facts for non-`secret-pointer` entities. `GET /workspaces/:hash/context-map/entities/:entityId` returns a selected entity detail bundle with aliases, facts, relationships, evidence refs, and entity-target audit events; `secret-pointer` entities withhold readable/fact/evidence fields and audit-event details. `PUT /workspaces/:hash/context-map/entities/:entityId` lets the user edit active entity fields (`name`, `typeSlug`, `status`, `sensitivity`, `summaryMarkdown`, `notesMarkdown`, `confidence`), validates existing entity types and allowed statuses/sensitivities, clamps confidence, records an entity `edited` audit event, and emits a Context Map update. `GET /workspaces/:hash/context-map/review` returns filtered `context_candidates`, per-status counts, and referenced/recent `context_runs`; its default `status=pending` is the Needs Attention queue, while dismissed-history views request `status=discarded` or `status=all`. `DELETE /workspaces/:hash/context-map` clears graph, candidate, evidence, run, cursor, and audit state while preserving the workspace enablement/settings and seeded system entity types, but returns `409` while a scan is active so an in-flight scan cannot repopulate the cleared store. `POST /workspaces/:hash/context-map/scan` starts a manual pass with source `manual_rebuild` in the background, returns immediately with `{ ok:true, started:true, source:'manual_rebuild' }`, uses the same cursor/source-span deduplication and candidate extraction path as scheduled/final scans plus enabled high-signal workspace source packets, requires Context Map to be enabled, and returns `409` when the workspace is already being processed. `PUT /workspaces/:hash/context-map/enabled` stops any active run before disabling the feature. Progress and completion are communicated by the persisted `context_runs` rows and `context_map_update` fan-out rather than by holding the HTTP request open. `POST /workspaces/:hash/context-map/scan/stop` stops the current run even if the workspace was disabled after the run started, returns `409` when nothing is running, and leaves retryable work untouched. Disabled workspaces return an empty Needs Attention queue without opening a Context Map DB. `PUT /workspaces/:hash/context-map/candidates/:candidateId` lets the user edit a pending candidate payload and confidence before manual application, preserves source-span provenance, and records an `edited` audit event. `applyContextMapCandidate()` supports every processor candidate type currently emitted by `ContextMapService`: `new_entity_type`, `new_entity`, `entity_update`, `entity_merge`, `alias_addition`, `sensitivity_classification`, `new_relationship`, `relationship_update`, `relationship_removal`, `evidence_link`, and `conflict_flag`. Applying creates/reuses active entity types/entities/relationships, updates entity or relationship fields, marks merged entities and removed relationships with lifecycle statuses, adds aliases/facts/evidence links when supplied, updates entity sensitivity, links source-span evidence to the candidate and active target, marks the candidate `active` with `appliedAt`, and writes an `applied` audit event whose details include `appliedBy:'processor'` for auto-applied candidates and `appliedBy:'user'` for manual applies. Relationship candidates require existing subject/object entities or relationship ids. When a new-relationship candidate references a missing endpoint by exact name and a single pending `new_entity` candidate matches that endpoint in the same workspace, `applyContextMapCandidate()` raises a dependency error instead of mutating state; the route returns those dependency candidate ids/roles/names so the UI can confirm. A retry with `includeDependencies:true` applies the endpoint entity candidates first and the relationship second inside one transaction. Ambiguous pending endpoint matches require user attention. Other invalid payloads return an apply error rather than changing state. `POST /context-map/candidates/:candidateId/discard` marks only pending/stale/conflict/failed candidates `discarded`; `POST /context-map/candidates/:candidateId/reopen` returns only discarded candidates to `pending` while treating already-pending candidates idempotently. Those candidate-decision routes require Context Map to be enabled and write an `audit_events` row with the previous status when they change state.

`ChatService.getContextMapStatus(hash)` returns the compact `ConversationContextMapStatus` used by `GET /conversations/:id` and live `context_map_update` frames. It includes enablement, pending flag, candidate counts (`pending`, `stale`, `conflict`, `failed`), run counts (`running`, `failed`), and latest/last run metadata. Context Map routes, processor run start, and processor completion call the router's workspace fan-out helper after mutations so every connected conversation in the workspace patches `conv.contextMap` and can show or clear the composer notification without waiting for a full conversation reload.

`createContextMapMcpServer()` exposes read-only active graph retrieval to chat CLIs through the same bearer-token stdio-stub pattern as Memory and KB Search. `issueContextMapMcpSession(convId, hash)` returns an `agent-cockpit-context-map` MCP server config that launches `src/services/contextMap/stub.cjs` with `CONTEXT_MAP_TOKEN` and `CONTEXT_MAP_ENDPOINT`. The HTTP route `POST /mcp/context-map/call` checks `X-Context-Map-Token`, verifies Context Map is still enabled for the workspace, and dispatches only read-only tools: `entity_search`, `get_entity`, `get_related_entities`, and `context_pack`. Search is local lexical matching over active entity names and aliases, plus summaries, notes, and active facts for non-`secret-pointer` entities. Detail/pack responses include active relationships and optional evidence references; `secret-pointer` entities withhold summary, notes, facts, and evidence content from MCP results.

### Workspace Knowledge Base

Per-workspace **Knowledge Base** gives each workspace a curated store of documents the CLI can consult on demand. The feature is **opt-in per workspace** via `WorkspaceIndex.kbEnabled` (defaults to `false`) and is independent of the Memory toggle — enabling one does not enable the other. When the toggle is off, every KB code path (pointer injection, pipeline, UI browser) short-circuits and the `knowledge/` directory is never created on disk.

The feature is delivered across five sequential PRs. **PR 1** landed the storage layer, types, settings plumbing, feature toggles, and system-prompt pointer. **PR 2** added the ingestion stage: per-format handlers, background conversion, the upload endpoint, and the KB Browser UI for uploading/listing/deleting raw files. **PR 3** rewrites the metadata layer as SQLite (`knowledge/state.db`), introduces virtual folders with multi-location (Option B) ref-counted deletes, adds the per-raw digestion pipeline with its own CLI, an auto-digest toggle, a pending-delete state, a batch "Digest All Pending" runner, and the Entries tab in the KB Browser. **PR 4** adds the Dreaming pipeline — cross-entry synthesis that builds a knowledge graph of topics and connections, the Synthesis tab in the KB Browser, and the dream banner in the chat input area. **PR 5** (current) adds the Reflection layer — a 5th dreaming phase that identifies clusters of connected topics and generates cross-cluster reflections (patterns, contradictions, gaps, trends, insights) with entry citations, plus the `knowledge_search` MCP tool.

**Pipeline stages:**
1. **Ingestion** — Our code. Accepts a raw upload, stores it under `knowledge/raw/<rawId>.<ext>`, extracts text + media into `knowledge/converted/<rawId>/` via format-specific handlers. LibreOffice is used as an opt-in helper for PPTX slide rasterization when `Settings.knowledgeBase.convertSlidesToImages` is true.
2. **Digestion** (PR 3 active, KB vNext chunked path active) — Per-raw. Builds deterministic chunks from `kb_documents`/`kb_document_nodes` via `planDigestChunks()`, using both the structural unit budget and estimated converted-text size per unit, extracts the matching converted Markdown range from `converted/<rawId>/text.md`, then spawns the Digestion CLI profile (`Settings.knowledgeBase.digestionCliProfileId`, legacy fallback `digestionCliBackend`) once per chunk. Each prompt includes source metadata, chunk ID, chunk index/total, unit range, node titles, and only that chunk's Markdown. The CLI can still follow relative media links under `converted/<rawId>/`. If `Settings.knowledgeBase.kbGleaningEnabled` is true, the chunk runs through `runSessionShot([digestPrompt, gleaningPrompt], ...)` so the second prompt asks for missed entries against the same source range; otherwise digestion uses one `runOneShot()` call per chunk. Backends that implement `runOneShot()` but do not provide a native multi-prompt session still support gleaning through `BaseBackendAdapter`'s transcript replay fallback. Strict mode is used: every chunk must parse successfully before replacement entry files are staged and the old DB rows are replaced. Successful chunk outputs produce one or more entry documents (YAML frontmatter + markdown body) which are parsed, slug-deduped, written to `knowledge/entries/<entryId>/entry.md`, and indexed in the `entries` + `entry_tags` SQLite tables. Workspaces without document structure fall back to the previous whole-document prompt.
3. **Dreaming** (PR 4+5 — active) — Cross-entry synthesis, manual trigger. Spawns the Dreaming CLI profile (`Settings.knowledgeBase.dreamingCliProfileId`, legacy fallback `dreamingCliBackend`) in a five-phase pipeline: Routing → Verification → Synthesis → Discovery → Reflection. Produces a knowledge graph stored in SQLite (`synthesis_topics`, `synthesis_topic_entries`, `synthesis_connections`, `synthesis_reflections`, `synthesis_reflection_citations`) and materialized as markdown files under `knowledge/synthesis/` and `knowledge/synthesis/reflections/`. Entry→topic assignments are many-to-many. Topic→topic connections carry a relationship label and confidence level (`extracted | inferred | speculative`). Reflections are cross-cluster insights citing specific entries. See the **Dreaming pipeline** section below for full details.

**Document structure and chunk planning** (`documentStructure.ts`, `chunkPlanner.ts`, `sourceRange.ts`): After ingestion writes `converted/<rawId>/text.md` and `meta.json`, deterministic structure generation persists one `kb_documents` row plus ordered `kb_document_nodes`. PDF-style `## Page N` blocks become page nodes, PPTX-style `## Slide N` blocks become slide nodes, Markdown headings become hierarchical section nodes, and unknown documents get one fallback root range. Existing converted raws can be repaired without re-uploading: `KbIngestionService.backfillDocumentStructures(hash, { force? })` scans raw rows and builds missing structure from `converted/<rawId>/text.md`, while `rebuildDocumentStructure(hash, rawId)` force-rebuilds one raw for the Trace modal. Missing converted text is skipped, not fatal, so legacy failed raws do not block a workspace-wide backfill. `planDigestChunks(document, nodes, { maxUnitsPerChunk?, maxEstimatedTokensPerChunk?, unitTextLengths? })` preserves natural nodes when they fit, fills uncovered unit gaps with fallback chunks, merges adjacent small structure nodes up to both the unit budget and estimated token budget, splits oversized structure nodes deterministically with reason `split-large-node`, and falls back to complete ordered unit ranges when no nodes exist. Digestion passes `unitTextLengths` derived from `converted/<rawId>/text.md` so text-heavy DOCX/Markdown sections split before one prompt gets too large; callers that omit `unitTextLengths` keep the unit-only behavior. `extractSourceRange()` and `estimateSourceUnitTextLengths()` are shared by trace/source reads and chunked digestion so both paths slice page/slide/section/line units consistently.

**Glossary query expansion** (`glossary.ts`): `expandGlossaryQuery(query, db.listGlossary())` applies workspace terms to MCP `search_entries` and `search_topics` only. Matching is case-insensitive and whole-word (`employee` does not match `OEE`), multi-word terms allow whitespace between words, and each match is expanded inline as `term (expansion)`. Overlapping glossary terms are resolved against the original query by longest non-overlapping span first, so `OEE target` wins over `OEE`, and already-inserted expansion text is never expanded again. MCP responses include `query`, and when expansion occurred also include `expanded_query` plus `glossary_matches` for traceability. ID-based tools (`get_topic`, `find_similar_topics`, `find_unconnected_similar`, `get_document_structure`, `get_source_range`) do not expand because they do not accept free-form query text.

Both active stages run on a **per-workspace bounded-parallelism queue** (`WorkspaceTaskQueue` in `src/services/knowledgeBase/workspaceTaskQueue.ts`). The queue dispatches up to `Settings.knowledgeBase.cliConcurrency` tasks at a time (default 2) and is shared between `KbIngestionService` + `KbDigestionService` via a single `WorkspaceTaskQueueRegistry` constructed in `routes/chat.ts` — so the budget is unified across both pipelines per workspace (e.g. with `cliConcurrency=2`, a workspace can have 2 ingestions, or 1 ingestion + 1 digestion, but never 3 of any combination). Folder ops (`createFolder` / `renameFolder` / `deleteFolder`) dispatch via `runBarrier()`: the queue waits for every in-flight task to settle, runs the barrier alone, then resumes normal dispatch — this prevents an `UPDATE raw_locations` rewrite during a folder rename from racing against an ingestion writing a fresh location row. Each entry point reads the latest `cliConcurrency` from settings and pushes it to the registry on the next dispatch, so settings changes propagate without recreating the queue. Cross-workspace work runs in parallel — the registry is keyed by hash. Dreaming runs against its own session-driven runner that already reads `cliConcurrency` for its synthesis batches and stays independent of this queue.

**SQLite layer** (`src/services/knowledgeBase/db.ts`): Each workspace owns one `knowledge/state.db` (better-sqlite3, WAL mode, `foreign_keys = ON`). The `KbDatabase` class is the only code allowed to talk to the DB — everything else goes through it. Schema:
- **`meta`** — `{ schema_version, created_at }` key/value store. Schema is versioned via `KB_DB_SCHEMA_VERSION` and bumped on destructive migrations.
- **`raw`** — one row per content-addressed raw file: `{ raw_id, sha256, status, byte_length, mime_type, handler, uploaded_at, digested_at, error_class, error_message, metadata_json }`. Indexed on `status` and `sha256` (the dedupe lookup key).
- **`folders`** — virtual folder tree keyed on `folder_path` (root is the empty string `''`, always present after `_ensureRootFolder`). Folders are purely a metadata construct — no on-disk directories are created.
- **`raw_locations`** — `(raw_id, folder_path, filename)` junction table. Same `raw_id` can live in multiple folders (Option B multi-location) so uploading the same bytes under a different name or in a different folder doesn't duplicate the raw file; it just adds a location row. `ON DELETE CASCADE` from `raw` and `ON DELETE RESTRICT` from `folders` so the orchestrator has to explicitly empty a folder before dropping it.
- **`kb_documents`** — one document-structure root per raw file: `{ raw_id, doc_name, doc_description, unit_type, unit_count, structure_status, structure_error, created_at, updated_at }`. Cascades on raw delete. This is the durable range index root used by the KB vNext chunking/retrieval pipeline.
- **`kb_document_nodes`** — document range nodes keyed by `(raw_id, node_id)`: `{ node_id, raw_id, parent_node_id, title, summary, start_unit, end_unit, sort_order, source, metadata_json }`. `source` is `deterministic | ai | fallback`. Cascades from `kb_documents` and is ordered by `sort_order` for structure display and later chunk planning.
- **`entries`** — digested entry metadata: `{ entry_id, raw_id, title, slug, summary, schema_version, stale_schema, digested_at, needs_synthesis }`. `entry_id` is stable across redigests (format: `<rawId>-<slug>[-<n>]`). Cascades on raw delete. `stale_schema` is set when the current `entrySchemaVersion` is newer than the row's. `needs_synthesis` (INTEGER DEFAULT 1) is set on new digestion, re-digestion, or when a co-topic entry is deleted — tracks which entries the next dream run should process. Entry delete helpers mark surviving co-topic entries stale before SQLite cascades remove `synthesis_topic_entries`, preserving the topic-neighborhood signal needed for incremental Dreaming. Partial index on `(entry_id) WHERE needs_synthesis = 1`.
- **`entry_tags`** — `(entry_id, tag)` many-to-many, cascades on entry delete.
- **`kb_entry_sources`** — one row per source range that contributed to a digested entry: `{ entry_id, raw_id, node_id, chunk_id, start_unit, end_unit }` with primary key `(entry_id, raw_id, chunk_id, start_unit, end_unit)`. `node_id` is a representative node only when the chunk maps to exactly one document node; merged multi-node chunks store `NULL` and rely on the explicit unit range to avoid misleading first-node provenance. Cascades from both `entries` and `raw`, so redigestion and raw deletion clear old lineage automatically. `listEntrySources(entryId)` joins this table to `kb_documents` and `kb_document_nodes` to return `docName`, `unitType`, and `nodeTitle` for entry-detail provenance.
- **`kb_glossary`** — workspace-scoped query expansion table: `{ id, term, expansion, created_at, updated_at }`. `term` is `UNIQUE COLLATE NOCASE` so `OEE` and `oee` cannot coexist. The KB Settings tab owns CRUD, and MCP query tools apply whole-word, case-insensitive expansion before vector/keyword search.
- **`synthesis_meta`** — `(key, value)` key/value store for dream run metadata (`status`, `last_run_at`, `last_run_error`, `god_nodes`).
- **`synthesis_topics`** — `{ topic_id, title, summary, content, updated_at }`. One row per discovered topic.
- **`synthesis_topic_entries`** — `(topic_id, entry_id)` many-to-many junction. Cascades from both `synthesis_topics` and `entries`. Entry→topic membership IS the lineage — no separate manifest needed.
- **`synthesis_connections`** — `{ source_topic, target_topic, relationship, confidence, evidence }`. `UNIQUE(source_topic, target_topic)`. Cascades from `synthesis_topics` on either FK.
- **`synthesis_reflections`** — `{ reflection_id TEXT PRIMARY KEY, title TEXT NOT NULL, type TEXT NOT NULL, summary TEXT, content TEXT NOT NULL, created_at TEXT NOT NULL, original_citation_count INTEGER NOT NULL DEFAULT 0 }`. One row per reflection. `type` is one of `pattern | contradiction | gap | trend | insight`. `original_citation_count` stores the number of cited entries at insertion time — used by `listStaleReflectionIds()` to detect when cascade-deletes silently remove citation rows.
- **`synthesis_reflection_citations`** — `(reflection_id TEXT NOT NULL, entry_id TEXT NOT NULL)` many-to-many junction. `REFERENCES synthesis_reflections(reflection_id) ON DELETE CASCADE`, `REFERENCES entries(entry_id) ON DELETE CASCADE`. `PRIMARY KEY (reflection_id, entry_id)`. Index on `entry_id` for reverse lookups (finding all reflections that cite a given entry).

All writes go through prepared statements. Folder ops use `transaction()` wrappers so rename/cascade delete roll back atomically. Entry inserts also run in a transaction so the row and its tags land together. `normalizeFolderPath` strips leading/trailing slashes, collapses repeated slashes, rejects `..` / control chars / empty segments / segments longer than 128 chars / total paths longer than 4096 chars, and returns `''` for root — called from every DB method that touches a folder path so the rules live in one place.

**Migration from Phase 1/2 state.json**: `openKbDatabase({ dbPath, legacyJsonPath, rawDir })` handles the one-shot migration:
1. If `state.db` already exists → open and return (idempotent).
2. Else if `state.json` exists → open a fresh DB, read the legacy JSON, **re-hash every raw file from disk** (the legacy format only kept the 16-char `rawId`; the DB needs the full `sha256` to dedupe), insert `raw` + `raw_locations` rows in one transaction, snap any `ingesting`/`digesting` row to `failed` (those states mean the server died mid-run), and rename the old JSON to `state.json.migrated` as a one-release safety copy.
3. Else → open a fresh DB with the empty schema + root folder only.

Files missing on disk during migration are still inserted with `sha256 = rawId` and a warning log — we'd rather surface the broken row in the UI than silently drop it.

**`KbState` shape** surfaced by `GET /kb` (snapshot assembled on every read from the DB, not persisted separately):
```typescript
{
  version: number,              // DB schema version from meta
  entrySchemaVersion: number,   // KB_ENTRY_SCHEMA_VERSION from digest.ts
  autoDigest: boolean,          // mirrors WorkspaceIndex.kbAutoDigest
  autoDream: KbAutoDreamConfig, // mirrors WorkspaceIndex.kbAutoDream, normalized to { mode: 'off' }
  dreamingStatus: string,       // persisted synthesis_meta.status, overlaid to running by the route while the in-memory dream lock is active
  dreamProgress: {              // persisted synthesis_meta.dream_progress while dreaming
    phase: 'routing' | 'verification' | 'synthesis' | 'discovery' | 'reflection',
    done: number,
    total: number,
    startedAt?: number,
    phaseStartedAt?: number,
  } | null,
  needsSynthesisCount: number,  // entries with needs_synthesis = 1; Pipeline uses this as queued Dream work
  counters: {                   // workspace totals for header badges
    rawTotal: number,
    rawByStatus: Record<KbRawStatus, number>,
    failedByStage: {
      conversion: number,  // failed before a kb_documents row exists
      digestion: number,   // failed after conversion produced document structure
      unknown: number,     // defensive remainder
    },
    entryCount: number,
    pendingCount: number,       // ingested + pending-delete
    folderCount: number,
    topicCount: number,
    connectionCount: number,
    reflectionCount: number,
  },
  folders: KbFolder[],          // full flat folder tree sorted by folderPath
  raw: KbRawEntry[],            // ONE PAGE of the focused folder (not workspace-wide)
  digestProgress: KbDigestProgress | null,
  updatedAt: string,
}
```
`KbRawStatus` is `'ingesting' | 'ingested' | 'digesting' | 'digested' | 'failed' | 'pending-delete'`. `pending-delete` is set when the last location of a raw file is removed while auto-digest is off — the raw row survives until "Digest All Pending" runs and decides whether to digest-then-purge or just purge. `KbErrorClass` is `'timeout' | 'cli_error' | 'malformed_output' | 'schema_rejection' | 'unknown'`, stored on the raw row alongside the full `errorMessage` so the UI can surface what actually went wrong.

**Feature gating:** `getOrInitKbDb(hash)` on `ChatService` is the single entry point — it opens `state.db` (and runs migration) on first access for enabled workspaces, and returns `null` for disabled workspaces so nothing touches disk. `getKbStateSnapshot(hash, { folderPath, limit, offset })` assembles the `KbState` object from the DB for `GET /kb`, including the normalized per-workspace `kbAutoDream` config and persisted synthesis status/progress; the chat route overlays `dreamingStatus: 'running'` from `KbDreamService.isRunning(hash)` on `GET /kb` and `status: 'running'` on `GET /kb/synthesis` so a just-started Dream/Re-Dream shows as active before the first persisted progress frame and both Pipeline and Synthesis views agree. It returns an in-memory empty scaffold when KB is disabled. `getWorkspaceKbPointer(hash)` returns `null` when disabled, otherwise `mkdir -p`s `knowledge/entries/` so the CLI can read the directory without hitting ENOENT on a brand-new workspace, and returns a bracketed text block naming the absolute path.

**Pointer injection:** On new sessions, the chat route prepends three bracketed blocks to the outgoing user message — discussion history (always), memory (when enabled), knowledge base (when enabled) — see **Workspace Context Injection** above. KB content itself is never serialized into the system prompt; the CLI reads `state.db` / `entries/<entryId>/entry.md` on demand via its own file tools, so resumed sessions keep access and mid-session additions are visible on the next turn. Additionally, when KB is enabled, the system prompt receives a KB Tools addendum that teaches the CLI about the 6 MCP search/ingestion tools and the filesystem layout for direct reads — including `synthesis/reflections/*.md` for cross-cluster reflection files generated by the Dreaming pipeline's Reflection phase. The CLI uses search tools (Layer 1 — hybrid retrieval) to narrow relevant knowledge, then reads files directly (Layer 2 — agentic orchestration) for full content.

**LibreOffice detection** (`src/services/knowledgeBase/libreOffice.ts`): `detectLibreOffice()` caches the last `LibreOfficeStatus` (`{ available, binaryPath, checkedAt }`) at module level, but accepts `{ refresh: true }` for user-facing checks. It uses `which soffice` on POSIX and `where.exe soffice` on Windows — not `soffice --version`, because launching the full binary would spawn a user profile. Windows refresh probes merge the live process `PATH` with current user and machine `Path` values from the registry, then fall back to common install locations such as `C:\Program Files\LibreOffice\program\soffice.exe`; this lets Welcome, Install Doctor, and Settings detect a just-installed LibreOffice without restarting Agent Cockpit. The status is consulted by the ingestion stage when `Settings.knowledgeBase.convertSlidesToImages` is true; rasterization refreshes detection before falling back, so a newly installed LibreOffice can be used immediately. If LibreOffice remains unavailable, ingestion logs a warning and falls back to extracting text + speaker notes + embedded media only. `GET /kb/libreoffice-status` forces a fresh probe so the global Settings modal can validate the checkbox on click: when the user turns the "Convert PPTX slides to images" box on, the frontend fetches the endpoint, and if `available` is `false` the box auto-reverts to unchecked and a warning appears underneath telling the user to install LibreOffice.

**Pandoc detection** (`src/services/knowledgeBase/pandoc.ts`): Mirror of the LibreOffice module, with one key difference — pandoc is **required**, not optional, for DOCX ingestion. `detectPandoc({ refresh: true })` uses `which pandoc` / `where.exe pandoc` to locate the binary, then probes `pandoc --version` (cheap and side-effect-free) to surface the version string in the Settings UI. Windows refresh probes merge the live process `PATH` with current registry `Path` values and fall back to common Pandoc install locations such as `%LOCALAPPDATA%\Pandoc\pandoc.exe` and `C:\Program Files\Pandoc\pandoc.exe`. Result is cached as `PandocStatus` (`{ available, binaryPath, version, checkedAt }`) at module level for runtime calls. The shared `runPandoc(args, { timeoutMs, maxBuffer })` refreshes detection before throwing when the cached status reports unavailable, so a newly installed binary can be used without restarting Agent Cockpit. When pandoc is missing, the route layer rejects `.docx` uploads pre-flight with a `400` containing an install hint (`brew`/`apt`/`choco` one-liners and `https://pandoc.org/installing.html`). The frontend surfaces the same state in two places: the KB Browser Raw tab renders a persistent warning banner (`chat-kb-banner-warn`) above the upload zone when `GET /kb/pandoc-status` returns `available: false`, and the global Settings → Knowledge Base tab shows a "Pandoc (required for DOCX ingestion)" status row — green "Detected vX.Y.Z at /path" or red "Not found on PATH" — refreshed on tab open.

**Global settings:** `Settings.knowledgeBase` configures three CLI profiles separately — **Ingestion** (optional, vision-capable, used for AI-assisted page/slide/image conversion at ingest time), **Digestion**, and **Dreaming** — each with its own `*CliProfileId`, legacy `*CliBackend` fallback, `cliModel`, and `cliEffort`. It also exposes `cliConcurrency` (1–10, default 2) — a per-workspace cap on documents processed in parallel by all three pipelines (within a document, work stays sequential) — `kbGleaningEnabled` (default false) for the second digestion pass, and the `convertSlidesToImages` opt-in. Ingestion is opt-in (empty = skip AI conversion, falls back to image-only references). Digestion + Dreaming require a configured profile/backend before they run. The settings modal exposes cascading profile → model → effort pickers per role, a numeric input for CLI concurrency, a gleaning toggle, and the PPTX slide-image conversion toggle.

**Migration — `dreamingConcurrency` → `cliConcurrency`:** The hybrid-ingestion design (PR 1) renamed the per-pipeline-CLI concurrency setting from `dreamingConcurrency` to `cliConcurrency` so it can govern ingestion + digestion + dreaming uniformly. `SettingsService.getSettings()` performs a read-time copy: when an existing `settings.json` has `dreamingConcurrency` set but no `cliConcurrency`, the value is mirrored into `cliConcurrency` in memory before the settings object is returned. Disk state is **not** rewritten — the new key materializes on disk only when the user saves a setting through the UI. The deprecated `dreamingConcurrency` field stays on the `Settings` type for one release cycle, then is removed.

**Workspace settings:** The workspace Settings modal has a Knowledge Base tab with a single checkbox (`kbEnabled`). Once enabled, the full-screen **KB Browser** (see Section 7) becomes the upload UI and entry list. Saving the toggle hits `PUT /workspaces/:hash/kb/enabled` and does not touch `memoryEnabled`. The KB Browser owns the day-to-day per-workspace operational controls: `kbAutoDigest` is surfaced as a toolbar switch in Raw, and `kbAutoDream` is surfaced in Settings as Off / Every N hours / Time window. Auto-Dream is workspace-scoped because KB state is workspace-scoped; it is deliberately not part of global `Settings.knowledgeBase`.

**Ingestion pipeline:** `KbIngestionService` (`src/services/knowledgeBase/ingestion.ts`) is the single entry point the chat route calls for uploads, folder CRUD, and raw deletion. It is constructed with `{ chatService, emit, digestTrigger? }` where `emit(hash, frame)` is a workspace-scoped WS fan-out callback and `digestTrigger` is a late-bound reference to `KbDigestionService.enqueueDigest` — the circular dependency between the two services is broken via `setDigestTrigger()` after both are constructed.

- **Content addressing** — `sha256 = sha256(buffer)` is the full hash, `rawId = sha256.slice(0, 16)` is the stable URL/filename prefix. Dedupe looks up the **full sha256** in the `raw` table via `getRawBySha`, so renames don't break dedupe and collisions on the short prefix would still be caught (extremely unlikely in practice).
- **Multi-location (Option B)** — `enqueueUpload(hash, { buffer, filename, mimeType, folderPath })`:
  - If the sha256 already exists and the same `(folderPath, filename)` tuple is already a location for that raw → no-op, returns `{ deduped: true, addedLocation: false }`.
  - If the sha256 already exists but `(folderPath, filename)` is new → insert a single `raw_locations` row, re-stage the raw bytes on disk if they're missing (edge case), return `{ deduped: true, addedLocation: true }`. No reconversion.
  - If the sha256 is new but `(folderPath, filename)` is already taken by a different sha256 → throw `KbLocationConflictError`, which the route maps to `409`. The UI surfaces "a different file already exists at this path — rename and retry".
  - If both are new → write `raw/<rawId>.<ext>`, insert `raw` + `raw_locations` rows in a single transaction (status `ingesting`), and schedule background conversion.
- **Per-workspace FIFO queue** — A `Map<hash, Promise<unknown>>` chains every mutation (ingestion, conversion handler, delete, folder rename, digestion) so a single workspace can only have one in-flight operation at a time. Different workspaces run in parallel. The queue is shared with `KbDigestionService` via constructor injection — batch digests and uploads can't interleave. `waitForIdle(hash)` returns when the chain is empty (used by tests and by the delete path to avoid TOCTOU against a still-running handler).
- **Filename + folder validation** — Filenames can't contain path separators or be empty. Folder segments are validated through `normalizeFolderPath` in `db.ts` — reject `..`, control chars, `>128 chars` per segment, `>4096 chars` total. Validation failures surface as `KbValidationError` → `400`.
- **Handler dispatch** — `pickHandler(filename, mimeType)` matches on extension first (`.pdf`→pdfHandler, `.docx`→docxHandler, `.pptx`→pptxHandler, passthrough-supported text/image extensions→passthroughHandler), then falls back to a MIME-type map. It returns lazy wrapper handlers so importing `src/services/knowledgeBase/handlers/index.ts` does not load native PDF/canvas dependencies into unrelated chat-route code or tests. The selected format module is dynamically imported only when `ingestFile()` executes that handler; text passthrough imports remain lightweight, while image passthrough lazy-loads the shared `pageConversion` helper only when image conversion/downscaling is needed. Unknown files throw `UnsupportedFileTypeError`, caught by the background handler which lands the raw row in `status='failed'` with the error message stored on `raw.error_message` and `errorClass='unknown'`.
- **Conversion output** — Each handler writes into `converted/<rawId>/`:
  - `text.md` — extracted text (or image index) with an `# <filename>` header. PDF is a per-page hybrid (`## Page N` sections with `> source: pdfjs|artificial-intelligence|image-only` annotations and a `![Page N](pages/page-NNNN.png)` link on every page) — see the `pdf.ts` handler entry below for the full classification flow. DOCX is GFM-flavored markdown from pandoc (tables preserved); each embedded image reference is followed by a quoted `> Image description (source: artificial-intelligence): …` block when the Ingestion CLI was able to describe it (skipped for icons under 100 px wide and when no Ingestion CLI is configured). PPTX is a per-slide hybrid (`## Slide N` sections with `> source: xml-extract|artificial-intelligence|image-only` annotations, optional `### Speaker Notes`, and a `![Slide N](slides/slide-NNN.png)` link when the deck was rasterized) — see the `pptx.ts` handler entry below for the full classification flow. A trailing `## Embedded Media` block lists `ppt/media/*` files. Passthrough preserves markdown as-is or wraps other text in a code fence tagged with the file extension; full content is preserved regardless of file size.
  - `meta.json` — `{ rawId, handler, extractedAt, metadata }` where `metadata` is handler-specific (`pageCount`, `wordCount`, `slideCount`, `totalSlideCount`, `hiddenSlideCount`, `slidesToImagesRequested`, `slideImagesWarning`, `renderedSlideCount`, `sourceCounts`, `mediaCount`, …).
  - `media/` — copied embedded media (DOCX `word/media/`, PPTX `ppt/media/`, passthrough images). Text handlers that find no media omit the directory.
- **Progress substeps** — The ingestion pipeline emits `substep: { rawId, text }` frames at each stage so the UI can show live progress beneath the status badge: `"Converting…"` before `ingestFile()` runs, and `"Storing…"` after the handler returns while writing output files. The frontend also starts an elapsed-time ticker when a raw enters the `ingesting` state.
- **Completion** — On success the orchestrator updates the `raw` row to `status='ingested'`, writes the handler tag to `raw.handler`, stores handler metadata on `raw.metadata_json`, and emits a `kb_state_update` frame. If `kbAutoDigest` is true, the orchestrator fires `digestTrigger(hash, rawId)` to chain a digest job onto the same queue before returning. On failure it sets `status='failed'` with the error class and message, emits the frame, and does not trigger digestion.
- **Ref-counted delete** — `deleteLocation(hash, rawId, folder, filename)` removes a single `raw_locations` row. If `countLocations(rawId) > 0` after the delete, the raw row stays (another location still references it). If this was the last location, the raw is fully purged via `_purgeRawInternal` (removes raw bytes, converted dir, `entries/<entryId>/` dirs, DB rows, and stale entry embeddings best-effort) regardless of auto-digest setting. After purge, `_deleteOrphanTopics()` is called inside the transaction to remove any synthesis topics that lost all their entries due to the cascade delete.
- **Full purge** — `purgeRaw(hash, rawId)` (also aliased as the legacy `deleteRaw` for test compatibility) unconditionally drops every `raw_locations` row, the `raw` row, the raw bytes, `converted/<rawId>/`, every entry's `entries/<entryId>/` dir, and the DB rows for the entries + `entry_tags` (cascade). It also deletes those entries from the vector store best-effort when embedding is configured. After purge, orphan topic cleanup runs. Used by the "full purge" mode of `DELETE /kb/raw/:rawId` (no query string).
- **Folder CRUD** — `createFolder`, `renameFolder`, `deleteFolder({ cascade })` all run on the same queue and emit `folders: true` frames. `renameFolder` rewrites every `raw_locations` row under the subtree in a single SQLite transaction — no on-disk renames happen because folders are purely virtual. `deleteFolder` with `cascade: true` walks the subtree and removes every location following the same ref-counted purge rules as `deleteLocation` (always full purge on last location), then drops the folder rows deepest-first; without `cascade` it throws if the subtree still contains any locations.
- **WS fan-out** — The chat router instantiates `KbIngestionService` with an `emit` callback that walks `activeStreams` and, for every conversation whose workspace hash matches the frame, calls the WS module's `send`. The KB Browser is conversation-agnostic though, so it also polls `GET /workspaces/:hash/kb` every 1.5s while open to pick up changes even when no CLI stream is active in the target workspace.

**Digestion pipeline:** `KbDigestionService` (`src/services/knowledgeBase/digest.ts`) handles the `ingested/failed/digested → digesting → digested|failed` transition. It is constructed with `{ chatService, backendRegistry, emit, queueRegistry }` and shares the per-workspace `WorkspaceTaskQueue` with the ingestion service via the same `WorkspaceTaskQueueRegistry`, so a `cliConcurrency=2` budget covers any combination of ingestion + digestion tasks running together for that workspace. Single-file manual triggers accept `failed` raws as retries and `digested` raws as redigests; batch mode remains scoped to pending work (`ingested` + `pending-delete`) so "Digest All Pending" does not unexpectedly redigest the whole KB.

- **Prompt construction** — `buildDigestPrompt` renders a markdown prompt that names the filename + folder context, the handler tag, the MIME type, and the **relative path** to `converted/<rawId>/text.md` (so the CLI can read additional artifacts — slide PNGs, media, etc. — via its own file tools rather than having everything pasted into the prompt). Handler metadata is stringified into the prompt so the CLI knows up front how many pages/slides to expect. The rules section includes a **source-aware image-consultation rule** keyed on the `> source: <label>` blockquote that the hybrid ingestion handlers emit at the top of each `## Page N` / `## Slide N` section: `pdfjs` and `xml-extract` mean the text was deterministically extracted and the image is backup (consult only when the text seems incomplete or references a visual element); `artificial-intelligence` means the markdown was already reconstructed from the image at ingest time and is the primary source (open the image to verify table cells, figure details, or layout when accuracy matters); `image-only` means no text was extractable and the image IS the content (MUST open). Image paths resolve relative to the converted text file's directory. This replaces the blunt PR #207 rule that required the CLI to open every referenced image — now that `source: artificial-intelligence` and `source: pdfjs`/`xml-extract` already carry reliable text, opening every image was both wasteful (cost) and counter-productive (the AI-generated markdown captures more structure than a fresh AI read of the same pixels).
- **CLI invocation** — The Digestion CLI is resolved from `Settings.knowledgeBase.digestionCliProfileId` (falling back to legacy `digestionCliBackend`) through `ChatService.resolveCliProfileRuntime()`, then `BackendRegistry.get(runtime.backendId)`. The default call uses `runOneShot(prompt, { allowTools: true, workingDir: knowledgeDir, timeoutMs: digestTimeoutMs, cliProfile: runtime.profile })` so the CLI can follow relative paths inside `knowledge/` and account profiles use their isolated auth/config directory. When `kbGleaningEnabled` is true, the same options are passed to `runSessionShot([digestPrompt, gleaningPrompt], ...)`; the first output is parsed as the primary extraction and a non-empty second output is parsed as additional entries. **Adaptive timeout:** `digestTimeoutMs = Math.max(30 * 60_000, unitCount * 10 * 60_000)` where `unitCount` is the chunk's page/slide/section/line span. So small chunks and non-paginated formats bail at 30 min, while a 25-page chunk gets ~4 hours — enough that even Opus on a slow vision model can finish. Missing/unregistered CLI lands as `errorClass='unknown'`.
- **Progress substeps** — The digestion pipeline emits `substep: { rawId, text }` frames before each chunk's CLI analysis and again before parsing entries. It also updates `digestProgress.chunks` as the raw moves through `planning`, per-chunk `digesting`, per-chunk `parsing`, and final `committing`, so the Pipeline tab can show live planned/completed chunks instead of waiting for committed `kb_entry_sources` rows. Combined with the frontend's elapsed-time ticker (started when a raw enters `digesting`), this gives the user a sense of progress during long digestion runs.
- **Error classification** — Each failure mode lands a specific `KbErrorClass` on the raw row so the UI can render actionable copy:
  - **`timeout`** — `runOneShot` exceeded its adaptive `max(30 min, chunkUnitCount × 10 min)` timeout, or the backend adapter surfaced timeout wording such as `timeout`, `timed out`, or Kiro/Bedrock `TimedOut`.
  - **`cli_error`** — CLI exited non-zero or threw an adapter error.
  - **`malformed_output`** — stdout couldn't be parsed into entry-delimited frontmatter + body (also used for a missing `text.md`).
  - **`schema_rejection`** — parsed entries exist but failed field validation (missing title/slug/summary/tags).
  - **`unknown`** — catch-all (including "KB disabled mid-run" and config errors).
- **Entry parsing** — Output is split on `---`-separated entry boundaries. Each chunk is parsed as YAML frontmatter + markdown body and validated against `ParsedEntry` (`title`, `slug`, `summary`, `tags[]`, `body`). The body accumulation loop distinguishes entry-boundary `---` lines from markdown horizontal rules within the body by peeking ahead: when `---` is encountered, the parser scans forward past blank lines and checks whether the next non-blank line matches `title:` (the required first frontmatter field). If it does, the `---` is treated as a new entry boundary; otherwise it is included in the current entry's body as a horizontal rule. The parser is lenient about common CLI output quirks: leading/trailing triple-backtick code fences (e.g. `` ```yaml ... ``` ``) are stripped, preamble prose before the first `---` fence is skipped, and frontmatter keys are lowercased before lookup so `Title:` and `TITLE:` both resolve to `title`. Malformed chunks raise `DigestParseError`, schema violations raise `DigestSchemaError` — both are caught internally and classified. On `malformed_output` or `schema_rejection` failures, the raw CLI output is dumped to `knowledge/digest-debug/<rawId>-<iso>.txt` and the debug path is appended to the error message so the user can inspect exactly what the CLI returned.
- **Strict chunk mode** — The service parses every chunk's primary output and optional gleaning output before deleting stale entries. If any chunk fails, the raw is marked `failed` and previous entries/source lineage remain in place. The Raw tab's Retry button re-runs that raw after the underlying issue is fixed.
- **Entry lineage + dedupe** — Each parsed entry is paired with the chunk source range that produced it. Before writing, the digest run merges clear duplicates within the same raw: exact slug matches merge first, then normalized title matches. The first entry's title/slug remain canonical; summaries and bodies are appended when the later duplicate carries non-identical content, tags are unioned in first-seen order, and all distinct source ranges are preserved. Non-matching entries remain separate; there is no LLM semantic merge in this pass.
- **Entry serialization** — `stringifyEntry(entry, opts?)` converts a `ParsedEntry` back to YAML frontmatter + markdown body. The frontmatter always includes `title`, `slug`, `summary`, `tags[]`, and `schemaVersion` in a deterministic order. Optional temporal metadata — `uploadedAt` (ISO 8601 from the raw row's `uploaded_at`) and `digestedAt` (ISO 8601 timestamp of the digestion run) — is appended to the frontmatter when provided via `StringifyEntryOptions`. Values containing YAML-special characters (colons, quotes) are auto-quoted.
- **Entry frontmatter format** — Each `entry.md` on disk has the following YAML frontmatter fields:
  ```yaml
  ---
  title: "Entry Title"
  slug: entry-title
  summary: "One-line summary"
  tags: [tag1, tag2]
  schemaVersion: 1
  uploadedAt: "2026-04-10T08:30:00.000Z"
  digestedAt: "2026-04-10T09:15:00.000Z"
  ---
  ```
  `uploadedAt` records when the raw file was first uploaded; `digestedAt` records when this entry was produced by the Digestion CLI. Both are ISO 8601 strings. These timestamps are informational — they help users and downstream tooling understand the freshness of each entry.
- **Slug collision handling** — `entryId = <rawId>-<slug>`. During redigest, existing entry IDs for the same raw are treated as reusable replacement slots, while IDs already reserved by another entry in the same run or present for another raw get `-2`, `-3`, … suffixes until `entryIdTaken` returns false. Reusing same-raw IDs keeps stable links for unchanged slugs; stale entry embeddings are deleted from the vector store best-effort before replacement embeddings are upserted.
- **Entry writing** — Each entry is staged under `entries/.staging/<rawId>-<uuid>/<entryId>/entry.md` before any stale rows are replaced. After all replacement files stage successfully, existing same-raw entry files are moved to `entries/.backup/<rawId>-<uuid>/`, a `manifest.json` records stale IDs, replacement IDs, and the replacement `digestedAt`, staged directories are renamed into `entries/<entryId>/`, and `replaceEntriesForRawId()` deletes old rows plus inserts replacement `entries`, `entry_tags`, and `kb_entry_sources` rows in a single SQLite transaction. On staging or transaction failure the raw row lands in `failed`, old rows/files are preserved or restored, and no partial DB replacement is committed.
- **Completion** — On success the raw row moves to `status='digested'`, `digestedAt` is stamped, `errorClass/errorMessage` are cleared, and a `kb_state_update` frame is emitted with `raw: [rawId]` and `entries: [entryId, …]`.
- **Pending-delete handling** — When `_runDigest` encounters a `pending-delete` raw it skips the entire CLI pipeline and purges the raw immediately (raw bytes, converted dir, leftover entries, DB rows). The user already deleted the last location — there's nothing to digest. The result carries `purged: true`.
- **Top-level safety net** — The digest pipeline is wrapped in a top-level try/catch so unexpected errors land as `status='failed'` with `errorClass='unknown'` instead of leaving the raw stuck in `digesting` forever.
- **Batch mode** — `enqueueBatchDigest(hash)` lists every `ingested` + `pending-delete` raw (`pending-delete` first so purges happen before digestions) and dispatches each one onto the shared `WorkspaceTaskQueue` in parallel — up to `cliConcurrency` digests run at once. Failures inside the batch don't stop the run — they land on their raw row and the rest still complete. Per-task settles update the unified **digestion session counter** (see below) so the toolbar reports `done / total — ~ETA` across all in-flight tasks.
- **Digestion session counter** — The service holds a per-workspace `sessions` map keyed by hash with shape `{ entriesCreated, pending, total, done, totalElapsedMs, startedAt, chunkProgress, activeChunkKeys }`, created lazily when any digest task enqueues (single via `enqueueDigest`, batch via `enqueueBatchDigest`, or an auto-digest chained by the ingestion orchestrator). One session covers all three paths so concurrent single digests interleaving with a batch or auto-digest run share a single count-up.
  - `_trackPending(hash, +n)` opens the session on the first positive delta into an idle workspace, bumps `total` by the delta, and emits a `kb_state_update { digestProgress }` frame so the KB Browser toolbar can show the new totals immediately.
  - As each raw reaches chunk planning, `_recordChunkPlanning()` / `_recordChunksPlanned()` populate `digestProgress.chunks.total`; `_recordChunkStarted()`, `_recordChunkPhase()`, and `_recordChunkDone()` keep `chunks.active`, `chunks.done`, `chunks.phase`, and `chunks.current` live while a document is processed.
  - After each task settles, `_recordTaskDone(hash, elapsedMs)` bumps `done`, accumulates `totalElapsedMs`, persists the session row to `digest_session` in the KB DB (cheap — sqlite, sub-ms), and emits an updated `digestProgress` snapshot.
  - `_recordEntriesCreated(hash, n)` runs in parallel to bump `entriesCreated` and emit the existing `digestion: { active: true, entriesCreated }` frame.
  - When `pending` drops to zero the service emits a final `digestion: { active: false, entriesCreated }` frame **and** a `digestProgress: null` signal (so the frontend can clear the toolbar indicator without a truthy-check ambiguity), then deletes both the in-memory session and the persisted `digest_session` row.
- **Progress snapshot shape** — `computeDigestProgress({ total, done, totalElapsedMs, chunkProgress? })` is the single source of truth for the frame shape. It computes `avgMsPerItem = round(totalElapsedMs / done)` (0 until the first settle) and, once `done >= 2` and the average is non-zero, appends `etaMs = (total - done) * avgMsPerItem`. When chunk progress exists it also includes `chunks: { done, total, active, phase, current? }`, where `phase` is `planning | digesting | parsing | committing`; `total` grows as queued raws reach the planner, and `current` is the most recently updated raw/chunk because a workspace session can process more than one raw concurrently. Withholding ETA until the second sample lands avoids a wildly fluctuating estimate caused by the first task's CLI warm-up + filesystem cache cost. The same helper runs in `chatService.getKbStateSnapshot` so `GET /workspaces/:hash/kb` can return `digestProgress` consistent with the live WS frames (the snapshot path reads the persisted row and runs it through `computeDigestProgress`).
- **Crash recovery** — On `KbDatabase` construction, `_recoverFromCrash()` flips any stuck `status='digesting'` raws back to `ingested` (nothing can actually be digesting in a freshly-opened DB) and `DELETE FROM digest_session`. A server crash mid-session therefore can't leave the toolbar indicator stuck forever; the next enqueue opens a fresh session. Before the first digest enqueue for a workspace in a process, `KbDigestionService` also scans `entries/.backup/` manifests and `entries/.staging/`: if SQLite already contains the manifest's replacement IDs with the recorded replacement `digestedAt`, the replacement committed and the backup is discarded; otherwise replacement dirs are removed and backed-up stale entry files are restored before retrying. This closes the filesystem/SQLite crash window around redigest replacement.

**Dreaming pipeline:** `KbDreamService` (`src/services/knowledgeBase/dream.ts`) handles the cross-entry synthesis phase that turns digested entries into a knowledge graph of topics and connections. It is constructed with `{ chatService, backendRegistry, emit, kbSearchMcp }` and holds a per-workspace lock (`running` Set) to prevent concurrent dream runs. The lock is acquired synchronously before any async pre-flight checks, so manual endpoint calls and scheduler starts cannot both pass the "not running" check during the initial DB/settings awaits.

- **Two modes:** Incremental (`dream(hash)`) processes only entries with `needs_synthesis = 1`. Full Rebuild (`redream(hash)`) wipes current synthesis graph data (`wipeSynthesis()` removes topics, topic-entry assignments, connections, reflections, and reflection citations) and topic vector embeddings, then reprocesses every entry. Re-Dream deliberately preserves entry vector embeddings so retrieval remains available while synthesis is rebuilt; entries processed during the run refresh their entry vectors, which also repairs KBs whose entry vectors were previously removed by an older Re-Dream path. Re-Dream also preserves `synthesis_runs` and `synthesis_topic_history` so historical topic evolution remains auditable. Both modes are fire-and-forget — the endpoint returns 202 immediately.
- **Retrieval-based pipeline (Phase B):** Replaced the flat-file Discovery→Synthesis two-phase approach with a three-phase retrieval pipeline. No more `_dream_tmp/` directory or `all-topics.txt` dumps — topic discovery is now server-side via embeddings + hybrid search.
  1. **Routing** — Embeds all pending entries (`title — summary`) in batches (50/batch) via the embedding service, upserts those entry vectors into the PGLite vector store, then performs `hybridSearchTopics(text, embedding, 10)` for each entry. Classifies each entry by top match score:
     - **Strong match** (score ≥ `dreamingStrongMatchThreshold`, default 0.75): routed directly to synthesis with all topics above the borderline threshold.
     - **Borderline** (score ≥ `dreamingBorderlineThreshold`, default 0.45): queued for LLM verification.
     - **No match** (score < borderline or no results): queued for new topic creation.
  2. **Verification** — Lightweight LLM call (no MCP tools, `baseRunOptions`) for borderline entries. Batched at 10 entries. Prompt asks yes/no per entry-topic pair. `parseVerificationOutput()` extracts `{ verified: [...], rejected: [...] }` from the CLI response. Verified entries join synthesis groups; rejected entries join the unmatched pool.
  3. **Synthesis** — CLI calls with MCP search tools (`runOptionsWithMcp`) for topic updates. `buildRetrievalSynthesisPrompt()` lists pre-matched topic IDs (CLI uses `get_topic` MCP tool to fetch content) and instructs the CLI to use `search_topics`/`find_similar_topics` for connection discovery. All 10 operations are available. Unmatched entries go through `buildNewTopicCreationPrompt()` with MCP tools enabled.
- **Cold start path** — When no topics exist or no embedding config is available, all entries go through `_runColdStart()`: if an embedding config exists, entry vectors are refreshed before synthesis; entries are then sorted by tags and batched at 10. First batch uses `buildNewTopicCreationPrompt()` without MCP tools (nothing to search). Topics from batch 1 are embedded immediately, so subsequent batches get MCP tools and can search existing topics.
- **Per-batch topic embedding** — After each synthesis batch's `applyOperations()`, `_embedBatchTopics()` embeds only the affected topics (extracted by `extractAffectedTopicIds()` from create/update/delete/merge/split operations). Makes new/updated topics immediately searchable by subsequent batches. A final `_embedTopics()` sweep at run end cleans up stale embeddings.
- **KB Search MCP session** — At dream start, `kbSearchMcp.issueKbSearchSession(hash, hash)` mints a bearer token scoped to the dream run (session key = workspace hash). The token is passed to CLI calls via `mcpServers` in `RunOneShotOptions`. Revoked in the `finally` block after the run. For conversation sessions, the session key is the `convId` — see KB Search MCP server section below.
- **File-reference prompts** — All prompts are lightweight instruction sheets that point the CLI to files on disk. Prompts stay small regardless of KB scale; the CLI reads files using its own tools (`allowTools: true`). Each prompt includes an `EXECUTION_STRATEGY` block instructing multi-agent parallelism when supported.
- **Concurrency** — `Settings.knowledgeBase.cliConcurrency` (default 2, max 10) controls parallel CLI calls within each batch. Batches within a phase are sequential; CLI calls within a batch are parallel. The same setting governs ingestion + digestion + dreaming pipelines uniformly (renamed from `dreamingConcurrency`; legacy key auto-migrated on read).
- **Operation types** — The CLI returns `{ "operations": [...] }` JSON with 10 supported ops: `create_topic`, `update_topic`, `merge_topics`, `split_topic`, `delete_topic`, `assign_entries`, `unassign_entries`, `add_connection`, `update_connection`, `remove_connection`. All ops are parsed and validated by `parseDreamOutput()` then applied transactionally by `applyOperations()` (`src/services/knowledgeBase/dreamOps.ts`). Connection operations pre-check that both endpoint topics exist and return explicit `connection skipped` warnings instead of surfacing raw SQLite foreign-key failures. Dream runs pass `{ runId }` into `applyOperations()`, and every mutating topic op writes a `synthesis_topic_history` row: create (`created`), update (`updated` with old/new content and current entry IDs), merge (`merged_into` rows for removed source topics), split (`split_from` on the source topic with JSON-encoded replacement topic summaries), and delete (`deleted`).
- **Run tracking** — `_run()` generates a UUID `run_id` once the workspace DB is available and inserts `synthesis_runs(status='running', mode='incremental'|'redream')` before the MCP session and CLI work. The `finally` block finishes the row as `completed`, `failed`, or `stopped` with `completed_at`; fatal errors and nonfatal warning summaries are recorded in `error_message`. Pre-flight failures before a workspace DB is available do not create run rows.
- **Error handling** — Each batch is retried once on thrown failure (`_runCliWithRetry`). Every parse call site additionally goes through `_runCliAndParseWithRetry<T>(adapter, prompt, options, parse, hash, phase)`, which also retries once when the parser reports a JSON-extraction or JSON-parse failure (detected by `hasParseFailure(warnings)`, which matches `/json parse error|no json (object )?found/i`). Structural/schema warnings (e.g. "missing X array", unknown op types) are **not** retried — a retry would not meaningfully change them. When both attempts return a parse-failure result, the raw outputs are written to `<kb-knowledge-dir>/_dream_debug/parse-failure-<phase>-<ISO-timestamp>.txt` (best-effort; never throws). `<phase>` is one of `verification`, `synthesis`, `new-topics`, `cold-start`, `discovery`, `reflection`. If the first CLI attempt returns no output at all, the wrapper returns `null` — callers preserve their existing empty-output skip behavior (entries stay `needs_synthesis = 1`). Failed batches are skipped; their entries keep `needs_synthesis = 1` for the next run. Dream timeout: 1 200 000 ms (20 minutes per CLI call).
- **Pre-flight check** — At run start, `checkOllamaHealth(cfg)` is called. If Ollama is unreachable, a warning is logged and retrieval routing is skipped (falls through to cold start).
- **God-node detection** — After each dream run, `detectGodNodes()` identifies topics with entry count or connection count > 3× the average (minimum 10). Flagged topic IDs are stored in `synthesis_meta.god_nodes` and surfaced in the Synthesis tab UI with a star badge.
- **Stale cascade** — When entries are deleted (re-digestion, raw purge), `deleteEntriesByRawId()` / `deleteRaw()` mark all surviving entries sharing a topic with the deleted ones as `needs_synthesis = 1` before SQLite cascades remove topic assignments. This preserves the co-topic signal and ensures the next dream run updates affected topics.
- **Materialized views** — After each dream run, `regenerateSynthesisMarkdown()` (`src/services/knowledgeBase/dreamMarkdown.ts`) wipes and recreates `knowledge/synthesis/` with `index.md` (topic table), `topics/<id>.md` (per-topic prose + related + entries), and `connections.md` (full connection graph). The `reflections/` subdirectory is **preserved** during the wipe — reflection markdown is generated separately by `regenerateReflectionMarkdown()`, which writes individual files at `synthesis/reflections/<id>.md`.
- **Connection Discovery (Phase C):** A 4th phase that runs after synthesis and topic embedding, before god-node detection. `_runConnectionDiscovery()` sweeps for missing connections using three strategies:
  1. **Embedding-based sweep** — For each topic, `findSimilarTopics(topicId, 10)` returns similar topics; pairs that already have a connection are filtered out.
  2. **Entry-driven propagation** — `db.listTopicPairsBySharedEntries()` finds topic pairs sharing assigned entries but no existing connection. SQL joins `synthesis_topic_entries` on `entry_id`, groups by topic pair, excludes already-connected pairs (both directions).
  3. **Transitive discovery (2-hop)** — `db.listTransitiveCandidates()` finds pairs (A, C) where A→B and B→C exist but A↔C doesn't. Uses a CTE that unions both directions of connections, joins 2 hops, and returns the intermediate topic + both relationship labels. Constrained to `topicA < topicC` to avoid duplicate pairs.
  - **Deduplication:** Candidates from all three strategies are merged into a `Map<normalizedKey, DiscoveryCandidate>` keyed by sorted topic ID pair. Each candidate accumulates: `embeddingSimilarity`, `sharedEntryCount`, `transitiveSignal` (boolean), `transitivePath` (string description).
  - **Ranking:** Candidates are scored: `embeddingSimilarity * 0.5 + (sharedEntryCount / maxEntryCount) * 0.3 + (transitiveSignal ? 0.2 : 0)`. Top 50 candidates are selected (`DISCOVERY_CANDIDATE_CAP = 50`).
  - **LLM verification:** Candidates are batched at 5 (`DISCOVERY_BATCH_SIZE = 5`). Each batch sends a `buildConnectionDiscoveryPrompt()` with rich context per candidate pair: both topic titles + summaries + full content, existing connections for both topics, assigned entry titles + summaries, shared entries, and the discovery signal description. The LLM returns `{ results: [{ topic_a, topic_b, accept, source_topic, target_topic, relationship, confidence, evidence }] }` — the LLM decides directionality. `parseDiscoveryOutput()` extracts accepted connections. Accepted connections are persisted via `db.upsertConnection()`.
  - **Post-discovery:** Markdown is regenerated a second time (connections may have been added). God-node detection runs after discovery.
  - **Constants:** `DISCOVERY_CANDIDATE_CAP = 50`, `DISCOVERY_BATCH_SIZE = 5`, `DISCOVERY_EMBEDDING_TOP_K = 10`.
- **Reflection (Phase E):** A 5th phase that runs after discovery as the final stage before god-node detection. `_runReflection()` generates cross-cluster insights by examining groups of connected topics.
  - **Deterministic checks** — Before LLM calls, three cleanup passes run: (0) **Orphan topic cleanup** — `_deleteOrphanTopics()` removes topics with zero entry assignments (may have been orphaned by cascade-deletes since the last run). (1) **Stale topic detection** — identifies topics whose prose (`content`) is older than the newest assigned entry's `digested_at`, marking them for re-synthesis in a future run. (2) **Delete stale reflections** — any existing reflection whose cited entries have been changed, deleted, or lost via cascade since the reflection was created is removed via `db.deleteReflections(staleIds)` (using `db.listStaleReflectionIds()`, which checks three conditions: orphaned entries via LEFT JOIN, re-digested entries via timestamp comparison, and lost citations via `COUNT(c.entry_id) < r.original_citation_count`).
  - **Cluster identification** — `identifyTopicClusters()` uses BFS on the `synthesis_connections` graph to find connected components containing 2+ topics. Each cluster is a set of topic IDs that are transitively connected. Capped at 20 clusters per run (`REFLECTION_CLUSTER_CAP = 20`) to bound LLM cost.
  - **LLM reflection** — For each cluster, `buildReflectionPrompt()` assembles a prompt containing the cluster's topic summaries, inter-topic connections, and summaries of assigned entries. The CLI returns reflections with inline `[Entry: title](entry-id)` citations. Each reflection has a `type` field: `pattern` (recurring theme), `contradiction` (conflicting information), `gap` (missing coverage), `trend` (directional change), or `insight` (general observation). `parseReflectionOutput()` extracts the structured reflections and their citations from the CLI response.
  - **Storage** — Existing reflections are wiped before regenerating (`db.wipeReflections()`). New reflections are stored in the `synthesis_reflections` table (with `original_citation_count` set to the number of cited entries at insertion time) and their citations in `synthesis_reflection_citations`. Markdown files are written at `synthesis/reflections/<id>.md` via `regenerateReflectionMarkdown()`.
  - **Progress** — Emitted via `_emitDreamProgress('reflection', done, total, hash)` so the frontend dream banner and Synthesis tab can track reflection progress.
  - **Exported functions** — `dream.ts` exports three reflection helpers for testability: `parseReflectionOutput()` (extracts structured reflections + citations from CLI output), `identifyTopicClusters()` (BFS connected-component finder), and `buildReflectionPrompt()` (assembles per-cluster LLM prompt).
  - **KbDatabase methods** — `insertReflection()` writes a reflection + its citations in a single transaction, storing `citedEntryIds.length` as `original_citation_count`. `listReflections()` returns all reflections with citation counts and stale flags. `getReflection(id)` returns a single reflection with its `citedEntryIds` array. `wipeReflections()` drops all rows from `synthesis_reflections` and `synthesis_reflection_citations`. `listStaleReflectionIds()` returns reflection IDs where: cited entries were deleted (LEFT JOIN NULL), re-digested after the reflection's `created_at`, or lost via cascade (`COUNT < original_citation_count`). `_countStaleReflections()` returns the stale count for the synthesis snapshot. `deleteReflections(ids)` removes specific reflections by ID. `_deleteOrphanTopics()` removes topics with zero entry assignments (called inside `deleteRaw()` and at dream pipeline start).
- **WS events** — `_emitSynthesisChange(hash)` emits `kb_state_update { changed: { synthesis: true } }` on run start/end. `_emitDreamProgress(phase, done, total, hash)` persists `synthesis_meta.dream_progress` and emits progress frames with `dreamProgress: { phase: 'routing' | 'verification' | 'synthesis' | 'discovery' | 'reflection', done, total, startedAt?, phaseStartedAt? }` for the frontend dream banner, Synthesis tab, and Pipeline tab. An initial `done: 0` frame is emitted immediately before the first CLI call so the frontend exits the "Starting…" state within seconds. `requestStop(hash)` emits a frame with `changed: { synthesis: true, stopping: true }` immediately so the UI can render the "Stopping…" state before the in-flight batch finishes.
- **Cooperative stop** — `requestStop(hash)`, `isStopRequested(hash)` public methods. Backed by a private `stopRequested: Set<string>` alongside `running`. `_checkStop(hash, result)` is called between top-level phases in `_run` (after synthesis, after topic embedding, after discovery, before reflection) and between batches in every phase loop: retrieval synthesis batch groups, unmatched new-topic batches, cold-start batches, discovery batches, and reflection clusters. When a stop is honored it sets `result.stopped = true`, writes `synthesis_meta.status = 'idle'` and `synthesis_meta.stopped_at = <now>`, and returns early — the `finally` block then marks the current `synthesis_runs` row `stopped`, clears `running` and `stopRequested`, revokes the MCP session, and emits the final `synthesis: true` frame. **`last_run_at` is deliberately not updated on a stop** (it's not a natural completion). Already-committed per-batch work (`applyOperations` + `clearNeedsSynthesis`) persists, so the next `dream()` call is incremental and picks up any entries still flagged `needs_synthesis = 1`. `DreamResult.stopped: boolean` lets callers distinguish stop from normal completion or error. Stop latency is bounded by `DREAM_TIMEOUT_MS` (20 min) in the worst case — the current CLI call runs to completion before the next boundary check. Not a pause/resume: next Dream/Re-Dream click is the resume mechanism.
- **Auto-Dream scheduler** — `KbDreamScheduler` (`src/services/knowledgeBase/autoDream.ts`) starts from `server.ts` after `chatService.initialize()` and stops during chat-route shutdown. It scans `chatService.listKbEnabledWorkspaceHashes()` every 60 seconds, reads each workspace's normalized `kbAutoDream` config, and opens the workspace KB DB with `getKbDb(hash)` only for enabled workspaces whose schedule is not off. See [ADR-0029](adr/0029-schedule-knowledge-base-auto-dream-per-workspace.md).
  - **Off** does nothing and cancels scheduler ownership of any window-started run for that workspace.
  - **Interval** starts incremental `dream(hash)` when pending synthesis exists and `last_run_at + intervalHours` is due. If a dream is already running when the next interval arrives, the scheduler skips starting another run and lets the current run continue.
  - **Window** uses local server time and starts incremental dreaming only while the current time is inside `[windowStart, windowEnd)`, including overnight windows such as `22:00` to `02:00`. If the scheduler started the current run and the window closes, it calls `requestStop(hash)`. Manual dream runs are never stopped by the scheduler.
  - **Continuation model** uses the existing cooperative stop semantics instead of persisted checkpoints. Committed batches clear `needs_synthesis`; unprocessed or failed batches keep `needs_synthesis = 1`; the next scheduled incremental run resumes from those pending entries. The important caveat is stop latency: an in-flight CLI/model call is allowed to finish before the stop is observed.
- **Stale status recovery** — On DB construction, any `synthesis_meta.status = 'running'` is reset to `'idle'`. This prevents the UI from showing a stale "Dreaming in progress" state after a server restart that killed a mid-dream process.

**KB Search MCP server** (`src/services/kbSearchMcp/index.ts`, `src/services/kbSearchMcp/stub.cjs`): Exposes knowledge base search and ingestion tools to CLIs during both dreaming and conversation sessions via the MCP stdio protocol. Follows the same two-process pattern as the Memory MCP server.

- **Stub** (`stub.cjs`) — Dependency-free CommonJS process spawned by CLIs as an MCP server. Env vars: `KB_SEARCH_TOKEN`, `KB_SEARCH_ENDPOINT`. Server name: `agent-cockpit-kb-search`. Implements minimal MCP protocol (`initialize`, `tools/list`, `tools/call`). Forwards tool calls as `POST { tool, arguments }` to the HTTP endpoint with `X-KB-Search-Token` header.
- **10 tools:**
  - `search_topics({ query, limit? })` — Hybrid search (semantic + keyword) over all topics. Handler: embed query → `store.hybridSearchTopics()`. Falls back to `store.keywordSearchTopics()` silently when Ollama is unavailable. Returns `{ topics: [{ topic_id, title, summary, score }] }`.
  - `get_topic({ topic_id })` — Full topic content + connections + entries. Handler: `db.getTopic()` + `db.listConnectionsForTopic()` + `db.listTopicEntryIds()`. Returns `{ topic_id, title, summary, content, connections[], entries[] }`.
  - `get_topic_neighborhood({ topic_id, depth?, limit?, min_confidence?, include_entries? })` — Bounded graph traversal over synthesized topic connections. Handler: `db.getTopic()` for the seed, `db.listAllConnections()` filtered by confidence, then undirected BFS capped at `depth <= 2` (default 1). Paths preserve stored edge direction even though discovery treats edges as undirected. Default `min_confidence='inferred'` includes `extracted` + `inferred` and excludes `speculative`; `speculative` is opt-in. `db.detectGodNodes()` supplies a score penalty so broad topics rank lower. Returns `{ seed_topic, topics: [{ topic_id, title, summary, distance, score, path, entries? }] }`.
  - `find_similar_topics({ topic_id, limit? })` — Embedding-based similarity. Handler: `store.findSimilarTopics()`. Returns empty results silently when embeddings are unavailable (pure embedding-based, no keyword fallback). Returns `{ topics: [{ topic_id, title, summary, score }] }`.
  - `find_unconnected_similar({ topic_id, limit? })` — Similar topics that have NO existing connection to the given topic. Handler: `store.findSimilarTopics()` filtered against `db.listConnectionsForTopic()`. Over-fetches by `limit + 20` from the vector store to compensate for filtered-out connected topics. Returns empty results silently when embeddings are unavailable. Returns `{ topics: [{ topic_id, title, summary, score }] }`.
  - `search_entries({ query, limit? })` — Hybrid search over entries. Handler: embed query → `store.hybridSearchEntries()`. Falls back to `store.keywordSearchEntries()` silently when Ollama is unavailable. Returns `{ entries: [{ entry_id, title, summary, score }] }`.
  - `list_documents({ query?, limit? })` — Lists converted source documents from `kb_documents`, optionally filtered by filename/description. `limit` is clamped to `1..100` with default `20`. Returns `{ documents: [{ raw_id, doc_name, doc_description, unit_type, unit_count, structure_status, updated_at }] }`.
  - `get_document_structure({ raw_id, offset?, limit? })` — Returns one document's structure metadata and a bounded page of `kb_document_nodes` rows without full body text. `offset` defaults to `0`; `limit` is clamped to `1..500` with default `200`; the response includes `{ node_count, offset, limit, truncated }` so agents can request later pages for very large PDFs and slide decks.
  - `get_source_range({ raw_id, start_unit, end_unit })` — Reads a bounded range from `converted/<rawId>/text.md`. Page and slide documents are sliced by `## Page N` / `## Slide N` blocks, section documents by heading order, line documents by line number, and fallback documents only allow range `1..1`. Ranges are capped at 25 units and 80,000 returned characters. The response includes referenced relative media paths.
  - `kb_ingest({ file_path })` — Ingest a local file into the workspace knowledge base. Handler: reads the file from disk, copies it to `knowledge/raw/` under a `conversation-documents` folder, and calls `kbIngestion.enqueueUpload()`. Respects the workspace's `kbAutoDigest` toggle (auto-digest triggers if enabled). Returns `{ ok, raw_id, filename, deduped }` on success, `{ error }` on failure. Requires `KbIngestionService` to be passed to the MCP factory.
- **Session registry** — `Map<token, { workspaceHash }>` keyed by bearer token, with a reverse `Map<sessionKey, token>` for revocation. `issueKbSearchSession(sessionKey, hash)` mints a token and returns `{ token, mcpServers: McpServerConfig[] }`. `revokeKbSearchSession(sessionKey)` removes the token. Reuses existing tokens for the same session key. For dreaming, `sessionKey` = workspace hash. For conversations, `sessionKey` = `convId` — tokens are issued alongside Memory MCP on message send, and revoked on session reset or conversation delete.
- **Ollama fallback** — When `embedText()` throws (Ollama down, model not found, etc.), `search_topics` and `search_entries` silently degrade to keyword-only search via `keywordSearchTopics`/`keywordSearchEntries`. `find_similar_topics` and `find_unconnected_similar` return empty results since they are purely embedding-based. Fallbacks are logged at `console.warn` level.
- **Router** — `POST /kb-search/call` (mounted at `/mcp/kb-search/call` via `router.use('/mcp', ...)`). Auth via `x-kb-search-token` header. Dispatches by `tool` field to handler functions. 401 for invalid/missing token, 400 for unknown tool, 500 for handler errors.

**Embedding infrastructure** (`src/services/knowledgeBase/embeddings.ts`, `src/services/knowledgeBase/vectorStore.ts`): Phase A of the three-layer retrieval architecture (#126). Provides per-workspace vector + full-text search over entries and topics using PGLite (embedded WASM PostgreSQL) with pgvector and Ollama for embeddings.

- **Embedding service** (`embeddings.ts`) — Stateless Ollama client wrapping the `/api/embed` endpoint. `embedText(text, cfg?)` embeds a single string; `embedBatch(texts, cfg?)` embeds multiple in one HTTP call (Ollama accepts `input: string[]`). `checkOllamaHealth(cfg?)` performs a quick connectivity + model probe. `resolveConfig(cfg?)` applies defaults: model `nomic-embed-text`, host `http://localhost:11434`, dimensions `768`.
- **PGLite vector store** (`vectorStore.ts`) — `KbVectorStore` class, one instance per workspace cached on `ChatService._kbVectorStores`. Database directory: `knowledge/vectors/` (PGLite uses a directory, not a single file). Schema:
  - **`store_meta`** — `(key, value)` key/value table tracking `model` and `dimensions` for mismatch detection.
  - **`entry_embeddings`** — `(entry_id PK, title, summary, embedding vector(N), tsv tsvector GENERATED)`. tsvector is weighted: title=A, summary=B. HNSW index on `embedding vector_cosine_ops`. GIN index on `tsv`.
  - **`topic_embeddings`** — Same schema shape as entries, keyed on `topic_id`.
- **Search operations:**
  - `vectorSearchEntries/Topics(queryEmbedding, topK)` — cosine-distance nearest neighbours via `<=>` operator.
  - `keywordSearchEntries/Topics(query, topK)` — `plainto_tsquery` + `ts_rank` BM25-style full-text search.
  - `hybridSearchEntries/Topics(query, queryEmbedding, topK)` — reciprocal rank fusion (RRF, k=60) of vector + keyword results. Fetches 2×topK from each, merges by RRF score.
  - `hybridSearch(query, queryEmbedding, topK)` — searches both entries and topics, interleaved by score.
  - `findSimilarTopics(topicId, topK)` — topic-to-topic similarity by embedding distance.
- **Lifecycle hooks:**
- **After digestion** — `_embedEntries()` in `KbDigestionService` embeds each new entry's `title — summary` text via `embedBatch()` and upserts into the vector store. Redigestion and raw purge paths call `store.deleteEntry(entryId)` for removed entries before/alongside DB and file cleanup so stale vectors do not outlive deleted entry rows. Best-effort: failures are logged but don't fail the digestion or purge.
  - **During dreaming (per-batch)** — `_embedBatchTopics()` embeds only the topics affected by each synthesis batch's operations, making them immediately searchable by subsequent batches.
  - **After dreaming (final sweep)** — `_embedTopics()` in `KbDreamService` re-embeds all topics (topics may be created/updated/merged/split), removes stale topic embeddings for deleted topics. Best-effort.
  - **Dreaming routing** — `_runWithRetrieval()` embeds pending entries, upserts those entry vectors, and uses `hybridSearchTopics()` to classify matches by score threshold (strong/borderline/none) for retrieval-based topic routing. `_runColdStart()` also refreshes entry vectors when embedding is configured, so Re-Dream can repair missing entry vector coverage without re-digesting files. Full Re-Dream clears only `topic_embeddings` via `wipeTopicEmbeddings()`; `wipeAllEmbeddings()` remains reserved for provider/model dimension resets.
- **Dimension change handling** — `KbVectorStore._initSchema()` reads the stored `dimensions` from `store_meta` on construction. If it differs from the requested dimensions, both `entry_embeddings` and `topic_embeddings` tables are dropped and recreated with the new vector size. `ChatService.setWorkspaceKbEmbeddingConfig()` detects model/dimension changes and closes + evicts the cached vector store so the next access rebuilds with the new schema.
- **Per-workspace config** — `WorkspaceIndex.kbEmbedding` stores `{ model?, ollamaHost?, dimensions? }`. Defaults: `nomic-embed-text`, `http://localhost:11434`, `768`. Config is read by the digest and dream services via `chatService.getWorkspaceKbEmbeddingConfig(hash)`. When no config is set, embedding is skipped (entries/topics are still created in SQLite, just not embedded).

**Per-image AI conversion + downscaling helper (`src/services/knowledgeBase/ingestion/pageConversion.ts`):** Scaffolding for the hybrid AI-assisted ingestion design (see `docs/design-kb-ingestion-hybrid.md`). Two exports:

- `convertImageToMarkdown(imagePath, { adapter, model?, effort?, timeoutMs? }) → { markdown, retried }` — a single retry-once wrapper over `adapter.runOneShot` that converts one image (PDF page render, slide render, embedded DOCX figure, or standalone uploaded image) into clean Markdown using the **unified image-to-Markdown prompt** (design §8). Behavior:
  - Sets `workingDir` to the image's parent directory and `allowTools: true` so the CLI can use its `Read` tool. The prompt references the image by basename so it works regardless of the absolute path.
  - Retries once if `runOneShot` throws **or** returns empty/whitespace output. Throws the last error after a second failure — handlers decide whether to fall back to `source: image-only`. The returned `retried` flag is `true` iff the second attempt was the one that succeeded; handlers store it as `aiRetries` on per-page metadata so 0/1 reflects what actually happened (not a constant).
  - Default per-image timeout is 10 minutes (vs. an adaptive `max(30 min, pageCount × 10 min)` for whole-document digestion).

- `ensureAiReadyImage(srcPath, destPath) → string` — downscaling helper handlers run **before** `convertImageToMarkdown`. If the source's long edge exceeds **`MAX_LONG_EDGE_PX = 2576`**, the helper decodes via `@napi-rs/canvas`, draws a proportionally-scaled copy, encodes to PNG, writes it to `destPath`, and returns `destPath`. Otherwise (or when `loadImage` throws — SVG, video, unsupported codec, malformed bytes) it returns `srcPath` unchanged with no write. The 2576 cap keeps each image under ~4,800 vision tokens for typical aspect ratios and avoids provider-side rejections on multi-MB inputs. The returned path is what the caller hands to `convertImageToMarkdown` **AND** embeds as the markdown link in `text.md`, so the same downscaled file serves both ingestion-time AI **and** digestion-time CLI reads. Re-encoding to PNG drops EXIF/color-profile/animation metadata on the downscaled copy, but the original at `srcPath` is never modified — KB Browser still serves the full-fidelity file for human viewing. Convention across handlers: the destination path is always `<srcPath>.ai.png` (sibling on disk), so a 5 MB `figure.jpg` becomes `figure.jpg` (preserved) + `figure.jpg.ai.png` (downscaled, ≤2576 px long edge). When a sibling is written, handlers add it to `mediaFiles[]` alongside the original.

- `HandlerInput` exposes `ingestionAdapter?`, `ingestionModel?`, `ingestionEffort?` so handlers (PRs 3–6 of the design sequence) can plumb the configured Ingestion CLI through to `convertImageToMarkdown`. When `ingestionAdapter` is undefined, hybrid handlers will skip the AI conversion path entirely and produce image-only output, but `ensureAiReadyImage` still runs for every image so digestion-time CLI reads stay under the cap regardless of when a CLI gets configured.

**Format handlers (`src/services/knowledgeBase/handlers/`):**
- **`pdf.ts`** — Hybrid per-page pipeline: each page is (1) rasterized to a 196 DPI PNG (always), (2) probed for layout signals via `pdfjs`, then (3) classified as **safe-text** or **needs-ai**. The classifier sits in `src/services/knowledgeBase/ingestion/pdfSignals.ts` and combines two heuristics — `figureCount` from the page's operator list (counts `paintImageXObject` / `paintInlineImageXObject` / `paintInlineImageXObjectGroup` / `paintImageXObjectRepeat` ops, resolved from `unpdf.getResolvedPDFJS().OPS`) and `tableLikely` from the text-item layout (buckets `TextItem`s by Y within `rowTolerance=2.0`, requires ≥`minTableRows=3` rows of ≥`minColumnsPerRow=2` items each, with column X-coordinates aligning across rows within `xTolerance=5.0`). Either probe throwing produces conservative needs-ai signals (`figureCount=1`). A page is **safe-text** when `figureCount === 0 && !tableLikely` — its body in `text.md` is the `pdfjs` extracted text. Otherwise it's **needs-ai** and the handler calls `convertImageToMarkdown(imagePath, { adapter, model, effort })` against the configured Ingestion CLI. When AI conversion fails (after retry-once) or no Ingestion CLI is configured, the page falls back to `image-only` — markdown contains just the image link. Each page section is a `## Page N` block prefixed with a single-line annotation `> source: pdfjs|artificial-intelligence|image-only | figures: N | table-likely: T/F`, followed by the body (when any) and **always** the `![Page N](pages/page-NNNN.png)` image link. After every successful rasterize, the handler also runs `ensureAiReadyImage(absImagePath, absImagePath + '.ai.png')` — typical 8.5×11 pages at 196 DPI render to 1666×2156 px, fit under the 2576 px cap, and need no sibling; oversized formats such as Legal, B-size, or A3 pages get a `.ai.png` sibling capped to 2576 px, the markdown image link is rewritten to point at the sibling, both originals and siblings are tracked in `mediaFiles[]`, and the AI call (when on the needs-ai path) reads from the sibling. Per-page render failures are isolated (a single bad page doesn't kill the whole doc); the section is forced to `image-only` with a `> source: image-only | note: failed to rasterize this page (...)` annotation, and the page numbers are surfaced in `metadata.failedPages`. Metadata: `pageCount`, `renderedPageCount`, `rasterDpi` (always 196), `sourceCounts` (object: `{ pdfjs?: N, "artificial-intelligence"?: N, "image-only"?: N }`), and `pages` (array of `{ pageNumber, source, figureCount, tableLikely, extractedChars, aiCallDurationMs, aiRetries, renderError? }`). Handler tag: `pdf/rasterized-hybrid`. Media files include rendered page images plus any `.ai.png` siblings produced by the downscale step.
- **`docx.ts`** — Hybrid `pandoc + per-image AI description` pipeline. Step 1 shells out to `pandoc` via the shared `runPandoc` helper (`--from=docx --to=gfm --extract-media=<outDir>/media --wrap=none <tempDocx>`); pandoc is the only tool in the stack that preserves semantic tables from OOXML, so the previous `mammoth`-based path's flat-prose tables are gone. The handler buffers the upload to a temp `.docx`, runs pandoc, flattens pandoc's nested `media/media/` output into a single `media/` dir, and prepends `# <filename>`. Pandoc emits image references in **two forms** that both get rewritten by basename: relative-path markdown (`![](./media/media/foo.png)`) for plain images, and absolute-path raw HTML (`<img src="/abs/.../media/media/foo.png" style="…" />`) for images that carry inline width/height styling — the markdown spec can't represent inline sizing, so pandoc falls back to HTML and absolutizes the `src`. The handler matches both by basename via a shared `findFlattenedRel` helper, so a single image referenced multiple ways still resolves to one collision-safe `media/<basename>` path. Step 2 walks each extracted image: it measures the natural width via `@napi-rs/canvas.loadImage` and skips icons/logos/dividers (`width < 100px` → `source: too-small`). Otherwise, when an Ingestion CLI is configured, it calls `convertImageToMarkdown(absolutePath, { adapter, model, effort })` and **appends a quoted `> Image description (source: artificial-intelligence): …` block** right after every reference to that image in the markdown (multi-line descriptions get `> ` on every line so they render as a single blockquote). Both reference forms (`![alt](mediaRel)` and `<img ... src="mediaRel" ...>`) get the description appended. Per-image failures fall back to the bare image link (`source: ai-failed`); when no Ingestion CLI is configured, the bare link stays put with `source: no-adapter`. Width-measurement decode failures fall through to the AI step (vision can sometimes still read a bytes-corrupt image). Each over-cap image also goes through `ensureAiReadyImage(abs, abs + '.ai.png')` before the AI call; when a `.ai.png` sibling gets written, a small `rewriteRefsToSibling` helper rewrites **both** reference forms — markdown `![alt](media/foo.png)` and pandoc's HTML fallback `<img src="media/foo.png" style="…">` — by escaping the rel and running two `RegExp.replace` passes against the markdown buffer, so the inline-style attributes are preserved. The sibling rel is also pushed into `mediaFiles[]` alongside the original. Returns `handler: 'docx/pandoc-hybrid'`. Metadata: `wordCount`, `mediaCount`, `sourceCounts` (`{ artificial-intelligence?: N, too-small?: N, no-adapter?: N, ai-failed?: N }`), and `images` (array of `{ mediaPath, width, source, aiCallDurationMs, aiRetries }`). **Pandoc is required** — the route layer rejects `.docx` uploads pre-flight when `detectPandoc()` reports missing. Legacy `.doc` (OLE binary format) is unsupported end-to-end: the route returns a `400` with the message `Legacy .doc format is not supported. Please resave the document as .docx…`.
- **`pptx.ts`** — Hybrid per-slide pipeline: each visible slide is (1) parsed for body + speaker notes from `ppt/slides/slideN.xml` and `ppt/notesSlides/notesSlideN.xml` via `adm-zip` + `fast-xml-parser` (`removeNSPrefix: true`), (2) probed for layout signals on the **raw namespaced XML** (regex-scanned *before* `removeNSPrefix` strips prefixes), then (3) classified as **safe-text** or **needs-ai**. The signal extractor sits in `src/services/knowledgeBase/ingestion/pptxSignals.ts` and combines two heuristics — `figureCount` (count of `<p:pic>` + `<c:chart>` tags; charts are particularly important because XML extraction loses chart data entirely — chart series live in a separate chart part, not in the slide's `<a:t>` runs) and `tableLikely` (`<a:tbl>` present, since tables flatten to a sequence of disconnected text runs without geometry, which the digestion CLI can't reconstruct into a Markdown table). A slide is **safe-text** when `figureCount === 0 && !tableLikely` — its body is the joined `<a:t>` text. Otherwise it's **needs-ai**: when a rasterized slide image exists AND an Ingestion CLI is configured, the handler calls `convertImageToMarkdown(absolutePath, { adapter, model, effort })` and uses the AI markdown as the body (`source: artificial-intelligence`). On AI failure (after retry-once), the slide falls back to `image-only` with an empty body and the image link — **the XML body is intentionally dropped** because it's the same text we already declared insufficient (charts/tables flatten poorly), so keeping it would contradict why we routed to AI. When no Ingestion CLI is configured but the rasterized image exists, the slide is `image-only` directly (empty body, image link). When **no rasterized image is available** (rasterization off or that specific slide failed to render), needs-ai slides fall back to `xml-extract` because there's nothing else to offer; the section carries `note: no slide image available; using XML extraction`. **Speaker notes are preserved on every slide regardless of source** — they're a separate text stream from the slide body, so the chart/table issue that drove `needs-ai` doesn't apply to author-written notes. Each section is a `## Slide N` block prefixed with a single-line annotation `> source: xml-extract|artificial-intelligence|image-only | figures: N | table-likely: T/F` (with an optional `| note: …` for fallback context), followed by the body (when any), the optional `### Speaker Notes` block, and the `![Slide N](slides/slide-NNN.png)` link when the slide was rasterized. **Hidden slides** (those with `show="0"` on the `<p:sld>` root element) are filtered out and the survivors are renumbered 1..N — this is required because LibreOffice's PDF export *also* skips hidden slides when rasterizing, so keeping them in the markdown with their original numbers would drift "Slide N" out of sync with `slides/slide-NNN.png` and attach notes/body to the wrong image. A short `> **Note:** X of Y slides in this deck are marked hidden…` line is injected near the bottom of `text.md` whenever `hiddenSlideCount > 0` so downstream digestion sees the skip. The trailing `## Embedded Media` block continues to list `ppt/media/*` files (the deck's original embedded images, separate from the slide rasterizations). Slide rasterization itself lives in a sibling module `src/services/knowledgeBase/ingestion/pptxSlideRender.ts` so it can be stubbed via `jest.spyOn` for AI-path tests; it shells out to `soffice --headless --convert-to pdf --outdir <tmp>` and then uses `unpdf`'s `renderPageAsImage` backed by a statically-imported `@napi-rs/canvas` (with a pre-built `CanvasFactory` passed to `getDocumentProxy` so pdfjs's internal auxiliary canvases also go through the native backend) to rasterize each slide into `slides/slide-NNN.png` at `PPTX_SLIDE_RASTER_SCALE = 7 / 3` (168 DPI). Common 16:9 slides exported as 960x540 PDF points render to 2240x1260 px, exact 28 px multiples under the 2576 px long-edge cap. When `convertSlidesToImages` is requested at ingestion time, the handler refreshes `LibreOfficeStatus`; if unavailable, it records `slideImagesWarning` in metadata and injects a "Slide rasterization note" into `text.md` explaining LibreOffice is missing. Both image streams — rasterized slides AND the `ppt/media/*` embedded media — flow through a shared `downscaleAll(rels, outDir)` helper that calls `ensureAiReadyImage(abs, abs + '.ai.png')` per file in parallel. The result is a `DownscaledImage` record per source image (`{ storeRel, linkRel, absLinkPath }`); when a sibling was written, `linkRel`/`absLinkPath` point at the `.ai.png`, otherwise they alias the original. The handler uses `linkRel` for both the per-slide markdown image link AND the trailing **`## Embedded Media`** block (`![<basename>](linkRel)`), and `absLinkPath` for the AI call when the slide is on the needs-ai path. `mediaFiles[]` aggregates both originals and any siblings. Returns `handler: 'pptx/hybrid'`. Metadata: `slideCount` (visible), `totalSlideCount` (visible + hidden), `hiddenSlideCount`, `embeddedMediaCount`, `slidesToImagesRequested`, `slideImagesWarning?`, `renderedSlideCount`, `sourceCounts` (`{ "xml-extract"?: N, "artificial-intelligence"?: N, "image-only"?: N }`), and `slides` (array of `{ slideNumber, source, figureCount, tableLikely, extractedChars, hasImage, aiCallDurationMs, aiRetries, note? }`).
- **`passthrough.ts`** — Handles text types (`txt, md, json, yaml, csv, tsv, log, xml, html, rst, markdown, htm, yml`) and image types (`png, jpg, jpeg, gif, webp, svg, bmp`). Markdown is inlined verbatim under the filename header; other text gets wrapped in a code fence tagged with the file extension. Full content is preserved regardless of file size — no truncation. **Image uploads run through the same hybrid path as PDF pages / PPTX slides / DOCX figures:** the bytes are written to `media/<filename>` first, then — when an Ingestion CLI is configured AND the extension is in the AI-eligible set (`png/jpg/jpeg/gif/webp/bmp`) — the on-disk image is sent through `convertImageToMarkdown(absolutePath, { adapter, model, effort })` and the AI's Markdown is inlined above the `![filename](media/filename)` reference (`source: artificial-intelligence`). Without an adapter, on AI failure (after retry-once), or for `.svg` (always skipped — text-based, no vision benefit), the body falls back to `source: image-only` with a `> note: …` annotation explaining why and the bare image link. For non-SVG image uploads, the on-disk PNG/JPEG/etc. is also passed through `ensureAiReadyImage(mediaPath, mediaPath + '.ai.png')` — when the long edge exceeds 2576 px, a `.ai.png` sibling is written, the markdown link is rewritten to point at the sibling, and both originals + siblings end up in `mediaFiles[]`. The same downscaled file feeds both the ingestion-time AI call and digestion-time CLI reads. Output structure: `# <filename>` → `> source: artificial-intelligence|image-only [| note: …]` → AI markdown (when any) → `![<filename>](media/<filename>[.ai.png])`. Metadata: `byteLength` always; for image uploads also `source`, `sourceCounts` (`{ artificial-intelligence?: 1, image-only?: 1 }`), `aiCallDurationMs`, `aiRetries`, optional `note`, and `downscaled: true` when a sibling was written. Handler tag: `passthrough/text` for text and `passthrough/image` for images. Exports `passthroughSupports(filename)` for the dispatcher.

**Multer configuration:** The `POST /kb/raw` route uses `multer({ storage: multer.memoryStorage(), limits: { fileSize: 1024 * 1024 * 1024, files: 1 } }).single('file')`. The 1 GB cap is generous enough for real-world PPTX decks and media-heavy PDFs; the separate conversation-attachment endpoint keeps its smaller limit. Memory storage is required because the orchestrator needs to hash the buffer before touching disk (so dedup works and we never write a file the user already has). Multer errors (file-too-large, etc.) are wrapped in a JSON-returning middleware shim so the KB Browser surfaces a useful message instead of an HTML 500 — `LIMIT_FILE_SIZE` becomes `400 { error: "File exceeds the 1 GB upload limit." }`.

### Migration

On first startup after upgrade, `initialize()` detects legacy `conversations/` directory:
1. Reads all conversation JSON files, groups by workspace
2. Writes workspace index + session files to `workspaces/{hash}/`
3. Renames old dirs to `*_backup/`

## 4.2 Backend Adapter System

The CLI backend layer uses a **pluggable adapter pattern**. New CLI tools can be added without modifying routes, chat service, or frontend.

`src/services/cliProfiles.ts` defines helpers that map internal backend ids to physical CLI vendors, deterministic server-configured CLI profiles, and profile-derived runtime backends. The registry remains backend-keyed, but profile-based message sends, one-shot OCR calls, auto-title generation, session-summary generation, reset cleanup, memory capture, Memory MCP processing, and KB ingestion/digestion/dreaming choose the adapter through the profile resolver when a profile id is present. Claude Code profiles add `protocol: "standard" | "interactive"` so the same physical Claude CLI profile can select either `claude-code` or `claude-code-interactive` without a separate composer/provider picker. Server-configured profiles still use the server user's existing CLI state. Codex and Claude Code profiles apply profile-specific `command`, `configDir`, and `env` at spawn/path-resolution time. Kiro stays self-configured only: no command/config/env/runtime-protocol isolation is applied because `kiro-cli` has no dedicated profile directory override and isolating via `HOME` was rejected.

### BaseBackendAdapter (`src/services/backends/base.ts`)

Abstract base class. Every backend must implement:
- **`get metadata`** — returns `{ id, label, icon, capabilities, resumeCapabilities, models? }` where capabilities includes booleans for `{ thinking, planMode, agents, toolActivity, userQuestions, stdinInput }` plus optional `goals`. `goals` may be absent/false, legacy `true`, or a structured `{ set, clear, pause, resume, status }` capability where `status` is `'native'`, `'transcript'`, or `'none'`. Route and frontend code normalize legacy `true` as full native Codex-style support. `resumeCapabilities` is `{ activeTurnResume, activeTurnResumeReason, sessionResume, sessionResumeReason }`: `activeTurnResume` answers whether the cockpit can safely reattach to the exact in-flight backend turn after a cockpit server restart without resending the prompt, while `sessionResume` answers the weaker next-turn context-continuity question. `models` is an optional array of `{ id, label, family, description?, costTier?, default?, supportedEffortLevels? }` for backends that support model selection. `supportedEffortLevels` is an optional `('none' | 'minimal' | 'low' | 'medium' | 'high' | 'xhigh' | 'max')[]`; omit it when the model does not support adaptive reasoning effort. Values are backend/model-specific: Codex may expose `none` / `minimal`, while Claude Code exposes `max` on supported Opus models.
- **`getMetadata(options?)`** — async metadata resolver. Base class returns `this.metadata`; profile-aware adapters can use `options.cliProfile` to return profile-specific model catalogs without changing the static registry list. `GET /api/chat/cli-profiles/:profileId/metadata` calls this method.
- **`sendMessage(message, options)`** — returns `{ stream, abort, sendInput }` where `stream` is an async generator yielding events matching the stream event contract in Section 3. For generated files/images that do not appear in assistant text, adapters yield `{ type: 'artifact', sourcePath | dataBase64, filename?, mimeType?, title?, sourceToolId? }`; `processStream` persists the bytes and forwards a normalized `{ type: 'artifact', artifact }` frame. `options` includes `{ sessionId, conversationId, cliProfileId?, cliProfile?, isNewSession, workingDir, systemPrompt, externalSessionId, model?, effort?, serviceTier? }`. `conversationId` is the stable conversation ID (does not change on session reset) — used by backends like Kiro that key long-lived processes by conversation. `cliProfileId` identifies the selected profile for persistence and diagnostics; `cliProfile` is the resolved profile object passed to profile-aware adapters. Codex consumes `cliProfile.command`, `cliProfile.configDir`, and `cliProfile.env`; Claude Code consumes `cliProfile.command`, `cliProfile.configDir → CLAUDE_CONFIG_DIR`, and `cliProfile.env`; Kiro still spawns from server-configured state until its phase lands. `model` is the full model ID to use for this invocation (e.g. `claude-opus-4-7`); backends that don't support model selection ignore it. `effort` is the adaptive reasoning level for this turn; backends ignore it when the selected model doesn't declare the requested level in `supportedEffortLevels`. `serviceTier` is currently Codex-only; `fast` forces Codex Fast mode for the turn/thread, while absence leaves the selected Codex profile/config in control.
- **`generateSummary(messages, fallback, options?)`** — returns a one-line summary string. `options.cliProfile` carries the resolved profile for adapters that execute a one-shot CLI call.
- **`generateTitle(userMessage, fallback, options?)`** — returns a short conversation title. Base class provides a default that truncates the user message to 80 chars. `options.cliProfile` carries the resolved profile for adapters that execute a one-shot CLI call. `chatService.generateAndUpdateTitle()` applies an additional 8-word hard-cut on the adapter's return value before persisting, so adapters don't need to enforce a word count themselves.
- **`shutdown()`** — called during server shutdown. Override to kill long-lived processes. No-op by default.
- **`onSessionReset(conversationId)`** — called when user resets a session. Override to clean up per-conversation state. No-op by default.
- **`extractMemory(workspacePath, options?)`** — returns a `MemorySnapshot` for the backend's native memory system, or `null` if unsupported / no memory exists. Called by `ChatService.captureWorkspaceMemory` from the reset memory finalizer and real-time memory watcher paths. `options.cliProfile` carries the resolved profile for adapters whose native memory lives under a profile-specific config directory. Base class returns `null`.
- **`getMemoryDir(workspacePath, options?)`** — resolves the absolute path to the backend's native memory directory for a workspace, without reading contents. Returns `null` when the backend has no memory system or no memory directory exists yet. Used by the real-time `MemoryWatcher` to know which directory to watch. `options.cliProfile` carries the resolved profile for profile-aware memory dirs. Subclasses that implement `extractMemory` should also implement this. Base class returns `null`.
- **`runOneShot(prompt, options?)`** — executes a one-shot LLM call without streaming. Used by the Memory MCP handler, post-session extraction, OCR, auto-title/session-summary generation, the default Digestion CLI path, and Context Map processing. Options: `{ model?, effort?, serviceTier?, timeoutMs?, allowTools?, workingDir?, mcpServers?, cliProfile?, abortSignal? }`. Adapters should honor `abortSignal` by terminating their child process and rejecting when possible; callers such as Context Map also check the signal between work units. Returns the trimmed stdout. Default base class throws `Error('<Adapter>.runOneShot is not implemented')`.
- **`runSessionShot(prompts, options?)`** — executes multiple prompts in one backend session and returns one output per prompt. KB digestion uses it only when `Settings.knowledgeBase.kbGleaningEnabled` is true so the second prompt can ask for missed entries after the primary extraction. The default base class provides a functional fallback by replaying prior prompt/response pairs into subsequent `runOneShot()` calls; adapters can override it with a native session implementation when the backend supports one.

### Shared Tool Utilities (`src/services/backends/toolUtils.ts`)

Shared helpers used by all backend adapters. Extracted for cross-adapter reuse — adapters import from here, never from each other.

- `sanitizeSystemPrompt(prompt)` — strips control characters, truncates to 50K max
- `isApiError(text)` — detects `API Error: NNN` patterns
- `shortenPath(filePath)` — truncates long paths to `.../{last}/{two}`
- `extractToolOutcome(toolName, content)` — classifies tool results as success/error/warning
- `extractToolDetails(block)` — converts Claude Code tool_use blocks into ToolDetail objects
- `extractUsage(event)` — normalizes usage/cost data into UsageEvent

### BackendRegistry (`src/services/backends/registry.ts`)

- `register(adapter)` — stores by `metadata.id`. First registered becomes default. Validates `instanceof BaseBackendAdapter`.
- `get(id)` — returns adapter or `null`
- `list()` — returns metadata array
- `getDefault()` — returns first registered or `null`
- `shutdownAll()` — calls `shutdown()` on all registered adapters (used in graceful server shutdown)

The backend registry is keyed by internal backend id. Profile resolution sits above it: `CliProfile.vendor` chooses the physical CLI identity, and Claude Code's optional `protocol` chooses either the standard `claude-code` adapter or the `claude-code-interactive` adapter. Codex and Claude Code receive resolved profile command/config/env today; Kiro always uses server-configured spawning.

### Backend Resume Capability Matrix

Backend metadata distinguishes **active-turn resume** from **session resume**:

| Backend | Active-turn resume after cockpit restart | Session resume for later turns |
|---|---|---|
| Claude Code | `unsupported` — `--resume` starts a new CLI invocation from session history and does not reattach to a lost process/stdout stream. | `supported` — follow-up turns pass the cockpit session id through `--resume`. |
| Kiro | `unsupported` — ACP `session/load` restores session history for a later prompt, but the adapter cannot reattach to the exact in-flight `session/prompt` response stream. | `supported` — `external_session` persists the Kiro session id and `session/load` rehydrates it after process respawn. |
| Codex | `unsupported` — `thread/resume` restores the thread for a later turn, but the adapter cannot safely reattach to an already-running turn after the cockpit process loses its notification iterator. | `supported` — `external_session` persists the Codex thread id and `thread/resume` rehydrates it after process respawn. |

Because every current backend advertises `activeTurnResume: 'unsupported'`, startup reconciliation continues to persist an interrupted assistant `streamError` and remove the durable job. The server never re-sends an accepted prompt automatically. Current adapters still emit diagnostic runtime identifiers where available: Claude Code, Claude Code Interactive, Kiro, and Codex emit their local child `processId`; Codex records the app-server turn id from the `turn/start` response path, emits it as `backend_runtime.activeTurnId` from that path, and dedupes `turn/started` if the notification is also emitted. Future backend-specific work must update the matrix, persist any additional required runtime identifiers, and add tests proving reattach does not duplicate tool execution before changing reconciliation behavior.

### ClaudeCodeAdapter (`src/services/backends/claudeCode.ts`)

**Metadata:** `id: 'claude-code'`, capabilities: `thinking: true, planMode: true, agents: true, toolActivity: true, userQuestions: true, stdinInput: true, goals: { set: true, clear: true, pause: false, resume: false, status: 'transcript' }`. `resumeCapabilities` reports active-turn resume as unsupported and later-session resume as supported via `--resume`. The goal capability means Claude Code can start and clear goals through the CLI's `/goal` slash command and can expose transcript-derived status when `goal_status` attachments exist, but Agent Cockpit does not emulate pause or resume. Exposes `models` array with full model IDs (no aliases — IDs pin a specific version so the dropdown is stable across CLI updates):

| `id` | `label` | `family` | `costTier` | `default` |
|---|---|---|---|---|
| `claude-opus-4-7` | Opus 4.7 | opus | high | — |
| `claude-opus-4-6` | Opus 4.6 | opus | high | — |
| `claude-sonnet-4-6` | Sonnet 4.6 | sonnet | medium | ✓ |
| `claude-haiku-4-5` | Haiku 4.5 | haiku | low | — |

Adaptive reasoning effort support (`supportedEffortLevels`):
- `claude-opus-4-7`: `['low', 'medium', 'high', 'xhigh', 'max']` — the `xhigh` level is Opus 4.7-only
- `claude-opus-4-6`: `['low', 'medium', 'high', 'max']` — the `max` level is Opus-only
- `claude-sonnet-4-6`: `['low', 'medium', 'high']`
- `claude-haiku-4-5`: field omitted (no effort support)

**`sendMessage(message, options)`:**
- `options`: `{ sessionId, isNewSession, workingDir, systemPrompt, model?, effort?, cliProfile? }`
- Returns `{ stream, abort, sendInput }`
- `abort()` sends SIGTERM to CLI process
- `sendInput(text)` writes to stdin (safe after abort)
- Per-request state: each call creates its own `state` object (no shared mutable state)
- Guards the `--effort` flag: only forwards it when the model declares the requested level in `supportedEffortLevels`. This protects against stale conversation state after a model swap.
- Resolves `cliProfile` through `resolveClaudeCliRuntime(profile?)`: `command` defaults to `claude`, `env` is merged into `process.env`, and `configDir` maps to `CLAUDE_CONFIG_DIR` (taking precedence over any `env.CLAUDE_CONFIG_DIR`). On Windows, the runtime resolver prefers installer-managed package entrypoints under `<installDir>\cli-tools` or `%APPDATA%\npm` before falling back to `.cmd` shims through `cmd.exe`; macOS/Linux keep direct command execution. Non-Claude profiles are rejected before spawning.

**CLI invocation:**
```bash
<profile.command || claude> --print \
  --permission-mode bypassPermissions \
  --output-format stream-json \
  --verbose \
  [--model <id>]                     # if model specified (e.g. claude-opus-4-7, claude-sonnet-4-6)
  [--effort <level>]                # if effort specified AND model supports that level
  [--session-id <uuid>]              # if isNewSession
  [--append-system-prompt <prompt>]  # if isNewSession and systemPrompt
  [--resume <uuid>]                  # if not isNewSession
  -p "<user message>"
```
The child process env is the resolved profile env. Server-configured profiles leave `CLAUDE_CONFIG_DIR` unset, so Claude Code uses the server user's default `~/.claude`. Account profiles set `CLAUDE_CONFIG_DIR=<configDir>`, isolating credentials, settings, sessions, plugins, and native memory under that directory.

**Goals:** `setGoalObjective(objective, options)` delegates to the existing streaming path by sending `/goal <objective>` as the prompt text. The chat route uses the same durable stream supervision as normal messages but does not save a user message for the slash command; instead it persists a system `goalEvent` message so the objective is visible in the transcript without becoming ordinary backend input. `clearGoal(options)` runs `/goal clear` through `_createStream()` and drains it to completion; the route only permits this while the conversation is idle so a second Claude CLI invocation does not race the active turn. `pauseGoal()` and `resumeGoal()` intentionally throw because Claude Code does not expose reliable pause/resume controls for goals.

`getGoal(options)` is best-effort. It resolves the Claude project directory candidates for the selected workspace/profile, reads the active session JSONL from the first candidate that contains that session transcript, and parses the newest `goal_status` attachment. A status with `met: true` becomes `complete`; otherwise it is `active`. The returned backend-neutral goal includes `backend: 'claude-code'`, `source: 'transcript'`, `sessionId`, `objective` derived from the attachment condition, `lastReason`, `iterations`, `turns`, duration/token counters, timestamps when present, and `supportedActions: { clear: true, stopTurn: true, pause: false, resume: false }`. If a later `/goal clear` command or "Goal cleared" transcript text appears, parsing returns `null`.

**`extractToolDetails(block)`** — parses `tool_use` content blocks into enriched detail objects:

| Tool | Description format | Extra fields |
|------|-------------------|--------------|
| `Read` | `Reading \`{path}\`` or `Reading file` | — |
| `Write` | `Writing \`{path}\`` | `isPlanFile` if path contains `.claude/plans/`; plan writes also expose `planContent` and `planFilePath` |
| `Edit` | `Editing \`{path}\`` or `Editing file` | — |
| `Bash` | `input.description`, or `Running: \`{cmd}\`` (truncated 60 chars), or `Running command` | — |
| `Grep` | `Searching for \`{pattern}\` in {glob}` or `Searching files` | — |
| `Glob` | `Finding files matching \`{pattern}\`` or `Finding files` | — |
| `Agent` | Uses `input.description` | `isAgent: true`, `subagentType` (default `'general-purpose'`) |
| `TodoWrite` | `Updating task list` | — |
| `WebSearch` | `Searching: \`{query}\`` or `Searching the web` | — |
| `WebFetch` | `Fetching: {url}` or `Fetching web content` | — |
| `EnterPlanMode` | `Entering plan mode` | `isPlanMode: true`, `planAction: 'enter'` |
| `ExitPlanMode` | `Plan ready for approval` | `isPlanMode: true`, `planAction: 'exit'`, `planContent` from `input.plan`/`input.content`, and `planFilePath` from `input.planFilePath`/`input.plan_file_path`/`input.filePath`/`input.file_path` when present |
| `AskUserQuestion` | `Asking a question` | `isQuestion: true`, `questions` array |
| (unknown) | `Using {name}` | — |

All detail objects include `tool`, `id` (block id or null), and `description`. Long file paths are shortened to `.../{last}/{two}` when >3 segments.

**`extractUsage(event)`** — parses `result` events for usage data. Returns `{ type: 'usage', usage: { inputTokens, outputTokens, cacheReadTokens, cacheWriteTokens, costUsd } }` or `null` if no usage data is present. Field mapping: `input_tokens` → `inputTokens`, `output_tokens` → `outputTokens`, `cache_read_input_tokens` → `cacheReadTokens`, `cache_creation_input_tokens` → `cacheWriteTokens`, `cost_usd` → `costUsd`.

**`generateSummary(messages, fallback, options?)`** — calls `runOneShot(prompt, { timeoutMs: 30000, cliProfile: options.cliProfile })`. Falls back gracefully.

**`generateTitle(userMessage, fallback, options?)`** — calls `runOneShot(prompt, { timeoutMs: 30000, cliProfile: options.cliProfile })` to generate a short title (prompt asks for max 8 words) from the user's first message. Falls back to truncated user message. The returned title is further bounded to 8 words by `chatService.generateAndUpdateTitle()`.

**`runOneShot(prompt, options?)`** — spawns `<profile.command || 'claude'> --print` with the given prompt and options. Builds args: `['--print', '-p', prompt, '--model', model]`, plus `--permission-mode bypassPermissions` when `allowTools` is set, `--mcp-config <json>` when `mcpServers` are set, and `--effort <level>` only when the selected model supports the requested effort. Uses `execFile` with `{ timeout: timeoutMs, maxBuffer: 4MB, cwd, env }`, where `env` is the resolved profile env and includes `CLAUDE_CONFIG_DIR=<configDir>` for account profiles. Closes stdin immediately via `child.stdin?.end()` to prevent the CLI's "no stdin data received" warning. Error handling filters the stdin warning from stderr and classifies errors: killed processes report timeout, non-empty stderr is preferred over generic exit codes. Default timeout: 60 seconds unless callers raise it for digestion/dreaming.

**Text / thinking event emission:** The adapter parses the CLI's `stream-json` output and yields `{ type: 'text', content }` / `{ type: 'thinking', content }` events for each `text` / `thinking` content block inside an `assistant` event. These whole-block events are emitted **without** a `streaming: true` flag. Delta-level events (`content_block_delta`) are only produced when the CLI is invoked with `--include-partial-messages`, which this adapter does not pass, so in practice the delta branch is dead code and every live text segment arrives as a whole block. The chat router (`processStream` in `src/routes/chat.ts`) treats both shapes identically: every text/thinking event is forwarded to the client and accumulated into the in-flight assistant message, and the accumulator is persisted at `turn_boundary` or `done` whenever it is non-empty — the `streaming` flag is not consulted. This matters because turns frequently interleave pre-tool-call text → tool use → post-tool-call text, and gating the save on delta-style streaming would silently drop the pre-tool-call text (the original intermediate-save bug).

### ClaudeCodeInteractiveAdapter (`src/services/backends/claudeCodeInteractive.ts`)

**Metadata:** `id: 'claude-code-interactive'`, label `Claude Code Interactive`, icon shared with Claude Code, same model catalog as `ClaudeCodeAdapter`, capabilities `thinking: true, planMode: true, agents: true, toolActivity: true, userQuestions: true, stdinInput: true, goals: { set: true, clear: true, pause: false, resume: false, status: 'transcript' }`. `resumeCapabilities` reports active-turn resume as unsupported because Agent Cockpit cannot reattach to a hidden terminal UI after process loss, and later-session resume as supported through Claude Code's session history. See [ADR-0058](adr/0058-support-claude-code-interactive-through-transcript-watched-pty.md).

The adapter is a separate internal backend, not a separate physical CLI profile vendor. `cliProfiles.ts` maps a Claude Code profile with `protocol: "interactive"` to `claude-code-interactive`, while `protocol: "standard"` maps to `claude-code`; both protocols share the physical `claude-code` vendor, `command`, `env`, `CLAUDE_CONFIG_DIR`, auth state, native memory paths, plan usage, and CLI update target. Conversations still store `backend: 'claude-code-interactive'` for compatibility and transcript rendering, but the user-facing selection is the CLI profile's Protocol field. Background one-shot jobs (`runOneShot`, title generation, summaries, Memory extraction, KB processors, Context Map processors, and OCR helpers) delegate to the existing Claude Code headless path rather than driving a hidden PTY.

Before the first PTY spawn, `claudeInteractivePty.ts` verifies the bundled `node-pty` `spawn-helper` executable on Unix-like platforms. Some npm installs unpack the prebuilt helper without executable bits, which causes every `node-pty.spawn()` call to fail with `posix_spawnp failed` before Claude starts. The adapter repairs the helper mode in place when possible and otherwise surfaces a Claude Code Interactive startup error that names the PTY startup failure.

**Transport:** `sendMessage(message, options)` starts a hidden local PTY through `node-pty`, launches `<profile.command || 'claude'>` without `-p`, sends the prompt into the terminal via bracketed paste followed by a delayed Enter so multi-line workspace-context prefixes remain one prompt submission, and never renders or parses terminal escape output as chat content. After Claude readiness is detected, the adapter waits a short prompt-ready delay before bracketed paste because resumed Claude Code interactive screens can fire `SessionStart` before the input box is accepting paste. Terminal output is control-only: `claudeInteractiveTerminal.ts` answers Claude/Ink DEC/XTerm startup queries for DA1, DA2, DSR cursor position, XTVERSION, and `18t` window size; the PTY controller keeps a rolling stripped-terminal buffer so split workspace trust prompts can be auto-confirmed once, sends queued user answers, sends Escape on abort, and asks the CLI to exit with `/exit` during cleanup before killing the PTY if needed. Public `done` is emitted only after the PTY cleanup path has requested exit and waited for a short post-exit settle, so the browser composer cannot start a fast `--resume` follow-up against a session whose previous interactive process is still closing. A process-local `ClaudeInteractiveSessionManager` tracks active controllers by conversation id so shutdown and route cleanup can stop them.

**Hook lifecycle:** `claudeInteractiveHooks.ts` creates a per-turn temp hook sink and passes inline `--settings` JSON that registers Claude Code `SessionStart`, `PreToolUse`, and `Stop` command hooks. Hook setup is best-effort and optional: if the platform or temp-file setup cannot support hooks, the adapter logs a warning and falls back to transcript polling. When hooks are available, the adapter waits for `SessionStart` before submitting the prompt; if the hook does not arrive before the readiness timeout, the prompt is submitted through the old fallback path. Production `PreToolUse` is registered narrowly for `AskUserQuestion` so the browser can surface a clarifying-question card from the hook payload when Claude's transcript line has not flushed yet; the transcript remains authoritative and later duplicate question tool events are suppressed by tool id/question signature. The hook harness also supports a diagnostic mode used only by the real-Claude compatibility suite; that mode records broader current-CLI hook events such as `PostToolUse`, `SessionEnd`, `SubagentStart`, `SubagentStop`, `PermissionRequest`, and `PermissionDenied` into test artifacts without changing production stream semantics. `Stop` is the preferred turn-completion signal. Its payload can provide the exact `transcript_path` and `last_assistant_message`; the former pins the transcript tailer to the current turn's file, and the latter is used only as a non-duplicating fallback when no assistant text was emitted from the transcript.

**Transcript source:** Structured events come from Claude Code's JSONL transcripts under the selected Claude config project directory, resolved through the same sanitized-workspace candidates used by `ClaudeCodeAdapter.getGoal()` or pinned from a `Stop` hook `transcript_path`. Candidate resolution checks both the lexical workspace path and its filesystem realpath so symlinked paths such as macOS `/var/...` temp workspaces can still find transcripts that Claude stores under `/private/var/...`. `ClaudeTranscriptTailer` tails the active `{sessionId}.jsonl`, tracks byte offsets and seen UUIDs, and handles the common race where the file does not exist until after the PTY starts without skipping the first transcript entries. After `Stop`, the adapter flushes any buffered partial line and retries final transcript reads briefly so Stop-before-flush races still emit the final assistant/tool/usage events before a single `done`. The tailer filters by `sessionId` when the transcript provides one.

**Event mapping (`src/services/backends/claudeTranscriptEvents.ts`):**
- assistant text blocks → `text`
- assistant thinking blocks → `thinking`
- assistant `tool_use` blocks → enriched `tool_activity` using the same Claude Code tool-detail extraction, including `Agent`, `AskUserQuestion`, plan-mode tools, MCP-style names, and `.claude/plans/` writes
- user `tool_result` blocks → `tool_outcomes` plus `turn_boundary`
- system progress/notification entries → `task_progress` / `task_notification`
- duration/result entries → `turn_duration`, `usage`, and terminal `done` behavior
- `goal_status` attachments → backend-neutral `goal_updated` snapshots with `backend: 'claude-code-interactive'`
- `/goal clear` local-command stdout containing `Goal cleared` → `goal_cleared`

When a `.claude/plans/` write completes in plan mode, the streaming route may emit a synthetic `ExitPlanMode` approval before Claude's transcript flushes the real `ExitPlanMode` tool event. If the real transcript event arrives later, the route uses it to flush any deferred approval input and then suppresses the duplicate plan card so an already-approved plan cannot reopen after the final response.

`sendInput(text)` maps live responses back into the PTY. For `AskUserQuestion`, the controller chooses the numbered option when the user's text matches an option label/value, including object-shaped options with `label` fields, otherwise it sends the raw text. Option selection waits briefly before sending arrow/Enter keystrokes so the interactive question menu is ready even when the question card came from `PreToolUse` before the transcript flushed. Current real Claude Code question menus are option-driven, so browser typed-input coverage verifies option-label-to-menu-index mapping rather than relying on arbitrary free-text `AskUserQuestion` payloads. `abort()` sends Escape to stop the active turn. Because transcript parsing is private-API dependent, the adapter is version-gated by compatibility warnings rather than assumed stable across arbitrary Claude CLI releases.

**Goals:** `setGoalObjective()` sends `/goal <objective>` through the interactive PTY stream. `getGoal()` reads the transcript JSONL and parses the latest goal attachment with `backend: 'claude-code-interactive'`. `clearGoal()` sends `/goal clear` through an idle interactive stream, treats Claude's local-command stdout `Goal cleared` transcript marker as the completion signal, and fails if that marker is not observed before timeout or stream completion because current Claude Code clears can finish without a normal assistant `Stop`/`turn_duration` completion. The route rejects clear while a Claude Code or Claude Code Interactive stream is active. Pause/resume intentionally throw unsupported errors, matching Claude Code.

### KiroAdapter (`src/services/backends/kiro.ts`)

**Metadata:** `id: 'kiro'`, capabilities: `thinking: true, planMode: false, agents: true, toolActivity: true, userQuestions: false, stdinInput: false`. `resumeCapabilities` reports active-turn resume as unsupported and later-session resume as supported via `external_session` + `session/load`. Exposes a hardcoded `models` array so the selector renders immediately on page load (Kiro's ACP `session/new` response also carries a model list, but that's only available after the first message — reading it lazily caused the dropdown to appear empty and then pop in "after the fact"). The hardcoded list mirrors [kiro.dev/docs/cli/models](https://kiro.dev/docs/cli/models/):

| ID | Family | Cost Tier | Default |
|---|---|---|---|
| `auto` | router | medium | ✓ |
| `claude-opus-4.7` | opus | high | — |
| `claude-opus-4.6` | opus | high | — |
| `claude-opus-4.5` | opus | high | — |
| `claude-sonnet-4.6` | sonnet | medium | — |
| `claude-sonnet-4.5` | sonnet | medium | — |
| `claude-sonnet-4.0` | sonnet | medium | — |
| `claude-haiku-4.5` | haiku | low | — |
| `deepseek-3.2` | other | low | — |
| `minimax-m2.5` | other | low | — |
| `minimax-m2.1` | other | low | — |
| `glm-5` | other | low | — |
| `qwen3-coder-next` | other | low | — |

No `supportedEffortLevels` — Kiro does not expose effort tuning. When Kiro adds or removes models upstream, update this list manually.

**Integration protocol:** ACP (Agent Client Protocol) — JSON-RPC 2.0 over stdin/stdout via `kiro-cli acp`.

**Model selection:** After session setup (`session/new` or `session/load`), calls `session/set_model({ sessionId, modelId })` if `options.model` is set. Non-fatal — continues with default model on failure. Model can be changed mid-session without process restart (unlike Claude Code).

**ACP process lifecycle:** Lazy spawn + idle timeout + transparent recovery.
- First message → spawn `kiro-cli acp` → `initialize` handshake → `session/new(cwd)` → **yield `external_session` stream event** → `[session/set_model]` → `session/prompt`
- Subsequent messages → reuse process → `[session/set_model]` → `session/prompt`
- Idle timeout (configurable via `KIRO_ACP_IDLE_TIMEOUT_MS` env var, default 1 hour) → kill process. The timer resets on every ACP notification received during a turn, so long-running multi-agent runs don't get killed mid-flight; "idle" means literally no activity from the app-server, not "time since last user message".
- Next message after timeout → respawn → `initialize` → `session/load(sessionId, cwd)` → **drain replayed notifications** → `session/prompt`

**Stream termination:** Empirically, Kiro holds the `session/prompt` JSON-RPC response until the turn is fully done — after all subagents, tool calls, permission requests, and streaming chunks complete. The response body's `stopReason` (`end_turn` | `cancelled` | `refusal` | `max_tokens`) IS the end-of-turn signal; Kiro does **not** emit a `session/update` with `sessionUpdate: 'turn_end'` on the ACP channel. The adapter calls `AcpClient.stopNotifications()` immediately when the `session/prompt` response resolves (same on `.catch()`), which ends the notification iterator and the stream loop exits with a `done` event. The `turn_end` branch in the notification handler is kept as defensive handling in case Kiro ever starts emitting it.

**Abort handling:** Kiro active-turn resume is unsupported, so a user abort is treated as a hard boundary for the per-conversation ACP process. The adapter best-effort sends `session/cancel`, then stops notification iteration, kills the active `kiro-cli acp` process, and removes it from the process map. This prevents the next user message from reusing a client that still has the aborted `session/prompt` response or notification iterator attached. The next send respawns ACP, rehydrates the persisted `externalSessionId` via `session/load`, drains replayed notifications, and sends a fresh `session/prompt`.

**`session/load` replay handling:** Per the ACP spec, `session/load` streams the full prior session history as `session/update` notifications before returning. The adapter's `AcpClient.notificationQueue` buffers these alongside any future notifications, and the notification consumer loop doesn't start until after `session/prompt` is sent. To prevent the replayed `agent_message_chunk` frames from being consumed as text for the current turn (which would concatenate every prior assistant response into the new reply), `AcpClient.drainNotifications()` is called immediately after the `session/load` promise resolves. The drain count is logged.

**Session mapping:** Agent Cockpit session IDs map to Kiro ACP session IDs via in-memory `sessionMap`. Immediately after `session/new` returns, the adapter also yields an `external_session` `StreamEvent` carrying the Kiro session ID. `processStream` consumes that event and calls `chatService.setExternalSessionId(convId, id)`, which writes it to the active `SessionEntry.externalSessionId` on disk. On the next turn after a cockpit server restart, `SendMessageOptions.externalSessionId` carries the persisted ID back into the adapter; the rehydrate branch puts it into the fresh in-memory `sessionMap` and uses it for `session/load`. Without this, the in-memory map would be empty on restart and the adapter would reject the next turn with `"No Kiro session ID available for this conversation"`.

**Tool name normalization:** Kiro uses lowercase tool names (`read`, `shell`, `delegate`, etc.). The adapter normalizes to Agent Cockpit display names (Read, Bash, Agent, etc.) via `extractKiroToolDetails()`.

**Thinking tool:** Kiro's `thinking` tool is special-cased — output is emitted as `ThinkingEvent` (displayed in thinking UI) rather than `ToolActivityEvent`.

**Permission handling:** All `session/request_permission` messages are auto-approved with `allow_always`.

**Usage tracking:** Kiro's `_kiro.dev/metadata` notifications are parsed for `credits` (accumulated) and `contextUsagePercentage` (snapshot, overwritten each update). These are persisted on the conversation and session `Usage` objects but **excluded from the daily usage ledger** — Kiro's credit-based billing is not comparable with token-based backends. The frontend header's per-CLI chip surfaces the context percentage for `kiro` (see `chip-renderers.jsx` in spec-frontend). Because `contextUsagePercentage` is a Kiro-only snapshot and `chatService._addToUsage` only overwrites the field when the incoming source includes it, both `updateConversationBackend` and `resetSession` explicitly clear the field on the conversation entry (and `updateConversationBackend` also clears it on the active session) so a stale Kiro value cannot leak into a subsequent Claude Code / generic-backend chip.

**ACP error diagnostics:** `AcpClient` rejects pending requests with an `Error` enriched with the JSON-RPC `error.code` (as a numeric `code` property) and `error.data` (preserved as-is). Both `_acpOneShot` and the streaming `sendMessage` path format a `[code=N] | data: <json>` suffix into the rejection / terminal stream error / log message (data is JSON-stringified and clamped to 500 chars). A streaming `session/prompt` rejection yields a backend `error` event before `done` so the frontend shows a durable stream error instead of an empty assistant turn. This is critical for diagnosing upstream-model rejections like `-32603 Internal error`, where the JSON-RPC `data` field carries the actual reason ("Prompt is too long", token counts, etc.) that the bare `message` discards.

**One-shot LLM calls (`runOneShot`, `generateTitle`, `generateSummary`):** All three callers funnel through one private helper, `_acpOneShot(prompt, { model?, timeoutMs? })`, which speaks ACP via an **ephemeral** `kiro-cli acp` process — separate from any per-conversation ACP process — so one-shot calls never pollute streaming session state. The flow: spawn `kiro-cli acp` → `initialize` → `session/new({ cwd, mcpServers })` → optional `session/set_model({ modelId })` → `session/prompt`. Permission requests are auto-approved with `{ outcome: 'selected', optionId: 'allow_always' }`. The buffer rule for collecting the assistant's final answer is: append `agent_message_chunk` text only when no tool is currently active, **clear** the buffer on each `tool_call`, and remove the tool from the active set on `tool_call_update`. This discards any pre-tool reasoning ("Let me read the image first.") and retains only the post-tool final answer, exactly as `agent_message_chunk` text. The process is killed on `session/prompt` resolution, on timeout (`timeoutMs`, default 60s; `generateTitle`/`generateSummary` use 30s), or on spawn error. `generateTitle` and `generateSummary` truncate the result (80 chars / 200 chars) and fall back gracefully if kiro-cli is not installed or not authenticated. ACP is used uniformly because the legacy `kiro-cli chat --no-interactive` path interleaves tool-call narration (`Reading images: …`, ` (using tool: …)`, ` ✓ Successfully …`, ` - Completed in Xs`) inline with the answer text whenever the agent decides to call a tool — and parsing that back out is fragile and CLI-version-dependent. ACP gives us structurally distinct frame types so no string parsing is required.

**Image content-block auto-attach (`collectImageContentBlocks`):** Before sending `session/prompt`, `_acpOneShot` scans `workingDir` for image files (`.png`/`.jpg`/`.jpeg`/`.gif`/`.webp`) whose basename appears in the prompt text and appends them as ACP `{ type: 'image', mimeType, data: <base64> }` content blocks alongside the existing `{ type: 'text' }` block (capped at 5 attachments). This sidesteps Kiro's `fs_read` Image-mode path, which base64-inlines image bytes into the conversation transcript and overflows the upstream model's prompt budget — observed as JSON-RPC `-32603 Internal error: Prompt is too long` even after KB ingestion downscaled the image to fit Kiro's documented 10 MB attachment cap. The basename match uses **token-boundary** matching (`basenameAppearsAsToken`), not loose `String.includes`: a candidate matches only when the surrounding characters are not filename characters (`[A-Za-z0-9._-]`). This is required because filename chars flow into each other — e.g. `foo.png` is a substring of `foo.png.ai.png`, so naive `includes` would attach both files when only the `.ai.png` sibling is the intended target, blowing past Kiro's 10 MB attachment cap and re-triggering the same `-32603` error. The attach is **fully encapsulated inside the Kiro adapter**: callers supply plain text prompts and a `workingDir` exactly as before, and other adapters (Claude Code, Codex) are unaffected. KB ingestion benefits because `convertImageToMarkdown` already sets `workingDir` to the image's parent directory and references the image by basename in the prompt template (see `pageConversion.ts`), so the matching is automatic. If `workingDir` is undefined or unreadable the helper returns no blocks — the original text-only `session/prompt` shape is preserved.

**Bedrock-safe re-encode on Kiro attach (`pngHasAlpha` / `reencodeForKiro` / `KIRO_MAX_LONG_EDGE_PX = 1568`):** Before base64-encoding any image attachment, `collectImageContentBlocks` calls `reencodeForKiro(buf)` which loads the image through `@napi-rs/canvas` and re-encodes to JPEG@92 over a white background when **either** of two Bedrock-parity gates triggers: (1) **alpha channel present** — `pngHasAlpha` inspects the IHDR color-type byte at offset 25 and returns true for color types 4 (grayscale + alpha) or 6 (RGBA), since Bedrock's Opus 4.7 deployment rejects RGBA inputs upstream; (2) **long edge exceeds `KIRO_MAX_LONG_EDGE_PX = 1568`** — KB ingestion's `MAX_LONG_EDGE_PX = 2576` is sized for Opus 4.7 on the Anthropic API, but Bedrock's Opus 4.7 has a documented parity gap and rejects images at that resolution with `ValidationException` (wrapped as JSON-RPC `-32603 Internal error`); 1568 px is the prior-Opus limit which Bedrock has had longer to support. When triggered, the helper proportionally downscales while preserving aspect ratio, flattens alpha onto white, and emits JPEG (which has no alpha channel) — the resulting content block's `mimeType` flips to `image/jpeg`. When neither gate fires (in-range RGB image, or any non-image format), `reencodeForKiro` returns `null` and the caller sends the original bytes unchanged. A cheap pre-flight `buf.length < 64` short-circuit skips the canvas decoder for synthetic test buffers and obviously-malformed files. If `napiCanvas.loadImage` throws (corrupt or unsupported source), the helper also returns `null` and the original bytes are sent as-is. This is a Kiro-only concession: KB ingestion's `MAX_LONG_EDGE_PX = 2576` is preserved unchanged for the Anthropic-API path used by Claude Code.

### CodexAdapter (`src/services/backends/codex.ts`)

**Metadata:** `id: 'codex'`, capabilities: `thinking: true, planMode: false, agents: true, toolActivity: true, userQuestions: true, stdinInput: true, goals: { set: true, clear: true, pause: true, resume: true, status: 'native' }`. `resumeCapabilities` reports active-turn resume as unsupported and later-session resume as supported via `external_session` + `thread/resume`. The `agents` capability covers Codex's `collabAgentToolCall` ThreadItem (an opaque single-card MVP — see "Subagents" below); `userQuestions` covers the EXPERIMENTAL `item/tool/requestUserInput` request (see "User questions" below); `goals` covers the Codex thread-goal RPCs surfaced through the desktop composer (see "Thread goals" below and [ADR-0032](adr/0032-use-codex-thread-goals-for-goal-mode.md)). Models are discovered dynamically via `model/list` against a `codex app-server`; until that fetch lands the picker shows a hardcoded `FALLBACK_MODELS` set covering `gpt-5.5` (default), `gpt-5.5-codex`, and `gpt-5.5-mini`. Fallback Codex models declare `supportedEffortLevels: ['low', 'medium', 'high', 'xhigh']` so chat, Memory CLI, and KB CLI settings can show an effort selector before discovery completes. The fallback also stays in effect when the CLI is missing or `model/list` fails. The constructor kicks off `_refreshModels()` once at construction using the server-configured Codex runtime; under Jest (`process.env.JEST_WORKER_ID` is set) the refresh is skipped so unit tests don't spawn real processes. `getMetadata({ cliProfile })` refreshes and returns a profile-specific model catalog, cached by the resolved Codex runtime key; server-configured models continue to use the existing `metadata.models` cache. Streaming and one-shot effort filtering use the cached catalog for the selected profile when available, otherwise the fallback list. Dynamic model discovery maps Codex's `supportedReasoningEfforts[].reasoningEffort` (or debug-style `supported_reasoning_levels[].effort`) into the shared `supportedEffortLevels` field, including Codex protocol-only `none` and `minimal` when a model advertises them.

**Integration protocol:** Codex App Server (JSON-RPC 2.0) over stdin/stdout via `codex app-server --enable goals`. The protocol is documented inline in the OpenAI codex repo and ships TypeScript types via `codex app-server generate-ts --out <DIR>`. The adapter hand-types only the request/response/notification shapes it actually uses to avoid vendoring 400+ generated files; drift is checked manually by re-running the generator and diffing. App-server goal RPCs require `initialize({ capabilities: { experimentalApi: true } })`, so every app-server process uses that capability even when the current turn is a normal message.

**Auth model:** A server-configured Codex profile leaves `CODEX_HOME` unset, so Codex authenticates via the server user's default `~/.codex/auth.json` — written by `codex login` (ChatGPT OAuth) or `codex login --api-key` (API-key billing). A Codex profile with `configDir` maps that directory to `CODEX_HOME` for every spawned Codex process, isolating auth, config, sessions, rollout storage, plugins, skills, and MCP collision reads under that directory. `cliProfile.env` is merged into the spawned process environment, then `configDir` wins by setting `CODEX_HOME` after the merge; `cliProfile.command` overrides the executable (`codex` by default). The adapter never imports `@openai/codex-sdk` because that path requires API-key billing and would break ChatGPT subscription users. Both `codex app-server` (streaming and plan usage) and `codex exec` (one-shot) read the same profile-selected `auth.json`, so subscription and API-key users work identically.

**App-server process lifecycle:** Lazy spawn + idle timeout + transparent recovery, mirroring Kiro.
- First message or first goal → spawn `codex app-server --enable goals` → `initialize({clientInfo, capabilities: { experimentalApi: true }})` → `thread/start({cwd, approvalPolicy, sandbox, experimentalRawEvents: false, persistExtendedHistory: false, model?, developerInstructions?})` → **yield `external_session` stream event** → `turn/start({threadId, input, model?, effort?})` for messages or `thread/goal/set({threadId, objective, status: 'active'})` followed by a goal-owned `turn/start({threadId, input: [{ type:'text', text:'Work on this active goal...' }], model?, effort?})` for goals.
- Subsequent messages → reuse process → drain queued stale notifications, inspect `thread/read({ includeTurns: true })` for an orphaned `inProgress` turn, interrupt that orphan with `turn/interrupt` when one is found, then `turn/start`.
- Idle timeout (configurable via `CODEX_IDLE_TIMEOUT_MS`, default 10 min) → SIGTERM → cleanup. The timer resets on every JSON-RPC notification received during a turn so long-running multi-agent runs don't get killed mid-flight; "idle" means literally no activity from the app-server, not "time since last user message".
- Next message or goal action after timeout → respawn → `initialize` → `thread/resume({threadId, cwd, approvalPolicy, sandbox, excludeTurns: true, persistExtendedHistory: false, model?})` → drain replayed notifications, rejecting any queued server-to-client requests instead of approving them → `turn/start` or goal RPC. If `thread/resume` fails (e.g. persisted threadId no longer exists on disk), the adapter falls back to a fresh `thread/start` so the conversation isn't dead-ended.
- Process reuse is keyed by conversation id, MCP-server hash, and Codex profile runtime key. Changing the selected profile before an empty session starts forces a new `app-server` process; mid-session profile switching is blocked at the route layer.

**Execution policy:** `approvalPolicy` and `sandbox` default to `never` and `danger-full-access` so Codex runs as a trusted local agent inside Agent Cockpit. `server.ts` passes `config.CODEX_APPROVAL_POLICY` and `config.CODEX_SANDBOX_MODE` into `CodexAdapter`; `src/config/index.ts` reads them from the matching environment variables and validates them against Codex's CLI values (`untrusted`, `on-failure`, `on-request`, `never` for approval policy; `read-only`, `workspace-write`, `danger-full-access` for sandbox). The default starts interactive Codex threads with full elevated permissions and makes `codex exec` one-shots use `--dangerously-bypass-approvals-and-sandbox`. Restricted deployments can set stricter env values such as `CODEX_APPROVAL_POLICY=on-request` and `CODEX_SANDBOX_MODE=workspace-write`; non-default restricted one-shots pass explicit `--ask-for-approval` / `--sandbox` flags. See [ADR-0062](adr/0062-default-codex-to-full-local-access.md) for the deployment-level policy decision.

**Turn input:** Codex's `turn/start` expects an array of typed input items: `[{ type: 'text', text: <message>, text_elements: [] }]`. The cockpit currently sends a single text element; image attachments and other element types are not surfaced through the cockpit UI. When the selected Codex model declares the requested `effort` in `supportedEffortLevels`, the adapter passes it as `turn/start.effort`. Unsupported or missing efforts are omitted so existing conversations without a stored effort continue to use Codex's model default.

**Turn ownership:** Codex app-server notifications and server-to-client requests are turn-owned. For normal chat and goal streams, the `turn/start` response is the authoritative source of the active parent `turnId`; `turn/started` and `thread/goal/updated.turnId` are accepted only as duplicate runtime notifications for that same id. The adapter extracts turn ownership from `params.turnId` or `params.turn.id` and drops any text, thinking, tool activity, tool outcomes, artifacts, token usage, errors, request-user-input, or completion event whose `threadId` and `turnId` do not match the active parent turn. Stale command/file approvals are answered with `{ decision: 'cancel' }`, stale permissions and user-input requests receive JSON-RPC errors, and legacy v1 approvals are aborted when drained before a new run. Child-thread tool events are allowed only after a current parent-turn `spawnAgent` item records the child `threadId`; stale child events cannot populate that map because stale parent items are dropped before `recordSpawnAgentReceivers()` runs. This rule is captured in [ADR-0055](adr/0055-own-codex-app-server-events-by-turn.md).

**Turn termination:** Codex emits a `turn/completed` notification when the turn finishes. The notification includes per-turn token usage (`thread/tokenUsage/updated` arrives separately). On a matching parent-thread and active-turn `turn/completed`, the stream loop yields a `done` event and stops the notification iterator. Stale completions and child-thread completions are ignored. Goal streams also treat a matching terminal `thread/goal/updated` (`status: 'complete'` or `'budgetLimited'`) as terminal, because Codex can finish a native goal and omit the matching `turn/completed`; this prevents an achieved/budget-limited goal from leaving the cockpit stream and composer stuck in running state. While a goal stream is open, the adapter also polls `thread/goal/get` once per second. If polling discovers terminal native goal state before any matching notification arrives, the stream follows the same terminal path and, when no assistant text has appeared, starts the follow-up report turn described below. `turn/interrupt({threadId, turnId})` is the abort path.

**Thread goals:** Codex goals are enabled by app-server's `goals` feature flag and require the experimental API capability at initialize time. The adapter exposes `getGoal`, `setGoalObjective`, `resumeGoal`, `pauseGoal`, and `clearGoal` on top of `thread/goal/get`, `thread/goal/set`, and `thread/goal/clear`.

Codex goal state is normalized to the backend-neutral `ThreadGoal` shape with `backend: 'codex'`, `source: 'native'`, and `supportedActions: { clear: true, pause: true, resume: true, stopTurn: true }`. Codex-native fields are preserved:

```ts
{
  backend: 'codex';
  threadId: string;
  objective: string;
  status: 'active' | 'paused' | 'budgetLimited' | 'complete';
  supportedActions: { clear: true; pause: true; resume: true; stopTurn: true };
  source: 'native';
  tokenBudget: number | null;
  tokensUsed: number;
  timeUsedSeconds: number;
  createdAt: number;
  updatedAt: number;
}
```

`setGoalObjective(objective, options)` resolves or creates the Codex thread using the same `_resolveThread` path as normal streaming, then calls `thread/goal/set({ threadId, objective, status: 'active' })`. `resumeGoal(options)` calls `thread/goal/set({ threadId, status: 'active' })` without changing the objective. After the native goal state is set, both methods explicitly start a goal-owned Codex turn. New goals send a text input shaped as "Work on this active goal and report the result in this chat" plus the objective; resumes without an objective send "Continue working on the active goal and report the result in this chat." That explicit turn is what produces normal `item/agentMessage/delta` output, so `processStream` can persist the goal result as an assistant message in the transcript instead of only showing lifecycle cards. The selected model and supported effort are passed through exactly like normal chat turns. If Codex sends a terminal goal update before any assistant text is emitted, the adapter treats that terminal update as closing the goal turn, clears the active turn id, and starts one follow-up report turn in the same thread with "The goal has completed, but no final report was emitted..." so the transcript still receives a final assistant report when Codex separates goal completion from chat output.

Both methods return `{ stream, abort, sendInput }` and the route attaches that stream through the normal `StreamJobSupervisor` + `processStream` path. The stream yields `external_session` and `backend_runtime` when applicable, forwards current-thread `thread/goal/updated` as `goal_updated`, forwards current-thread `thread/goal/cleared` as `goal_cleared`, and uses the goal-owned `turn/start` response to establish the active goal turn before forwarding turn-scoped text, usage, tool, error, or completion events. Current Codex notifications may carry the turn id as `params.turn.id`, so the adapter accepts both `params.turnId` and `params.turn?.id`.

`pauseGoal(options)` calls `thread/goal/set({ threadId, status: 'paused' })` and returns the updated goal. `clearGoal(options)` calls `thread/goal/clear({ threadId })`. Pause and clear are state mutations, not abort operations: they reuse an already-running app-server process when one exists and do not call `turn/interrupt` or detach the active stream. If no stream is active, the chat route sends the corresponding goal WebSocket frame itself so connected clients still update. Route-level set/pause/resume/clear actions and terminal stream goal statuses are also persisted as system `goalEvent` messages, so Codex goal objectives/results remain visible even when the native goal run produces no assistant text.

The adapter preserves Codex's `budgetLimited` status and budget-related fields when Codex returns them, but Agent Cockpit v1 does not send token budget parameters and the desktop composer does not expose token budget controls.

**Token usage (`thread/tokenUsage/updated`):** The notification carries `{ total, last, modelContextWindow }`, where `last` is the just-finished API call and `total` is the running cumulative across the thread. Both `total.inputTokens` and `last.inputTokens` include the cached portion of the prompt (Codex re-sends the full conversation history every turn, mostly as cache reads). The cockpit's `deriveCodexUsage()` helper (exported from `codex.ts`) maps the notification to a cockpit `Usage` event using *only* the `last` counters: `inputTokens = max(0, last.inputTokens − last.cachedInputTokens)` (fresh, uncached input — same semantics as Anthropic's `input_tokens`, so it accumulates meaningfully via the chatService `+=` aggregator), `outputTokens = last.outputTokens`, `cacheReadTokens = last.cachedInputTokens`, and `contextUsagePercentage = round(last.totalTokens / modelContextWindow × 100)`. Using `last.totalTokens` (not `total.totalTokens`) is essential — context % is a snapshot of *this turn's* window utilization, capped at 100%; using cumulative caused 1000%+ readings on long sessions.

**Duplicate usage filter:** After turn ownership passes, Codex can still re-emit a `thread/tokenUsage/updated` notification at a later boundary that mirrors the prior turn's final state byte-for-byte (same `last`, same `total`). Without filtering, the chatService `+=` aggregator double-counts one turn's worth of tokens at every transition — sessions with N turns end up showing roughly (N−1) extra "fresh inputs" stacked on top of the real total. The stream handler tracks `entry.lastTotalTokens` on `CodexProcessEntry` (per-conversation, persists across user-message turns within the same `app-server` process) and drops the notification when `tokenUsage.total.totalTokens` matches the previous emitted value. Genuine intra-turn updates always advance the cumulative total, so the filter never suppresses a real event.

**Mid-turn input (`sendInput`):** The `sendInput(text)` callback returned from `sendMessage()` routes input one of two ways depending on per-turn state:
- **If a `tool/requestUserInput` is pending** (see "User questions" below), the adapter responds to that JSON-RPC request with `{answers: {<firstQuestionId>: {answers: [text]}}}` and clears the pending state. The cockpit only surfaces the first question, so only that question's id receives the user's text — the server tolerates partial answers (its `HashMap` is unconstrained on the wire).
- **Otherwise**, the adapter issues a fire-and-forget `turn/steer({threadId, input: [{ type: 'text', text }], expectedTurnId})` request, appending the text to the currently in-flight turn (the model receives it as part of the same turn rather than starting a new one). `expectedTurnId` is required by the protocol; the adapter passes the tracked `turnId` so the server rejects stale UI input rather than silently applying it to a turn the user didn't intend.

Guards on both paths: no-op when (a) the turn has been aborted, (b) `text` is empty/non-string, or (c) the client/threadId state hasn't been populated yet. `turn/steer` rejections are logged via `console.warn` and otherwise swallowed so a stale steer never crashes the stream.

**Approvals:** Codex uses server-to-client JSON-RPC requests (not notifications) for approvals — `item/commandExecution/requestApproval`, `item/fileChange/requestApproval`, and `item/permissions/requestApproval`. The adapter auto-approves these only when the request belongs to the active parent turn or a child thread already mapped from that parent turn:
- `commandExecution` / `fileChange` → `{ decision: 'acceptForSession' }`
- `permissions` → `{ scope: 'session' }`

Stale command/file approvals are canceled, stale permissions requests are rejected with a JSON-RPC error, and drained legacy approvals are aborted. A stale request is never approved under a later user message.

Other server-side requests we do **not** answer (we reply with rejection so the turn doesn't hang on UI we don't expose):
- `mcpServer/elicitation/request` — MCP server elicitations
- `item/tool/call` — out-of-protocol dynamic client-side tool calls

**User questions (`item/tool/requestUserInput`):** EXPERIMENTAL, API v2 only. The server pauses the turn and sends a JSON-RPC request with `{threadId, turnId, itemId, questions: [{id, header, question, isOther?, isSecret?, options?}]}`. The adapter accepts the request only when its thread/turn ownership matches the active stream, records `{reqId, itemId, turnId, questions}` in per-turn `pendingUserInput` state, and yields a `tool_activity` event with `tool: 'AskUserQuestion'`, `isQuestion: true`, and `questions` reshaped for the cockpit's `pendingInteraction` UI (`{question, options}` per entry, where each option is `{label, description}`). When the user answers, `sendInput()` (see above) responds to the request with `{answers: {<firstQuestionId>: {answers: [text]}}}` and clears the pending state. If the server cancels the pending request (e.g. on turn interrupt) it emits `serverRequest/resolved` with the same `requestId`, and the dispatch clears `pendingUserInput` only when that pending request belongs to the active turn so a stale `sendInput()` doesn't try to respond to a dead request. Multi-question requests aren't surfaced — the cockpit's UI is single-question, matching Claude Code's behavior; the unanswered questions are omitted from the response and the server tolerates the partial answer.

**Tool name normalization:** Codex thread items are typed (`type: 'commandExecution' | 'fileChange' | 'mcpToolCall' | 'dynamicToolCall' | 'webSearch' | 'imageView' | 'imageGeneration' | 'collabAgentToolCall' | 'agentMessage' | 'reasoning' | 'plan' | …`), not named. The adapter dispatches on `item.type` via `extractCodexToolDetails()` and `ITEM_TYPE_TO_TOOL`. For `mcpToolCall` and `dynamicToolCall` the actual tool name on the item is preserved verbatim.

**Image generation artifacts:** Codex `imageGeneration` items still emit an `ImageGen` tool card, and on `item/completed` the adapter also tries to produce a vendor-neutral `artifact` event. It first searches the completed item for a base64 image payload (`result`, `image`, `data`, `base64`, `b64_json`, `content`, or nested equivalents, including image data URLs). If the protocol item does not carry bytes, it searches for an existing image path in the item and then falls back to the selected profile's `CODEX_HOME/generated_images/**/{item.id}.png` layout. The artifact event carries `dataBase64` or `sourcePath`, `filename`, `mimeType`, `title: 'Generated image'`, and `sourceToolId: item.id`; `processStream` copies/decodes it into the conversation artifacts dir so a first-turn generated image displays even when Codex's final assistant text is empty.

**Subagents (`collabAgentToolCall`):** Codex emits `collabAgentToolCall` ThreadItems on the parent connection whenever the parent thread invokes a child via the collab protocol. The item shape is `{ id, tool, status, senderThreadId, receiverThreadIds, prompt?, model?, agentsStates }`, where `tool` is one of `spawnAgent | sendInput | resumeAgent | wait | closeAgent`, item-level `status` is `inProgress | completed | failed`, and per-receiver lifecycle (`pendingInit | running | interrupted | completed | errored | shutdown | notFound`) lives in `agentsStates`, keyed by child thread id. Each call is rendered as a top-level cockpit Agent card (`tool: 'Agent'`, `isAgent: true`, `subagentType: <op>`, `description` derived from the operation kind plus prompt preview). Outcome derivation: a receiver `errored` / `notFound` state surfaces as a card error even when the call's item-level `status` is `completed`, so subagent failures don't silently render green.

**Child-thread demultiplexing:** Every `item/*` and `item/*/delta` notification from `codex app-server` carries `threadId` and `turnId` at the params envelope (alongside the `item` object) — this is not stated in the README and was verified by capturing raw JSON-RPC traffic. The adapter maintains a per-stream `Map<childThreadId, topLevelCallId>` (`subagentByThreadId`) that's populated whenever a current parent-turn `collabAgentToolCall(spawnAgent)` reaches `item/completed` with non-empty `receiverThreadIds[]`: each child threadId is recorded against the spawning call's id. Grand-children (a child thread that itself spawns) flatten to the same top-level call id since the cockpit UI nests one level deep — when the sender threadId is already in the map, the new receivers inherit its top-level call id rather than the nested call's id. Subsequent `item/started` notifications from a mapped child thread are tagged with `parentAgentId: <topLevelCallId>` on the emitted `tool_activity` event so the existing UI nesting (used by Claude Code and Kiro via `parentAgentId`) attributes the activity to the right Agent card. Child-thread `agentMessage` and `reasoning` deltas are dropped because the cockpit UI has no place to render per-child message/thinking streams under an Agent card; the child's final summary surfaces via `agentsStates[childTid].message` on the closing wait/closeAgent collabAgentToolCall and is rendered as the Agent card's outcome. The demux and ownership helpers (`extractCodexThreadId`, `extractCodexTurnId`, `eventBelongsToActiveParentTurn`, `recordSpawnAgentReceivers`, `lookupParentAgentId`, `eventIsFromChildThread`, `isParentTurnCompleted`) are exported from `codex.ts` for unit testing.

**Per-thread turn lifecycle:** `turn/completed` (and `turn/failed`) is emitted **once per thread** during a multi-agent run — each child has its own turn lifecycle, and the parent's `turn/completed` arrives last (after the parent finishes synthesizing its final summary from the children's results, sometimes ~10s after the first child's `turn/completed`). The dispatch filters via `isParentTurnCompleted(params, state.threadId, activeTurnId)`: it only marks the cockpit turn as ended when `params.threadId === state.threadId` (the parent thread) and `params.turnId` / `params.turn.id` matches the active parent turn. If `activeTurnId` is omitted by a legacy caller, the helper preserves its old thread-only behavior for unit tests and single-thread compatibility. Without this filter, the cockpit would close the notification stream on a stale or child `turn/completed`, and a follow-up turn could surface buffered text instead of being sent to the model.

**Thinking:** `item/reasoning/textDelta` and `item/reasoning/summaryTextDelta` are emitted as `ThinkingEvent` so they render in the dedicated thinking UI rather than the main message body.

**Session mapping:** The Codex `thread.id` is yielded as an `external_session` `StreamEvent` immediately after `thread/start`. `processStream` persists it on the active `SessionEntry.externalSessionId` so `thread/resume` can reattach across cockpit restarts. The Codex `turn/start` response carries the app-server turn id; the adapter records it, emits `backend_runtime.activeTurnId` from that response path, and dedupes `turn/started` only when that notification carries the same thread and turn. The `excludeTurns: true` flag on `thread/resume` suppresses the history replay (Codex would otherwise re-emit every prior assistant turn as notifications, which would be double-counted in the cockpit transcript).

**MCP injection:** Codex configures MCP servers via `[mcp_servers.<name>]` sections in its config.toml. The adapter injects cockpit-managed servers per-spawn via repeated `-c` config overrides on `codex app-server` (and `codex exec` for one-shots). For each server the adapter emits three (or two if no env) `-c` pairs:

```
-c mcp_servers.<name>.command="<cmd>"
-c mcp_servers.<name>.args=["<arg1>", "<arg2>", …]
-c mcp_servers.<name>.env={ KEY = "value", … }
```

Values are TOML-escaped via small inline helpers (`tomlEscapeString` / `tomlBareKey`). Codex's `-c` parser overrides the matching keys on top of whatever config file the selected runtime loads. For server-configured profiles this is the normal `~/.codex/config.toml`; for Codex profiles with `configDir` it is `<configDir>/config.toml` via `CODEX_HOME`. The selected config directory is otherwise untouched — auth, sessions, plugins, skills, and any user-defined config keys all load normally — which means cockpit-managed threads share rollout storage with the matching standalone `codex` CLI usage and `thread/resume` works across process respawns.

Before injecting, the adapter reads the selected runtime's `config.toml` (`~/.codex/config.toml` or `<configDir>/config.toml`) and skips any server whose name collides with a `[mcp_servers.<name>]` already present there (warning logged) so user-defined MCP servers always win. To handle MCP set or profile changes within a conversation, the adapter hashes the `mcpServers` list, tracks the profile runtime key, and respawns the app-server when either differs from the previous spawn for that conversation.

**Codex Fast service tier:** Fast mode is represented as a service-tier override rather than a model id. When `serviceTier === 'fast'`, the adapter adds `-c service_tier="fast"` and `-c features.fast_mode=true` to `codex app-server --enable goals` spawns and to `codex exec` one-shots. The app-server process cache key includes the service-tier key, so changing a conversation between Default and Fast respawns the Codex app-server for that conversation. Absence does not force a standard tier; Codex falls back to the selected profile/config.

**One-shot LLM calls (`runOneShot`, `generateTitle`, `generateSummary`):** All three funnel through `_execOneShot(prompt, options)`, which resolves `options.cliProfile`, builds the platform invocation, and runs Codex `exec --dangerously-bypass-approvals-and-sandbox --skip-git-repo-check -C <cwd> [-c service_tier="fast"] [-c features.fast_mode=true] [-c mcp_servers.…] [-c model_reasoning_effort="<effort>"] [-m <model>] "<prompt>"` by default via `child_process.execFile` with the selected profile environment. On macOS/Linux the invocation is the direct Codex command; on Windows an installer-managed Codex package runs as `node.exe <codex.js> exec ...` and `.cmd` shims are wrapped through `cmd.exe`. When the server is configured with a restricted non-default policy, the one-shot uses `--ask-for-approval <policy> --sandbox <mode>` instead of the full-access flag. When `cliProfile.configDir` is set, `CODEX_HOME=<configDir>` is present in the child env; otherwise the child inherits the server-configured Codex home. The `service_tier` / `features.fast_mode` overrides are emitted only when `serviceTier: 'fast'` is supplied. The `model_reasoning_effort` override is emitted only when `{ model, effort }` is supplied and the selected model declares that effort in `supportedEffortLevels`; otherwise the one-shot uses Codex's model default. `codex exec` is a dedicated non-interactive subcommand that prints the model's final answer on stdout (no streaming-protocol parsing required). MCP injection reuses the same `-c`-flag path as the app-server and reads collisions from the selected config dir. The adapter enforces `timeoutMs` with its own timer, sends `SIGTERM`, then sends `SIGKILL` after the standard process-kill grace period; this keeps a timed-out Codex child that exits cleanly on `SIGTERM` from being misreported as successful empty stdout. ENOENT is mapped to a friendly "Codex CLI is not installed" error; all other errors are surfaced as `codex exec failed: <stderr or exit code>`. `generateTitle` / `generateSummary` truncate the result (80 chars / 200 chars) and fall back to a substring-of-userMessage / `Session (N messages)` placeholder if Codex is missing or unauthenticated.

**Regenerating protocol types (drift check):** When upgrading the Codex CLI, run `codex app-server --enable goals generate-ts --out /tmp/codex-ts` and diff the relevant interfaces (`v2/ThreadStartParams.ts`, `TurnStartParams.ts`, `ItemStartedNotification.ts`, `ThreadItem.ts`, `ServerRequest.ts`, `ToolRequestUserInputParams.ts`, `ToolRequestUserInputResponse.ts`, `Model.ts`, `ModelListResponse.ts`, `ThreadGoal.ts`, `ThreadGoalStatus.ts`, `ThreadGoalUpdatedNotification.ts`, `ThreadGoalClearedNotification.ts`) against the hand-typed shapes in `codex.ts`. The hand-typed subset is intentionally narrow — only fields the adapter reads. For effort support, the important fields are `Model.supportedReasoningEfforts`, `Model.defaultReasoningEffort`, and `TurnStartParams.effort`. For goals, the generated notification shapes are authoritative; current goal client request params are still hand-typed because some generated client-request unions lag the shipped app-server RPC methods even though the binary accepts them. Note that `item/tool/requestUserInput` and the goal API are marked EXPERIMENTAL upstream and could change shape without notice.

### Adding a New Backend

1. Create `src/services/backends/myBackend.ts` extending `BaseBackendAdapter`
2. Implement `metadata`, `sendMessage()`, `generateSummary()`, and optionally `generateTitle()`, `shutdown()`, `onSessionReset()`
3. Import shared helpers from `toolUtils.ts` (never import from another adapter)
4. Register in `server.ts` — no other changes needed
5. Use the generic `externalSessionId` field on `SessionEntry`/`SendMessageOptions` if the backend manages its own session IDs

## 4.3 UpdateService

**File:** `src/services/updateService.ts`

`UpdateService` routes update checks and update execution by the install channel
reported by `InstallStateService`. Dev installs (`channel: "dev",
source: "git-main"`) compare local `package.json` against
`origin/main:package.json`, pull `main`, install dependencies, rebuild generated
assets, verify PM2 configuration, and restart. Production installs
(`channel: "production", source: "github-release"`) check the external GitHub
Release manifest, download and verify the platform app archive, stage the new
release under the install root, run dependency/build preflight work there,
activate the release, then restart through PM2 with a health-check rollback
script. macOS activates by switching the `current` symlink. Windows activates by
recording the active versioned release directory in `install.json`.
[ADR-0054](adr/0054-adopt-mac-installer-and-release-channels.md) records the
channel decision and [ADR-0063](adr/0063-adopt-per-user-windows-installer.md)
records the Windows install/update decision. Production updates also enforce
release-declared Node runtime requirements from `release-manifest.json`; private
installer-managed runtimes are refreshed before dependency installation when a
release raises the required Node major.

- `start()` — runs `_checkRemoteVersion()` immediately, then polls every 15 minutes (unref'd interval)
- `stop()` — clears polling interval
- `getStatus()` — returns cached `{ localVersion, remoteVersion, updateAvailable, lastCheckAt, lastError, updateInProgress, installChannel, installSource, installStateSource }`
- `checkNow()` — immediate channel-aware version check, returns status
- `triggerUpdate({ hasActiveStreams })` — full update sequence with shared guards:
  1. Concurrent guard (`_updateInProgress` flag)
  2. Active/pending turn guard (refuses if CLI streams are active or a message send is accepted but still preparing)

Dev-channel sequence:

1. Dirty tree guard (`git status --porcelain`, ignoring `data/`, `.env`, `ecosystem.config.js`, `.DS_Store`, `.claude/`, `coverage/`, `plans/`)
2. `git checkout main` (30s timeout)
3. `git pull origin main` (60s timeout)
4. `npm install` (120s timeout)
5. `npm --prefix mobile/AgentCockpitPWA install` (120s timeout) so the mobile app's separate lockfile is installed before its production build.
6. Force V2 web build via `WebBuildService.ensureBuilt({ force:true })`, reported as an `npm run web:build` update step. A build failure returns a failed update result and does not restart, even when a previous V2 build exists.
7. Force mobile PWA build via `MobileBuildService.ensureBuilt({ force:true })`, reported as an `npm run mobile:build` update step. A build failure returns a failed update result and does not restart, even when a previous mobile build exists.
8. Verify interpreter — reads `ecosystem.config.js` fresh from disk (via `fs.readFileSync`, not `require`, to avoid stale cache), checks the configured interpreter exists. Path-based interpreters (starting with `.` or `/`) are checked on disk; bare commands (e.g. `npx`, `node`) are resolved via `which` on POSIX or `where` on Windows.
9. Delegates to `_launchRestartScript()` (shared with `restart()` below)

Production-channel sequence:

1. Require stored `installDir` and `appDir`; on macOS resolve the current symlink
   target as the rollback target, and on Windows use the stored versioned
   `appDir` as the rollback target.
2. Download `release-manifest.json` and `SHA256SUMS` from
   `https://github.com/<repo>/releases/latest/download/`.
3. Verify the manifest checksum against `SHA256SUMS`.
4. Parse the manifest, find the platform archive artifact (`role:
   "app-tarball"` on macOS, `role: "app-zip"` or `platform: "win32",
   format: "zip"` on Windows), download it, verify its `SHA256SUMS` entry, and
   verify its manifest `sha256`.
5. Refuse when the manifest version is not newer than the running package
   version.
6. Extract the archive into a staging directory under `<installDir>/releases`
   (`tar -xzf` on macOS, PowerShell `Expand-Archive` on Windows), require
   `server.ts`, then atomically rename it to the manifest `packageRoot`.
7. Resolve the release-required Node major from
   `manifest.requiredRuntime.node.minimumMajor`, falling back to the extracted
   release's `package.json` `engines.node`, then to Node 22. If the current
   runtime already satisfies the major, record a `verify Node.js runtime` step;
   installer-owned Windows runtimes must also contain `node.exe`, `npm.cmd`,
   `npx.cmd`, and npm's `npm-cli.js`, `npx-cli.js`, and `npm-prefix.js`
   entrypoints, and must be able to report both Node and npm versions through
   `node.exe`. Incomplete Windows runtimes are treated as failed verification
   and repaired before dependency installation. Otherwise download the latest
   official Node archive for that major from Node.org, verify it against
   `SHASUMS256.txt`, install it under the install root, prepend it to the
   updater process `PATH`, persist that PATH into the copied `.env` and
   `ecosystem.config.js`, record the bundled npm version when it can be
   observed, and report an `install Node.js runtime` step. macOS uses the Darwin
   tarball, extracts to `<installDir>/runtime/node-v<version>`, and repoints
   `<installDir>/runtime/node`. Windows uses the Windows ZIP and keeps the
   versioned runtime directory directly, without a symlink or junction. This
   migrates system-Node production installs to an installer-owned private
   runtime without mutating global Node.
8. Run root `npm ci` and mobile `npm --prefix mobile/AgentCockpitPWA ci` inside
   the extracted release. When a Windows private Node runtime is active, npm is
   invoked as `node.exe node_modules\npm\bin\npm-cli.js ...` so update-time
   dependency installs and builds do not depend on global `PATH` or `.cmd` shim
   quoting. Other Windows npm commands still resolve as `npm.cmd`; because
   Node's `execFile` does not execute Windows `.cmd`/`.bat` shims directly, the
   updater invokes those shims through `cmd.exe /d /s /c`.
9. Run V2 and mobile build preflight services for the extracted release. Fresh
   markers skip builds; missing/stale markers or missing assets run the
   corresponding build with the same platform-specific npm command resolution.
   Any build error fails the update before the symlink changes.
10. Confirm `public/v2-built/index.html` and `public/mobile-built/index.html`
   exist.
11. Copy `.env` from the previous current release into the new release. macOS
    also copies `ecosystem.config.js`; Windows regenerates `ecosystem.config.js`
    so PM2 monitors `<installDir>\bin\run-agent-cockpit.vbs` through
    `wscript.exe //B //NoLogo` and the config points at the new versioned app
    directory. The Windows runner launches `node --import tsx server.ts` hidden
    through `%ComSpec% /d /s /c`, redirects stdout/stderr to
    `<installDir>\pm2\logs\agent-cockpit-runner-*.log`, and the config sets
    `windowsHide: true` so PM2-managed Node processes do not open a desktop
    console window.
12. Activate the release. macOS switches `<installDir>/current` to the new
    release; the path must either be absent or a symlink, and a non-symlink
    current path is refused. Windows records the new versioned release path as
    `appDir` in `install.json`.
13. Update `install.json` through `InstallStateService.writeState()` when the
    service is available, including the current `nodeRuntime` metadata and the
    release version.
14. Launch `_launchRestartScript({ appRoot, healthUrl, rollbackTarget })`. The
    health URL is derived from the new release `.env` `PORT` value and defaults
    to `3334`. Windows also passes the target `expectedVersion` and the previous
    install status so the generated PowerShell rollback script can require the
    live `/api/chat/version.version` response to match the target release before
    it accepts the update, then restore `install.json` before restarting the old
    app if the target restart fails or a stale old process keeps answering the
    port.

- `restart({ hasActiveStreams })` — plain server restart (no git pull / npm install / interpreter verification). Applies the same concurrent + active/pending turn guards as `triggerUpdate()`, then calls `_launchRestartScript()`. Used by the Server tab in Global Settings so users can re-trigger startup-time detection (e.g. pandoc) after installing external binaries. The guard delegates to the stream supervisor's in-flight check and still treats accepted/preparing sends as active even though they now have durable stream jobs; planned/manual restarts should be blocked before work is interrupted.
- `_launchRestartScript()` (private) — on POSIX writes
  `<dataRoot>/restart.sh` (sets PATH to `node_modules/.bin`, sleeps 2s,
  `pm2 delete` + `pm2 start` against `ecosystem.config.js`), then launches it
  via double-fork (`nohup ... &` in subshell) to survive PM2 treekill. On
  Windows it writes `<dataRoot>/restart.ps1`, sets install-local `PM2_HOME`,
  prepends the private runtime path when present, resolves the active release's
  app-local `node_modules\.bin\pm2.cmd`, restarts the app through a checked
  native-command wrapper that deletes the existing PM2 entry before
  `pm2 startOrRestart`, and throws on failed `pm2 startOrRestart` or `pm2 save`,
  and launches the script detached with PowerShell
  `-WindowStyle Hidden` and Node `windowsHide: true`.
  Output is logged to `<dataRoot>/update-restart.log`. Shared by
  `triggerUpdate()` and `restart()`. Production callers also pass a health URL
  and rollback target; the script probes the health URL for up to 20 seconds and
  rolls back (relinking `current` on macOS, restoring the previous install
  manifest and starting the old versioned app on Windows) before exiting with
  failure. Windows production updates additionally parse the health JSON and
  require `version` to match the target release, so an older stale server on the
  same port cannot satisfy the health check. Windows rollback switches the script
  PATH and PM2 CLI invocation back to the previous install runtime and previous
  release's app-local `pm2.cmd` when that runtime was recorded, then saves PM2
  state after restarting the old ecosystem config.

Both `triggerUpdate()` and `restart()` return `{ success, steps: [{ name, success, output }] }`. On failure, includes `error` field.

- `_checkRemoteVersion()` — production downloads and parses the latest release
  manifest version; dev runs `git fetch origin main` + `git show
  origin/main:package.json`.
- `_isNewer(remote, local)` — three-part numeric semver comparison

## 4.3.2 InstallDoctorService

**File:** `src/services/installDoctorService.ts`

`InstallDoctorService` is the shared diagnostic layer for installer follow-up,
the V2 welcome flow, and future Settings health panels. It is constructed in
`server.ts` with `{ appRoot, dataRoot, installStateService, updateService }`.
Tests can inject a command runner and optional-tool detectors so doctor coverage
does not execute real local CLIs.

`getStatus()` returns:

```ts
{
  generatedAt: string,
  overallStatus: 'ok' | 'warning' | 'error',
  install: InstallStatus,
  checks: Array<{
    id: string,
    label: string,
    status: 'ok' | 'warning' | 'error',
    required: boolean,
    summary: string,
    detail?: string,
    remediation?: string,
    installActions?: Array<{
      id: string,
      kind: 'command' | 'link',
      label: string,
      description?: string,
      command?: string[],
      href?: string,
    }>,
  }>,
}
```

Required checks currently cover Node.js 22+, npm, local PM2, writable
`<AGENT_COCKPIT_DATA_DIR>`, and the desktop V2 build shell at
`public/v2-built/index.html`. Optional/warning checks cover app-directory
writability for future updates, Windows logon startup when running on Windows,
the mobile PWA build shell, Claude Code CLI, Codex CLI, Kiro CLI, Pandoc,
LibreOffice, and install/update channel metadata. Windows PM2 probes run
`node.exe <activeAppDir>\node_modules\pm2\bin\pm2 --version` directly, preferring
the installer-recorded runtime and `install.appDir`, so the welcome screen does
not trigger npm package-resolution prompts; non-Windows PM2 probes continue to
run `npx --no-install pm2 --version`. npm probes resolve from
`install.nodeRuntime.binDir` when the installer recorded a private or system
runtime, so Windows private-Node installs do not depend on the inherited process
`PATH` for the welcome checks. When the recorded Windows runtime contains the npm
CLI JavaScript entrypoints, Install Doctor uses
`node.exe node_modules/npm/bin/npm-cli.js` for npm probes and npm-backed install
actions. That avoids `npm.cmd` shim quoting entirely for the installer-managed
path. Windows Claude/Codex welcome actions run npm with
`--prefix <installDir>\cli-tools` so the CLIs land in an Agent Cockpit-owned
per-user prefix instead of depending on npm's global Windows prefix. After a
successful Claude/Codex install action, the service prepends
`<installDir>\cli-tools` to the current server `PATH`, persists the same entry
to the current user's Windows `Path`, repairs Agent Cockpit-managed terminal
wrappers, and broadcasts an environment change before returning the refreshed
doctor result. User `Path` persistence is a warning step after npm succeeds:
if writing the current user's `Path` fails, the install action still repairs
wrappers, refreshes detection, and can return success with the failed PATH step
captured in `steps[]`. The repaired `codex.ps1`/`codex.cmd` wrappers call the
recorded private `node.exe` plus the installed `codex.js` entrypoint directly,
so a user does not need a separate global Node install for terminal `codex` to start.
That makes immediate authentication/profile checks work and makes
`claude`/`codex` available to new PowerShell or terminal sessions. Windows
Claude/Codex probes share the backend resolver:
`@anthropic-ai\claude-code\bin\claude.exe` is executed directly,
`@openai\codex\bin\codex.js` is executed through `node.exe`, and npm-created
`claude.cmd`/`codex.cmd` shims in installer/user prefixes remain fallbacks. If
the CLIs were installed outside Agent Cockpit, the resolver also probes bare
`claude.exe`/`claude.cmd` and `codex.exe`/`codex.cmd` commands from the user's
Windows PATH. The
fallback Windows command runner still launches
`.cmd` shims through `cmd.exe /d /s /c` and wraps the full command line in outer
quotes so absolute runtime paths under `Agent Cockpit` directories survive
`cmd.exe /s` quote stripping. The Windows logon startup check queries the
`AgentCockpit` scheduled task unless
`install.startup.kind` is `manual`, in which case the disabled startup state is
reported as intentional. Missing CLI warnings are optional and tell the user to
install only the backend they plan to use; Claude Code remediation includes
`npm i -g @anthropic-ai/claude-code`, Codex remediation includes
`npm i -g @openai/codex`, and Kiro remediation uses the official docs on
Windows while retaining the official shell installer action on POSIX. Pandoc and
LibreOffice warnings are platform-aware: macOS mentions Homebrew only as an
already-installed option and Windows points at the official Windows
installer/download pages. Pandoc/LibreOffice warnings tell the user to refresh
checks after installing because those detectors are refreshed on every doctor
request. A corrupt install manifest downgrades the
update-channel check to warning while still returning inferred install metadata.
`overallStatus` is `error` when any required check errors, `warning` when any
check warns/errors but required checks are usable, and `ok` otherwise.

Missing optional dependency checks can also include install actions. Command
actions are fixed server-side allowlist entries; the browser receives the
command array for transparency but can only execute it by POSTing the action id
to `/api/chat/install/actions/:id/run`. The service never accepts arbitrary
command text from the browser, refuses duplicate action runs, and refuses all
install commands while any conversation turn is active or pending. Supported
command actions are:

- `claude-cli:npm-install` -> `npm i -g @anthropic-ai/claude-code@latest`
  (Windows adds `--prefix <installDir>\cli-tools` and uses the installer
  private runtime's npm CLI when available)
- `codex-cli:npm-install` -> `npm i -g @openai/codex@latest`
  (Windows adds `--prefix <installDir>\cli-tools` and uses the installer
  private runtime's npm CLI when available)
- `kiro-cli:official-install` -> `sh -c "curl -fsSL https://cli.kiro.dev/install | bash"`, exposed only outside Windows
- `pandoc:brew-install` -> `brew install pandoc`, exposed only when Homebrew is detected
- `libreoffice:brew-install` -> `brew install --cask libreoffice`, exposed only when Homebrew is detected

Link actions open official documentation/download pages and are not runnable on
the server. Pandoc and LibreOffice always expose official download link actions
when missing so fresh Macs without Homebrew still have a first-class path.
Successful command actions return the captured step output and a refreshed
install-doctor payload. Pandoc/LibreOffice doctor checks force fresh binary
detection every time, so official installer link actions can be followed by the
Welcome screen's Refresh button without requiring a server restart.

## 4.3.3 WebBuildService

**File:** `src/services/webBuildService.ts`

`WebBuildService` owns the generated-asset build preflight used by `server.ts` and the self-update path. The default instance covers the main V2 web app: it compares source/config/package hashes against `public/v2-built/.agent-cockpit-build.json`, runs the root `npm run web:build` script when needed, and writes a fresh marker after successful builds. The default source root is `web/AgentCockpitWeb/`; the default output root is `public/v2-built/`. The default runner builds into a hidden sibling staging directory (`public/.v2-built-staging-*`) via Vite's `--outDir` override, verifies the staged `index.html`, then swaps the staging directory over `public/v2-built/` only after success. This keeps the last known-good build available if a rebuild fails.

Public surface:

- `getBuildDir()` — returns the static directory Express mounts at `/v2`.
- `getMarkerPath()` — returns the build marker path.
- `ensureBuilt({ force? } = {})` — coalesces concurrent calls on the same service instance, computes expected hashes, and returns `WebBuildStatus`.

Freshness inputs:

- Every file under `web/AgentCockpitWeb/`, excluding `node_modules`, `dist`, and `.vite`.
- Root `package.json`.
- Root `package-lock.json`.
- Current git SHA is recorded in the marker for diagnostics but is not used by `markerMatches`; source and package hashes determine freshness so equivalent rebuilds are stable across commits.

Modes and failure policy:

- `auto` (default outside tests): build when missing, stale, or `force:true`.
- `skip` (default in `NODE_ENV=test` through config): compute status but do not build for normal startup preflight. `force:true` overrides skip mode so self-update still refreshes generated assets before restart.
- Missing/stale build success writes `{ sourceHash, packageJsonHash, packageLockHash, gitSha, builtAt }` to `.agent-cockpit-build.json`.
- Build failure with no previous `index.html` throws so startup can fail loudly.
- Build failure with an existing `index.html` returns a non-fresh status with `error`, allowing startup to serve the previous build while surfacing the stale-build warning. `UpdateService` treats that same error status as a failed update and refuses to restart.

## 4.3.4 MobileBuildService

**File:** `src/services/mobileBuildService.ts`

`MobileBuildService` is a focused wrapper around `WebBuildService` for the generated `/mobile/` asset tree. It uses `mobile/AgentCockpitPWA/` as the source root, ignored `public/mobile-built/` as the build output, `public/mobile-built/.agent-cockpit-build.json` as the freshness marker, and a distinct staging prefix (`public/.mobile-built-staging-*`). The command it runs is:

```bash
npm --prefix mobile/AgentCockpitPWA run build -- --outDir <stagingDir>
```

It is constructed in `server.ts` with the same `WEB_BUILD_MODE` as the main V2 web build service. ## 4.3.5 CliUpdateService

**File:** `src/services/cliUpdateService.ts`

`CliUpdateService` detects and updates the local backend CLIs that Agent Cockpit can spawn through configured CLI profiles. It is deliberately separate from `UpdateService`: `UpdateService` updates Agent Cockpit itself, while this service updates vendor CLI binaries such as Codex, Claude Code, and Kiro.

**Lifecycle:**
- `server.ts` constructs `new CliUpdateService(__dirname)` next to `UpdateService`, passes it into `createChatRouter`, and starts it after `chatService.initialize()` with `cliUpdateService.start(() => chatService.getSettings())`.
- `start(loadSettings)` runs an immediate background check and then polls every 15 minutes on an unref'd interval.
- `stop()` clears the interval; `chatShutdown()` calls it with the rest of the router-owned services.

**Target resolution:**
- Reads `Settings.cliProfiles`; if a persisted legacy `defaultBackend` exists without a matching profile, the settings read migration supplies that profile. Fresh installs with no profiles produce no update targets.
- Skips disabled profiles and groups active profiles by resolved vendor, command, invocation prefix, and `PATH`, producing one update item for each real runtime target.
- Codex profiles resolve through `resolveCodexCliRuntime(profile)` so `command`, merged `env`, and `CODEX_HOME` are honored. On Windows, installer-managed Codex resolves to `node.exe <installDir>\cli-tools\node_modules\@openai\codex\bin\codex.js`.
- Claude Code profiles resolve through `resolveClaudeCliRuntime(profile)` so `command`, merged `env`, and `CLAUDE_CONFIG_DIR` are honored. On Windows, installer-managed Claude resolves to the package `claude.exe`.
- Kiro uses the server-configured `kiro-cli` runtime because Kiro profiles do not expose profile-local command/config/env overrides.

**Status checks:**
- Runs the vendor version command (`codex --version`, `claude --version`, `kiro-cli version`) and extracts the first semver-looking value.
- Resolves the command path using `which` (`where` on Windows), or by checking an explicit path command directly.
- For Codex and Claude Code, calls `npm root -g` and realpath-checks whether the resolved command lives under the expected global package directory (`@openai/codex` or `@anthropic-ai/claude-code`). Only then is the install method marked `npm-global`.
- For npm-global installs, calls `npm view <package> version` and compares three-part semver numerically.
- Kiro is marked `self-update` with an update command, but there is no latest-version source in the current service, so `updateAvailable` remains false unless a future detector adds one.
- Missing/unknown/native installs remain visible in Settings with `lastError`, but they are not actionable from the composer notification.
- Claude Code targets also attach `interactiveCompatibility` for `claude-code-interactive` by comparing the installed Claude CLI version with the adapter's tested version (`2.1.142` at this release). Newer installed versions warn that Interactive mode may still work but can have compatibility issues, that Standard mode remains fully supported, and link to Anthropic's Claude Agent SDK plan-usage support article. Older versions warn that expected transcript fields may be missing. If the latest available npm version is newer than the tested version, the row also carries `updateCaution` so the web UI can warn before updating the shared Claude binary. These warnings never create a second physical update target and do not block the standard Claude Code updater. The separate `scripts/check-claude-interactive-release.js` GitHub Actions helper reads the same tested-version constant and opens at most one support-validation issue per newer npm version so real-Claude compatibility can be checked before the constant is advanced.

**Update path:**
- `triggerUpdate(itemId, { loadSettings, hasActiveStreams, onUpdated })` refuses when another update is in progress for the same item or when `hasActiveStreams()` reports any active/pending conversation turn.
- Supported commands are:
  - Codex npm-global: `npm i -g @openai/codex@latest`
  - Claude Code npm-global: `npm i -g @anthropic-ai/claude-code@latest`
  - Kiro self-update: `<kiro command> update --non-interactive`
- Each command runs through `execFile` with the target profile env, a 120-second timeout, and capped output.
- When the updated Windows target is the Agent Cockpit-managed
  `<installDir>\cli-tools` prefix, the service repairs the terminal wrapper
  after npm completes so `codex.ps1`/`codex.cmd` keep calling the private
  `node.exe` rather than npm's default global-Node shim.
- On success the route-level caller invokes `backendRegistry.shutdownAll()` so long-lived backend child processes restart on next use with the updated executable.
- After the update command, the service re-probes the target and returns `{ success: true, steps, item }`. Failures return `{ success: false, steps, error, item? }` without mutating unrelated status items.

See [ADR-0027](adr/0027-manage-cli-updates-from-web-cockpit.md) for the web-only product decision and supported-install policy.

## 4.4 ClaudePlanUsageService

**File:** `src/services/claudePlanUsageService.ts`

Fetches and caches the Claude Code account-wide plan usage snapshot (5-hour session %, weekly %, per-model breakdown, extra credits) from Anthropic's undocumented `/api/oauth/usage` endpoint using the OAuth access token that Claude Code writes to the user's machine. Read-only — the service never refreshes the token or writes back to the credentials store. Surfaces in the V2 ContextChip tooltip for Claude Code and Claude Code Interactive conversations because both providers share the same Claude Code account/profile credentials. The default cache uses the server-configured Claude Code credentials. Claude Code profile caches use the selected profile's `configDir`/`CLAUDE_CONFIG_DIR` credential file and are stored separately.

**Constants:**
- `BASE_API_URL = 'https://api.anthropic.com'`
- `ANTHROPIC_BETA = 'oauth-2025-04-20'` (required `anthropic-beta` header)
- `CREDENTIALS_PATH = ~/.claude/.credentials.json` for the default server-configured cache; profile caches with `configDir` read `<configDir>/.credentials.json`.
- `KEYCHAIN_SERVICE = 'Claude Code-credentials'` (macOS only)
- `REFRESH_MIN_INTERVAL_MS = 10 * 60 * 1000` — minimum gap between fetch **attempts** (not successes) so transient failures back off cleanly instead of hammering. Token-expired skips do **not** count as attempts — they reset the cooldown so the next trigger can retry as soon as Claude Code refreshes the OAuth token.
- `STALE_AFTER_MS = 15 * 60 * 1000` — client-visible stale threshold (slightly above the refresh floor)
- `EXPIRY_BUFFER_MS = 5 * 60 * 1000` — token is treated as expired when `Date.now() + buffer >= expiresAt`

**Constructor:** `new ClaudePlanUsageService(appRoot: string, options?: { dataRoot?: string })` — stores the default cache at `<options.dataRoot || appRoot/data>/claude-plan-usage.json` and profile-specific caches under `<options.dataRoot || appRoot/data>/claude-plan-usage/<encodeURIComponent(profile.id)>.json`.

**Methods:**
- `init()` — loads the default persisted snapshot and any profile snapshots off disk on server startup. Silently ignores `ENOENT`; logs any other read/parse failure and keeps the in-memory snapshot as its initial empty shape.
- `getCached(profile?: CliProfile)` — returns `{ fetchedAt, planTier, subscriptionType, rateLimits, lastError, stale }`. Does not trigger a refresh. Without a profile, or with the plain `server-configured-claude-code` profile that has no custom command/config/env, reads the default cache. With an account/custom profile, reads that profile's isolated cache. `stale` is computed as `now - fetchedAt > STALE_AFTER_MS` (or `true` when `fetchedAt` is null). This is the exact shape returned by `GET /api/chat/plan-usage`.
- `maybeRefresh(reason: string, profile?: CliProfile)` — triggers a refresh for the default cache or selected profile cache if all of: no in-flight refresh for that cache key, and at least `REFRESH_MIN_INTERVAL_MS` have passed since the last attempt for that key. Returns the shared in-flight promise when a fetch is already running, or an immediately-resolved promise when the throttle is active. The `reason` string is logged alongside success/failure (`server-start`, `turn-done`).
- `_refresh(reason)` / `_refreshProfile(reason, profile)` (private) — reads credentials, short-circuits with `lastError: 'token-expired'` if the token is within the 5-minute buffer (and resets the relevant last-attempt timestamp to `0` so the throttle does not gate the next trigger — no API call was made, so this is not a real attempt), otherwise calls `GET https://api.anthropic.com/api/oauth/usage` with `Authorization: Bearer <token>` + `anthropic-beta: oauth-2025-04-20` and a 10-second abort signal. On success stores `{ fetchedAt: now, planTier: creds.rateLimitTier, subscriptionType: creds.subscriptionType, rateLimits: <body>, lastError: null }` into the selected cache. On non-2xx or network failure preserves the prior `fetchedAt`/`planTier`/`subscriptionType`/`rateLimits` values and only overwrites `lastError` with the failure message — the UI can still render the last-known snapshot.
- `_persist()` / `_persistSnapshot(file, snapshot)` (private) — writes the default snapshot to `<dataRoot>/claude-plan-usage.json` or profile snapshots to `<dataRoot>/claude-plan-usage/<encoded-profile-id>.json` via `atomicWriteFile`. Failures are logged and swallowed.

**Credential resolution (`readStoredCredentials`):** The OAuth tokens Claude Code writes have two storage backends, tried in order:
1. Profile with `configDir` — reads `<configDir>/.credentials.json` only. There is intentionally no Keychain fallback for account profiles because the default Keychain service is not profile-isolated.
2. Server-configured/default path — reads `~/.claude/.credentials.json`, preferred when present (works on Linux and any Mac where Claude Code fell back from the Keychain).
3. macOS Keychain (service name `Claude Code-credentials`) — fallback only for the default path, read via `/usr/bin/security find-generic-password -s "Claude Code-credentials" -w` with a 10-second timeout. Only attempted on `process.platform === 'darwin'`. No Keychain permission prompt: the `security` CLI reuses the same ACL that Claude Code itself installed, so the user sees no OS-level dialog.

Parsed shape (`parseCredsBlob`):
```ts
{ claudeAiOauth: {
    accessToken: string,
    expiresAt?: number,        // epoch ms; null ⇒ treat as non-expiring
    subscriptionType?: string,  // e.g. "max"
    rateLimitTier?: string,     // e.g. "default_claude_max_20x"
} }
```
Missing `accessToken` throws `credentials missing claudeAiOauth.accessToken`, surfaced as `lastError`.

**Integration points:**
- `server.ts` — instantiated with `__dirname` plus `config.AGENT_COCKPIT_DATA_DIR`, `init()` on startup then immediate `maybeRefresh('server-start')`. Passed into `createChatRouter` via the `claudePlanUsageService` dependency.
- `src/routes/chat/statusRoutes.ts`:
  - `GET /plan-usage` route returns `getCached()` verbatim. `GET /plan-usage?cliProfileId=<id>` resolves the profile, rejects missing/disabled/non-Claude profiles with `400`, and returns `getCached(profile)` without triggering a refresh.
- `src/routes/chat.ts` stream orchestration:
  - `onDone` stream callback calls `maybeRefresh('turn-done', runtime.profile)` when `backendId === 'claude-code'`. Plain server-configured profiles write the default cache so the current ContextChip UI continues to work without a query parameter; account/custom profiles write their profile cache. No other backend triggers a refresh.

**Error taxonomy (`lastError` values):**
- `'token-expired'` — access token at/within the 5-minute expiry buffer.
- `'credentials missing claudeAiOauth.accessToken'` — creds blob parsed but missing field.
- `'no credentials found at <path>'` — file not found on non-darwin and thus nothing to fall back to.
- `'keychain read failed: <msg>'` — `security` CLI returned non-zero.
- `'usage API <status>: <body>'` — upstream HTTP 4xx/5xx (body truncated to 200 chars).
- Any network error message verbatim (timeout, DNS, TLS).

## 4.5 KiroPlanUsageService

**File:** `src/services/kiroPlanUsageService.ts`

Fetches and caches the Kiro (Amazon Q Developer) account-wide plan usage snapshot (subscription tier, monthly credits used / cap, overage status + dollar charges, bonus credits, reset date) by reading the kiro-cli's SQLite auth store read-only and calling `AmazonCodeWhispererService.GetUsageLimits` directly. Read-only — the service never writes to the SQLite DB, never refreshes the token, and never mutates the active CodeWhisperer profile. Surfaces in the V2 ContextChip tooltip for Kiro conversations only.

**Why direct-call instead of an ACP piggyback:** an earlier attempt to fetch this via the ACP `initialize` / `authenticate` stream on a background worker conversation caused cross-conversation message leakage (the ACP client routed frames to whichever conversation happened to be streaming at the time). The direct REST path against the Amazon Q endpoint avoids the ACP transport entirely — the cadence gained from this approach is what makes the server-startup refresh viable without disturbing any live conversation.

**Constants:**
- `Q_API_URL = 'https://q.us-east-1.amazonaws.com/'`
- `Q_TARGET = 'AmazonCodeWhispererService.GetUsageLimits'` (`X-Amz-Target` header)
- `Q_ORIGIN = 'KIRO_CLI'` (query param + body field — identifies this as a kiro-cli caller)
- `REFRESH_MIN_INTERVAL_MS = 10 * 60 * 1000` — minimum gap between fetch **attempts** (not successes) so transient failures back off cleanly instead of hammering. Token-expired skips do **not** count as attempts — they reset the cooldown so the next trigger can retry as soon as kiro-cli rotates its IdC token.
- `STALE_AFTER_MS = 15 * 60 * 1000` — client-visible stale threshold (slightly above the refresh floor)
- `EXPIRY_BUFFER_MS = 30 * 1000` — AWS IdC tokens rotate every ~8 min, so the buffer here is much tighter than Claude's 5 min; if <30s remain the service skips and waits for the CLI to refresh on its next real invocation

**Constructor:** `new KiroPlanUsageService(appRoot: string, options?: { dbPath?: string; dataRoot?: string })` — stores the on-disk cache at `<options.dataRoot || appRoot/data>/kiro-plan-usage.json`; `options.dbPath` is test-only (defaults to the platform-specific kiro-cli data file — see below).

**Methods:**
- `init()` — loads the persisted snapshot off disk on server startup. Silently ignores `ENOENT`; logs any other read/parse failure and keeps the in-memory snapshot as its initial empty shape.
- `getCached()` — returns `{ fetchedAt, usage, lastError, stale }`. Does not trigger a refresh. `stale` is computed as `now - fetchedAt > STALE_AFTER_MS` (or `true` when `fetchedAt` is null). This is the exact shape returned by `GET /api/chat/kiro-plan-usage`.
- `maybeRefresh(reason: string)` — triggers a refresh if all of: no in-flight refresh, and at least `REFRESH_MIN_INTERVAL_MS` have passed since the last attempt. Returns the shared in-flight promise when a fetch is already running, or an immediately-resolved promise when the throttle is active. The `reason` string is logged alongside success/failure (`server-start`, `turn-done`).
- `_refresh(reason)` (private) — opens the kiro-cli SQLite DB read-only via `readKiroAuth(dbPath)`, short-circuits with `lastError: 'token-expired'` if the access token is within the 30-second expiry buffer (and resets `_lastAttemptAt = 0` so the throttle does not gate the next trigger — no API call was made, so this is not a real attempt), otherwise `POST`s `https://q.us-east-1.amazonaws.com/?profileArn=<enc>&origin=KIRO_CLI` with headers `Content-Type: application/x-amz-json-1.0`, `X-Amz-Target: AmazonCodeWhispererService.GetUsageLimits`, `Authorization: Bearer <accessToken>`, `X-Amzn-Codewhisperer-Optout: false`, `Accept: */*` and body `{"profileArn":"<arn>","origin":"KIRO_CLI"}`, with a 10-second abort signal. On success stores `{ fetchedAt: now, usage: normalizeUsage(body), lastError: null }`. On non-2xx or network failure preserves the prior `fetchedAt` / `usage` values and only overwrites `lastError` — the UI still renders the last-known snapshot.
- `_persist()` (private) — `mkdir -p <dataRoot>/` then writes the snapshot to `<dataRoot>/kiro-plan-usage.json`. Failures are logged and swallowed.

**SQLite credential resolution (`readKiroAuth`):** opens the kiro-cli data file with `better-sqlite3` using `{ readonly: true, fileMustExist: true }`, then reads two rows with prepared statements and closes the DB in a `finally`:
- `SELECT value FROM auth_kv WHERE key = 'kirocli:odic:token'` — JSON blob with `{ access_token, expires_at }` (`expires_at` is ISO-8601, parsed to epoch ms; treated as non-expiring when absent or unparseable).
- `SELECT value FROM state WHERE key = 'api.codewhisperer.profile'` — JSON blob with `{ arn }` (the active CodeWhisperer profile ARN, threaded into both the query string and the JSON body as required by the Q endpoint).

**Default DB path (`defaultKiroDbPath`):** platform-specific — `~/Library/Application Support/kiro-cli/data.sqlite3` on darwin, `%APPDATA%/kiro-cli/data.sqlite3` on win32 (falls back to `~/AppData/Roaming/…` when `APPDATA` is unset), and `$XDG_DATA_HOME/kiro-cli/data.sqlite3` on Linux (falls back to `~/.local/share/…`).

**Response normalization (`normalizeUsage`):** pulls `subscriptionInfo`, `overageConfiguration.overageStatus`, `nextDateReset`, and `usageBreakdownList[0]` from the raw body, coercing every field through `strOrNull` / `numOrNull` so missing or non-primitive values become `null` rather than crashing the renderer. `bonuses` passes through as an opaque array (`unknown[]`) — the frontend surfaces the count rather than the contents. See [spec-api-endpoints.md § 3.15](spec-api-endpoints.md#315-kiro-plan-usage) for the full `KiroUsageData` shape.

**Integration points:**
- `server.ts` — instantiated with `__dirname` plus `config.AGENT_COCKPIT_DATA_DIR`, `init()` on startup then immediate `maybeRefresh('server-start')`. Passed into `createChatRouter` via the `kiroPlanUsageService` dependency.
- `src/routes/chat/statusRoutes.ts`:
  - `GET /kiro-plan-usage` route returns `getCached()` verbatim.
- `src/routes/chat.ts` stream orchestration:
  - `onDone` stream callback calls `maybeRefresh('turn-done')` when `backendId === 'kiro'`. No other backend triggers a refresh for this service.

**Error taxonomy (`lastError` values):**
- `'token-expired'` — access token at/within the 30-second expiry buffer.
- `'kiro-cli DB unavailable at <path>: <msg>'` — SQLite open failed (file missing when `fileMustExist: true`, permission error, corruption). Typical when the user has never run kiro-cli auth on this machine.
- `'kiro-cli auth_kv missing access token'` — DB opens but the `kirocli:odic:token` row is absent or has an empty `value`.
- `'kiro-cli state missing profile'` — DB opens but the `api.codewhisperer.profile` row is absent or has an empty `value`.
- `'kiro-cli token blob missing access_token'` — token row present but the parsed JSON has no `access_token` string.
- `'kiro-cli profile blob missing arn'` — profile row present but the parsed JSON has no `arn` string.
- `'GetUsageLimits <status>: <body>'` — upstream HTTP 4xx/5xx (body truncated to 200 chars).
- Any network error message verbatim (timeout, DNS, TLS).

## 4.6 CodexPlanUsageService

**File:** `src/services/codexPlanUsageService.ts`

Fetches and caches the OpenAI Codex (ChatGPT) account snapshot — plan tier (Plus / Pro / Business / Enterprise / etc.), 5-hour rate-limit window utilization, weekly rate-limit window utilization, and optional credit balance — by spawning a one-shot `codex app-server` process and calling its `account/read` and `account/rateLimits/read` JSON-RPC methods over stdio. Surfaces in the V2 ContextChip tooltip for Codex conversations only. The default cache uses the server-configured Codex runtime; Codex profile caches use the selected profile's `command`, `env`, and `CODEX_HOME`.

**Why one-shot spawn instead of the long-lived `app-server` already used by `CodexAdapter`:** the `app-server` instance owned by `CodexAdapter` is bound to a specific conversation thread; reusing it for plan-usage queries would couple two unrelated lifecycles (refresh on server start when there's no active turn, refresh on turn-done from any conversation). A throwaway process per refresh costs ~hundreds of milliseconds and runs at most once every 10 minutes, which is well below the per-turn budget. App-server *also* emits `account/updated` and `account/rateLimits/updated` notifications on long-lived sessions, but those don't help at server boot when no Codex conversation is active.

**Constants:**
- `REFRESH_MIN_INTERVAL_MS = 10 * 60 * 1000` — minimum gap between refresh attempts; matches the Claude/Kiro floor.
- `STALE_AFTER_MS = 15 * 60 * 1000` — client-visible stale threshold.
- `REFRESH_TIMEOUT_MS = 15_000` — hard ceiling on the spawned `codex app-server`. The two RPCs are sub-second in practice; a stuck process gets `SIGKILL`'d at this point.
- `PROCESS_KILL_GRACE_MS = 1_000` — after the fetch finishes the service first sends `SIGTERM` and gives the process this long to exit cleanly before `SIGKILL`. The fallback timer is `.unref()`'d so it never holds the event loop open.

**Constructor:** `new CodexPlanUsageService(appRoot: string, options?: { dataRoot?: string })` — stores the default on-disk cache at `<options.dataRoot || appRoot/data>/codex-plan-usage.json` and profile-specific cache files under `<options.dataRoot || appRoot/data>/codex-plan-usage/<encodeURIComponent(profile.id)>.json`.

**Methods:**
- `init()` — loads the default persisted snapshot and every profile snapshot off disk on server startup. Silently ignores `ENOENT`; logs any other read/parse failure and keeps the in-memory snapshot as its initial empty shape.
- `getCached(profile?: CliProfile)` — returns `{ fetchedAt, account, rateLimits, lastError, stale }`. Does not trigger a refresh. Without a profile, or with the plain `server-configured-codex` profile that has no custom command/config/env, reads the default server-configured cache. Account/custom Codex profiles read their isolated cache. `stale` is computed as `now - fetchedAt > STALE_AFTER_MS` (or `true` when `fetchedAt` is null). This is the exact shape returned by `GET /api/chat/codex-plan-usage`.
- `maybeRefresh(reason: string, profile?: CliProfile)` — triggers a refresh for the default runtime or selected profile if all of: no in-flight refresh for that cache key, and at least `REFRESH_MIN_INTERVAL_MS` have passed since the last attempt for that key. Returns the shared in-flight promise when a fetch is already running, or an immediately-resolved promise when the throttle is active. The `reason` string is logged alongside success/failure (`server-start`, `turn-done`).
- `_refresh(reason, profile?)` (private) — calls `fetchFromAppServer(profile)` (see below). On success stores `{ fetchedAt: now, account, rateLimits, lastError: null }` into the default snapshot or that profile's snapshot. On failure preserves the prior `account` / `rateLimits` values and only overwrites `lastError` — the UI still renders the last-known snapshot.
- `_persist(profile?)` (private) — writes the default snapshot to `<dataRoot>/codex-plan-usage.json` or a profile snapshot to `<dataRoot>/codex-plan-usage/<encoded-profile-id>.json` via `atomicWriteFile`. Failures are logged and swallowed.

**App-server interaction (`fetchFromAppServer`):**
1. Resolve the Codex CLI runtime. Without a profile the command is `codex` and the env inherits `process.env`. With a profile, `profile.command || 'codex'` is used, `profile.env` is merged into the child env, and `profile.configDir` sets `CODEX_HOME`. On Windows, installer-managed Codex resolves to `node.exe` plus the package `codex.js` as the first argument before `.cmd` shim fallback.
2. Build the platform invocation and spawn it with `app-server`: `spawn(invocation.command, invocation.args, { stdio: ['pipe', 'pipe', 'pipe'], env: runtime.env })`. Errors from `spawn` itself (rare — only thrown for invalid args) bubble up as `'spawn <command> app-server failed: <msg>'`.
3. Register an `'error'` listener on the proc to capture `ENOENT` (no selected command on `PATH`).
4. `await new Promise(r => setImmediate(r))` — yield once so the async `'error'` event has a chance to fire before any `stdin.write` (which would otherwise crash with `EPIPE` on a non-existent process).
5. If `spawnFailed` is set at this point, throw `'codex app-server unavailable: <msg>'`.
6. Construct an `RpcClient` (see below) bound to the proc and call `initialize` with `{ clientInfo: { name: 'agent-cockpit', title: null, version: '1.0.0' }, capabilities: null }`.
7. Issue `account/read` (params `{ refreshToken: false }` — we don't want to trigger an OAuth refresh from a passive read) and `account/rateLimits/read` (no params) in parallel via `Promise.all`.
8. Normalize both results (see below) and return `{ account, rateLimits }`.
9. **`finally`:** clear the 15s kill timer; if the proc isn't already killed, `SIGTERM` then schedule a `SIGKILL` after `PROCESS_KILL_GRACE_MS` (the fallback timer is `.unref()`'d).

**`RpcClient` (private nested class):** minimal newline-delimited JSON-RPC over `proc.stdin` / `proc.stdout`.
- `nextId` starts at 1; each `request` allocates a fresh id and stores `{ resolve, reject }` in a `pending` map keyed by id.
- The stdout listener buffers partial lines, splits on `\n`, and for each complete line: `JSON.parse`, then if the message has an `id` and no `method` (i.e. it's a response, not a server-to-client notification or request), look up `pending[id]` — `reject(new Error(error.message))` on RPC error, `resolve(result)` otherwise.
- Server-to-client notifications and requests are dropped silently (the cockpit only cares about its own responses).
- On `proc.close` the client rejects every pending promise with `'codex app-server closed'` and clears the map. Subsequent `request()` calls reject immediately with `'codex app-server is closed'`.

**Response normalization:** every accessor goes through `strOrNull` / `numOrNull` so missing or wrong-typed fields become `null` rather than crashing the renderer.
- `normalizeAccount(raw)` reads `raw.account` and returns `{ type, email, planType }` (or `null` if `account` is absent).
- `normalizeRateLimits(raw)` reads `raw.rateLimits` and returns `{ limitId, limitName, primary, secondary, credits, planType, rateLimitReachedType }` (or `null` if `rateLimits` is absent). Each `primary` / `secondary` window is normalized to `{ usedPercent, windowDurationMins, resetsAt }` — `resetsAt` is **epoch seconds** (not ISO string like Claude's `resets_at`).
- `normalizeCredits(raw)` returns `{ hasCredits, unlimited, balance }` where `hasCredits` and `unlimited` are coerced to strict booleans (`=== true`) and `balance` is preserved as a string (Codex returns it pre-formatted, e.g. `'0'`).

**Window semantics:** Codex's API returns two windows per limit: `primary` (5-hour, `windowDurationMins: 300`) and `secondary` (weekly, `windowDurationMins: 10080`). The frontend uses `windowDurationMins` to label each bar — it does **not** assume the slot order, so a future Codex API change that swaps slots won't mislabel the bars.

**Integration points:**
- `server.ts` — instantiated with `__dirname` plus `config.AGENT_COCKPIT_DATA_DIR`, `init()` on startup then immediate `maybeRefresh('server-start')`. Passed into `createChatRouter` via the `codexPlanUsageService` dependency.
- `src/routes/chat/statusRoutes.ts`:
  - `GET /codex-plan-usage` route returns `getCached()` verbatim. `GET /codex-plan-usage?cliProfileId=<id>` resolves the profile, rejects missing/disabled/non-Codex profiles with `400`, and returns `getCached(profile)` without triggering a refresh.
- `src/routes/chat.ts` stream orchestration:
  - `onDone` stream callback calls `maybeRefresh('turn-done', runtime.profile)` when the resolved backend is Codex. No other backend triggers a refresh for this service.

**Error taxonomy (`lastError` values):**
- `'spawn <command> app-server failed: <msg>'` — synchronous throw from `child_process.spawn`.
- `'<command> app-server unavailable: <msg>'` — async `'error'` event (typically `ENOENT` when the selected command is not on `PATH`).
- `'codex app-server closed'` — the process exited before responding to a pending RPC.
- `'codex app-server is closed'` — a request was issued after the proc had already been closed.
- Any RPC error message verbatim (e.g. `'auth required'`, `'invalid params'`).
- Any network/TLS error verbatim if `codex` itself fails to reach OpenAI's account API.
