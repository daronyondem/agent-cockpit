# Agent Cockpit v0.4.37

## Shipped For Users

- Windows Claude setup now matches the terminal experience: after Agent Cockpit verifies Claude Code login, plain `claude` in a new PowerShell window should open as an authenticated CLI instead of returning to the first-run login selector.
- Windows Codex installed from the Welcome screen now works from terminal windows without requiring a separate global Node.js install.
- Windows self-updates and Install Doctor repairs now refresh Agent Cockpit-managed Claude/Codex terminal wrappers, so existing installs can recover CLI launch behavior without a full reinstall.
- Legacy Welcome setup profiles are migrated back to system CLI auth more completely, preventing Agent Cockpit chats from using an isolated auth directory while the user's terminal sees a different login state.

## Developer Details

- Added Windows-only Claude Code setup-auth repair in `CliProfileAuthService`: when system-home `claude auth status --json` reports `loggedIn: true`, Agent Cockpit ensures `%USERPROFILE%\.claude.json` has `hasCompletedOnboarding: true`. Isolated profiles with `CLAUDE_CONFIG_DIR` remain isolated and are not modified.
- Tightened setup-auth profile selection and migration so stale `configDir`, `CLAUDE_CONFIG_DIR`, and `CODEX_HOME` values are removed for setup profiles, while explicit non-setup isolated profiles are not reused as system-home setup profiles.
- Added `windowsCliToolWrappers` to repair Agent Cockpit-managed Windows wrappers. Codex wrappers call the recorded private `node.exe` plus `@openai/codex/bin/codex.js`; Claude wrappers call the package `claude.exe` directly when present.
- Wired wrapper repair into Install Doctor CLI install actions, Windows production self-update, and Windows CLI update flows.
- Updated deployment/API/backend/testing specs, project memory, and `AGENTS.md` with the Windows-specific CLI/auth rules. Mobile PWA impact: none; changes are server/installer/update-path behavior only.
