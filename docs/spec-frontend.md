## V2 ŌĆö Default Frontend

**Paths:**
- `/` ŌĆö redirects to `/v2/` (handled in `server.ts`).
- `/v2/` ŌĆö real app entry (`public/v2/index.html` ŌåÆ `shell.jsx`). The default UI.

V2 is the production UI. The legacy vanilla-JS UI and mockup screens have been removed; only `/` and `/v2/` are supported frontend entry paths. All app paths are behind `requireAuth`. The V2 bundle is a React 18 + Babel Standalone prototype (no build step ŌĆö see the production build issue on GitHub). Source tree:

The sidebar footer reads `GET /api/me` and supports first-party local owner sessions (`provider: "local"`) as the default auth identity. Legacy Google/GitHub provider values are still tolerated for transitional OAuth sessions, but the login surface no longer depends on third-party provider buttons by default.

- `public/v2/src/streamStore.js` ŌĆö per-conversation stream store exposed as `window.StreamStore`. Holds one `ConvState` entry per conversation the user has visited in the current tab (`{ convId, conv, messages, input, sending, streaming, streamError, streamErrorSource, streamingMsgId, usage, loaded, loadError, ws, wsOpening, wsReconnectAttempts, wsReconnectTimer, lastFrameAt, replayActive, reconcileTimer, uiState, pendingInteraction, respondPending, composerBackend, composerModel, composerEffort, pendingAttachments, queue, resetting }`) with its own WebSocket, accumulated messages, and streaming flags. `streamErrorSource` mirrors terminal stream-error provenance (`'backend'`, `'transport'`, `'abort'`, `'server'`, or null) so the UI can distinguish intentional user cancellation from retryable failures while keeping `streamError` as the display text. `uiState` is one of `'streaming' | 'awaiting' | 'error' | null`; `pendingInteraction` is either `{ type: 'planApproval', planContent }` or `{ type: 'userQuestion', question, options }` or `null`. The three `composer*` fields hold the user's picker selections ŌĆö hydrated from the conv record on first load (`data.backend/model/effort`) and sticky once the user touches them, so flipping convs and back doesn't rewind the pickers. `pendingAttachments` is an array of `PendingAttachment` entries (`{ id, file, status: 'uploading'|'done'|'error', progress, result: AttachmentMeta | null, xhr, error }`) driving the composer's attachment tray ŌĆö each entry tracks one file's lifecycle from user-drop through server-upload to final send. `result` holds the typed `AttachmentMeta` returned by `POST /upload` (including server-inferred `kind` and optional `meta` sublabel like "12 pages") so the chip can render without a second fetch. The tray clears on successful POST `/message` and is restored on POST failure. `queue` is the client mirror of the persisted `messageQueue` (array of `QueuedMessage` ŌĆö see `spec-data-models.md`); the store auto-drains the queue head on every `done` frame when the conversation is idle (no `pendingInteraction`, no `streamError`). Public surface: `getState(convId)`, `load(convId)`, `ensureWsOpen(convId)`, `revalidateAllSockets()` (forces every live WS to close + reopen so the server can replay the buffered events ŌĆö used by the visibility/online listeners; see *Network-change revalidation* below), `send(convId, text)`, `respond(convId, text)` (delivers an answer to the pending plan/question interaction ŌĆö see *Plan approval & clarifying questions* below), `setInput(convId, value)`, `setComposerBackend(convId, value)` / `setComposerModel(convId, value)` / `setComposerEffort(convId, value)` (composer picker setters ŌĆö values flush to the server with the next POST `/message`, not immediately), `addAttachments(convId, files)` / `removeAttachment(convId, attachmentId)` / `clearPendingAttachments(convId)` (tray setters ŌĆö `addAttachments` kicks off one XHR upload per file, `removeAttachment` aborts in-flight uploads and `DELETE`s already-uploaded ones from the artifacts dir, `clearPendingAttachments` empties the tray without deleting the uploads ŌĆö used when the user enqueues a message and the attachments are detached onto the queue entry). Queue helpers: `loadQueue(convId)` (GET `/queue`), `enqueue(convId, content, attachments)` (optimistic append + PUT), `removeFromQueue(convId, index)`, `reorderQueue(convId, fromIndex, toIndex)`, `updateQueueItem(convId, index, patch)`, `clearQueue(convId)` (DELETE). Backwards-compat helpers exposed for legacy transcript rendering: `parseUploadedFilesTag(content)` (returns `{ content, attachments } | null`), `attachmentKindFromPath(path)`. Lifecycle: `reset(convId) ŌåÆ Promise<boolean>` (sets `state.resetting = true` for the duration of the call so `ChatLive` can surface an in-feed "Archiving sessionŌĆ”" progress bubble ŌĆö the server-side reset archives the session, captures memory, and re-initialises which can take several seconds; resolves `true` on success so callers can fire a success toast, `false` when the conv is streaming/sending/already-resetting or the POST fails), `archive(convId)`, `closeWs(convId)`, `subscribe(convId, listener)` (returns unsubscribe), `subscribeGlobal(listener)` (fires whenever any conv's `uiState` transitions ŌĆö used by the sidebar), `convStates()` (returns the compact `{ [convId]: 'streaming' | 'awaiting' | 'error' }` map consumed by `<Sidebar>`). React views read these through the `useConversationState(convId)` / `useConvStates()` hooks in `shell.jsx`. Because state lives *above* the `<ChatLive>` view, switching conversations does **not** abort an in-flight stream: the prior conv keeps collecting frames, the sidebar lights up every row whose `uiState` is non-null, and switching back to the conv immediately shows the accumulated messages. WebSockets are opened lazily on first visit and left open for the session. Unexpected active-stream closes reconnect with backoff; transport errors do not become backend stream errors. Failed WebSocket opens clear `ws/wsOpening` so the next reconnect attempt creates a fresh socket instead of reusing a rejected `CONNECTING` promise. The server's replay buffer handles immediate recovery, while `GET /active-streams` reconciliation clears stale local streaming state if replay has already been cleaned. `/active-streams` may include operational `runtime` identifiers for backend supervision, but the UI treats those as server diagnostics and only uses the stream IDs/states to clear stale local flags. `archive(convId)` is the only path that closes the WS and removes the entry.
  CLI profile state extends the same picker lifecycle: each entry also carries `composerCliProfileId`, hydrated from `data.cliProfileId`, and `setComposerCliProfile(convId, profileId, backendId)` updates the selected profile plus its vendor backend while clearing model/effort so the cascade reconciles against the new vendor. `send()` and queued auto-drain include `cliProfileId` with the next `/message` POST and mirror the confirmed value back onto `conv.cliProfileId`.
  Codex goal state is held in the same per-conversation entry as `goal` (`ThreadGoal | null`) and `goalMode` (`boolean`). `load(convId)` fetches `GET /goal` only for Codex conversations that already have an external session id; non-Codex conversations and new Codex conversations start with `goal: null`. `handleFrame` updates `goal` on `goal_updated` and clears it on `goal_cleared`. Public goal helpers are `setGoal(convId, text)`, `setGoalMode(convId, value)`, `pauseGoal(convId)`, `resumeGoal(convId)`, and `clearGoal(convId)`. `setGoal` composes uploaded-file markers from completed attachments into the objective, creates only an assistant streaming placeholder, clears the composer/tray on acceptance, posts to `POST /goal`, and does not optimistically append a user message. Selecting a non-Codex backend/profile clears `goalMode`.

- `public/v2/src/codexPlanUsageStore.js` ŌĆö OpenAI Codex account plan usage store exposed as `window.CodexPlanUsageStore`.   **Per-conversation state lift (PR 4b)** ŌĆö `App` reads the sidebar's per-row indicator map from `useConvStates()`, which is a thin subscription over `StreamStore.subscribeGlobal` (fires when any conv's `uiState`, `unread`, or first-touch transitions). Every visited conv has an entry in the store and contributes to `StreamStore.convStates()`, so multiple rows can light up simultaneously ŌĆö e.g. the user can kick off a stream in conv A, switch to conv B, kick off a second stream there, and both rows show `.streaming` in the sidebar while only one `<ChatLive>` is mounted. The store flips the entry's `uiState` to `'streaming'` on send, `'error'` on stream error, and `null` on `done`. `convStates()` also surfaces `'unread'` for entries with the unread flag set, and `'idle'` as a sentinel for any touched entry that is currently neither active-state nor unread (used by the sidebar to override the stale `c.unread` it may still be holding from the cached server list ŌĆö see *Unread state* below).

  **Refresh-time hydration (`StreamStore.hydrateActiveStreams`)** ŌĆö a page refresh wipes the whole in-memory `StreamStore` (new JS context), but accepted CLI streams keep running on the server because the chat router, not browser WebSocket state, owns their lifetime. Without intervention the sidebar would show gray dots on those live conversations until the user clicked into each one (which triggers `ensureWsOpen` -> buffer replay/reconcile -> `uiState` rebuilt). To close that gap, `Sidebar` calls `StreamStore.hydrateActiveStreams()` once on mount: it fetches `GET /api/chat/active-streams` (returns compatibility `ids: string[]` plus `streams[]` summaries merged from the stream supervisor's runtime `activeStreams` map and durable `stream-jobs.json`; summaries also include operational-only `runtimeAttached` and `pending` booleans plus diagnostic backend `runtime` metadata) and seeds `streaming: true`, `uiState: 'streaming'`, and `streamError: null` on each ID that doesn't already have an open WS. For seeded convs the user never opens, the dot stays blue until the next hydrate or conversation-list refresh. The first WS replay or `/active-streams` reconciliation corrects the state to `'awaiting'`/`null`/`'error'` as appropriate. After a full server restart, the backend iterator is gone; startup reconciliation removes leftover durable jobs and appends a persisted assistant `streamError` (`source:'server'`, "Interrupted by server restart") for turns whose user message was already saved. After a graceful external server shutdown, reconciliation uses the recorded terminal reason (`source:'server'`, "Interrupted by server shutdown"). When `load()` or reconciliation refetches a conversation after the replay buffer is gone, the store derives active `streamError` from the final persisted assistant message if that message has `streamError`; an older stream-error message followed by a normal user/assistant message is historical and does not re-enter error state. A refetched durable stream error sets `uiState: 'error'` before queue draining is considered, so restored queued messages stay paused for explicit user action. Non-fatal on network/auth errors - hydration just silently no-ops so the sidebar still renders with what's in the server-cached `conversations` list.

  **Unread state** ŌĆö a conv is "unread" when a response stream completed on it while the user was viewing a different conv. Tracked in three places that all stay in sync:
  - **Server (authoritative)** ŌĆö `unread: boolean | undefined` on the workspace-index entry. Toggled via `PATCH /api/chat/conversations/:id/unread` (see API spec). Persisted across reload.
  - **Server-cached list** ŌĆö every `conversations` summary returned by `AgentApi.listConversations()` carries the boolean as `c.unread`. The sidebar uses this for cold rows (rows the user hasn't visited this session, so there's no `StreamStore` entry for them).
  - **`StreamStore` (per-conv, in-memory)** ŌĆö `ConvState.unread` (`boolean`) on every visited conv. The sidebar uses `StreamStore.convStates()` for any conv that has an entry, falling back to `c.unread` only when no entry exists. This split is what lets the sidebar clear the dot **immediately** on selection without waiting for a `/conversations` refetch.

  **Auto-mark on done** ŌĆö `StreamStore.handleFrame`'s `done` branch flags `unread = true` on the target conv whenever `convId !== activeConvId` and the stream landed in idle (no `pendingInteraction`, no `streamError`). The shell pushes the active id into the store via `StreamStore.setActiveConvId(id)` from a `useEffect` on `activeConvId`, so the store always knows which conv (if any) is currently visible.   **Manual mark / clear** ŌĆö selecting a conv (`onSelectConv` in `shell.jsx`) calls `StreamStore.markRead(id)`, which always touches local state (creating a blank entry on cold convs so `convStates()` returns `'idle'` to override `c.unread`) and POSTs `unread: false`.   **Send flow (all logic lives in `StreamStore.send(convId, text)`; the view just invokes it):**
  1. Ensure the conv's WS is open (`ensureWsOpen` resolves the pending open promise or opens a new socket; a 5 s connect timeout is surfaced as an error).
  2. Optimistically append a pending user message and an empty pending-assistant placeholder (with an empty `contentBlocks` array); mark the placeholder's id as the entry's `streamingMsgId`; flip `uiState` to `'streaming'`.
  3. Before building the POST body, gather all completed file attachments (`pendingFiles` entries whose `status === 'done'`); when any exist, append `[Uploaded files: <abs-path-1>, <abs-path-2>, ŌĆ”]` to the content (on a fresh line separated by `\n\n`; the tag alone if the textarea was empty). The attachments were already uploaded to the conv's artifacts dir by `addFiles` ŌĆö the server paths are what the LLM sees, and the message body is their only channel. Snapshot `pendingFiles` before clearing so a POST failure can restore the chips intact (the files are still on disk server-side, so there's no re-upload). `POST /api/chat/conversations/:id/message` with `{ content, cliProfileId?, backend?, model?, effort? }` via `AgentApi.fetch` (CSRF-token header is attached automatically; a 401 triggers the registered session-expired handler). The optional picker fields ride along only when the composer pickers have a value (either hydrated from the conv record or touched by the user). The server-side route (see Chat Routes spec) conditionally updates the conversation row when any of these differs from what's currently stored, so the chosen picker values become the new defaults for the next turn. On success, swap the pending user message for the server-authoritative `userMessage` returned in the response and mirror the picker values back onto the entry's `conv.cliProfileId/backend/model/effort` so future renders see the persisted truth.
  4. WebSocket frame handling runs inside `StreamStore.handleFrame(convId, frame)` ŌĆö every frame updates the target conv's entry regardless of which view is currently mounted:
     - `text` ŌĆö `{ content }` delta appended to the streaming placeholder's ordered `contentBlocks`. Consecutive `text` deltas merge into the last text block so one contiguous paragraph produces one block.
     - `thinking` ŌĆö `{ content }` delta appended as a `thinking` block (same merge rule as text).
     - `tool_activity` ŌĆö pushes a new `{ type: 'tool', activity: { tool, description, id, duration: null, startTime, isAgent?, subagentType?, parentAgentId? } }` block onto `contentBlocks`. Plan-mode markers (`isPlanMode`) and clarifying-question frames (`isQuestion`) are *not* pushed as tool blocks; instead they update `pendingInteraction` + `uiState='awaiting'` (see *Plan approval & clarifying questions* below).
     - `tool_outcomes` ŌĆö patches the matching tool block(s) in the streaming placeholder by `activity.id === toolUseId`, setting `outcome` and `status`.
     - `artifact` ŌĆö pushes a new `{ type: 'artifact', artifact }` block onto the streaming placeholder and uses the artifact title/filename as the placeholder's legacy `content` fallback when no text has streamed. The server has already copied/decoded the artifact into `data/chat/artifacts/{convId}/`, so the client only needs the normalized descriptor.
     - `assistant_message` ŌĆö authoritative persisted message; replaces the placeholder **and clears `streamingMsgId`** so the next delta opens a fresh placeholder for the next segment. This is the *real* segment boundary on the server (the server only emits `assistant_message` when it just persisted an intermediate/final message). Intermediate segments land with `turn: 'progress'`, the final segment with `turn: 'final'`. The feed render collapses runs of consecutive `turn: 'progress'` assistant messages into a progress breadcrumb (see *Progress-breadcrumb collapse* below). Before deciding replace-vs-append, the handler drops any prior entry in `messages` whose `id` matches `frame.message.id` (excluding the current placeholder). This collapses the duplicates produced by the server's replay path: when a fresh WS connects after a turn already completed (page reload, sleep wake, `online` event, visibility revalidation Ōēź30 s) the buffered events are re-emitted, and either (a) the persisted message is already in `messages` from `load()` and the replayed frame would append a copy, or (b) `streamingMsgId` is null so the replayed `text` deltas spin up a *new* placeholder and the replayed `assistant_message` would otherwise leave both the original entry and the freshly-replaced placeholder in place ŌĆö both with the same id.
     - `turn_complete` ŌĆö **explicit no-op** (documented branch in `handleFrame`, not a silent drop). The server emits `turn_complete` right after every `turn_boundary` CLI event, which V1 uses to archive `activeTools`ŌåÆ`toolHistory` and stop a shared activity timer. V2 doesn't need that archival: tools live as `contentBlocks[]` on the streaming assistant message and `tool_outcomes` mutates them in place, turn splits already happen on `assistant_message` (placeholder swap + `streamingMsgId = null`), and live activity timers are per-block and self-clear when `duration` lands. Kiro never emits `turn_boundary`, so this frame is Claude-Code-only; the Kiro adapter yields `done` directly from its `turn_end` handler.
     - `replay_start` ŌĆö wipes the streaming placeholder's `contentBlocks` (and `content`, and any `pendingInteraction`/`planModeActive`) so the server's buffered frames can rebuild state cleanly on reconnection. Keeps `streamingMsgId` intact so replayed `text`/`tool_activity` frames land on the same placeholder message id. No-op when `streamingMsgId` is null (nothing to wipe). Backend-agnostic: the server's per-conv ring buffer (`ws.ts`, 5000-event cap) captures every frame regardless of adapter, so replay affects both Claude Code and Kiro conversations identically. Without this wipe, navigating away from and back to an in-progress conversation would duplicate text + tool rows in the placeholder until a live `assistant_message` overwrote it.
     - `replay_end` ŌĆö explicit no-op. State was rebuilt by the normal handlers during replay; React subscribers already re-render on each update, so there's no additional reconcile step to run. V1 uses this frame to trigger a DOM re-render and re-show pending interaction dialogs, but V2's declarative view layer handles that implicitly.
     - `title_updated` ŌĆö updates the entry's `conv.title` in-place and invalidates the cached conversations list so the sidebar re-fetches on next mount.
     - `usage` ŌĆö refreshes the entry's `usage` from the frame's `sessionUsage` field (session-scoped totals; conversation-level totals on the same frame are intentionally ignored here so the topbar chip reflects only the active session and zeroes on reset).
     - `assistant_message` ŌĆö also calls `bumpConvListActivity(convId, frame.message.timestamp)` so the sidebar row's relative-time label and list ordering reflect the just-persisted server timestamp without a full list re-fetch (see *Sidebar live activity* below).
     - `error` ŌĆö flips `streamError` and `uiState` to `'error'`, clears `pendingInteraction` (a dead stream can't accept an answer).
     - `done` ŌĆö clears `streaming` and `streamingMsgId`; `uiState` resolves to `'error'` if one already landed, `'awaiting'` if `pendingInteraction` is still set, else `null`. `pendingInteraction` itself is preserved across `done` so the user can still answer after the stream's finally block runs. When the stream lands in idle (no `streamError`, no `pendingInteraction`) and `convId !== activeConvId`, the entry's `unread` is also flipped to `true` and `AgentApi.markConversationUnread(convId, true)` fires (see *Unread state* below).
  5. On POST failure, the pending user and assistant placeholders are rolled back and the error is shown.

  **Network-change revalidation (`StreamStore.revalidateAllSockets`):** browsers commonly leave a `WebSocket` in `readyState === OPEN` for several minutes after the underlying TCP socket has died (laptop sleep, NAT rebind on Wi-Fi switch). Without intervention, a POST `/message` from the stale tab would reach the server while the conversation has no live WS, the chat router would buffer the user message, and the conv would look "in-progress" forever after a refresh ŌĆö a real bug we hit when waking the laptop in a different location. To unstick this, `streamStore.js` registers two browser-event listeners on module load:
  - `window.online` ŌåÆ calls `revalidateAllSockets()` unconditionally. Coming back from offline implies a fresh interface, so any prior socket is suspect.
  - `document.visibilitychange` ŌåÆ records `lastHiddenAt = Date.now()` on hide; on show, calls `revalidateAllSockets()` only when the gap is `>= 30_000` ms (`VISIBILITY_REVALIDATE_THRESHOLD_MS`). Below the threshold the socket is left alone ŌĆö replay would otherwise wipe the streaming placeholder's `contentBlocks` on every brief tab switch.

  `revalidateAllSockets` iterates every entry in the store with a non-null `s.ws`, calls `s.ws.close(4000, 'revalidating')`, nulls out `ws/wsOpening`, and immediately invokes `ensureWsOpen(convId)` to create a fresh socket. The close is transport-only: the server keeps any accepted CLI stream alive through the router's `activeStreams` map and preserves the event buffer for replay. The server replays the buffered events on the new connection, the existing `replay_start` handler wipes the placeholder so frames rebuild cleanly, and live streaming resumes. If the CLI finished while the browser was away and the replay buffer has already been cleaned up, the delayed `/active-streams` reconciliation refetches persisted messages and clears local streaming state. A `POST /message` submitted before a WS is connected still starts the CLI turn server-side and marks the event buffer disconnected so frames are available on the next socket.

  **Assistant body rendering (`AssistantBody` ŌåÆ `groupBlocksForRender` ŌåÆ `renderSegment`):** `contentBlocks` is authoritative when present. For legacy messages saved before the field existed, a fallback in `deriveBlocks` synthesizes blocks from the message's `thinking`/`toolActivity`/`content` fields (thinking first, then tools, then the text body) so older conversations render without special cases. `groupBlocksForRender` then walks the blocks and splits them into four kinds of segments: `text`, `thinking`, `artifact`, and `tool-run` (consecutive tool blocks merge into one run so they can be grouped as a unit). Each segment renders as:
     - `text` ŌåÆ `.prose` container with the markdown pipeline (`marked.parse` + `DOMPurify.sanitize`). Fenced code blocks land as `<pre><code>ŌĆ”</code></pre>`; `.prose pre` renders them as a monospace panel with `overflow-x: auto`, `white-space: pre`, and the `--surface-2` background so the block is visually distinct and selectable/copy-able. Markdown tables (`.prose table`) use `border-collapse: collapse` with `1px solid var(--border)` cell borders and `6px 10px` padding so rows and columns are visually separated; `<th>` cells carry a `--surface-2` background and `font-weight: 600` to mark the header row.
     - `thinking` ŌåÆ `ThinkingBlock` ŌĆö a dashed-border pill button that, when clicked, toggles a below-pill panel showing the raw thinking text (`white-space: pre-wrap`, dimmed). The collapsed label shows a short preview (first 80 chars of line 1) so the user can tell segments apart at a glance.
     - `artifact` ŌåÆ `GeneratedArtifact`. Image artifacts render an inline thumbnail backed by `GET /conversations/:id/files/:filename` and click through to `ImageLightbox`; all artifacts also render a file card with View/Download actions using the conversation-file endpoint. Non-image artifacts render only the file card.
     - `tool-run` ŌåÆ `ToolRun`, which feeds its activities into `partitionToolRun` and then renders each segment via `SubagentCard` (for agents) or `ToolGroup` (for parallel/sequential clusters of non-agent tools). See *Tool-run grouping* below.

  **Tool-run grouping (`partitionToolRun` + `partitionParallel`):**
  - Cross-message subagent index (`buildAgentIndex` + `AgentIndexContext`): a subagent's `Agent` tool_use and its internal children often end up persisted on *different* messages (the Agent's accompanying "Now spawning..." text triggers an early save, then the subagent's child tool_uses accumulate on the next saved message). To reassemble the subagent card, the feed-level `AgentIndexProvider` walks the entire `messages` list before rendering and produces `{ agentIds, childrenByAgent }`: `agentIds` is the set of every `isAgent` tool ID seen, and `childrenByAgent` maps each agent ID to the flat list of child activities (from *any* message) whose `parentAgentId` points at it, in chronological order. The provider wraps the whole feed so every `ToolRun` down the tree reads the same index via `React.useContext(AgentIndexContext)`.
  - `partitionToolRun(tools, agentIndex)` walks the run in order. Activities whose `isAgent === true` become `{type:'agent'}` segments whose `children` come from `agentIndex.childrenByAgent.get(activity.id)` (partitioned themselves via `partitionParallel`). Activities whose `parentAgentId` appears in `agentIndex.agentIds` are skipped at the top level ŌĆö they render inside the parent card wherever that card lives, even if it's on a different message. Everything else accumulates into a buffer that flushes through `partitionParallel` whenever an agent interrupts (preserving the source order).
  - `partitionParallel(tools)` merges consecutive activities that share a server-assigned `batchIndex` (see Backend Services spec) into `{type:'parallel'}` clusters; activities across distinct `batchIndex` values become separate `{type:'sequential'}` clusters. The server bumps `batchIndex` on every CLI `turn_boundary` event, so consecutive tools tagged with the same index are exactly the parallel tool_uses of a single LLM assistant turn ŌĆö this is authoritative regardless of how tool execution time spaces the events. **Legacy fallback:** tools persisted before the server started tagging `batchIndex` lack the field; for those, `partitionParallel` falls back to the 500ms `PARALLEL_THRESHOLD_MS` startTime-proximity rule used by the original `chatGroupParallelItems()` helper. Single activities become a single-item `sequential` group.
  - `ToolGroup` renders the cluster as a `.tools` card. Parallel clusters add the `.parallel` modifier (left rail) and always show a `.tools-head` with a `PARALLEL` tag plus step count; sequential clusters only show the header when they have two or more items (a lone tool renders as a bare row inside a plain `.tools` card, per the mock).
  - `SubagentCard` renders the agent activity as a `.subagent` card. The header shows `.chip` (subagentType), `.title` (the agent's description/tool), and `.elapsed` (duration or `ŌĆ”`). The body contains nested `ToolGroup` cards for the agent's partitioned children; CSS strips the child cards' outer border/rounding so the parent card supplies the chrome.
  - `ToolRow` renders an individual activity. The leading marker is `.run` (pulsing amber) while the tool is in flight, `.done` (green) once `outcome` or `duration` lands, and `.err` (red) when `status === 'error'`. The right-hand cell shows the outcome text when available, else the duration in `ms`, else `ŌĆ”` while running. When the server sends `tool_outcomes` without a `duration` (some backends don't), `patchToolOutcomes` computes elapsed from the activity's local `startTime` so the row flips to `.done` with a usable timing.

In the streaming-placeholder case, the in-flight multi-segment turn stays in a single agent presence ŌĆö the breadcrumb of already-saved segments on top, the currently streaming content below ŌĆö instead of splitting into a standalone breadcrumb bubble above a separate streaming bubble. Rendered as its own `ProgressBreadcrumbBubble` ŌĆö a stand-alone agent bubble whose head reads `progress ┬Ę N segment{s}` and whose body is the `ProgressBreadcrumb`.

  **Plan approval & clarifying questions (PR 4c)** ŌĆö the server emits two interactive `tool_activity` frames that require a user response rather than being executed like normal tools:
  - `{ tool: 'ExitPlanMode', isPlanMode: true, planAction: 'exit', planContent }` ŌĆö the LLM has finished drafting a plan and is asking for approval to proceed. `planContent` is the markdown body of the plan; the server falls back to the accumulated assistant text when no explicit plan file was captured.
  - `{ tool: 'AskUserQuestion', isQuestion: true, questions: [{ question, options: [{label, description?}] }], description }` ŌĆö the LLM needs a clarifying answer before continuing. Only the first question in the array is surfaced; `options` may be empty (free-form answer only) or contain labelled choices.

  - `pendingInteraction = { type: 'planApproval', planContent }` or `{ type: 'userQuestion', question, options }`
  - `uiState = 'awaiting'` ŌĆö the sidebar row flips to the amber halo dot (`.sb-row.awaiting`) and every visible conv shows its own pending state independently.

  `tool_activity` frames with `planAction: 'enter'` are ignored (they mark the *start* of plan mode, which isn't actionable). Neither interactive frame is pushed onto the streaming message's `contentBlocks` ŌĆö they are presented entirely through the dedicated card UI below.

  **Stream error card:** when `state.streamError` is non-null (set by a terminal WS `error` frame, a failed `/message` POST, or reconnect exhaustion/connect-open failure), `<ChatLive>` renders a `<StreamErrorCard>` at the foot of the feed (after `InteractionCard`). Non-terminal backend warning frames (`terminal: false`) are logged and do not show this card; transient socket `onerror`/`onclose` events during an active stream schedule reconnect instead of turning into backend-looking errors. The store preserves `state.streamErrorSource` from durable messages and terminal WS frames; `source: 'abort'` renders as an **Operation aborted** cancellation notice using the amber `.err-card-abort` variant, with no **Retry** action. Other sources render as **Stream error** and may show **Retry** when a prior user message can be re-sent. The server still persists terminal stream failures as assistant `streamError` messages for durable reload/restart recovery, but while the matching error is active, `<ChatLive>` suppresses that latest persisted marker from the normal transcript so the user sees only the styled card instead of a duplicate assistant bubble. Durable abort markers are always suppressed from the normal transcript, even after the cancellation notice is dismissed, because they are recovery markers rather than assistant output. Older non-abort `streamError` messages followed by normal user/assistant activity remain historical transcript entries. The card uses the mock's `.err-card` / `.err-head` / `.err-actions` classes: accent left border, uppercase mono head with the message in dim mono on the right, a prose body that says how many queued messages are paused, and actions ŌĆö **Dismiss** (always shown, clears the error), **Retry** (non-abort errors only when there is a last user message), and **Resume queue** (only rendered when `queue.length > 0`). These buttons call `StreamStore.clearStreamError(convId, opts)` before retrying/resuming; clearing nulls `streamError` and `streamErrorSource`, recomputes `uiState` (`'awaiting'` if a `pendingInteraction` is still set, else `null`), and ŌĆö when `opts.resumeQueue === true` ŌĆö immediately calls `drainQueueIfReady(convId)` so the head of the queue fires as a fresh `/message` POST. Because `drainQueueIfReady` checks `streamError` before acting, the error must be cleared first.

  **Card rendering (inside `<ChatLive>`):** when `state.pendingInteraction` is set, `<ChatLive>` renders an `<InteractionCard>` at the bottom of the feed (after messages, before `streamError`). The card dispatches on interaction type:
  - `PlanApprovalCard` ŌĆö amber-top `.plan` card with a `NEEDS YOU` chip, the rendered `planContent` inside a `.prose` container (via the same `marked + DOMPurify` pipeline used for message bodies), and `.plan-actions` with `Reject` (danger) / `Approve & run` (primary) buttons. Clicking either calls `StreamStore.respond(convId, 'yes' | 'no')`.
  - `QuestionCard` ŌĆö accent-top `.plan` card with a `Clarifying question` header, the question text as `.plan-title`, an optional row of option buttons (each pre-fills the answer input when clicked; `title` attribute carries the option description), a focused `<input>` for free-form typing, and a `Send` button. Enter submits. The submit calls `StreamStore.respond(convId, text)`.

  While `pendingInteraction` is set, the composer's textarea is disabled with placeholder `"Answer the prompt above to continueŌĆ”"` and the send button is disabled (`awaiting` joins `sending`/`streaming` in `canSend`'s guard). Users cannot send an ordinary composer message while an interaction is pending ŌĆö they must click Approve/Reject or answer the question.

  **Answer delivery (`StreamStore.respond`):**
  1. Set `respondPending = true` (disables card buttons; shows `SendingŌĆ”` in the primary button).
  2. `POST /api/chat/conversations/:id/input` with `{ text, streamActive: streaming || sending }`. The `streamActive` hint tells the server whether the client still considers the CLI stream live, so the server can avoid racing a just-finished stream.
  3. On `{ mode: 'stdin' }` ŌĆö the server piped the answer straight into the CLI's stdin; the existing stream will produce more frames. Clear `pendingInteraction`, reset `respondPending`, and set `uiState` back to `'streaming'` (if still streaming), `'error'` (if already errored), or `null`.
  4. On `{ mode: 'message' }` ŌĆö the stream has already ended. Clear `pendingInteraction`, reset `respondPending`, then delegate to `StreamStore.send(convId, text)` which runs the normal send flow (optimistic user message, POST `/message`, fresh stream via the existing WS).
  5. On network/HTTP error ŌĆö surface via `streamError` and flip `uiState` to `'error'`.

  **Lifecycle edges:**
  - `reset(convId)` clears `pendingInteraction` and `uiState` along with the rest of the session.
  - `send(convId, text)` clears `pendingInteraction` as part of the optimistic update ŌĆö if the user bypasses the card by typing into the composer (which only happens when we're not correctly guarding ŌĆö see above), we don't leave a stale interaction behind.
  - `archive(convId)` deletes the whole entry so `pendingInteraction` naturally goes away.
  - A new `error` frame mid-interaction clears `pendingInteraction` (a dead stream can't accept the answer).
  - A `done` frame *preserves* `pendingInteraction` ŌĆö the stream's `finally` block runs but the user can still answer; the next `respond()` call will get `mode: 'message'` and transparently start a new send.

  **Cascade reconciliation** ŌĆö a `useEffect` inside `ComposerPicks` watches `(backends, selectedProfileBackend, composerCliProfileId, composerBackend, composerModel, composerEffort)` and pushes reconciled values to the store whenever the catalog shifts under the user's selection:
  - If the selected CLI profile exists, its `vendor` is the effective backend. If there is no selected profile and `composerBackend` is not in the registry (e.g. a deleted backend still persisted on the conv record), snap to the first backend.
  - If the current `composerModel` isn't in the effective backend's `models[]`, reset to the backend's `default` model (falling back to the first model).
  - If the current `composerEffort` isn't in the selected effective model's `supportedEffortLevels`, reset to `'high'` when supported, else the first level.
  The reconciler only writes to the store when a value actually needs to change ŌĆö it doesn't re-fire on every render. Because `streamStore.send` reads the reconciled values out of the same store fields, the POST body always carries a self-consistent `(backend, model, effort)` triple even when the user hadn't touched the chips since the catalog last shifted.

  **State lifecycle** ŌĆö the pickers live in `StreamStore`'s per-conv entry (`composerCliProfileId / composerBackend / composerModel / composerEffort`), not in component state, so:
  - First load of a conv hydrates the chips from `conv.cliProfileId/backend/model/effort` via `StreamStore.load`.
  - Once the user picks a value, the `composer*` fields become sticky ŌĆö switching to another conv and back doesn't rewind them.
  - On successful `POST /message`, `StreamStore.send` mirrors the sent values back onto `conv.cliProfileId/backend/model/effort` so the next hydration (or a new subscriber) sees the persisted truth.
  - `reset(convId)` / `archive(convId)` wipe the entry entirely; the next mount starts fresh from the conv record.

  Composer-level setters (`setComposerCliProfile` / `setComposerBackend` / `setComposerModel` / `setComposerEffort`) only mutate the store and never POST directly ŌĆö picker changes flush to the server with the **next** `/message`, by design. This avoids chatty PATCHes for a user who flips through the chips before typing, and leans on the `/message` endpoint's existing conditional-update behavior. The profile/backend chip is locked once the active session has messages; model and effort remain selectable for future turns.

Restored entries qualify too since the file still lives on the server. Clicking it calls `ocrAttachment(entry)` in `shell.jsx`, which delegates to `StreamStore.ocrAttachment(convId, attachmentId)`. The store flips the entry's `ocrStatus` to `'running'` (the chip swaps the icon for a small spinner via `.att-ocr-spin` and disables the button) and POSTs to `/conversations/:id/attachments/ocr` with the attachment's absolute `path`. On success the returned Markdown is cached on the entry as `ocrMarkdown` (with `ocrStatus: 'done'`); on failure `ocrStatus: 'error'` + `ocrError: msg` and a red `toast.error('OCR failed: ' + msg)` surfaces ŌĆö the composer is left untouched. Either way the result Markdown (or cached value on a re-click) is spliced into the composer at the cursor through the same shared `insertAtComposerCursor(text)` helper that dissolve uses, with caret positioned at the end of the insert. **Cache semantics** ŌĆö the cache lives on the `PendingAttachment` (composer-state, not server-persisted), so a re-click is instant with no second CLI spin-up. The button title flips to "Insert OCR Markdown (cached)" once the cache is populated. The cache is dropped with the rest of the composer state when the message ships or the conversation is left. **Original image stays put** ŌĆö OCR never removes the attachment; the user decides whether to keep the image alongside the inlined text (mixed text+diagram screenshots) or remove it (text-only screenshots) before sending. **Re-entrancy guard** ŌĆö clicking again while `ocrStatus === 'running'` throws inside the store (the button is also `disabled` in the running state, so this is defense-in-depth for keyboard activations / programmatic callers).

  `StreamStore.addAttachments(convId, files)` creates one `PendingAttachment` entry per file with a unique `id`, initial `status: 'uploading'`, `progress: 0`, and an `xhr: null` slot. Each entry then fires `AgentApi.conv.uploadFile(convId, file, onProgress, onXhr)`:
  - `onXhr(xhr)` is called synchronously at XHR creation ŌĆö the store patches `xhr` onto the entry so `removeAttachment` can abort it cleanly.
  - `onProgress(loaded, total)` updates the entry's `progress` percentage (0ŌĆō100) so the chip's spinner ring fills.
  - On resolve (server returns `{ files: AttachmentMeta[] }` ŌĆö the helper picks the first entry since we send one file per request), the store flips the entry to `status: 'done'`, sets `result` to the full typed `AttachmentMeta` (including `kind` + optional `meta` like "12 pages"), and clears `xhr`.
  - On reject with `err.aborted === true`, the store does nothing (the chip was already removed). Other errors flip the entry to `status: 'error'` with the error message stashed for the chip's `title`.

  `StreamStore.removeAttachment(convId, attachmentId)` branches on status: aborts the XHR when `uploading`, fires `AgentApi.conv.deleteUpload(convId, result.name)` when `done` (fire-and-forget ŌĆö the chip disappears from the UI regardless of the DELETE outcome), else drops the entry directly. `StreamStore.clearPendingAttachments(convId)` empties the tray without deleting the uploads ŌĆö used when the user enqueues a message so the attachments detach onto the queue entry and the server copies remain on disk.

  - `size: 'md'` ŌĆö composer tray card.   **Thumbnail rehydration safety.** `PendingAttChip`'s image-thumb effect branches on `entry.restored`. For live entries it calls `URL.createObjectURL(file)` on the in-memory `File` blob (revoked on unmount). For entries rehydrated from `localStorage` by `hydrateAttachmentsFromDraft` ŌĆö where `file` is a `{ name, size }` stub, not a `Blob` ŌĆö the effect skips `createObjectURL` entirely and falls back to the server-side URL `AgentApi.chatUrl('conversations/:id/files/:filename')` so the thumb still renders after a session-expired full reload. A `file instanceof Blob` guard covers the edge case of a restored entry that lacks the `restored` flag for any reason (defense-in-depth; never reached in practice). V1's composer got this right implicitly by omitting the `type` field on restored files, which short-circuited the legacy `f.type && f.type.startsWith('image/')` check; V2's refactor into `kindFromFile` dropped that implicit guard, so V2 uses explicit branching instead.

  **Send-time behavior (`StreamStore.send`):**
  - **Guard** ŌĆö when any `pendingAttachments` entry is `uploading`, `send()` returns early. `canSend` computes the same guard so the send button is disabled while uploads are in flight.
  - **Content mutation** ŌĆö completed attachments' absolute server paths are joined with `", "` and appended as `[Uploaded files: path1, path2, ŌĆ”]` via `composeWireContent(queuedMessage)`. If the textarea is empty but attachments are present, the tag becomes the full content (and `canSend` allows this path). The CLI still reads the files from disk ŌĆö the typed chips are a UI concern only and the wire format is unchanged.
  - **Snapshot / rollback** ŌĆö `send()` captures both the current `pendingAttachments` array AND the `input` string before the optimistic update; it clears both immediately (so the UI repaints with an empty composer and no tray) and restores both if the POST `/message` fails (including the 401/`Session expired` path). No re-upload is needed on rollback ŌĆö the server still holds the files under the conv's artifacts dir. Restoring `input` is what keeps the user's typed message visible after a mid-send session-expired 401, so they don't have to retype.
  - **Persisted draft flush before wipe** ŌĆö `send()` calls `flushDraftNow(convId)` right before the optimistic state wipe. This cancels any pending debounced `scheduleDraftSave` timer and writes the current (pre-wipe) state to `localStorage` synchronously. Without this, a timer firing between wipe and POST resolution would see `input: ''` + `pendingAttachments: []` and delete the `ac:v2:draft:<convId>` key ŌĆö which would leave the full-reload path (popup-blocked fallback for silent re-auth) with no draft to rehydrate. On POST failure `clearDraft` is *not* called (only on server-confirmed success), so the flushed copy remains the source of truth for a reload.
  - **`clearAllStreamErrors()`** ŌĆö iterates every tracked ConvState and calls `clearStreamError(convId)` on any with a non-null `streamError`. Called by the shell after a successful silent re-auth (see `spec-server-security.md` ┬¦ "V2 frontend silent re-auth") so stale "session expired" error cards disappear across every open conv the moment the session is back.

  **Message queue (`QueueStack` / `QueueRow`):** the v2 composer supports queueing follow-up messages during a live run. State mirrors the server-persisted `messageQueue` (typed `QueuedMessage[]`) via `StreamStore.queue`; the store auto-drains the head on every `done` frame when the conversation is idle (see `drainQueueIfReady` in `streamStore.js`).
  - **Enqueue paths** ŌĆö three ways to enqueue: (a) Ōīś/Ctrl + Enter in the composer textarea always enqueues, even while idle; (b) plain Enter with content while streaming enqueues; (c) clicking the composer's primary button while streaming with content in the composer enqueues the current draft and clears it. `doEnqueue()` in `shell.jsx` snapshots the composer's done `pendingAttachments` onto the queue entry's `attachments[]`, calls `StreamStore.enqueue`, then `setInput('')` + `clearPendingAttachments()`. The uploads stay on disk ŌĆö the queue entry owns their `AttachmentMeta` references.
  - **Composer primary button (dual-purpose while streaming)** ŌĆö when the agent is idle the button is the standard blue **Send** affordance (`.send`, `Ico.up`). When `streaming === true` it splits on `hasContent`: with text or done attachments it stays as the blue **Send** glyph (`.send`, `Ico.up`, title `"Queue behind current run"`) and clicking enqueues; with an empty composer it flips to the red **Stop** affordance (`.send.stop`, `Ico.stop`, title `"Stop agent"`) and clicking calls `StreamStore.stopStream(convId)`. `stopStream` posts `POST /api/chat/conversations/:id/abort` through `AgentApi.abortConversation`, so Stop works even if the browser WebSocket is disconnected. On `{ aborted:true }`, the store immediately applies a local `'Aborted by user'` cancellation state (`streamErrorSource:'abort'`, `streaming:false`, `uiState:'error'`, assistant placeholder removed) so the UI unsticks even if the follow-up refetch fails; the refetch then replaces optimistic placeholders with the server's durable abort messages when available. Abort notices render as non-retryable cancellations: **Retry** is hidden, **Dismiss** returns the conversation to idle, and the user can send a new message normally. If the refetch succeeds but does not contain a final persisted `streamError`, the local abort error still keeps queue draining paused. Replayed terminal abort frames are deduped by message id if they arrive after the refetch. On `{ aborted:false }`, the store refetches the conversation because the server stream already ended naturally. No-op when the conv isn't streaming. Restores parity with V1's `chatStopStreaming()` behavior without making Stop depend on browser transport.
  - **`canSend` / `canEnqueue` guards** ŌĆö `canSend = hasContent && !sending && !streaming && !awaiting && !hasUploadingFiles`, `canEnqueue = hasContent && !sending && !awaiting && !hasUploadingFiles && streaming`. The stop button has no separate guard ŌĆö it's enabled whenever the streaming branch renders it (which only happens with `streaming && !hasContent`). "Content" means non-empty textarea **or** at least one done attachment.
  - **`QueueStack`** renders below the tray (and above `.composer-foot`) when `queue.length > 0`. Header row shows "Queued (N)" + a "Clear all" link. Each entry maps to a `QueueRow`.
  - **`QueueRow` (collapsed)** ŌĆö compact two-line layout: top row shows the 1-based position badge (`.qrow-num`), the message text truncated to 2 lines, and the action buttons (Ōåæ/Ōåō reorder, Ō£Ä edit, ├Ś remove). Upload failures render inline above the footer. Only one row may be expanded at a time; if the auto-drain pops the expanded index, the `QueueStack` collapses back to the list.
    - **Paste-to-attachment (parity with composer)** ŌĆö the editor's textarea has its own `onPaste` handler that mirrors the main composer's behavior: file-kind clipboard items are renamed `<basename>-<ts>-<n>.<ext>` and uploaded; text pastes Ōēź 1000 chars synthesise `pasted-text-YYYYMMDD-HHMMSS.txt` and upload it. Failures surface in the inline error area like file-picker uploads.
    - **Dissolve (within-session)** ŌĆö when paste-to-attachment fires inside the editor, the original pasted text is also stashed in a `dissolveMapRef` (`Map<attachmentName, originalText>`) keyed by the server-returned `result.name`. Chips whose name appears in that map render the same `.att-dissolve` button as the composer; clicking it splices the recovered text into the editor's textarea at the current cursor (or appends if no selection), removes the chip from `atts` (and from the dissolve map), and refocuses the textarea with the caret at the end of the splice. The same > 50,000-char `dialog.confirm` gate applies. **Lifetime** ŌĆö the dissolve map is component state, so it lasts only until the editor row is closed (Cancel / Save / collapse). The `removeAtt` helper clears the matching dissolve-map entry whenever an attachment is dropped, dissolve or otherwise.
  - **Persistence** ŌĆö every mutator (`enqueue`, `removeFromQueue`, `reorderQueue`, `updateQueueItem`) optimistically updates `state.queue` then schedules a full-queue PUT; `clearQueue` schedules DELETE. Queue persistence is serialized per conversation with a single pending snapshot: if another mutation occurs while a PUT/DELETE is in flight, the pending request is replaced with the latest queue shape (or latest clear), and the drainer sends that newest request after the current request settles. Client-only `inFlight` markers are stripped before PUT. This guarantees the final server queue matches the latest local queue even when auto-drain success/rollback persistence overlaps with user edits.
  - **Auto-drain on `done`** ŌĆö when a stream completes with no `pendingInteraction` and no `streamError`, the store marks `queue[0]` as in-flight and composes the entry's wire-format content via `composeWireContent` (content + `[Uploaded files: ŌĆ”]` tag rebuilt from `attachments[]`). The content is sent via the internal `sendWireContent(convId, queuedMessage)` helper (bypasses the pendingAttachments pipeline since attachments were already uploaded at enqueue time). The server-side queue is shortened only after `/message` accepts the send; on success the follow-up PUT persists the current non-in-flight queue tail, not the stale tail captured at drain start. If `/message` returns 409 because the server still owns an active stream, or the send fails before the POST can start, the original queued item, including attachments, is restored at the head and the restored queue is PUT back to the server. Rollback merges that original head with the current non-in-flight queue tail, so edits to later queued items made while the head was in flight are preserved locally and across reloads.

  **Backwards compatibility (historical transcripts):** messages saved before the typed queue existed may carry `[Uploaded files: /abs/path1, /abs/path2]` trailing text on `Message.content` (user messages). The renderer feeds these through `StreamStore.parseUploadedFilesTag(content)`, which strips the tag and returns `{ content, attachments }` where each `AttachmentMeta` has `name` + `path` + `kind` (inferred from extension via `attachmentKindFromPath`). `size`/`meta` are unavailable for legacy entries ŌĆö the `md` chip shows only the filename and kind badge in that case. Users receive the same typed-chip UX for old transcripts at read-time; the on-disk bytes are untouched.
