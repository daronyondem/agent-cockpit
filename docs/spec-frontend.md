## V2 ‚Äî Default Frontend

**Paths:**
- `/` ‚Äî redirects to `/v2/` (handled in `server.ts`).
- `/v2/` ‚Äî real app entry (`public/v2/index.html` ‚Üí `shell.jsx`). The default UI.
- `/v2/deck.html` ‚Äî static mockup slide deck (all 16 screens from the design handoff, navigable with ‚Üê/‚Üí). Preserved during the rewrite so the target design can be referenced in-place; driven by `public/v2/src/app2.jsx`.
- `/legacy/` ‚Äî redirects to `/index.html` (the V1 UI ‚Äî see [¬ß Legacy UI (V1)](#legacy-ui-v1) below).

V2 is the production UI. The legacy vanilla-JS UI at `/index.html` is preserved as a fallback during the cutover and will be removed once V2 proves itself; `/v2/deck.html` keeps the full design deck around as reference. All paths are behind `requireAuth`. The V2 bundle is a React 18 + Babel Standalone prototype (no build step ‚Äî see the production build issue on GitHub). Source tree:

- `public/v2/index.html` ‚Äî real app entry. Loads React/ReactDOM/Babel, `marked` (markdown parser), and `DOMPurify` (HTML sanitizer) from unpkg, the four stylesheets (`tokens.css`, `app.css`, `dialog.css`, `tooltip.css`), then `api.js`, `streamStore.js`, `planUsageStore.js` (Claude Code account plan usage singleton ‚Äî see below), `kiroPlanUsageStore.js` (Kiro account plan usage singleton ‚Äî see below), `tabIndicator.js` (plain JS, self-initialising ‚Äî mounts the browser-tab favicon-dot subscriber on top of `StreamStore.subscribeGlobal`), `icons.jsx`, `dialog.jsx`, `toast.jsx`, `tooltip.jsx`, `chip-renderers.jsx`, `primitives.jsx`, `screens/kbBrowser.jsx` (the real KB Browser), `screens/filesBrowser.jsx` (the real Files browser), `screens/settingsScreen.jsx` (the real Settings screen), `folderPicker.jsx`, `workspaceSettings.jsx`, `sessionsModal.jsx`, `updateModal.jsx`, and `shell.jsx`. Load order matters: `dialog.jsx` must come after `icons.jsx` (uses `Ico.*`) and before `shell.jsx` / the screens (which call `useDialog`); `toast.jsx` must come before `shell.jsx` and the screens that call `useToasts`; `tooltip.jsx` must come before `primitives.jsx` / `shell.jsx` / the screens (which may mount `<Tip>`); `settingsScreen.jsx` must come before `shell.jsx` because shell references `<SettingsScreen>`; `updateModal.jsx` must come before `shell.jsx` because shell references `<UpdateModal>` and `<RestartOverlay>`; `workspaceSettings.jsx` must come before `shell.jsx` because shell references `<WorkspaceSettingsModal>`; `sessionsModal.jsx` must come before `shell.jsx` because `ChatLive` references `<SessionsModal>`. Does **not** load the mock deck screens (`screens/kb.jsx`, `screens/files.jsx`, and the other `screens/*.jsx` stubs) ‚Äî those live only in `deck.html`. Each script/stylesheet URL carries a `?v=N` cache-buster ‚Äî bump when any of the imported JSX/JS/CSS files change so browsers fetch the new version instead of a cached copy.
- `public/v2/deck.html` ‚Äî slide deck entry. Loads the same base bundle plus `screens/*.jsx` and `app2.jsx` (the `SLIDES` scaffold + `Frame`/`Tweaks` overlay).
- `public/v2/src/tokens.css`, `public/v2/src/app.css`, `public/v2/src/dialog.css`, `public/v2/src/tooltip.css` ‚Äî design tokens + cockpit chrome styles + anchored-popover dialog styles + tooltip primitive styles. Driven by `data-theme` / `data-density` / `data-radius` / `data-direction` attributes on the root element.
- `public/v2/src/api.js` ‚Äî CSRF-aware fetch helper exposed as `window.AgentApi` (`apiUrl`, `chatUrl`, `chatWsUrl`, `fetch`, `listConversations`, `createConversation`, `restoreConversation`, `getVersion`, `getUpdateStatus`, `checkVersion`, `triggerUpdate`, `browseDir`, `mkdirDir`, `rmdirDir`, `getBackendsCached`, `setSessionExpiredHandler`, `kb`, `explorer`, `settings`, `conv`). The conversation list is no longer cached at the API layer ‚Äî `StreamStore.getConvList()`/`subscribeConvList()`/`loadConvList()` own that state instead, so callers that need the list subscribe to the store rather than calling a cache helper. The four version helpers back the sidebar version indicator: `getVersion()` fires `GET /api/chat/version` (returns `{ version, remoteVersion, updateAvailable }` ‚Äî the current server version plus whatever remote/update flags the background updater has cached), `getUpdateStatus()` fires `GET /api/chat/update-status` (read-only poll ‚Äî safe to run every 5 min), `checkVersion()` fires `POST /api/chat/check-version` (forces a fresh upstream `git ls-remote`), and `triggerUpdate()` fires `POST /api/chat/update-trigger` (the server pulls, installs, and pm2-restarts; the in-flight fetch may reject with `TypeError: Failed to fetch` because the process dies mid-request ‚Äî the UI treats that shape as success, same as V1). `restoreConversation(id)` fires `PATCH /api/chat/conversations/:id/restore` and is the inverse of `StreamStore.archive(convId)` ‚Äî used by the sidebar's archive-view per-row restore button. `AgentApi.conv` bundles the per-conversation helpers driving the composer attachment chips and the Sessions modal: `uploadFile(convId, file, onProgress?, onXhr?)` (XHR-based single-file upload to `POST /api/chat/conversations/:id/upload` ‚Äî `onProgress(loaded, total)` feeds the chip progress bar, `onXhr(xhr)` hands the caller the raw XHR so it can abort in-flight uploads when the user removes the chip, and the promise resolves with the first entry from the server's `{ files: [{name, path, size}] }` payload), `deleteUpload(convId, filename)` (`DELETE /api/chat/conversations/:id/upload/:filename` ‚Äî fired when the user removes an already-uploaded chip), `getSessions(convId)` (`GET /api/chat/conversations/:id/sessions` ‚Üí `{ sessions: SessionHistoryItem[] }` ‚Äî list of past sessions with `number` / `startedAt` / `endedAt` / `messageCount` / `summary` / `isCurrent`; drives the Sessions modal's list view), `getSessionMessages(convId, sessionNumber)` (`GET /api/chat/conversations/:id/sessions/:num/messages` ‚Üí `{ messages: Message[] }` ‚Äî persisted messages for a past session; the Sessions modal detail view skips this fetch and reuses the in-memory `currentMessages` when the user opens the session they're already viewing), and `sessionDownloadUrl(convId, sessionNumber)` (URL builder only ‚Äî pointed at `GET /api/chat/conversations/:id/sessions/:num/download` for `window.open`, the server returns markdown with `Content-Disposition: attachment`). `getClaudePlanUsage()` fires `GET /api/chat/plan-usage` and resolves to the cached snapshot `{ fetchedAt, planTier, subscriptionType, rateLimits, lastError, stale }` ‚Äî consumed by `window.PlanUsageStore` (see below). The endpoint never forces a refresh; the server throttles the actual Anthropic `/api/oauth/usage` fetch at one every 10 min behind the route. `getKiroPlanUsage()` fires `GET /api/chat/kiro-plan-usage` and resolves to the parallel cached snapshot `{ fetchedAt, usage, lastError, stale }` ‚Äî consumed by `window.KiroPlanUsageStore` (see below). Same 10-min server-side throttle and same never-force-refresh contract, but the underlying call is a direct JSON-1.0 POST to `AmazonCodeWhispererService.GetUsageLimits` using the bearer token the server reads read-only from the kiro-cli SQLite auth store. `getBackendsCached()` resolves to the `BackendMetadata[]` array from `GET /api/chat/backends` (the registry is effectively static so one fetch per session is plenty; every `<ChatLive>` mount consults it for the composer pickers). `chatWsUrl(convId)` derives the WebSocket URL from the same base as REST (same protocol swap ws‚Üîwss), so reverse-proxy deployments that serve `/v2/` under a subpath pick up the prefix automatically. `AgentApi.kb` bundles the KB endpoint helpers consumed by the v2 KB Browser. Reads: `getState(hash, {folder, limit, offset})` (extra params optional ‚Äî defaults to root folder, 500-row page), `getEntries(hash, {search, tags, limit, offset, folder, rawId, ...})`, `getEntry(hash, entryId)`, `getTags(hash)`, `getSynthesis(hash)`, `getTopic(hash, topicId)`, `getReflections(hash)`, `getReflection(hash, reflectionId)`, `rawDownloadUrl(hash, rawId)` (URL builder only, no fetch), `rawMediaUrl(hash, rawId, mediaPath)` (URL builder for the per-raw media endpoint ‚Äî splits `mediaPath` on `/`, `encodeURIComponent`s each segment, and rejoins so embedded slashes stay structural), and `pandocStatus()` (top-level `/api/chat/kb/pandoc-status` ‚Äî not workspace-scoped ‚Äî drives the install banner above the Raw tab), and `libreOfficeStatus()` (top-level `/api/chat/kb/libreoffice-status` ‚Üí `{ available: boolean, version?: string }`; consumed by the Settings screen's KB tab to gate the "Convert PPTX slides to images" toggle and to surface a status pill). Mutations: `setAutoDigest(hash, enabled)` (`PUT /kb/auto-digest`), `digestRaw(hash, rawId)` (`POST /kb/raw/:rawId/digest`), `digestAll(hash)` (`POST /kb/digest-all`), `deleteRaw(hash, rawId, folder?, filename?)` (when both `folder` and `filename` are passed it deletes only that location, else fully purges the raw), and `uploadRaw(hash, file, folder?, onProgress?)` (XHR-based single-file upload that resolves with the server's `{ entry, deduped, addedLocation }` payload and surfaces progress via the `onProgress(loaded, total)` callback; mirrors the `explorer.upload` pattern), `createFolder(hash, folderPath)` (`POST /kb/folders`, idempotent), `renameFolder(hash, fromPath, toPath)` (`PUT /kb/folders`), `deleteFolder(hash, folderPath, cascade)` (`DELETE /kb/folders?folder=‚Ä¶&cascade=‚Ä¶`; `cascade` is always passed `true` from the V2 UI because the confirm copy already warns the user about nested content), `getEmbeddingConfig(hash)` (`GET /kb/embedding-config` ‚Üí `{ embeddingConfig: {model?, ollamaHost?, dimensions?} | null }`), `setEmbeddingConfig(hash, cfg)` (`PUT /kb/embedding-config` with `{ model, ollamaHost, dimensions }` ‚Üí `{ embeddingConfig: <result> }`), and `embeddingHealth(hash)` (`POST /kb/embedding-health` ‚Üí `{ ok: boolean, error?: string }`; runs server-side via `checkOllamaHealth(cfg)`). `AgentApi.explorer` bundles the workspace file explorer helpers consumed by the v2 Files Browser: `tree(hash, relPath)`, `preview(hash, relPath)` (always `mode=view`), `rawUrl(hash, relPath)` / `downloadUrl(hash, relPath)` (URL builders for `mode=raw` image tags and `mode=download` links), `mkdir(hash, parent, name)`, `createFile(hash, parent, name, content)`, `saveFile(hash, relPath, content)`, `rename(hash, from, to, overwrite)`, `deleteEntry(hash, relPath)`, and `upload(hash, destPath, file, overwrite, onProgress)` (XHR-based single-file upload that resolves with the server's `{ ok, entry }` payload and surfaces progress via the `onProgress(loaded, total)` callback). `AgentApi.settings` bundles the global app-settings helpers consumed by the v2 Settings screen: `get()` (`GET /settings` ‚Üí full `Settings` object with all defaults filled in), `save(settings)` (`PUT /settings` with the full merged shape ‚Äî server returns the persisted result), `backends()` (`GET /backends` ‚Üí `{ backends: BackendMetadata[] }` with nested `models[].supportedEffortLevels`), `usageStats()` (`GET /usage-stats` ‚Üí `{ days: [{ date, records: [{ backend, model, usage }] }] }`), `clearUsageStats()` (`DELETE /usage-stats`), and `restartServer()` (`POST /server/restart` ‚Äî server replies 200 on success or 409 if any conversation stream is currently active). Errors include a `.status` field on the thrown `Error` so the UI can branch on 409 conflicts without re-parsing the response body; the 401 path still triggers the session-expired handler. Each JSON helper JSON-parses the response. `AgentApi.workspace` bundles the per-workspace settings helpers consumed by the Workspace Settings modal: `getInstructions(hash)` (`GET /workspaces/:hash/instructions` ‚Üí `{ instructions }`), `saveInstructions(hash, instructions)` (`PUT /workspaces/:hash/instructions`), `getMemory(hash)` (`GET /workspaces/:hash/memory` ‚Üí `{ enabled, snapshot: { capturedAt, files: [{ filename, type, name, description, content }] } | null }`), `setMemoryEnabled(hash, enabled)` (`PUT /workspaces/:hash/memory/enabled` with `{ enabled }`), `deleteMemoryEntry(hash, relPath)` (`DELETE /workspaces/:hash/memory/entries/:relPath` ‚Äî server broadcasts a `memory_update` WS event to any connected streams so open memory panels refresh in lockstep), `clearMemory(hash)` (`DELETE /workspaces/:hash/memory/entries` ‚Äî same WS broadcast, but with `changedFiles: []`), `getKb(hash)` (`GET /workspaces/:hash/kb` ‚Üí `{ enabled, state }` ‚Äî the modal only reads `enabled`, the full snapshot drives the KB Browser), and `setKbEnabled(hash, enabled)` (`PUT /workspaces/:hash/kb/enabled`). All helpers go through `chatFetch` so CSRF + session-expired handling stays consistent.
- `public/v2/src/streamStore.js` ‚Äî per-conversation stream store exposed as `window.StreamStore`. Holds one `ConvState` entry per conversation the user has visited in the current tab (`{ convId, conv, messages, input, sending, streaming, streamError, streamingMsgId, usage, loaded, loadError, ws, wsOpening, uiState, pendingInteraction, respondPending, composerBackend, composerModel, composerEffort, pendingAttachments, queue, resetting }`) with its own WebSocket, accumulated messages, and streaming flags. `uiState` is one of `'streaming' | 'awaiting' | 'error' | null`; `pendingInteraction` is either `{ type: 'planApproval', planContent }` or `{ type: 'userQuestion', question, options }` or `null`. The three `composer*` fields hold the user's picker selections ‚Äî hydrated from the conv record on first load (`data.backend/model/effort`) and sticky once the user touches them, so flipping convs and back doesn't rewind the pickers. `pendingAttachments` is an array of `PendingAttachment` entries (`{ id, file, status: 'uploading'|'done'|'error', progress, result: AttachmentMeta | null, xhr, error }`) driving the composer's attachment tray ‚Äî each entry tracks one file's lifecycle from user-drop through server-upload to final send. `result` holds the typed `AttachmentMeta` returned by `POST /upload` (including server-inferred `kind` and optional `meta` sublabel like "12 pages") so the chip can render without a second fetch. The tray clears on successful POST `/message` and is restored on POST failure. `queue` is the client mirror of the persisted `messageQueue` (array of `QueuedMessage` ‚Äî see `spec-data-models.md`); the store auto-drains the queue head on every `done` frame when the conversation is idle (no `pendingInteraction`, no `streamError`). Public surface: `getState(convId)`, `load(convId)`, `ensureWsOpen(convId)`, `send(convId, text)`, `respond(convId, text)` (delivers an answer to the pending plan/question interaction ‚Äî see *Plan approval & clarifying questions* below), `setInput(convId, value)`, `setComposerBackend(convId, value)` / `setComposerModel(convId, value)` / `setComposerEffort(convId, value)` (composer picker setters ‚Äî values flush to the server with the next POST `/message`, not immediately), `addAttachments(convId, files)` / `removeAttachment(convId, attachmentId)` / `clearPendingAttachments(convId)` (tray setters ‚Äî `addAttachments` kicks off one XHR upload per file, `removeAttachment` aborts in-flight uploads and `DELETE`s already-uploaded ones from the artifacts dir, `clearPendingAttachments` empties the tray without deleting the uploads ‚Äî used when the user enqueues a message and the attachments are detached onto the queue entry). Queue helpers: `loadQueue(convId)` (GET `/queue`), `enqueue(convId, content, attachments)` (optimistic append + PUT), `removeFromQueue(convId, index)`, `reorderQueue(convId, fromIndex, toIndex)`, `updateQueueItem(convId, index, patch)`, `clearQueue(convId)` (DELETE). Backwards-compat helpers exposed for legacy transcript rendering: `parseUploadedFilesTag(content)` (returns `{ content, attachments } | null`), `attachmentKindFromPath(path)`. Lifecycle: `reset(convId) ‚Üí Promise<boolean>` (sets `state.resetting = true` for the duration of the call so `ChatLive` can surface an in-feed "Archiving session‚Ä¶" progress bubble ‚Äî the server-side reset archives the session, captures memory, and re-initialises which can take several seconds; resolves `true` on success so callers can fire a success toast, `false` when the conv is streaming/sending/already-resetting or the POST fails), `archive(convId)`, `closeWs(convId)`, `subscribe(convId, listener)` (returns unsubscribe), `subscribeGlobal(listener)` (fires whenever any conv's `uiState` transitions ‚Äî used by the sidebar), `convStates()` (returns the compact `{ [convId]: 'streaming' | 'awaiting' | 'error' }` map consumed by `<Sidebar>`). React views read these through the `useConversationState(convId)` / `useConvStates()` hooks in `shell.jsx`. Because state lives *above* the `<ChatLive>` view, switching conversations does **not** abort an in-flight stream: the prior conv keeps collecting frames, the sidebar lights up every row whose `uiState` is non-null, and switching back to the conv immediately shows the accumulated messages. WebSockets are opened lazily on first visit and left open for the session (the server's 60 s grace buffer + replay handles transient drops); `archive(convId)` is the only path that closes the WS and removes the entry.
- `public/v2/src/planUsageStore.js` ‚Äî Claude Code account plan usage singleton exposed as `window.PlanUsageStore`. Plain JS (not JSX), loaded after `streamStore.js` and before `tabIndicator.js`. Holds one `cached` snapshot (`{ fetchedAt, planTier, subscriptionType, rateLimits, lastError, stale } | null`) and a `Set<Fn>` of subscribers. Public surface: `get()` (returns the cached snapshot or `null` before the first fetch), `refresh()` (coalesces concurrent calls ‚Äî returns the shared in-flight promise; on resolution updates `cached` and fans out to every subscriber via `subscriber(cached)`; preserves `cached` on fetch failure so the UI can keep rendering the prior snapshot), `subscribe(fn)` (returns an unsubscribe). The fetch goes through `AgentApi.getClaudePlanUsage()`; the server-side 10-min throttle behind `/api/chat/plan-usage` means calling `refresh()` on every chip mount + every Claude Code `done` frame is safe and cheap. `streamStore.js` calls `window.PlanUsageStore.refresh()` from the `done` frame handler whenever the completing conversation's backend is `claude-code`; other backends never trigger a refresh. The `ContextChip` in `shell.jsx` is the only UI consumer ‚Äî it subscribes on mount (only for Claude Code conversations) and forwards the snapshot into `ClaudeCodeTokenCard` via the renderer registry's opts bag.
- `public/v2/src/kiroPlanUsageStore.js` ‚Äî Kiro account plan usage singleton exposed as `window.KiroPlanUsageStore`. Plain JS (not JSX), loaded right after `planUsageStore.js` and before `tabIndicator.js`. Mirrors `planUsageStore.js` shape: holds one `cached` snapshot (`{ fetchedAt, usage, lastError, stale } | null`) and a `Set<Fn>` of subscribers. Public surface: `get()` (returns the cached snapshot or `null` before the first fetch), `refresh()` (coalesces concurrent calls ‚Äî returns the shared in-flight promise; on resolution updates `cached` and fans out to every subscriber via `subscriber(cached)`; preserves `cached` on fetch failure so the UI keeps rendering the prior snapshot), `subscribe(fn)` (returns an unsubscribe). The fetch goes through `AgentApi.getKiroPlanUsage()`; the server-side 10-min throttle behind `/api/chat/kiro-plan-usage` means calling `refresh()` on every chip mount + every Kiro `done` frame is safe and cheap. `streamStore.js` calls `window.KiroPlanUsageStore.refresh()` from the `done` frame handler whenever the completing conversation's backend is `kiro`; other backends never trigger a refresh. The `ContextChip` in `shell.jsx` subscribes on mount (only for Kiro conversations) and forwards the snapshot into `KiroTokenCard` via the renderer registry's opts bag.
- `public/v2/src/tabIndicator.js` ‚Äî browser-tab favicon-dot indicator for v2. Plain `<script>` (not JSX) loaded after `streamStore.js` and before the Babel-transpiled scripts; exposes `window.TabIndicator.init()` and auto-initialises in its IIFE. Reads state from `StreamStore.convStates()` via `StreamStore.subscribeGlobal`; the semantics match the V1 module described under *Browser Tab Status Indicator* below (identical four-state machine, identical favicon canvas rendering ‚Äî 64 px canvas, 22 % dot radius, white halo, same blue/green/red palette ‚Äî with the same sticky error flag and `visibilitychange` auto-clear). The only source-level difference is the data source: instead of V1's `state.chatStreamingConvs` + explicit `chatTabOnStreamChange({ error })` calls, the v2 module diffs the previous and current error-ID sets from `convStates()` to detect newly-errored conversations and recomputes from scratch on every global subscription fire.
- `public/v2/src/icons.jsx` ‚Äî shared icon functions on `window.Ico` (each takes a size, returns an SVG).
- `public/v2/src/dialog.jsx` ‚Äî reusable anchored-popover dialog system. Replaces native `window.alert` / `window.confirm` / `window.prompt` across the v2 app. Exports `window.Dialog` (low-level primitive), `window.DialogProvider` (wraps the React tree, hosts the single active dialog), `window.useDialog` (hook returning `{ confirm, prompt, alert, choice }`), and a `window.DialogApi` global escape hatch for non-React modules. Each hook method returns a Promise. **Anchoring:** every method accepts an optional `anchor` HTMLElement (typically `e.currentTarget` from the trigger button); the dialog computes `getBoundingClientRect()` placement above or below with viewport-edge clamping (32 px margin) and renders a small `.dlg-arrow` pointing at the anchor. When `anchor` is `null`/`undefined`, the dialog falls back to viewport-center with `data-anchored="false"` and the arrow is omitted. Window resize/scroll re-runs the position computation. **Variants** (each picks an eyebrow label, accent color, and icon ‚Äî overridable via `eyebrow`): `confirm` (default neutral), `warn` (red, "Are you sure?"), `yesno` (accent, "Question"), `text` (neutral, prompt input), `error` (red, alert glyph), `info` (accent), `choice` (accent, radio list). Panel widths: 380 (text/info), 420 (error), 360 (choice), 400 (default). **Hook API & cancel sentinels:**
  - `dialog.confirm({ anchor, title, body, confirmLabel, cancelLabel, destructive })` ‚Üí `Promise<boolean>`. Cancel resolves `false`. `destructive: true` auto-maps to variant `'warn'` (unless an explicit `variant` is given) and renders the primary button red (`.dlg-btn.primary.danger`).
  - `dialog.prompt({ anchor, title, inputLabel, inputDefault, placeholder, confirmLabel, cancelLabel })` ‚Üí `Promise<string | null>`. Cancel resolves `null`; the input is auto-focused and pre-selected.
  - `dialog.alert({ anchor, variant, title, body, confirmLabel })` ‚Üí `Promise<undefined>`. Single-action (no Cancel button); default `confirmLabel` is `'Got it'`. Use `variant: 'error'` for failures, `'info'` (default) for advisories.
  - `dialog.choice({ anchor, title, body, choices, defaultChoice, confirmLabel })` ‚Üí `Promise<string | null>`. `choices` is `[{ id, label, hint? }]`; the resolved value is the picked `id`.
  Cancel paths: clicking the scrim, Escape, or the Cancel button. Enter (without Shift) confirms. Only one dialog is shown at a time ‚Äî opening a new one resolves the existing one with its `cancelValue` and replaces it. The `useDialog` hook throws `Error('useDialog must be used within a DialogProvider')` when called outside the provider, so `<App/>` must be wrapped in `<DialogProvider>` at the React root.
- `public/v2/src/dialog.css` ‚Äî styles for the dialog system. `.dlg-scrim` is a fixed full-viewport blurred backdrop at `z-index: 2000` with a fade-in animation; `.dlg-panel` sits on top at `z-index: 2001` with a slide-and-fade animation (`top` for below-anchor, `bottom` for above-anchor ‚Äî origin tracks placement). `.dlg-arrow` is a 12 √ó 12 rotated square positioned 6 px outside the panel's top or bottom edge depending on `data-placement`; only rendered when `data-anchored="true"`. Component classes follow the `.dlg-*` prefix (`.dlg-head`, `.dlg-eyebrow`, `.dlg-esc`, `.dlg-title`, `.dlg-body`, `.dlg-detail`, `.dlg-input-wrap`, `.dlg-label`, `.dlg-input`, `.dlg-choices`, `.dlg-choice`, `.dlg-radio`, `.dlg-foot`, `.dlg-hint`, `.dlg-btn` with `ghost` / `primary` / `primary danger` modifiers).
- `public/v2/src/toast.jsx` ‚Äî non-blocking toast notification system. Complements the anchored-popover `Dialog` system: blocking confirms/errors use dialogs, transient confirmations (save-success, reset-success, etc.) use toasts. Exports `window.ToastProvider` (wraps the React tree below `DialogProvider`; hosts the toast queue + auto-dismiss timers), `window.useToasts` (hook returning `{ info, success, warn, error, dismiss }`), and `window.ToastApi` (global escape hatch mirrored onto `window` via `useEffect` so non-React modules can fire toasts without wiring context). **Hook API:** each level method accepts either a string (becomes the toast title) or an object `{ title, message, duration }`; default `duration` is 4000 ms (pass `0` to disable auto-dismiss). Returns the numeric toast id so the caller can dismiss it early via `toast.dismiss(id)`. Toasts render as a fixed stack in the bottom-right corner (`.toast-stack` at `z-index: 1900`, `role="status"` + `aria-live="polite"`); each card has a `.toast-close` (√ó) button and slides/fades in via the `toast-in` keyframe. Variants `info | success | warn | error` recolor the 3 px left border via the status tokens (`--status-running` / `--status-done` / `--status-awaiting` / `--status-error`). **Wired call sites (success path):** `ChatLive.handleReset` in `shell.jsx` fires `toast.success('Session reset')` after `StreamStore.reset` resolves truthy; `SettingsScreen.save` fires `toast.success('Settings saved')`; `KbSettingsTab.save` fires `toast.success('Embedding configuration saved')`. Error paths in all three still use `dialog.alert` because errors warrant the blocking anchored popover with the failing button as the anchor. The CSS block lives in `app.css` at the tail (before the mobile media query); toasts sit below the dialog scrim/panel (2000/2001) so dialogs always cover them, and above the mobile hamburger (1850) and sidebar (1800).
- `public/v2/src/updateModal.jsx` ‚Äî `window.UpdateModal` + `window.RestartOverlay`. `UpdateModal` is the Update-Now dialog triggered from the sidebar version indicator's amber "v{remoteVersion} available" badge.   **Per-conversation state lift (PR 4b)** ‚Äî `App` reads the sidebar's per-row indicator map from `useConvStates()`, which is a thin subscription over `StreamStore.subscribeGlobal` (fires when any conv's `uiState`, `unread`, or first-touch transitions). Every visited conv has an entry in the store and contributes to `StreamStore.convStates()`, so multiple rows can light up simultaneously ‚Äî e.g. the user can kick off a stream in conv A, switch to conv B, kick off a second stream there, and both rows show `.streaming` in the sidebar while only one `<ChatLive>` is mounted. The store flips the entry's `uiState` to `'streaming'` on send, `'error'` on stream error, and `null` on `done`. `convStates()` also surfaces `'unread'` for entries with the unread flag set, and `'idle'` as a sentinel for any touched entry that is currently neither active-state nor unread (used by the sidebar to override the stale `c.unread` it may still be holding from the cached server list ‚Äî see *Unread state* below).

  **Unread state** ‚Äî a conv is "unread" when a response stream completed on it while the user was viewing a different conv. Tracked in three places that all stay in sync:
  - **Server (authoritative)** ‚Äî `unread: boolean | undefined` on the workspace-index entry. Toggled via `PATCH /api/chat/conversations/:id/unread` (see API spec). Persisted across reload.
  - **Server-cached list** ‚Äî every `conversations` summary returned by `AgentApi.listConversations()` carries the boolean as `c.unread`. The sidebar uses this for cold rows (rows the user hasn't visited this session, so there's no `StreamStore` entry for them).
  - **`StreamStore` (per-conv, in-memory)** ‚Äî `ConvState.unread` (`boolean`) on every visited conv. The sidebar uses `StreamStore.convStates()` for any conv that has an entry, falling back to `c.unread` only when no entry exists. This split is what lets the sidebar clear the dot **immediately** on selection without waiting for a `/conversations` refetch.

  **Auto-mark on done** ‚Äî `StreamStore.handleFrame`'s `done` branch flags `unread = true` on the target conv whenever `convId !== activeConvId` and the stream landed in idle (no `pendingInteraction`, no `streamError`). The shell pushes the active id into the store via `StreamStore.setActiveConvId(id)` from a `useEffect` on `activeConvId`, so the store always knows which conv (if any) is currently visible. The flag is mirrored to the server via `AgentApi.markConversationUnread(convId, true)` (fire-and-forget ‚Äî failures are intentionally swallowed; the next `/conversations` fetch reconciles).

  **Manual mark / clear** ‚Äî selecting a conv (`onSelectConv` in `shell.jsx`) calls `StreamStore.markRead(id)`, which always touches local state (creating a blank entry on cold convs so `convStates()` returns `'idle'` to override `c.unread`) and POSTs `unread: false`. The user can left-click the dot on any non-active, non-active-state row to manually re-flag the conv as unread; the sidebar passes the click through `onMarkUnread` ‚Üí `StreamStore.markUnread(id)`. The dot click uses `event.stopPropagation()` so it doesn't also select the row.

  **Sidebar render priority** ‚Äî for each row, `Sidebar` resolves the dot like this: live state (`'streaming'` / `'awaiting'` / `'error'`) wins; otherwise unread is derived from `state === 'unread'` (touched, currently unread) or `state == null && c.unread === true` (untouched fallback to server). `state === 'idle'` (touched, currently read) wipes the dot to neutral. The unread dot is rendered as a `<button class="dot dot-btn">` only when the row is non-active and not in any other state ‚Äî otherwise it's a non-interactive `<span class="dot">`. The CSS rule `.sb-row.unread .dot { background: var(--text); }` paints a theme-adaptive ink dot (near-black on light themes, near-white on dark).

  **Send flow (all logic lives in `StreamStore.send(convId, text)`; the view just invokes it):**
  1. Ensure the conv's WS is open (`ensureWsOpen` resolves the pending open promise or opens a new socket; a 5 s connect timeout is surfaced as an error).
  2. Optimistically append a pending user message and an empty pending-assistant placeholder (with an empty `contentBlocks` array); mark the placeholder's id as the entry's `streamingMsgId`; flip `uiState` to `'streaming'`.
  3. Before building the POST body, gather all completed file attachments (`pendingFiles` entries whose `status === 'done'`); when any exist, append `[Uploaded files: <abs-path-1>, <abs-path-2>, ‚Ä¶]` to the content (on a fresh line separated by `\n\n`; the tag alone if the textarea was empty). The attachments were already uploaded to the conv's artifacts dir by `addFiles` ‚Äî the server paths are what the LLM sees, and the message body is their only channel. Snapshot `pendingFiles` before clearing so a POST failure can restore the chips intact (the files are still on disk server-side, so there's no re-upload). `POST /api/chat/conversations/:id/message` with `{ content, backend?, model?, effort? }` via `AgentApi.fetch` (CSRF-token header is attached automatically; a 401 triggers the registered session-expired handler). The optional `backend/model/effort` fields ride along only when the composer pickers have a value (either hydrated from the conv record or touched by the user). The server-side route (see Chat Routes spec) conditionally updates the conversation row when any of the three differs from what's currently stored, so the chosen picker values become the new defaults for the next turn. On success, swap the pending user message for the server-authoritative `userMessage` returned in the response and mirror the picker values back onto the entry's `conv.backend/model/effort` so future renders see the persisted truth.
  4. WebSocket frame handling runs inside `StreamStore.handleFrame(convId, frame)` ‚Äî every frame updates the target conv's entry regardless of which view is currently mounted:
     - `text` ‚Äî `{ content }` delta appended to the streaming placeholder's ordered `contentBlocks`. Consecutive `text` deltas merge into the last text block so one contiguous paragraph produces one block.
     - `thinking` ‚Äî `{ content }` delta appended as a `thinking` block (same merge rule as text).
     - `tool_activity` ‚Äî pushes a new `{ type: 'tool', activity: { tool, description, id, duration: null, startTime, isAgent?, subagentType?, parentAgentId? } }` block onto `contentBlocks`. Plan-mode markers (`isPlanMode`) and clarifying-question frames (`isQuestion`) are *not* pushed as tool blocks; instead they update `pendingInteraction` + `uiState='awaiting'` (see *Plan approval & clarifying questions* below).
     - `tool_outcomes` ‚Äî patches the matching tool block(s) in the streaming placeholder by `activity.id === toolUseId`, setting `outcome` and `status`.
     - `assistant_message` ‚Äî authoritative persisted message; replaces the placeholder **and clears `streamingMsgId`** so the next delta opens a fresh placeholder for the next segment. This is the *real* segment boundary on the server (the server only emits `assistant_message` when it just persisted an intermediate/final message). Intermediate segments land with `turn: 'progress'`, the final segment with `turn: 'final'`. The feed render collapses runs of consecutive `turn: 'progress'` assistant messages into a progress breadcrumb (see *Progress-breadcrumb collapse* below).
     - `turn_complete` ‚Äî **explicit no-op** (documented branch in `handleFrame`, not a silent drop). The server emits `turn_complete` right after every `turn_boundary` CLI event, which V1 uses to archive `activeTools`‚Üí`toolHistory` and stop a shared activity timer. V2 doesn't need that archival: tools live as `contentBlocks[]` on the streaming assistant message and `tool_outcomes` mutates them in place, turn splits already happen on `assistant_message` (placeholder swap + `streamingMsgId = null`), and live activity timers are per-block and self-clear when `duration` lands. Kiro never emits `turn_boundary`, so this frame is Claude-Code-only; the Kiro adapter yields `done` directly from its `turn_end` handler.
     - `replay_start` ‚Äî wipes the streaming placeholder's `contentBlocks` (and `content`, and any `pendingInteraction`/`planModeActive`) so the server's buffered frames can rebuild state cleanly on reconnection. Keeps `streamingMsgId` intact so replayed `text`/`tool_activity` frames land on the same placeholder message id. No-op when `streamingMsgId` is null (nothing to wipe). Backend-agnostic: the server's per-conv ring buffer (`ws.ts`, 1000-event cap, 60 s disconnect grace) captures every frame regardless of adapter, so replay affects both Claude Code and Kiro conversations identically. Without this wipe, navigating away from and back to an in-progress conversation would duplicate text + tool rows in the placeholder until a live `assistant_message` overwrote it.
     - `replay_end` ‚Äî explicit no-op. State was rebuilt by the normal handlers during replay; React subscribers already re-render on each update, so there's no additional reconcile step to run. V1 uses this frame to trigger a DOM re-render and re-show pending interaction dialogs, but V2's declarative view layer handles that implicitly.
     - `title_updated` ‚Äî updates the entry's `conv.title` in-place and invalidates the cached conversations list so the sidebar re-fetches on next mount.
     - `usage` ‚Äî refreshes the entry's `usage` (and therefore the topbar context chip when the conv is active).
     - `error` ‚Äî flips `streamError` and `uiState` to `'error'`, clears `pendingInteraction` (a dead stream can't accept an answer).
     - `done` ‚Äî clears `streaming` and `streamingMsgId`; `uiState` resolves to `'error'` if one already landed, `'awaiting'` if `pendingInteraction` is still set, else `null`. `pendingInteraction` itself is preserved across `done` so the user can still answer after the stream's finally block runs. When the stream lands in idle (no `streamError`, no `pendingInteraction`) and `convId !== activeConvId`, the entry's `unread` is also flipped to `true` and `AgentApi.markConversationUnread(convId, true)` fires (see *Unread state* below).
  5. On POST failure, the pending user and assistant placeholders are rolled back and the error is shown.

  **Assistant body rendering (`AssistantBody` ‚Üí `groupBlocksForRender` ‚Üí `renderSegment`):** `contentBlocks` is authoritative when present. For legacy messages saved before the field existed, a fallback in `deriveBlocks` synthesizes blocks from the message's `thinking`/`toolActivity`/`content` fields (thinking first, then tools, then the text body) so older conversations render without special cases. `groupBlocksForRender` then walks the blocks and splits them into three kinds of segments: `text`, `thinking`, and `tool-run` (consecutive tool blocks merge into one run so they can be grouped as a unit). Each segment renders as:
     - `text` ‚Üí `.prose` container with the markdown pipeline (`marked.parse` + `DOMPurify.sanitize`). Fenced code blocks land as `<pre><code>‚Ä¶</code></pre>`; `.prose pre` renders them as a monospace panel with `overflow-x: auto`, `white-space: pre`, and the `--surface-2` background so the block is visually distinct and selectable/copy-able.
     - `thinking` ‚Üí `ThinkingBlock` ‚Äî a dashed-border pill button that, when clicked, toggles a below-pill panel showing the raw thinking text (`white-space: pre-wrap`, dimmed). The collapsed label shows a short preview (first 80 chars of line 1) so the user can tell segments apart at a glance.
     - `tool-run` ‚Üí `ToolRun`, which feeds its activities into `partitionToolRun` and then renders each segment via `SubagentCard` (for agents) or `ToolGroup` (for parallel/sequential clusters of non-agent tools). See *Tool-run grouping* below.

  **Tool-run grouping (`partitionToolRun` + `partitionParallel`):**
  - Cross-message subagent index (`buildAgentIndex` + `AgentIndexContext`): a subagent's `Agent` tool_use and its internal children often end up persisted on *different* messages (the Agent's accompanying "Now spawning..." text triggers an early save, then the subagent's child tool_uses accumulate on the next saved message). To reassemble the subagent card, the feed-level `AgentIndexProvider` walks the entire `messages` list before rendering and produces `{ agentIds, childrenByAgent }`: `agentIds` is the set of every `isAgent` tool ID seen, and `childrenByAgent` maps each agent ID to the flat list of child activities (from *any* message) whose `parentAgentId` points at it, in chronological order. The provider wraps the whole feed so every `ToolRun` down the tree reads the same index via `React.useContext(AgentIndexContext)`.
  - `partitionToolRun(tools, agentIndex)` walks the run in order. Activities whose `isAgent === true` become `{type:'agent'}` segments whose `children` come from `agentIndex.childrenByAgent.get(activity.id)` (partitioned themselves via `partitionParallel`). Activities whose `parentAgentId` appears in `agentIndex.agentIds` are skipped at the top level ‚Äî they render inside the parent card wherever that card lives, even if it's on a different message. Everything else accumulates into a buffer that flushes through `partitionParallel` whenever an agent interrupts (preserving the source order).
  - `partitionParallel(tools)` merges consecutive activities that share a server-assigned `batchIndex` (see Backend Services spec) into `{type:'parallel'}` clusters; activities across distinct `batchIndex` values become separate `{type:'sequential'}` clusters. The server bumps `batchIndex` on every CLI `turn_boundary` event, so consecutive tools tagged with the same index are exactly the parallel tool_uses of a single LLM assistant turn ‚Äî this is authoritative regardless of how tool execution time spaces the events. **Legacy fallback:** tools persisted before the server started tagging `batchIndex` lack the field; for those, `partitionParallel` falls back to the 500ms `PARALLEL_THRESHOLD_MS` startTime-proximity rule used by the original `chatGroupParallelItems()` helper. Single activities become a single-item `sequential` group.
  - `ToolGroup` renders the cluster as a `.tools` card. Parallel clusters add the `.parallel` modifier (left rail) and always show a `.tools-head` with a `PARALLEL` tag plus step count; sequential clusters only show the header when they have two or more items (a lone tool renders as a bare row inside a plain `.tools` card, per the mock).
  - `SubagentCard` renders the agent activity as a `.subagent` card. The header shows `.chip` (subagentType), `.title` (the agent's description/tool), and `.elapsed` (duration or `‚Ä¶`). The body contains nested `ToolGroup` cards for the agent's partitioned children; CSS strips the child cards' outer border/rounding so the parent card supplies the chrome.
  - `ToolRow` renders an individual activity. The leading marker is `.run` (pulsing amber) while the tool is in flight, `.done` (green) once `outcome` or `duration` lands, and `.err` (red) when `status === 'error'`. The right-hand cell shows the outcome text when available, else the duration in `ms`, else `‚Ä¶` while running. When the server sends `tool_outcomes` without a `duration` (some backends don't), `patchToolOutcomes` computes elapsed from the activity's local `startTime` so the row flips to `.done` with a usable timing.

  **Progress-breadcrumb collapse (`collapseProgressRuns`):** the feed render walks `messages` through `collapseProgressRuns()` before mapping into bubbles. The helper produces three kinds of entries:
  - `plain` ‚Äî any message that isn't an assistant `turn: 'progress'` and isn't preceded by a progress run. Rendered as a normal `MessageBubble`.
  - `final-with-progress` ‚Äî a run of consecutive assistant `turn: 'progress'` messages immediately followed by any non-progress assistant bubble: either a persisted `turn: 'final'` message, or the live streaming placeholder (which has no `turn` field). In the streaming-placeholder case, the in-flight multi-segment turn stays in a single agent presence ‚Äî the breadcrumb of already-saved segments on top, the currently streaming content below ‚Äî instead of splitting into a standalone breadcrumb bubble above a separate streaming bubble. Rendered as the trailing bubble with the progress run passed in as `attachedProgress`, which renders inside the bubble (below the head, above the streaming/final text) as a collapsed `ProgressBreadcrumb` dashed pill: summary shows `N segment{s} ¬∑ <first-text-preview>` and a chevron; clicking expands a body that renders each progress message's full content (text / thinking / tool-runs) as a `.progress-row`. When the trailing bubble is the streaming placeholder, `isStreaming` is forwarded so the head reads `streaming‚Ä¶` and copy actions stay hidden.
  - `progress-trailing` ‚Äî a run with no following assistant bubble at all (stream aborted mid-turn before a new placeholder/final was attached). Rendered as its own `ProgressBreadcrumbBubble` ‚Äî a stand-alone agent bubble whose head reads `progress ¬∑ N segment{s}` and whose body is the `ProgressBreadcrumb`.

  **Streaming-target indicator:** the active streaming placeholder's bubble header shows `streaming‚Ä¶` in place of the relative timestamp.

  **Plan approval & clarifying questions (PR 4c)** ‚Äî the server emits two interactive `tool_activity` frames that require a user response rather than being executed like normal tools:
  - `{ tool: 'ExitPlanMode', isPlanMode: true, planAction: 'exit', planContent }` ‚Äî the LLM has finished drafting a plan and is asking for approval to proceed. `planContent` is the markdown body of the plan; the server falls back to the accumulated assistant text when no explicit plan file was captured.
  - `{ tool: 'AskUserQuestion', isQuestion: true, questions: [{ question, options: [{label, description?}] }], description }` ‚Äî the LLM needs a clarifying answer before continuing. Only the first question in the array is surfaced; `options` may be empty (free-form answer only) or contain labelled choices.

  `StreamStore.handleFrame` catches both and updates the target conv's entry with:
  - `pendingInteraction = { type: 'planApproval', planContent }` or `{ type: 'userQuestion', question, options }`
  - `uiState = 'awaiting'` ‚Äî the sidebar row flips to the amber halo dot (`.sb-row.awaiting`) and every visible conv shows its own pending state independently.

  `tool_activity` frames with `planAction: 'enter'` are ignored (they mark the *start* of plan mode, which isn't actionable). Neither interactive frame is pushed onto the streaming message's `contentBlocks` ‚Äî they are presented entirely through the dedicated card UI below.

  **Stream error card:** when `state.streamError` is non-null (set by a WS `error` frame, a failed `/message` POST, or an `onerror`/`onclose` on the socket itself ‚Äî the literal string `"WebSocket error"`), `<ChatLive>` renders a `<StreamErrorCard>` at the foot of the feed (after `InteractionCard`). The card uses the mock's `.err-card` / `.err-head` / `.err-actions` classes: accent-red left border, uppercase mono head with the error message in dim mono on the right, a prose body that says how many queued messages are paused, and two action buttons ‚Äî **Dismiss** (always shown, clears the error) and **Resume queue** (only rendered when `queue.length > 0`). Both buttons call `StreamStore.clearStreamError(convId, opts)` which nulls `streamError`, recomputes `uiState` (`'awaiting'` if a `pendingInteraction` is still set, else `null`), and ‚Äî when `opts.resumeQueue === true` ‚Äî immediately calls `drainQueueIfReady(convId)` so the head of the queue fires as a fresh `/message` POST. Because `drainQueueIfReady` checks `streamError` before acting, the error must be cleared first.

  **Card rendering (inside `<ChatLive>`):** when `state.pendingInteraction` is set, `<ChatLive>` renders an `<InteractionCard>` at the bottom of the feed (after messages, before `streamError`). The card dispatches on interaction type:
  - `PlanApprovalCard` ‚Äî amber-top `.plan` card with a `NEEDS YOU` chip, the rendered `planContent` inside a `.prose` container (via the same `marked + DOMPurify` pipeline used for message bodies), and `.plan-actions` with `Reject` (danger) / `Approve & run` (primary) buttons. Clicking either calls `StreamStore.respond(convId, 'yes' | 'no')`.
  - `QuestionCard` ‚Äî accent-top `.plan` card with a `Clarifying question` header, the question text as `.plan-title`, an optional row of option buttons (each pre-fills the answer input when clicked; `title` attribute carries the option description), a focused `<input>` for free-form typing, and a `Send` button. Enter submits. The submit calls `StreamStore.respond(convId, text)`.

  While `pendingInteraction` is set, the composer's textarea is disabled with placeholder `"Answer the prompt above to continue‚Ä¶"` and the send button is disabled (`awaiting` joins `sending`/`streaming` in `canSend`'s guard). Users cannot send an ordinary composer message while an interaction is pending ‚Äî they must click Approve/Reject or answer the question.

  **Answer delivery (`StreamStore.respond`):**
  1. Set `respondPending = true` (disables card buttons; shows `Sending‚Ä¶` in the primary button).
  2. `POST /api/chat/conversations/:id/input` with `{ text, streamActive: streaming || sending }`. The `streamActive` hint tells the server whether the client still considers the CLI stream live, so the server can avoid racing a just-finished stream.
  3. On `{ mode: 'stdin' }` ‚Äî the server piped the answer straight into the CLI's stdin; the existing stream will produce more frames. Clear `pendingInteraction`, reset `respondPending`, and set `uiState` back to `'streaming'` (if still streaming), `'error'` (if already errored), or `null`.
  4. On `{ mode: 'message' }` ‚Äî the stream has already ended. Clear `pendingInteraction`, reset `respondPending`, then delegate to `StreamStore.send(convId, text)` which runs the normal send flow (optimistic user message, POST `/message`, fresh stream via the existing WS).
  5. On network/HTTP error ‚Äî surface via `streamError` and flip `uiState` to `'error'`.

  **Lifecycle edges:**
  - `reset(convId)` clears `pendingInteraction` and `uiState` along with the rest of the session.
  - `send(convId, text)` clears `pendingInteraction` as part of the optimistic update ‚Äî if the user bypasses the card by typing into the composer (which only happens when we're not correctly guarding ‚Äî see above), we don't leave a stale interaction behind.
  - `archive(convId)` deletes the whole entry so `pendingInteraction` naturally goes away.
  - A new `error` frame mid-interaction clears `pendingInteraction` (a dead stream can't accept the answer).
  - A `done` frame *preserves* `pendingInteraction` ‚Äî the stream's `finally` block runs but the user can still answer; the next `respond()` call will get `mode: 'message'` and transparently start a new send.

  **Composer pickers (`ComposerPicks` + `PickChip`):** the chip row below the textarea renders three cascading pickers ‚Äî **Backend**, **Model**, **Effort** ‚Äî sourced from `AgentApi.getBackendsCached()` (one fetch per session for the static `BackendMetadata[]` registry). Each chip is a `PickChip` that visually matches the legacy deck styling (label + bold current value + chevron glyph) but layers a transparent native `<select>` on top via absolute positioning so a single click opens the OS-native dropdown ‚Äî keeping the full keyboard/a11y behavior of `<select>` without reimplementing a popover. Options read off the registry:
  - **Backend** ‚Äî `backends[].label` as visible label, `backends[].id` as value. Disabled (no options) until the fetch resolves.
  - **Model** ‚Äî only the models of the currently selected backend (`backend.models[]`). Disabled when the backend exposes no models (some backends are model-less). Each option renders `costTierDot(tier)` (‚óè for `high`, ‚óê for `mid`, ‚óã for `low`/unknown) followed by `m.label` so users can gauge cost at a glance.
  - **Effort** ‚Äî only the `supportedEffortLevels` of the currently selected model. Disabled when the model doesn't expose effort levels. Human labels (`low` ‚Üí "Low", `xhigh` ‚Üí "Extra high", `max` ‚Üí "Max") come from a local map.

  **Cascade reconciliation** ‚Äî a `useEffect` inside `ComposerPicks` watches `(backends, composerBackend, composerModel, composerEffort)` and pushes reconciled values to the store whenever the catalog shifts under the user's selection:
  - If `composerBackend` isn't in the registry (e.g. a deleted backend still persisted on the conv record), snap to the first backend.
  - If the current `composerModel` isn't in the selected backend's `models[]`, reset to the backend's `default` model (falling back to the first model).
  - If the current `composerEffort` isn't in the selected model's `supportedEffortLevels`, reset to `'high'` when supported, else the first level.
  The reconciler only writes to the store when a value actually needs to change ‚Äî it doesn't re-fire on every render. Because `streamStore.send` reads the reconciled values out of the same store fields, the POST body always carries a self-consistent `(backend, model, effort)` triple even when the user hadn't touched the chips since the catalog last shifted.

  **State lifecycle** ‚Äî the pickers live in `StreamStore`'s per-conv entry (`composerBackend / composerModel / composerEffort`), not in component state, so:
  - First load of a conv hydrates the chips from `conv.backend/model/effort` via `StreamStore.load`.
  - Once the user picks a value, the `composer*` fields become sticky ‚Äî switching to another conv and back doesn't rewind them.
  - On successful `POST /message`, `StreamStore.send` mirrors the sent values back onto `conv.backend/model/effort` so the next hydration (or a new subscriber) sees the persisted truth.
  - `reset(convId)` / `archive(convId)` wipe the entry entirely; the next mount starts fresh from the conv record.

  Composer-level setters (`setComposerBackend` / `setComposerModel` / `setComposerEffort`) only mutate the store and never POST directly ‚Äî picker changes flush to the server with the **next** `/message`, by design. This avoids chatty PATCHes for a user who flips through the chips before typing, and leans on the `/message` endpoint's existing conditional-update behavior.

  **Composer attachment tray + typed chips (`AttTray` / `AttChip` / `PendingAttChip` + whole-view drop overlay):** the composer supports per-message file attachments via two entry points:
  - **Paperclip button** (`.composer-foot .attach`) ‚Äî inline in the picker row. Clicking it triggers a hidden `<input type="file" multiple>` (stashed below the textarea). Selected files go directly through `StreamStore.addAttachments(convId, files)`.
  - **Drag-and-drop** ‚Äî the entire `<section className="main">` wrapper attaches `onDragEnter` / `onDragOver` / `onDragLeave` / `onDrop`. The drag-state is tracked with a counter ref so child re-enters don't flicker the overlay. While dragging files over the pane, a `.drop-overlay` absolute-positioned overlay (dashed accent border, "Drop files to attach" label) renders on top of the feed and composer. `dataTransfer.types.includes('Files')` gates the handlers so text/URL drags don't trigger the overlay.
  - **Clipboard paste** ‚Äî the composer textarea has an `onPaste` handler (v1 parity). If `clipboardData.items` contains any file entries (screenshots, copied images, dragged-in files), the handler calls `preventDefault()`, rewrites each file's name to `<basename>-<ts>-<n>.<ext>` so concurrent pastes don't collide, and forwards them via `StreamStore.addAttachments`. Otherwise, if `clipboardData.getData('text/plain')` is ‚â• 1000 characters, the handler synthesises a `File` named `pasted-text-YYYYMMDD-HHMMSS.txt` (mime `text/plain`) and forwards that as an attachment instead of letting the textarea absorb a wall of text. Shorter text pastes fall through to the default textarea behavior.

  `StreamStore.addAttachments(convId, files)` creates one `PendingAttachment` entry per file with a unique `id`, initial `status: 'uploading'`, `progress: 0`, and an `xhr: null` slot. Each entry then fires `AgentApi.conv.uploadFile(convId, file, onProgress, onXhr)`:
  - `onXhr(xhr)` is called synchronously at XHR creation ‚Äî the store patches `xhr` onto the entry so `removeAttachment` can abort it cleanly.
  - `onProgress(loaded, total)` updates the entry's `progress` percentage (0‚Äì100) so the chip's spinner ring fills.
  - On resolve (server returns `{ files: AttachmentMeta[] }` ‚Äî the helper picks the first entry since we send one file per request), the store flips the entry to `status: 'done'`, sets `result` to the full typed `AttachmentMeta` (including `kind` + optional `meta` like "12 pages"), and clears `xhr`.
  - On reject with `err.aborted === true`, the store does nothing (the chip was already removed). Other errors flip the entry to `status: 'error'` with the error message stashed for the chip's `title`.

  `StreamStore.removeAttachment(convId, attachmentId)` branches on status: aborts the XHR when `uploading`, fires `AgentApi.conv.deleteUpload(convId, result.name)` when `done` (fire-and-forget ‚Äî the chip disappears from the UI regardless of the DELETE outcome), else drops the entry directly. `StreamStore.clearPendingAttachments(convId)` empties the tray without deleting the uploads ‚Äî used when the user enqueues a message so the attachments detach onto the queue entry and the server copies remain on disk.

  **AttChip component** (two size variants):
  - `size: 'md'` ‚Äî composer tray card. A ~120 px tall tile with: a typed-kind icon badge (color-coded per `AttachmentKind` ‚Äî see the hint row below the composer for the legend), filename (ellipsized, 2-line clamp), and a `meta` sublabel underneath (e.g. "PDF ¬∑ 12 pages", "TS ¬∑ 142 lines", "MD ¬∑ 84 lines"). For `kind: 'image'` the card shows the `blob:` thumbnail (from `URL.createObjectURL(file)`, revoked on unmount) edge-to-edge with a dark overlay for the filename. During upload, a spinner ring with the percent fills from 0 ‚Üí 100; on error, a red status strip + the error message.
  - `size: 'sm'` ‚Äî queue-row pill. A compact ~36 px tile with just the kind-colored icon + truncated filename. Used inside expanded queue rows.

  `AttTray` renders a CSS grid (`.att-tray`, `grid-template-columns: repeat(auto-fill, minmax(160px, 1fr))`) above the textarea when `pendingAttachments.length > 0`, one `PendingAttChip` (md) per entry, plus a trailing `.att-add` tile (dashed border, plus icon) that opens the file picker. `PendingAttChip` is a thin wrapper that derives `kind` from the file extension via `StreamStore.attachmentKindFromPath`, builds the blob thumbnail for images, and passes the right props into `AttChip`.

  **Send-time behavior (`StreamStore.send`):**
  - **Guard** ‚Äî when any `pendingAttachments` entry is `uploading`, `send()` returns early. `canSend` computes the same guard so the send button is disabled while uploads are in flight.
  - **Content mutation** ‚Äî completed attachments' absolute server paths are joined with `", "` and appended as `[Uploaded files: path1, path2, ‚Ä¶]` via `composeWireContent(queuedMessage)`. If the textarea is empty but attachments are present, the tag becomes the full content (and `canSend` allows this path). The CLI still reads the files from disk ‚Äî the typed chips are a UI concern only and the wire format is unchanged.
  - **Snapshot / rollback** ‚Äî the current `pendingAttachments` array is captured before the optimistic update; the store clears it immediately (so the UI repaints without the tray) and restores it if the POST `/message` fails. No re-upload is needed on rollback ‚Äî the server still holds the files under the conv's artifacts dir.

  The 50 MB per-file / 10-files-per-request limits come from the server's multer config; the client does not enforce them up front ‚Äî errors surface as `status: 'error'` chips with the server's message pinned to the title tooltip. Below the tray the `.ca-hint-row` shows a legend of typed kinds with their accent colors so the user can correlate chips to types at a glance.

  **Message queue (`QueueStack` / `QueueRow`):** the v2 composer supports queueing follow-up messages during a live run. State mirrors the server-persisted `messageQueue` (typed `QueuedMessage[]`) via `StreamStore.queue`; the store auto-drains the head on every `done` frame when the conversation is idle (see `drainQueueIfReady` in `streamStore.js`).
  - **Enqueue paths** ‚Äî three ways to enqueue: (a) ‚åò/Ctrl + Enter in the composer textarea always enqueues, even while idle; (b) plain Enter with content while streaming enqueues; (c) clicking the stop/enqueue button (`.send.stop`, red with the octagon-stop glyph) during streaming enqueues the current draft and clears it. `doEnqueue()` in `shell.jsx` snapshots the composer's done `pendingAttachments` onto the queue entry's `attachments[]`, calls `StreamStore.enqueue`, then `setInput('')` + `clearPendingAttachments()`. The uploads stay on disk ‚Äî the queue entry owns their `AttachmentMeta` references.
  - **`canSend` / `canEnqueue` guards** ‚Äî `canSend = hasContent && !sending && !streaming && !awaiting && !hasUploadingFiles`, `canEnqueue = hasContent && !sending && !awaiting && !hasUploadingFiles && streaming`. "Content" means non-empty textarea **or** at least one done attachment.
  - **`QueueStack`** renders below the tray (and above `.composer-foot`) when `queue.length > 0`. Header row shows "Queued (N)" + a "Clear all" link. Each entry maps to a `QueueRow`.
  - **`QueueRow` (collapsed)** ‚Äî compact two-line layout: top row shows the 1-based position badge (`.qrow-num`), the message text truncated to 2 lines, and the action buttons (‚Üë/‚Üì reorder, ‚úé edit, √ó remove). When the row has attachments, a second row (`.qrow-atts`) renders horizontally-scrollable `AttChip size="sm"` pills. First/last rows get their up/down buttons disabled via `first` / `last` props.
  - **`QueueRowExpanded` (inline edit)** ‚Äî clicking a row's pencil (`Ico.edit`) button swaps it for an expanded editor (`.qexp` surface). The header repeats the `#N` badge with a `¬∑ in N positions` suffix when the row is not at the head. The body is a pre-filled `<textarea className="qexp-text">` with `autoFocus`. Attachments render as full-width `AttChip size="md"` cards with per-chip remove (`onRemove` ‚Üí drops from the local draft). The footer has **Add attachment** (opens a hidden file input, uploads via `AgentApi.conv.uploadFile` per file, appends the `AttachmentMeta` result to the draft), **Move up** (if not first), **Cancel** (discards draft, collapses), **Save** (calls `StreamStore.updateQueueItem(convId, index, { content, attachments })` ‚Äî disabled unless dirty and not uploading), and a destructive **Remove** (calls the stack's remove handler, which collapses automatically). Upload failures render inline above the footer. Only one row may be expanded at a time; if the auto-drain pops the expanded index, the `QueueStack` collapses back to the list.
  - **Persistence** ‚Äî every mutator (`enqueue`, `removeFromQueue`, `reorderQueue`, `updateQueueItem`) optimistically updates `state.queue` then PUTs the full queue; `clearQueue` DELETEs. A stale update is overwritten by the next PUT; there's no coalescing layer.
  - **Auto-drain on `done`** ‚Äî when a stream completes with no `pendingInteraction` and no `streamError`, the store pops `queue[0]`, PUTs the shortened queue, and composes the popped entry's wire-format content via `composeWireContent` (content + `[Uploaded files: ‚Ä¶]` tag rebuilt from `attachments[]`). The content is sent via the internal `sendWireContent(convId, content)` helper (bypasses the pendingAttachments pipeline since attachments were already uploaded at enqueue time).

  **Backwards compatibility (historical transcripts):** messages saved before the typed queue existed may carry `[Uploaded files: /abs/path1, /abs/path2]` trailing text on `Message.content` (user messages). The renderer feeds these through `StreamStore.parseUploadedFilesTag(content)`, which strips the tag and returns `{ content, attachments }` where each `AttachmentMeta` has `name` + `path` + `kind` (inferred from extension via `attachmentKindFromPath`). `size`/`meta` are unavailable for legacy entries ‚Äî the `md` chip shows only the filename and kind badge in that case. Users receive the same typed-chip UX for old transcripts at read-time; the on-disk bytes are untouched.
- **Everything else** ‚Äî not in the real app at `/v2/`. Right rail, tab indicator, and the other mock screens remain accessible only via `/v2/deck.html` and will be wired in future PRs.

## Messaging & Streaming

- `chatSendMessage()` gathers completed file paths from pending uploads, appends `[Uploaded files: ...]` to content, opens WebSocket, POSTs message (with `backend`, `model`, and `effort`), receives stream events via WS. When the selected backend, model, or effort differs from the stored defaults, it auto-saves the new choice via `PUT /settings` (fire-and-forget) so future new conversations use it. `defaultEffort` only persists to settings when the chosen model matches the stored `defaultModel`, preventing a mid-flight model swap from clobbering the settings-level default.
- Streaming uses `fetch` with manual ReadableStream parsing (not EventSource API)
- **Streaming state persistence:** `chatStreamingState` Map stores per-conversation state (accumulated text, thinking, tools, agents, tool/agent history, pending interactions). State survives conversation switches ‚Äî on return, the streaming bubble is recreated and restored. A separate `chatPendingInteractions` Map (keyed by convId) mirrors the `pendingInteraction` data and acts as a fallback for `chatRenderMessages()` ‚Äî if the streaming state is missing or its `pendingInteraction` is cleared during navigation, the fallback Map ensures plan approval / user question UI is still rendered.
- **WebSocket auto-reconnect:** On unexpected WS close during streaming, the client automatically attempts reconnection with exponential backoff (1s base, up to 5 attempts). On reconnect, the server replays buffered events wrapped in `replay_start`/`replay_end`. The client resets streaming state on `replay_start` (clears accumulated text/thinking/tools) and reprocesses replayed events from scratch. `assistant_message` events are deduplicated by message ID. `done` events during replay are ignored to prevent stale streams from destroying the current streaming state. After max attempts exhausted, `_doneResolve` is called to clean up. `chatDisconnectWs()` clears reconnect attempts to prevent auto-reconnect on deliberate close. Session reset clears the server-side event buffer to prevent stale events from replaying into the new session.
- **Streaming avatar:** The streaming message bubble reads the backend from the `chat-backend-select` dropdown (not from `state.chatActiveConv.backend`) to ensure the correct icon is shown immediately when the user switches backends, even before the conversation object is updated.
- **Elapsed timer:** live timer in streaming bubble header, self-cleans on DOM disconnect and nulls out the interval reference so it can restart on conversation switch-back
- **Unified streaming content:** A single `chatUpdateStreamingContent()` function renders all streaming state (thinking, text, tool history, active tools, agents, plan mode) together in one stacked view. Text content and tool activity accumulate and remain visible simultaneously ‚Äî new progress updates stack below previous content rather than replacing it. Items are grouped by agent via `chatGroupItemsByAgent()`: standalone tools render flat, while each agent card is followed by a scrollable sub-activity panel showing its child tools. Completed items show checkmarks and elapsed durations; running agents show animated spinners with live timers that count up in real-time.
- **Tool activity on completed messages:** When a message has a `toolActivity` array, `chatRenderToolActivityBlock()` renders a collapsible `<details>/<summary>` block (same pattern as thinking blocks) with a summary line (e.g. "15 ops ¬∑ 2 agents ¬∑ 5 read, 2 edited") generated by `chatBuildActivitySummary()`. Collapsed by default; expands to show the full chronological tool/agent list. Agent entries render as agent cards, tool entries as history items.
- **Ordered content blocks on completed messages:** When a message carries a `contentBlocks` array (new format ‚Äî see Data Models ‚Üí ContentBlock), `chatRenderContentBlocks()` walks the blocks in source order and emits: text blocks as markdown, thinking blocks via `chatRenderThinkingBlock()`, and **runs of consecutive tool blocks** as a single `chatRenderToolActivityBlock()` (so agent nesting and parallel grouping still apply within each run). This preserves the interleaving the CLI emitted ‚Äî e.g. "text ‚Üí activity breadcrumb ‚Üí text ‚Üí activity breadcrumb ‚Üí text" instead of the legacy "all thinking + all tools + all text stacked". `chatRenderMessage()` uses `contentBlocks` when present and falls back to the legacy `thinkingHtml + toolActivityHtml + rendered` stack when absent, so session files written before this field existed render unchanged. The progress-breadcrumb row renderer (`chatRenderProgressBreadcrumb`) uses the same helper with a `textWrapClass: 'chat-progress-row-text'` option so text blocks keep the row's compact typography. The live streaming bubble (`chatUpdateStreamingContent`) continues to use the existing progressive renderer ‚Äî ordered blocks take effect once the stream-final `assistant_message` is persisted and rendered.
- **Progress group collapsing:** `chatRenderMessages()` walks the session messages and collapses runs of consecutive assistant messages with `turn === 'progress'` (same `backend`, not interrupted by a user/system/memory_update message) into a **single compact breadcrumb pill** via `chatRenderProgressBreadcrumb()` ‚Äî a `<details>` element with no avatar, no role header, and no action buttons. Collapsed by default: the summary shows a caret, the step count (e.g. "4 steps"), the elapsed time computed against the preceding user message using the *last* progress row's timestamp (so it keeps climbing as more rows land), and a truncated single-line preview of the last row's text. Expanded: reveals a vertical timeline (`.chat-progress-timeline`) where each row holds a per-row timestamp, the row's `chatRenderThinkingBlock`/`chatRenderToolActivityBlock` output, and the row's markdown body. **Placement rules** ‚Äî three cases:
  1. **Run followed by a persisted final assistant message (same backend):** the breadcrumb is rendered *inside* that final message's `.chat-msg-content` (above its Thinking block + body) so the whole turn is one visual block ‚Äî breadcrumb ‚Üí Thinking ‚Üí activity ‚Üí final text.
  2. **Run is the trailing tail of the session AND a stream is live for this conversation** (`state.chatStreamingState.has(convId)`): the feed walk skips emitting the breadcrumb; instead `chatUpdateStreamingContent()` calls `chatBuildTrailingProgressBreadcrumb()` on every update and prepends the result inside the streaming bubble's own `.chat-msg-content` (under the Assistant header, above thinking / tool activity / accumulating text). This keeps past progress steps visible under the Assistant header while the agent works on the next segment, and transitions seamlessly into case (1) once the final message lands (the breadcrumb helper returns empty once the tail message is no longer `turn: 'progress'`).
  3. **Run is the trailing tail with no live stream** (stream crashed, session ended mid-turn): the breadcrumb renders standalone as a feed-level sibling with a left offset (`#chat-messages > .chat-progress-breadcrumb`) to align with the avatar column.
  Assistant messages with `turn === 'final'` (or legacy messages without `turn` ‚Äî treated as final for back-compat) always render as their own full card via `chatRenderSingleMessage()`. Rationale: the final message is what matters; the intermediate chatter should take minimal vertical space but remain accessible on demand, and should always appear tucked under a single Assistant header rather than as a full-width feed item.
- **Tool outcome indicators:** Each tool/agent in the activity log shows a colored outcome badge when outcome data is available. `chatRenderStatusCheck()` renders status-colored checkmarks (green ‚úì for success, red ‚úó for error, amber ‚úì for warning). `chatRenderOutcomeBadge()` renders a small colored badge with the outcome text (e.g. "exit 0", "4 matches", "not found"). Outcomes are extracted from CLI `tool_result` blocks by `extractToolOutcome()` in the backend, correlated by `tool_use_id`, and persisted on the `toolActivity` entries.
- **Sticky active section:** During streaming, when both completed and running tools exist, a `chat-activity-panel` container wraps them: completed items scroll in a bounded area while running items with spinners stay pinned at the bottom, always visible.
- **Parallel group indicator:** `chatGroupParallelItems()` detects consecutive agent entries whose `startTime` values are within 500ms (`PARALLEL_THRESHOLD_MS`) and wraps them in a `chat-parallel-group` container with a "parallel" label and a left accent border. Works in both persisted activity blocks and streaming display.
- **Agent detail expansion:** Agent cards with long descriptions or outcome data render as expandable `<details>` elements (`chatRenderAgentCard()`). Summary shows agent type, description, outcome badge, and elapsed time; expanding reveals full outcome details.
- **Turn boundaries:** intermediate assistant messages saved, content reset. `turn_complete` event archives active tools/agents to history so spinners stop. On `assistant_message`, tool/agent history is cleared after archiving ‚Äî the saved message's `toolActivity` now owns those entries, preventing duplicates when the next turn adds new agents to the streaming bubble. Agents are only archived when they have received their `tool_outcomes` (outcome/status set) ‚Äî sub-tool `turn_complete` events within an agent do NOT prematurely archive the parent agent. This ensures agents show spinners and live timers throughout their full execution.
- **Post-completion processing indicator:** When all tools/agents have completed but the model is still working (no text content yet), a "Processing..." indicator with typing dots is shown below the completed activity log. This fills the gap between agent completion and text output, so users always see ongoing work.
- **Thinking events:** do NOT archive active tool/agent state ‚Äî `turn_complete` handles archiving. This prevents premature archiving that would kill agent spinners and timers.
- **Plan approval / user input:** renders plan as markdown with approve/reject buttons (and user-question prompts) and submits responses through `POST /conversations/:id/input`, including a `streamActive` hint from the client. The server uses that hint to avoid racing a just-finished stream: it only forwards to `sendInput()` (`mode: 'stdin'`) when the client still considers the stream live; otherwise it returns `mode: 'message'` and the client resumes the conversation by sending the response as a normal follow-up message in the same session. On plan exit, if no explicit plan file was captured, the server falls back to the accumulated assistant text for `planContent` so the approval card still shows the full plan body. Once a plan approval or user question is pending (`pendingInteraction` is set), all DOM-updating event handlers (`thinking`, `tool_outcomes`, `text`, `tool_activity`, `turn_complete`, `replay_end`, and the activity timer) must guard against calling `chatUpdateStreamingContent()`, which would overwrite the pending interaction UI. Additionally, `chatUpdateStreamingContent()` itself contains a belt-and-suspenders guard that returns immediately when `pendingInteraction` is set, preventing any unguarded call site from overwriting the interaction UI. The `replay_end` handler re-shows the pending interaction instead of rendering regular streaming content. The `finally` block in `chatSendMessage` does **not** disconnect the WebSocket when `pendingInteraction` is set ‚Äî the WS remains available for continued stream updates even though interaction submission itself no longer depends on the browser WS.
- **Auto title update:** handles `title_updated` event by updating the active conversation title, the header, and the sidebar list in-place (no full reload needed).
- **Usage display:** a small indicator in the conversation header shows **session-level** token count (with `ICON_TOKEN`) and USD cost. Updated in real-time when `usage` events arrive during streaming. Displays on hover a tooltip with session input/output/cache token breakdown and cost, plus conversation-level totals. Hidden when no usage data exists (e.g. new conversation). For **Kiro** conversations, shows credits consumed and context usage percentage instead of tokens/cost. The token icon replaces the text label ‚Äî no "tokens" or "credits" word in the display.
- **Stream cleanup:** `chatCleanupStreamState()` accepts `{ force }` option. When the stream ends (`finally` block) while a `pendingInteraction` is active, force cleanup is **skipped** and the WebSocket is kept open ‚Äî the streaming state and message element are preserved so the user can still interact and send their response. The interaction response handler checks `!chatStreamingConvs.has(convId)` and force-cleans after the user responds. Content updates in the interaction handlers (plan approval, user question) are applied **before** cleanup so they remain visible. When no interaction is pending, the `finally` block uses `force: true` as before.
- **Send button state:** shows `ICON_STOP` (octagon) when streaming with no text input, `ICON_SEND` (paper plane) when idle or when streaming with text input (to queue). Toggled via `innerHTML`. Disabled during uploads or session resets.
- **Message queue:** Users can compose and submit messages while the CLI is actively responding. Queued messages are stored client-side in `chatMessageQueue` (Map of convId ‚Üí array of `{ id, content, inFlight }`) and **persisted server-side** as `messageQueue` (array of content strings) on the conversation entry. On every queue mutation (add, edit, delete, shift, clear), a sequential coalescing PUT syncs the current state to the server ‚Äî at most one PUT in flight at a time, with a follow-up if mutations occur during the request. Queued messages appear inline in the chat after the streaming bubble, styled as user messages with reduced opacity and an accent left border. Each shows a "Queued" badge and has Edit and Delete buttons. In-flight messages show "Sending..." and cannot be edited or deleted. When a response completes successfully, the next queued message is automatically sent (FIFO). Queue has three states: **Active** (streaming, auto-execute on completion), **Paused** (error, banner with Resume/Clear), and **Suspended** (restored from server after page load). The `chatQueuePaused` Set tracks paused conversations; `chatQueueSuspended` tracks restored conversations. On loading a conversation with a non-empty persisted queue and no active stream, the queue is restored into client state and marked suspended. A banner reads "N queued messages from a previous session" with Resume and Clear buttons. Suspended queues do not auto-execute ‚Äî the user must explicitly resume. Queue is automatically cleared on session reset and archive.
- **Per-message actions:** Each rendered message has a `.chat-msg-actions` row (visible on hover) with two buttons: **Copy** (copies the rendered plain text via `content.textContent`) and **Copy MD** (copies the original markdown source from the `data-raw-content` attribute on the `.chat-msg` element). Both buttons show a "Copied!" confirmation for 1.5 seconds after clicking.

## Browser Tab Status Indicator

Implemented in `public/js/tab-indicator.js`. Overlays a colored dot-badge on the favicon so users can monitor task progress while working in other tabs. `document.title` is never modified ‚Äî the signal lives entirely on the favicon. Four states:

| State | Favicon dot color | Condition |
|-------|-------------------|-----------|
| `idle` | (no dot ‚Äî base favicon) | No conversations streaming |
| `running` | Blue `#3b82f6` | Any conversation is streaming (`state.chatStreamingConvs.size > 0`) |
| `done` | Green `#22c55e` | Last stream ended successfully **while the tab was hidden** |
| `error` | Red `#ef4444` | Last stream ended with an error **while the tab was hidden** |

- **Any-conversation semantics:** A single indicator reflects the aggregate ‚Äî if multiple conversations stream concurrently, the indicator is `running` until all finish.
- **Hidden-only completion:** If the tab is visible when a stream ends, the indicator returns directly to `idle` ‚Äî the user already sees the completion in the UI, so no notification is needed. `done` and `error` are purely passive-notification states.
- **Error flag sticky across concurrent streams:** If any of multiple concurrent streams errors, the error flag persists; the indicator stays `running` until the *last* stream ends, at which point (if still hidden) it shows `error`. The flag is cleared once consumed.
- **Auto-clear on focus:** A `visibilitychange` listener clears `done` / `error` back to `idle` when the tab becomes visible. `running` is unaffected by visibility changes.
- **Pending interactions do not end the run:** When a stream ends with `pendingInteraction` set (plan approval / user question), the tab indicator is **not** updated ‚Äî the task is logically still active, awaiting user input, so the `running` state persists.
- **Favicon rendering:** The base SVG is loaded once, drawn onto a 64√ó64 canvas, and overlaid with a colored dot (22% radius, bottom-right, white halo for contrast). The result is exported as a PNG data URL and assigned to `<link rel="icon">.href`. Reverting to `idle` restores the original `favicon.svg` href.
- **Hook points:**
  - `chatStartMessageRequest()` in `streaming.js` calls `chatTabOnStreamChange()` immediately after adding to `chatStreamingConvs`.
  - The `finally` block in `chatStartMessageRequest()` calls `chatTabOnStreamChange({ error: !!hadError })` in the non-pending-interaction path, after deleting from `chatStreamingConvs`.
- **Initialization:** `chatInitTabIndicator()` is called from `chatInit()` in `main.js`. Wires the visibility listener and preloads the base favicon image. Idempotent.

## File Delivery

- **Trigger:** The CLI outputs `<!-- FILE_DELIVERY:/path/to/file -->` markers when the user explicitly requests a deliverable file (e.g. "give me a CSV", "create a report"). The system prompt instructs all backends to use this marker only for user-requested deliverables, not for routine file edits.
- **Marker extraction:** `chatExtractFileDeliveries(text)` runs as a pre-processing step before markdown rendering. Strips all `FILE_DELIVERY` HTML comments from the text and returns the list of file paths. Runs in both streaming and static message rendering paths (via `chatRenderMarkdown`).
- **File cards:** `chatRenderFileCards(files)` renders inline cards for each delivered file, showing a file icon, filename, and two buttons: **View** (opens the side panel) and **Download** (direct browser download via `Content-Disposition: attachment`). Both buttons hit `GET /workspaces/:hash/files?path=...&mode=view|download`.
- **Viewer panel:** A right-side panel (`.chat-file-viewer`) that slides in when the user clicks View. Shows the file's text content with syntax highlighting (via highlight.js). Capped at 2 MB on the server side. Closed via the X button or automatically on conversation switch. On mobile (‚â§768px), the panel overlays the full chat area.
- **Multi-file support:** Multiple `FILE_DELIVERY` markers in one message render multiple file cards. The View panel swaps content when opening a different file.

## Draft Persistence

- **In-memory layer:** `state.chatDraftState` is a `Map` keyed by convId (or `'__new__'` for the welcome-screen buffer) that holds `{ text, pendingFiles }`. Source of truth inside a single tab.
- **localStorage mirror:** every draft is also written under `chat:draft:<convId|__new__>` so drafts survive the sign-in-redirect page reload triggered by the session-expired overlay. The persisted shape is `{ text, files: [{ path, name, size }] }` ‚Äî only completed uploads are serialized. In-progress `File` bytes cannot be persisted and are dropped.
- **Helpers (`public/js/conversations.js`):**
  - `chatSaveDraft()` ‚Äî mirrors the textarea + `state.chatPendingFiles` into both layers under the active convId (or `'__new__'`). Empty drafts trigger `chatDeleteDraft` rather than writing `{ text: '', files: [] }`.
  - `chatRestoreDraft(convId)` ‚Äî reads from `state.chatDraftState` and paints the textarea + pending-files UI. Clears both when no draft is present.
  - `chatDeleteDraft(key)` ‚Äî removes the draft from both layers. Used by `chatDeleteConversation`, `chatArchiveConversation`, the empty-draft branch of `chatSaveDraft`, and the success paths of `chatSendMessage` / `chatStartMessageRequest`.
  - `chatMigrateDraft(fromKey, toKey)` ‚Äî moves a draft between keys in both layers. Invoked by `chatEnsureConversation` when a `'__new__'` draft is promoted to a real convId after an upload-triggered conversation creation.
  - `chatHydrateDraftsFromStorage()` ‚Äî called once from `chatInit` on boot. Iterates `localStorage` for `chat:draft:*` entries, rebuilds `state.chatDraftState`, and reconstructs file-entry objects with `{ name, size }` (type is intentionally omitted so the chip renderer's `f.type &&` guard skips `URL.createObjectURL` on the non-`Blob` restored file). Corrupted entries are dropped.
- **Save sites:**
  - On conversation switch ‚Äî `chatSelectConversation` (saves the outgoing conv's draft before loading the new one).
  - On opening the folder picker for a new conversation ‚Äî `chatCreateConversationWithDir`.
  - At the start of `chatSendMessage` ‚Äî immediately before the textarea is cleared, so the draft reaches localStorage even if the function throws synchronously before the fetch's catch runs.
- **Restore sites:**
  - On conversation switch ‚Äî `chatSelectConversation`.
  - After folder-picker creation ‚Äî `chatCreateConversationWithDir` (restores any `'__new__'` draft migrated to the new convId).
  - On boot ‚Äî `chatInit` calls `chatHydrateDraftsFromStorage()` then `chatRestoreDraft(null)` to paint the welcome-screen textarea with any pre-reload `'__new__'` draft.
  - On failed send ‚Äî `chatSendMessage` installs a `restoreFailedSend()` closure that repaints the captured `originalText` + `originalFiles` into the UI and calls `chatSaveDraft()` again. Wired into both the conversation-create catch block and `chatStartMessageRequest`'s `onSendFailed` callback.
- **Clear-on-send ordering:** the per-conv delete in `chatStartMessageRequest` runs **after** `await response.json()` succeeds, not before the POST. The `'__new__'` delete in `chatSendMessage` runs after the conversation-create POST succeeds. This keeps the draft intact when the message POST returns 401 (session expired mid-send) so the overlay's "your draft is preserved" promise holds.
- **Session-expired overlay:** `chatShowSessionExpired` (in `public/js/state.js`) still reads "You have been signed out. Your draft message is preserved ‚Äî sign in to continue." The claim is now backed by the localStorage mirror + the delete-after-success ordering.

## Knowledge Base Browser

The **KB Browser** is a full-screen panel that swaps into the main chat area (hiding the active conversation) when the user clicks the **KB** icon on a workspace-group header in the sidebar. It is implemented inline in `public/js/main.js` ‚Äî look for the `chatKbBrowser*` functions.

- **Entry point:** `chatKbBrowser.chat-conv-group-kb-btn` click handler (wired in `chatWireEvents`) reads `data-kb-hash` and `data-kb-label` off the button and calls `chatOpenKbBrowser(hash, label)`. The flag is sourced from `ConversationListItem.workspaceKbEnabled` in the sidebar payload; toggling KB on/off via the settings modal calls `chatLoadConversations()` so the sidebar button appears/disappears without a page reload.
- **Layout:** A `<div class="chat-kb-browser">` sibling of `#chat-messages` with a header (workspace label + KB counter pills + close button), a tab row (**Raw** / **Entries** / **Synthesis** / **Reflections** / **Settings** ‚Äî all five wired), and a tab-specific body. - **Header counters:** The header renders pills for `raw: N`, `entries: N`, and `pending: N` sourced from `state.counters` on every refetch. Counter text updates in place during polling so the user sees counts tick up as uploads/digests complete without scrolling or tab-switching.
- **Polling:** While the browser is mounted, `chatKbBrowserState.pollTimer` runs `chatKbBrowserRefetch()` every **1500 ms** against `GET /workspaces/:hash/kb?folder=<selectedFolder>` (or `GET /kb/entries?folder=‚Ä¶` on the Entries tab). During an active batch upload, the effective poll interval is stretched to **3000 ms** to reduce DOM thrash (implemented via a `_lastPollTick` timestamp guard inside the 1500ms `setInterval`). Polling is required because WS frames are conversation-scoped ‚Äî the browser can be open for a workspace that has no active CLI stream. On close, the timer is cleared.
- **WS frame reactivity:** `streaming.js` dispatches `kb_state_update` frames to `window.chatHandleKbStateUpdate(convId, event)`. The handler checks whether the browser is open for the same workspace hash and, if so, triggers an immediate refetch (in addition to the polling cadence) so changes during active CLI streams render without waiting for the next poll tick.
- **Toolbar:** A single row above the tab body carries the workspace-wide controls:
  - **Auto-digest switch** ‚Äî reads `state.autoDigest`, flips via `PUT /kb/auto-digest`. Changing the switch does not kick off any work itself; it only changes how subsequent uploads and deletes behave.
  - **+ Folder** button ‚Äî prompts for a folder path (accepts nested paths like `ops/weekly`), hits `POST /kb/folders`, selects the new folder in the tree on success.
  - **Digest All Pending** button ‚Äî disabled when `counters.pendingCount === 0`. Label is `Digest All Pending (N)` so the badge count is always visible. Clicking fires `POST /kb/digest-all`. A **digest progress indicator** renders to the left of the button showing unified aggregate progress across batch, single-file manual, and auto-digest runs ‚Äî sourced from `kb_state_update` frames carrying `digestProgress: { done, total, avgMsPerItem, etaMs? }` (and from `GET /kb`'s `digestProgress` field on initial mount or after a reload, so a long-running digest rehydrates without losing accuracy). The indicator format is `"N / M items ‚Äî ~E min remaining"` when ETA is available, falling back to `"N / M items"` before two tasks have settled. Entity count (from `digestion: { active, entriesCreated }` frames) is appended as a `¬∑ N entities created` suffix during active digestion. When the queue drains the server emits `digestProgress: null`, which clears the indicator. `digestion: { active: false, entriesCreated }` still populates `chatKbBrowserState.digestComplete`, surfacing a green **"Digestion complete ‚Äî N entities created"** banner at the top of the Raw tab with an `√ó` dismiss button (clears `digestComplete`, re-renders). The banner persists across tab changes within the browser session and across raw-list polls; leaving the KB Browser resets the state.
    - **ETA formatting** ‚Äî `chatKbFormatEta(ms)` renders `< 60 s` as `${secs}s`, `< 60 min` as `${mins} min`, and longer as `${hours}h ${mins}m` (or `${hours}h` on the minute boundary). Paired with `chatKbFormatDigestProgress(dp)` which composes the full indicator string; returns `''` when `dp` is falsy so the caller can unconditionally concatenate.
- **Raw tab:** Two-column layout ‚Äî a **folder tree sidebar** on the left and a **main content area** on the right.
  - **Folder tree** ‚Äî rendered from `state.folders` as a flat list indented by `folder_path` depth. Root is the implicit top node. Each folder row has a **chevron toggle** (non-leaf rows only ‚Äî leaf rows render a fixed-width spacer for column alignment), its path label, and hover-revealed **Rename** / **Delete** buttons (rename prompts for a new path ‚Üí `PUT /kb/folders`; delete warns about cascading children ‚Üí `DELETE /kb/folders?folder=‚Ä¶&cascade=true`). Clicking a folder sets `chatKbBrowserState.selectedFolder` and immediately refetches the raw list scoped to that folder; selecting a deep folder auto-expands every ancestor so the selected row is always visible.
    - **Collapse / expand** ‚Äî `chatKbBrowserState.collapsedFolderPaths` is a `Set<string>` of paths whose children are hidden. Empty by default (everything expanded). Root (`''`) has no chevron and cannot be collapsed. Clicking the chevron toggles the path in the set and calls `chatKbBrowserRenderTab()`; descendants of any collapsed path are filtered out of the rendered tree (the render function walks each candidate path's proper ancestors and skips it if any ancestor is in the set). The chevron rotates 90¬∞ in the `expanded` state and carries `aria-expanded="true"|"false"`. State is in-memory only ‚Äî closing and reopening the KB browser resets to fully expanded.

Its `scrollTop` is saved before each full `innerHTML` swap and restored after, alongside the entries list and entry detail scroll positions, so the 1500 ms poll cadence and WS frames don't reset the user's scroll position while they browse.

HTTP 500, network errors, and timeouts auto-retry up to 2 times with backoff (1s, 3s), then mark as error. Responses with `{ deduped: true, addedLocation: false }` are marked as `'deduped'` and increment `skippedCount`.   ```
- **Layout:** A `<div id="chat-file-explorer" class="chat-file-explorer">` sibling of `#chat-messages`. Header row shows the workspace label and a close button. Body is a CSS grid with two columns: a **tree pane** (`.chat-fe-tree-pane`, `minmax(260px, 2fr)`) and a **preview pane** (`.chat-fe-preview-pane`, `minmax(320px, 3fr)`).
- **Tree pane:**
  - Toolbar with **Up**, **Refresh**, **New Folder**, **New File**, and **Upload** buttons (in that order). The New Folder button prompts for a name, validates client-side (rejects `/`, `\`, `.`, `..`, and empty names before round-tripping), then hits `POST /explorer/mkdir` with `{ parent: <upload target folder>, name }`. The New File button prompts with `untitled.md` as the default; the client trims, applies the same `/`, `\`, `.`, `..`, empty-name guards, auto-appends `.md` when the typed name has no `.` (so users can type bare names but any extension is still accepted), then hits `POST /explorer/file` with `{ parent, name }`. On success the new file is auto-selected, its preview is fetched, and the pane is flipped straight into **Edit** mode so the user lands in the editor ready to type. On 409 the UI surfaces the server's conflict message via the standard alert modal. The "upload target folder" used by Upload, New Folder, and New File is the currently-selected directory row when one is selected, else the current breadcrumb folder.
  - Breadcrumb row showing the current folder path as clickable segments; clicking an ancestor snaps `currentFolder` back.
  - Lazy-loaded directory tree. Expanding a folder row calls `_expandFolder(rel)` which hits `GET /explorer/tree?path=<rel>` and caches the result in `feState.children`. Collapse just removes the path from `feState.expanded` without dropping the cache. Hidden files (names starting with `.`) are **always** rendered ‚Äî explicit product requirement.
  - Row interactions: single-click selects and populates the preview pane; double-click on a directory expands it. Hover-revealed **Rename**, **Download** (files only), and **Delete** buttons on every row. Rename/Delete fire the `PATCH /explorer/rename` and `DELETE /explorer/entry` endpoints; on 409 conflict for rename, the user is prompted to confirm overwrite, which retries with `overwrite: true`.
  - Drag-and-drop source/target: the pane itself accepts dropped files via `_onDragOver` / `_onDrop`, and directory rows accept drops via `_onRowDragOver` / `_onRowDrop` so users can pick a specific destination folder by dropping onto it. While a drag is active over the pane, `.chat-fe-tree-pane.drag-over` adds an accent border; a row drop target paints `.chat-fe-row.drop-target` as a dashed outline.
  - Upload panel pinned to the bottom of the tree pane (`.chat-fe-upload-panel`). Each `UploadItem` renders a filename, a progress bar (driven by the XHR `upload.onprogress` event), and a status line (queued / uploading / complete / error / conflict-prompt). Uploads run with **concurrency 3** ‚Äî additional items wait in `queued` state until a slot frees. On 409 the row surfaces an inline "Overwrite?" prompt; confirming re-POSTs with `?overwrite=true`.
- **Preview pane:**
  - `preview.kind === 'text'` renders the file as a syntax-highlighted `<pre><code>` block via highlight.js using `preview.language` inferred from extension. Fetched via `GET /explorer/preview?path=<rel>&mode=view`.
  - `preview.kind === 'markdown'` renders via marked with the same text source.
  - `preview.kind === 'image'` renders an `<img src="/api/chat/workspaces/:hash/explorer/preview?path=<rel>&mode=raw">` inside `.chat-fe-preview-image-wrap`. A client-side **25 MB** size guard (read from the entry's `size` before issuing the fetch) demotes oversized images to `oversize-image` and shows a download-only stub.
  - `preview.kind === 'oversize-text'` appears when the server responded with `413` (file > 5 MB cap) or the entry's own size exceeds the same cap ‚Äî shown as a message + Download button.
  - `preview.kind === 'unsupported'` covers binary files outside the supported preview set. Shows filename, size, and a Download button.
  - The preview header always shows filename, size, and mtime; an explicit **Download** button sits next to the filename and hits `?mode=download`.
  - **Editor mode** ‚Äî when `kind` is `text` or `markdown` and the content has loaded, the header exposes an **Edit** button. Clicking it flips `preview.editing = true` and swaps the `<pre>`/markdown body for a full-height `<textarea.chat-fe-preview-editor>` seeded from `preview.content` into a `preview.draft` field (updated live on `input`). The header swaps the **Edit** button for **Save** + **Cancel**. **Save** issues `PUT /explorer/file` with `{ path, content: draft }`, shows the button in a disabled "Saving‚Ä¶" state while the request is in flight, then on success updates `preview.content`/`preview.size`, clears `editing`/`draft`, and reloads the current folder so the tree's size/mtime column reflects the new bytes. Save failures surface the server error via the alert modal and keep the editor open. **Cancel** prompts for confirmation via `chatShowConfirm` when `draft !== content`, then exits edit mode. Markdown files render as markdown in view mode and as raw source in the textarea during edit ‚Äî the textarea uses monospace + `white-space: pre` + `tab-size: 2`.
- **Upload UX details:** Multiple files can be selected from the file picker or dropped onto the pane. For each `File`, a fresh `UploadItem` is pushed to `feState.uploads` and enqueued; the drain loop (`_runUploadQueue`) keeps at most 3 XHRs in flight at a time. On any XHR `progress` event, the per-item bar updates; on `load` (HTTP 200) the row is marked complete, the tree pane refreshes the affected folder, and the queue advances. On HTTP 409 the row transitions to a prompt state and is excluded from the active-count until the user resolves it (overwrite or cancel).
- **Cleanup:** `chatCloseFileExplorer()` aborts any in-flight upload XHRs, re-shows `#chat-messages` and the input area, and clears `feState`. Also invoked on sign-out and on workspace deletion. Exposed on `window` so the close-button `onclick` in the static header markup can call it.

The related `ICON_FOLDER`, `ICON_FOLDER_OPEN`, `ICON_FILE`, and `ICON_TRASH` SVG constants are defined in `public/js/state.js` alongside the other `ICON_*` constants.

## Theme System

CSS custom properties on `:root` (light) and `[data-theme="dark"]`. Theme applied by setting `data-theme` on `<html>`. Persisted to `localStorage` under `agent-cockpit-theme`. Synced from server settings on init. Listens for system theme changes when set to "system".

## SVG Icon System

All UI icons are inline SVGs using `stroke="currentColor"` for automatic theme compatibility. No icon font or sprite sheet ‚Äî icons are defined as JavaScript string constants in `public/js/state.js` and imported into whichever module renders them. Source SVG files are also stored in `public/icons/` for reference, but are not loaded at runtime.

**Icon constants** follow the naming pattern `ICON_<NAME>` (e.g. `ICON_SETTINGS`, `ICON_SEND`, `ICON_USER`). Each constant is a complete `<svg>` string with `class="kb-icon" width="14" height="14" viewBox="0 0 24 24"` and `stroke="currentColor"` so they inherit the parent element's text color. KB pipeline icons use the `KB_ICON_` prefix; reflection type icons use the `KB_REF_ICON_` prefix.

**Static HTML elements** (in `index.html`) that cannot use JS constants embed the same SVG markup directly inline ‚Äî e.g. the sidebar buttons (New Chat, Sessions, Settings, Sign Out, Reset), the attach button, and the initial send button.

**Dynamic elements** rendered via JS template literals reference the constants ‚Äî e.g. `${ICON_SETTINGS}` in a button's innerHTML. Elements that toggle icons (send ‚Üî stop button) use `el.innerHTML = ICON_SEND` / `el.innerHTML = ICON_STOP` rather than `textContent`.

**User avatar** uses `ICON_USER` with a special CSS override: `.chat-msg.user .chat-msg-avatar-svg` restores the blue circle background (`var(--accent-chat)`) and white color so the avatar remains visible on both light and dark themes. Other avatar SVGs (backend icons) render without a background circle.
