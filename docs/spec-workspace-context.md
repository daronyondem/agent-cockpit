# 5. Workspace Context Feature

[← Back to index](SPEC.md)

Workspace Context is the workspace-level operating-memory feature. It replaces
the former Context Map graph system with the simpler model that proved useful in
real workspaces: a CLI-readable markdown folder plus clear instructions that
teach the workspace CLI how to maintain it.

Workspace Context deliberately has no graph database, no candidate review queue,
no Context Map MCP server, no Memory Review queue, and no structured
entity/fact schema. The canonical operating memory is markdown maintained by the
configured Workspace Context processor and read directly by normal chat CLIs.
Memory is the atomic capture/search inbox; Workspace Context maintenance may
consume accepted active Memory entries and delete them from that inbox once the
context markdown has absorbed them
([ADR-0081](adr/0081-make-workspace-context-consume-memory-inbox.md)).
Workspace Context also owns exact reusable reference documents and durable
non-executable assets so user-directed future-use prompts, templates, style
guides, images, and source files do not need a separate workspace-local memory
folder ([ADR-0082](adr/0082-add-workspace-context-references-and-assets.md)).

## Mental Model

Workspace Context answers:

- What does this workspace already know?
- What people, projects, decisions, commitments, open threads, and relationships
  should the CLI remember?
- What context should the CLI update after chats, transcripts, mail threads,
  Slack exports, and workspace documents?
- Where should a user look if they want to inspect the current durable learning?

The output should be more than atomic facts. Good Workspace Context markdown can
include sections such as current read, strategic read, how to engage, open
threads, people, projects, decisions, cross-references, and temporal status. The
processor is instructed to choose useful human-readable structure instead of
filling a rigid schema.

## Data Ownership

Per-workspace data lives under:

```
data/chat/workspaces/{storageKey}/workspace-context/
├── WORKSPACE_CONTEXT.md
├── context/
│   └── overview.md
├── references/
│   └── *.md
├── assets/
│   └── {durable reference files}
├── runs/
│   ├── latest.md
│   └── {timestamp}-{source}.md
└── state.json
```

`context/` is the canonical synthesized operating-memory folder. The CLI may
create, rename, reorganize, and edit markdown files inside that folder. Agent
Cockpit shows the files as a read-only preview in Workspace Settings.

`references/` stores exact reusable markdown/text guidance that future agents
should read directly: prompts, templates, style rules, formatting instructions,
checklists, and other user-directed reference documents. References are durable
workspace material, but they are not normal synthesized memory. Maintenance may
read and link them, but should not rewrite them merely to make them more concise.
Reference routes accept markdown/text files only.

`assets/` stores durable non-executable reference files that context or
reference markdown can link to. Supported asset categories are intentionally
conservative: common images, PDFs, plain text/markdown, JSON, CSV/TSV, and
similar document/reference formats. Executables, shell scripts, package
archives, hidden files, path traversal, and symlink escapes are rejected. Asset
metadata includes path, name, size, MIME type, updated time, and whether the UI
can preview the file inline.

`WORKSPACE_CONTEXT.md` is generated by Agent Cockpit and contains the operating
directive for the CLI. It points at the workspace path, `context/`,
`references/`, and `assets/`, then instructs the CLI to:

- read relevant context and reference files before answering when prior
  workspace knowledge or exact saved guidance matters;
- create, reorganize, or update context markdown directly when durable
  information appears in chat, attached files, transcripts, emails, Slack
  exports, project documents, or workspace files;
- create or update reference markdown/text when the user asks to preserve exact
  reusable prompts, templates, style guides, procedures, or future instructions;
- store or link durable non-executable future-use files under `assets/` and add
  context/reference pointers that explain when to use them;
- create focused markdown files when no existing file is the right home;
- connect people, projects, decisions, commitments, themes, and source material;
- preserve temporal perspective by using `as of YYYY-MM-DD` for status-like
  claims, retaining exact event dates/times when timing matters, distinguishing
  source time from ingestion time when useful, and updating older notes when
  newer source material supersedes them;
- update autonomously without asking for user approval;
- avoid filtering user-provided workspace material;
- preserve exact reference materials unless the user asks to revise them or the
  reference itself is clearly superseded;
- avoid storing source transcripts verbatim unless the user explicitly asks for
  raw transcript retention.

`runs/` stores processor run reports. `runs/latest.md` mirrors the newest
completed run report. These reports are operational audit records, not source of
truth for the learned context. Maintenance prunes run reports and run-history
entries older than 7 days; `runs/latest.md` is retained as the pointer to the
newest report.

`state.json` is the small operational sidecar:

```ts
{
  version: 1;
  contextDir: string;
  lastRun?: WorkspaceContextRunRecord;
  lastCompletedAt?: string;
  lastScanCompletedAt?: string;
  lastMaintenanceCompletedAt?: string;
  runs: WorkspaceContextRunRecord[];
}
```

`WorkspaceContextRunRecord` is:

```ts
{
  runId: string;
  source: 'initial_scan' | 'scheduled' | 'session_reset' | 'archive' | 'manual_catchup' | 'maintenance';
  status: 'running' | 'completed' | 'failed' | 'stopped' | 'skipped';
  startedAt: string;
  completedAt?: string;
  filesConsidered: number;
  summary: string | null;
  errorMessage?: string;
  skippedReason?: 'scan-running' | 'maintenance-running' | 'already-running';
}
```

The sidecar is used for settings badges, run history, scheduler cadence, and
live notifications only. It is not a second representation of the learned
context.

## Workspace Instructions

When Workspace Context is enabled, Agent Cockpit installs a managed block in the
workspace root `AGENTS.md`:

```md
<!-- AGENT_COCKPIT_WORKSPACE_CONTEXT_START -->
## Workspace Context

This workspace uses Agent Cockpit Workspace Context.
Read and follow: `/absolute/path/to/WORKSPACE_CONTEXT.md`
Durable context markdown lives in: `/absolute/path/to/context`
Reusable reference documents live in: `/absolute/path/to/references`
Durable reference assets live in: `/absolute/path/to/assets`

Use those instructions and markdown files as the workspace operating memory.
Create, reorganize, and update Workspace Context files directly when durable
context, reusable guidance, or future-use assets change.
<!-- AGENT_COCKPIT_WORKSPACE_CONTEXT_END -->
```

The managed block is replaced in place when repaired. User-authored `AGENTS.md`
content outside the markers is preserved. Disabling Workspace Context removes
the managed block and deletes `AGENTS.md` only when that managed block was the
only content in the file.

This is the active-chat integration. There is no Workspace Context MCP
injection. Chat CLIs discover the feature through normal workspace instruction
loading and use filesystem tools to read, create, reorganize, or update the
Workspace Context folders.

## Processor Runs

Workspace Context uses the same configured backend adapter one-shot primitive as
Memory, KB processors, summaries, titles, and OCR. The processor must be a CLI
profile that resolves through `ChatService.resolveCliProfileRuntime()`, and the
adapter must implement `runOneShot()`.

If source planning finds no source files for a scan or no markdown files for a
maintenance pass, the service records a completed zero-file run with a concise
summary and does not invoke the processor. Zero-file scan completions still
update `lastScanCompletedAt`, so the scheduler cadence remains visible and does
not retry the same empty window every minute.

Every enabled processor run is first persisted as a `running` run before source
planning. The run summary is then updated as progress changes: checking source
files, processing the planned file count, invoking the processor, and finally
completed/failed/stopped. Zero-source runs remain in the running state for a
short minimum visible window before completing so an open Workspace Settings
screen can show the run start instead of only the finished record.

Global defaults live under `Settings.workspaceContext`:

```ts
{
  cliProfileId?: string | null;
  cliBackend?: string | null; // legacy fallback only
  cliModel?: string | null;
  cliEffort?: EffortLevel | null;
  scanIntervalMinutes?: number; // 1..1440, default 5
  cliConcurrency?: number;      // 1..10, default 1
  maintenanceIntervalHours?: number;     // 1..8760, default 24
  maintenanceCliConcurrency?: number;    // 1..10, default 1
}
```

Workspace overrides live under `WorkspaceIndex.workspaceContext`:

```ts
{
  processorMode: 'global' | 'override';
  cliProfileId?: string | null;
  cliBackend?: string | null; // legacy fallback only
  cliModel?: string | null;
  cliEffort?: EffortLevel | null;
  scanIntervalMinutes?: number | null;
  maintenanceIntervalHours?: number | null;
}
```

Legacy settings are migrated at read/write boundaries:

- `Settings.contextMap` is normalized into `Settings.workspaceContext`.
- `WorkspaceIndex.contextMapEnabled` is normalized into
  `WorkspaceIndex.workspaceContextEnabled`.
- `WorkspaceIndex.contextMap` is normalized into
  `WorkspaceIndex.workspaceContext`.
- Existing `workspaces/{storageKey}/context-map/` folders are removed during workspace
  index migration because the old graph store is no longer used.
- Removed old processor fields such as source toggles and extraction/synthesis
  concurrency are stripped.

### Run Sources

`initial_scan`
: Fired when enabling Workspace Context for a workspace that was previously
  disabled, or by the scheduler when an enabled workspace has no prior run.

`scheduled`
: Fired by the scheduler after the scan interval has elapsed.

`manual_catchup`
: Fired from Workspace Settings by **Run scan**.

`maintenance`
: Fired from Workspace Settings by **Run maintenance**. It reviews current
  Workspace Context markdown files and improves organization, deduplication,
  cross-references, and temporal clarity without scanning conversations or
  external sources unless they are explicitly listed as context files.

`session_reset`
: Fired by the persisted session-finalizer queue after reset creates an
  archived session.

`archive`
: Fired by the persisted session-finalizer queue after a conversation is
  archived.

### Source Planning

Workspace scans are intentionally simple. The processor receives source file
paths and reads them itself:

- conversation-scoped finalizer runs receive the archived `session-N.json` path
  for that conversation/session, plus uploaded-file paths referenced inside that
  session's messages;
- initial and manual scan runs consider active conversation session files in
  the workspace plus uploaded-file paths referenced inside those session
  messages;
- scheduled runs consider active conversation session files updated after the
  last completed Workspace Context scan run, plus uploaded-file paths referenced
  inside those updated session messages;
- maintenance runs receive the current markdown files under
  `workspace-context/context/`, current markdown/text files under
  `workspace-context/references/` as stable supporting material, an inventory of
  supported files under `workspace-context/assets/`, plus active Memory inbox
  entries from `memory/files/` when Memory is enabled for the workspace; they do
  not receive conversation/session source files;
- source paths are deduplicated and capped at 80 per run.

Maintenance memory-inbox inputs are planned separately from context markdown.
Only active Memory entries are eligible. The service validates that every Memory
source path stays under the workspace's `memory/files/` directory and caps the
memory-inbox portion at 80 entries, newest first. Deleted, superseded, and
redacted entries that are outside the active default scope are not automatically
consumed by maintenance.

Workspace files are still useful to Workspace Context because the active chat CLI
can update context, references, or assets while working on them, and future
ingestion pipelines can write source material into conversations or workspace
files before the scan processor runs. The current Workspace Context processor
does not rebuild a graph from workspace files and does not crawl arbitrary
workspace folders.

### Prompt Contract

The scan prompt gives the processor:

- current time;
- workspace path;
- absolute `WORKSPACE_CONTEXT.md` path;
- absolute markdown context folder path;
- absolute references folder path;
- absolute assets folder path;
- run source;
- source-window label;
- source file paths.

The processor is told to read `WORKSPACE_CONTEXT.md`, review source files,
inspect session exports for uploaded-file paths or other user-mentioned local
file paths, create/reorganize/update the Workspace Context folders directly,
preserve temporal perspective, not ask for approval, not produce JSON, and then
reply with a concise markdown summary of what context/reference/asset files
changed or why no update was needed.

The maintenance prompt gives the processor the same current time, workspace
path, instruction path, context folder, references folder, assets folder, run
source, source-window label, and file path list. Context markdown files are
listed as normal maintenance inputs. Reference markdown/text files are listed in
a separate **Reference Files** section as stable supporting material. Asset files
are listed as metadata inventory in an **Asset Inventory** section; maintenance
does not read binary asset contents unless they are also explicitly listed as
source files in a scan. When active Memory inbox entries are present, they are
listed in a separate **Memory Inbox Files** section and the prompt requires
exactly one fenced `workspace-context-memory-actions` block:

````md
```workspace-context-memory-actions
{"acceptedMemoryFiles":[]}
```
````

The processor must list only Memory filenames it actually incorporated into the
Workspace Context markdown. The server parses and validates the JSON after the
processor exits. If any Memory entries were offered and the action block is
missing, malformed, not an object, has a non-array `acceptedMemoryFiles`, has a
non-string item, or names a file that was not in the plan, the run fails visibly
and no Memory entry is deleted. Accepted entries are deleted only by Agent
Cockpit after successful validation. The summary shown in run history strips the
action block and appends the consumed-entry count.

The processor is told this is maintenance, not general new-source ingestion; it
should not scan conversations, external source files, or workspace files unless
explicitly listed. It should merge duplicate notes, split oversized files where
useful, add useful headings and cross-references, preserve temporal/as-of
information, remove stale duplication, incorporate accepted Memory inbox items,
add useful links to reference documents and assets, preserve exact reusable
reference documents unless they are clearly obsolete or user-directed updates are
needed, and update markdown directly without approval.

Before maintenance source planning, Agent Cockpit prunes Workspace Context run
logs older than 7 days from both `state.json.runs` and
`workspace-context/runs/*.md`. `runs/latest.md` and scheduler cadence timestamps
such as `lastScanCompletedAt` are retained.

## Scheduler

`WorkspaceContextScheduler` starts during chat route initialization. It wakes
every 60 seconds for scan scheduling, reads global Workspace Context settings,
lists enabled workspaces, applies the per-workspace scan interval override when
present, and starts due `initial_scan`/`scheduled` runs until `cliConcurrency`
is reached.

Maintenance scheduling is separate. The same scheduler performs an hourly
maintenance eligibility check. For each enabled workspace, it applies the
per-workspace maintenance interval override when present, compares only
`lastMaintenanceCompletedAt` against `maintenanceIntervalHours`, and starts a
`maintenance` run when due until `maintenanceCliConcurrency` is reached. Recent
scan completions do not suppress maintenance. Workspaces with no previous
maintenance completion are due on the next maintenance scheduler pass.
Maintenance interval values are whole hours. The default is 24 hours.

Run concurrency is workspace-scoped. A workspace can have at most one active
Workspace Context run. Manual scan/maintenance routes return `409` when a run
is already active. Scheduled runs that are due while the same workspace is
already running record a `skipped` run with `skippedReason:
'maintenance-running'`, `scan-running`, or `already-running` so the run history
shows why that due pass did not execute. Skipped runs do not update scan or
maintenance completion timestamps.

Due scheduled scans that find no changed source files record a completed
zero-file run instead of a skipped run. This makes scheduler activity visible in
Workspace Settings and advances the scan timestamp to the time of the empty
check.

## REST API

Workspace Context routes are mounted under `/api/chat` and are owned by
`src/routes/chat/workspaceContextRoutes.ts`.

| Method | Path | CSRF | Description |
|--------|------|------|-------------|
| `GET` | `/workspaces/:workspaceId/workspace-context/settings` | No | Returns `{ enabled, settings, state, contextDir, referencesDir, assetsDir, instructionPath, files, references, assets }`. Disabled workspaces return empty material lists. |
| `PUT` | `/workspaces/:workspaceId/workspace-context/enabled` | Yes | Body `{ enabled }`. Enables/disables the workspace. Enabling ensures the folders and `AGENTS.md` managed block exist and starts `initial_scan` when transitioning from disabled. Disabling stops an active run and removes the managed block. |
| `PUT` | `/workspaces/:workspaceId/workspace-context/settings` | Yes | Body `{ settings }` or direct settings object. Saves workspace processor override settings. |
| `POST` | `/workspaces/:workspaceId/workspace-context/scan` | Yes | Starts a background `manual_catchup` run with `forceAll:true`. Returns `{ ok:true, started:true, source:'manual_catchup' }`. Disabled workspaces return `403`; already-running workspaces return `409`. |
| `POST` | `/workspaces/:workspaceId/workspace-context/maintenance` | Yes | Starts a background `maintenance` run over current context markdown, reference files, asset inventory, and accepted active Memory entries. Returns `{ ok:true, started:true, source:'maintenance' }`. Disabled workspaces return `403`; already-running workspaces return `409`. |
| `POST` | `/workspaces/:workspaceId/workspace-context/scan/stop` | Yes | Aborts the active run for the workspace. Returns `409` when no run is active. |
| `POST` | `/workspaces/:workspaceId/workspace-context/repair-instructions` | Yes | Recreates generated files and reinstalls the `AGENTS.md` managed block. |
| `GET` | `/workspaces/:workspaceId/workspace-context/files` | No | Lists markdown files under `workspace-context/context/` as `{ path, name, size, updatedAt }`. |
| `GET` | `/workspaces/:workspaceId/workspace-context/files/*` | No | Reads a markdown file under `workspace-context/context/`. Traversal and non-markdown paths are rejected. Disabled workspaces return `403`. |
| `GET` | `/workspaces/:workspaceId/workspace-context/references` | No | Lists markdown/text files under `workspace-context/references/` as `{ path, name, size, updatedAt }`. Disabled workspaces return an empty list. |
| `GET` | `/workspaces/:workspaceId/workspace-context/references/*` | No | Reads a markdown/text reference under `workspace-context/references/`. Traversal and non-reference paths are rejected. Disabled workspaces return `403`. |
| `PUT` | `/workspaces/:workspaceId/workspace-context/references/*` | Yes | Body `{ content }`. Creates or replaces a markdown/text reference file. Paths must stay under `references/`, use allowed text/markdown extensions, avoid hidden/path-traversal segments, and stay under the text edit size limit. |
| `DELETE` | `/workspaces/:workspaceId/workspace-context/references/*` | Yes | Deletes one reference file. Unknown files return `404`. |
| `GET` | `/workspaces/:workspaceId/workspace-context/assets` | No | Lists supported non-executable files under `workspace-context/assets/` as `{ path, name, size, updatedAt, mimeType, previewable }`. Disabled workspaces return an empty list. |
| `GET` | `/workspaces/:workspaceId/workspace-context/assets/*` | No | Previews or downloads an asset. Query param `mode=view\|download`; view returns JSON for text assets under the preview size limit and streams safe image assets inline, while download sets `Content-Disposition: attachment`. |
| `POST` | `/workspaces/:workspaceId/workspace-context/assets/*` | Yes | Multipart upload with file field `file`. Stores the file under `assets/` after path, symlink, hidden-file, size, and extension/MIME allowlist checks. Existing files are replaced only when the request path is valid and the upload succeeds. |
| `DELETE` | `/workspaces/:workspaceId/workspace-context/assets/*` | Yes | Deletes one asset file. Unknown files return `404`. |
| `GET` | `/conversations/:id/workspace-context-file` | No | Conversation-scoped preview/download endpoint for assistant Markdown links to Workspace Context files. Query params are `path` (absolute data-root path or context/reference/asset-relative path, with optional `:line[:column]` for text files) and `mode=view\|download`. The route resolves the conversation workspace, requires Workspace Context to be enabled, serves only files under that workspace's `workspace-context/context/`, `references/`, or `assets/`, rejects traversal, symlink escapes, unsupported files, and other workspaces, and returns `language:"markdown"` for markdown previews so the chat file viewer renders Markdown. |
| `DELETE` | `/workspaces/:workspaceId/workspace-context` | Yes | Clears Workspace Context context, references, assets, and run history, then repairs generated files/instructions. Enablement and settings remain. Active runs return `409`. |

The Workspace Context settings and file GET routes set
`Cache-Control: no-store`. The settings screen polls these routes for live run
state, so browser or intermediary caching must not hide newly recorded scheduled
runs.

Removed old Context Map endpoints are not supported:

- `/workspaces/:workspaceId/context-map/*`
- `/mcp/context-map/call`
- graph/detail/review/candidate/entity routes

## WebSocket Updates

Workspace-scoped live frames use `workspace_context_update`:

```ts
{
  type: 'workspace_context_update';
  updatedAt: string;
  workspaceContext: ConversationWorkspaceContextStatus;
}
```

`ConversationWorkspaceContextStatus` includes:

```ts
{
  enabled: boolean;
  pending: boolean;
  runningRuns: number;
  failedRuns: number;
  contextDir?: string;
  fileCount?: number;
  latestRunId?: string;
  latestRunStatus?: 'running' | 'completed' | 'failed' | 'stopped';
  latestRunSource?: WorkspaceContextRunSource;
  latestRunCreatedAt?: string;
  latestRunCompletedAt?: string;
  lastRunId?: string;
  lastRunStatus?: 'running' | 'completed' | 'failed' | 'stopped';
  lastRunSource?: WorkspaceContextRunSource;
  lastRunCreatedAt?: string;
  lastRunCompletedAt?: string;
}
```

The frontend patches `conv.workspaceContext`, dispatches
`ac:workspace-context-update`, and shows the composer notification only for
running or failed Workspace Context runs. The notification opens Workspace
Settings on the Workspace Context tab, usually the Runs section.

## Frontend UX

Global Settings has a **Workspace Context** tab. It configures default processor
profile/model/effort, scan interval/concurrency, and maintenance
interval/concurrency.

Workspace Settings has a **Workspace Context** tab with a left rail:

- **Overview**: enable toggle, context folder, references folder, assets folder,
  instruction file, context/reference/asset counts, latest scan, latest
  maintenance, Run scan, Run maintenance, Refresh, Repair instructions, Stop
  when a run is active.
- **Processor**: global/default mode selector, optional workspace override
  profile/model/effort/scan interval, and Save Changes only while dirty.
- **Context**: two-pane file browser for `workspace-context/context/` with a
  search/list rail and read-only markdown preview.
- **References**: the same two-pane browser pattern for
  `workspace-context/references/`; the rail is search/list only, and the preview
  pane holds new-reference path entry, editing, save, and delete controls for
  exact reusable guidance.
- **Assets**: the same two-pane browser pattern for `workspace-context/assets/`;
  the rail is search/list only, and the preview pane holds selected asset
  preview/download/delete controls or the new-upload path/file controls for
  supported non-executable reference files.
- **Runs**: latest-first run records with source, status, start time, files
  considered, markdown-rendered summaries, and errors. Ordering uses
  `startedAt`, matching the timestamp shown on each card. The section shows five
  logs at a time with **Newer**/**Older** arrow controls when more logs exist.
  Summary links that point to files under `workspace-context/context/` or
  `workspace-context/references/` open the matching section and select that file
  for reading.
- **Danger Zone**: Clear Workspace Context.

The UI does not expose graph search, entity editing, Needs Attention, candidate
approval queues, or raw database inspection.

While the Workspace Context tab is on screen, the page refreshes runtime state
from `GET /workspaces/:workspaceId/workspace-context/settings` with a no-store request
every second. WebSocket `workspace_context_update` frames also construct or
refresh the visible latest run row immediately from the workspace status fields.
The latest run always takes precedence over older cached run rows so running
progress and completion/stopped/failed transitions are visible without a manual
browser refresh. The settings route overlays any in-memory active run on top of
persisted `state.json`, including the short period before source planning has
written a full run record. If **Run scan** or **Run maintenance** returns an
already-running `409`, the frontend refreshes this display state instead of
leaving the user on stale status.

## Historical Context Map Compatibility

Older installs can contain `contextMap` settings, `contextMapEnabled` workspace
flags, and `context-map/` folders. The current code migrates settings/flags and
removes old graph folders as part of Workspace Context normalization. The old
graph database and route/module code is intentionally deleted.
